# Web3 入职法律风险

## 一、Web3 用工模式的特殊性

Web3 项目多采用境外注册结构（新加坡、开曼、BVI 等），常见特征：

-   项目方在中国境内无注册主体
    
-   远程办公为主（Telegram / Zoom 协作）
    
-   难以签订有效《劳动合同》
    
-   无法依法缴纳五险一金
    

由此带来的直接影响：

-   薪资纠纷、工伤事故难以适用《劳动合同法》维权
    
-   社保、公积金缺失将影响落户、购房、贷款、子女教育等现实权益
    
-   即使有书面合同，也可能因主体不合格被认定无效
    

部分项目采用 **EOR（名义雇主）** 模式解决合规问题，但在纠纷中，劳动关系认定仍存在不确定性。

**入职前需自我评估：**

-   是否接受灵活用工关系
    
-   是否能接受无社保、公积金
    
-   重要岗位或长期入职，建议提前咨询专业律师
    

* * *

## 二、薪酬结构及法律风险

Web3 常见薪酬模式：

-   人民币 + Token
    
-   全 USDT
    
-   Token 为主要收入来源
    

主要风险点：

-   法律上，工资应以人民币支付，虚拟币支付工资可能被认定无效
    
-   若 Token / USDT 不计入工资：
    
    -   实际工资基数降低
        
    -   影响社保、经济补偿计算
        
-   项目自发 Token 价格波动大，项目失败时可能直接归零
    

结论：  
**Token 薪酬更接近高风险激励，而非稳定收入。**

* * *

## 三、虚拟货币出金风险

由于薪资含虚拟币，员工通常需通过 C2C / OTC 出金：

-   场内挂单
    
-   场外担保交易
    
-   私下交易群
    

主要法律风险：

-   对手资金涉赌、涉诈 → 银行卡冻结
    
-   可能被要求退赔涉案金额
    
-   高价出 U、协助换汇 → 帮信罪风险
    
-   情节严重 → 洗钱、掩饰隐瞒犯罪所得罪
    

**风险防范建议：**

-   尽量保留部分法币薪资
    
-   出金选择相对合规、可追溯渠道
    
-   避免灰色交易、异常高价交易
    
-   留存交易记录与资金来源证明
    

* * *

## 四、项目合法性审查要点

即便不是核心负责人，参与非法项目仍可能被牵连。

入职前建议重点核查：

-   是否存在 Token 融资或收益承诺
    
-   Token 分发机制与经济模型
    
-   是否面向中国大陆用户
    
-   是否存在返利、多级推广结构
    
-   白皮书内容是否涉及投资回报
    

原则：  
**项目本身不合规，个人风险无法完全切割。**

* * *

## 五、学习小结

-   Web3 入职 ≠ 传统劳动关系
    
-   用工、薪酬、出金均存在法律灰区
    
-   高收益往往伴随高合规风险
    
-   入职前审查项目与用工结构至关重要

# Web3 合规性要求与常见法律风险

## 一、Web3 行业法律环境概览

自 2017 年起，中国监管部门持续加强对虚拟货币相关活动的监管，核心政策要点包括：

-   明确禁止 ICO（首次代币发行）等代币融资行为
    
-   禁止虚拟货币交易所运营及撮合服务
    
-   明确虚拟货币不具有法偿性，不得作为支付工具
    
-   禁止境外交易平台向中国境内居民提供服务
    
-   严厉打击以虚拟货币为载体的非法集资、传销、洗钱、赌博等行为
    

总体监管思路可以概括为：

> 全面封堵虚拟货币的金融属性，对区块链技术本身保持有限容忍。

因此，Web3 作为技术栈并未被整体否定，但一旦涉及代币发行、融资、交易、矿池运营等高风险场景，即可能面临行政或刑事风险。

* * *

## 二、主要法律风险类型

### 1\. 代币发行与交易相关风险

中国法律明确禁止任何单位或个人通过 ICO、IEO、IDO 等方式进行融资活动。

需要注意的是：

-   不论代币名称（治理 Token、积分、凭证等）
    
-   不论发行方式（空投、白名单、内部测试）
    

只要代币具备以下特征之一，均可能被认定为非法金融行为：

-   具备融资功能
    
-   具有公开或变相流通性
    
-   伴随收益预期
    

在实践中，若筹资规模扩大，项目方可能面临非法吸收公众存款罪等刑事风险。

此外，技术人员若参与以下环节：

-   代币经济模型设计
    
-   空投与分发逻辑配置
    
-   智能合约部署与运行
    

通常难以仅以“提供技术服务”为由免责，可能被认定为共同行为人。

* * *

### 2\. 赌博与传销相关刑事风险

Web3 项目的经济模型设计直接影响其合规风险。

常见高风险结构包括：

-   充值 → 随机收益 → 可提现
    
-   抽奖、盲盒、转盘等随机激励机制
    
-   链游中以虚拟币或稳定币参与的博弈结构
    

上述模式若形成“下注—随机回报—兑现”的闭环，可能被认定为开设赌场罪。

在 NFT 项目、DAO 或挖矿平台中，若存在以下机制：

-   邀请返利
    
-   多级分佣
    
-   团队计酬
    

一旦构成层级推广关系，可能触及组织、领导传销活动罪。

* * *

### 3\. 场外交易（OTC）与洗钱风险

虚拟货币在场外交易中，常被用于规避外汇监管。

典型风险路径包括：

-   使用人民币购买 USDT、ETH 等虚拟货币
    
-   通过链上或境外平台兑换外币
    
-   实现跨境资金转移
    

该行为可能构成非法经营外汇业务。

更严重的情形是：

-   交易对手或资金来源涉及电信诈骗、赌博、毒品等犯罪
    
-   参与者即便主观上不知情，也可能被认定为协助洗钱
    

因此，参与虚拟货币兑换需高度谨慎，避免成为非法资金流转链条的一环。

* * *

### 4\. 民商事纠纷风险

在司法实践中，虚拟货币相关交易多被认定为合同无效，风险自担。

主要表现为：

-   虚拟货币买卖合同不受法院支持
    
-   因价格波动、对价不明、交付失败产生的纠纷，维权困难
    

委托他人代投虚拟货币亦存在较高风险：

-   投资亏损或资金去向不明
    
-   可能引发民事赔偿责任
    
-   情节严重时，可能被认定为非法经营或非法集资
    

实践中建议：

-   委托方应明确委托范围、权限和投资限制，并保留证据
    
-   受托方不得擅自扩大投资范围，不宜接受不特定公众资金
    

* * *

## 三、小结

在当前中国监管环境下：

-   Web3 技术本身不被全面禁止
    
-   涉及代币、资金、收益分配的业务风险极高
    
-   技术人员、运营人员、资金参与者均可能承担法律责任
    

理解合规边界，是参与 Web3 项目的前提条件。

## 去中心化众筹（Crowdfunding DApp）挑战

### 核心目标

设计并部署一个 **基于智能合约的去中心化众筹应用**，让互不信任的用户在**无需中介**的情况下协作筹集资金：

-   **成功** → 资金自动转给目标合约
    
-   **失败** → 每位参与者可安全取回自己的钱
    

* * *

### 关键技能点

-   **状态机设计（State Machine）**  
    管理众筹生命周期（筹款中 / 成功 / 失败）
    
-   **ETH 资金处理**
    
    -   `payable` 函数接收 ETH
        
    -   合约向外部合约转账
        
-   **映射（mapping）**
    
    -   追踪每个地址的出资金额
        
-   **事件（events）**
    
    -   链上记录捐款行为
        
    -   前端监听并展示用户行为
        
-   **外部合约交互**
    
    -   成功后向 `FundingRecipient` 合约发送资金
        

* * *

### 🏗️ 合约核心功能（CrowdFund.sol）

1.  **contribute()**
    
    -   `payable` 函数，接收 ETH
        
    -   更新 `balances[address]`
        
    -   触发 `Contribution` 事件
        
2.  **截止时间（deadline） + 阈值（threshold）**
    
    -   判断众筹是否成功
        
3.  **complete()**
    
    -   成功时，一次性将资金发送给目标合约
        
4.  **withdraw()**
    
    -   失败时，允许用户取回各自的 ETH
        

* * *

### 事件设计（前端关键）

```
event Contribution(address contributor, uint256 amount);
```

-   前端通过监听事件展示所有出资记录
    
-   若使用命名参数（如 `indexed contributor`），前端需通过**参数名**读取，而不是索引
    

* * *

### 开发流程

1.  初始化项目（Scaffold-ETH / SpeedRunEthereum）
    
2.  启动本地区块链 (`yarn chain`)
    
3.  部署合约 (`yarn deploy`)
    
4.  启动前端 (`yarn start`)
    
5.  修改合约 → 重新部署 → 前端自动更新
    
6.  部署到 Vercel 并提交到 SpeedRunEthereum
    

* * *

### 最终成果

-   一个可交互的 **Crowdfunding DApp**
    
-   用户可：
    
    -   质押 ETH
        
    -   查看所有出资记录
        
    -   在成功/失败条件下自动执行对应逻辑
        
-   应用已部署到 **公共网络**

# Polymarket 架构与链上数据解码

## 一、学习目标

-   理解 Polymarket 的核心数据模型：**Event、Market、Condition、Collection、Position(TokenId)**，并能清晰说明它们的层级与关系。
    
-   理解链上日志在 **市场创建 → 交易 → 结算** 全生命周期中的作用，以及日志如何串联形成完整证据链。
    
-   掌握链上日志解析方法，能够实现：
    
    -   **交易解码（Trade Decoder）**：给定交易哈希，还原成交的价格、数量、方向等。
        
    -   **市场解码（Market Decoder）**：给定 conditionId 或创建日志，还原市场参数（问题标识、预言机、抵押品、YES/NO TokenId）。
        

* * *

## 二、核心概念与数据模型

### 1\. Event 与 Market

-   **Event（事件）**：一个预测主题，如“2024 年美国大选”。
    
-   **Market（市场）**：事件下的具体预测问题，通常是二元（Yes / No）。
    
    -   一个 Event 可以包含多个 Market。
        
    -   Market 是实际发生交易的最小单位。
        

### 2\. NegativeRisk（负风险机制）

-   适用于 **多结果、互斥市场**（赢家通吃）。
    
-   通过 **NegRiskAdapter 合约**，将同一事件下的多个 Market 关联起来。
    
-   关键思想：
    
    -   某一 Market 的 **NO 头寸 ≈ 其他所有 Market 的 YES 头寸**。
        
    -   提高资金利用率，避免为每个结果单独提供流动性。
        
-   支持 **NO → 多个 YES** 的头寸转换（convert）。
    

* * *

## 三、CTF（Conditional Token Framework）中的关键对象

### 1\. Condition（条件）

-   市场在链上的唯一登记身份。
    
-   创建方式：调用 `prepareCondition`。
    
-   **ConditionId 计算公式**：
    
    ```text
    conditionId = keccak256(oracle, questionId, outcomeSlotCount)
    ```
    
-   含义：
    
    -   绑定问题、预言机和结果数量，是市场结算的依据。
        

### 2\. Collection（集合）

-   表示在某个 Condition 下选中的结果集合。
    
-   **CollectionId 计算公式**：
    
    ```text
    collectionId = keccak256(parentCollectionId, conditionId, indexSet)
    ```
    
-   Polymarket 中：
    
    -   `parentCollectionId = 0`
        
    -   二元市场：
        
        -   YES → indexSet = 1 (01)
            
        -   NO → indexSet = 2 (10)
            

### 3\. Position / TokenId（头寸）

-   用户实际持有、可交易的资产。
    
-   实现为 **ERC-1155 Token**。
    
-   **TokenId 计算公式**：
    
    ```text
    tokenId = keccak256(collateralToken, collectionId)
    ```
    
-   每个 Market 对应 **两个 TokenId**：YES 与 NO。
    

### 4\. Collateral（抵押品）

-   Polymarket 使用 **USDC（Polygon 上为 USDC.e）** 作为唯一抵押品。
    
-   每 1 份 Outcome Token 背后对应 **1 USDC 的潜在兑付**。
    

### 5\. 价格含义

-   YES 价格 = 市场对“事件发生概率”的定价。
    
-   例：价格 0.60 USDC
    
    -   发生 → 兑付 1 USDC（盈利 0.40）
        
    -   不发生 → 价值归零（损失 0.60）
        

* * *

## 四、市场生命周期与链上证据链

### 1\. 市场创建（Creation）

-   调用：`ConditionalTokens.prepareCondition`
    
-   **关键日志**：`ConditionPreparation`
    
-   作用：
    
    -   确认问题、预言机、结果数量
        
    -   标志市场正式存在
        

### 2\. 初始流动性与拆分（Split）

-   调用：`splitPosition`
    
-   **关键日志**：`PositionSplit`
    
-   作用：
    
    -   锁定 USDC
        
    -   铸造等量 YES / NO 头寸
        

### 3\. 交易撮合（Trading）

-   合约：`CTFExchange` 或 `NegRisk_CTFExchange`
    
-   **关键日志**：`OrderFilled`
    
-   包含信息：
    
    -   maker / taker
        
    -   makerAssetId / takerAssetId（0 表示 USDC，其余为 TokenId）
        
    -   成交数量、手续费
        
-   价格计算：
    
    -   由 USDC 与 TokenId 的成交数量比例得出
        

去重注意

-   一次撮合可能产生多条 `OrderFilled`：
    
    -   maker 侧填单
        
    -   taker 汇总（taker = exchange）
        
-   常见处理方式：
    
    -   过滤 `taker == exchange` 的日志
        
    -   或使用 `OrdersMatched` 作为唯一成交记录
        

铸造 / 合并的隐含逻辑

-   YES 买单 + NO 买单 → 锁定 1 USDC → 铸造一对头寸
    
-   YES 卖单 + NO 卖单 → 销毁头寸 → 赎回 USDC
    
-   对应 `PositionSplit` / `PositionsMerge` + `OrderFilled`
    

### 4\. 结算（Resolution）

-   预言机调用：`reportPayouts(conditionId, payouts[])`
    
-   结果：
    
    -   胜出头寸 = 1 USDC
        
    -   失败头寸 = 0
        
-   用户调用：`redeemPositions`
    
-   市场生命周期结束
    

* * *

## 五、总结：链上可验证的完整证据链

1.  **ConditionPreparation** → 市场参数与预言机绑定
    
2.  **PositionSplit** → 资金注入与头寸生成
    
3.  **OrderFilled / OrdersMatched** → 交易与价格形成
    
4.  **reportPayouts / redeemPositions** → 结果确认与清算
    

> 通过解析这些链上日志，可以在不依赖 Polymarket 前端或数据库的情况下，**完全基于链上数据重建市场的全部历史行为**。

## Layer 2 笔记

Layer 2 通过将交易执行移至链下、结果提交至以太坊主网（L1），在保证安全性的同时 **显著降低 Gas 成本、提升吞吐量**。

* * *

## Rollup 技术比较

| 类型 | 原理 | 优点 | 缺点 |
| --- | --- | --- | --- |
| Optimistic Rollup | 先假设正确，事后欺诈证明 | EVM 兼容性强、成本低 | 提现延迟（约 1–2 周） |
| ZK Rollup | 零知识证明验证状态 | 安全性高、快速提现 | 开发复杂度高、EVM 兼容不完全 |

* * *

## 主流 L2 平台

-   **Starknet**
    
    -   语言：Cairo（非 Solidity）
        
    -   特点：ZK-STARK，极高扩展性
        
-   **zkSync**
    
    -   工具链：支持 Solidity
        
    -   特点：ZK Rollup，开发体验友好
        
-   **Arbitrum**
    
    -   架构：Optimistic Rollup
        
    -   特点：完全兼容 EVM 工具链（Hardhat / Foundry）
        
-   **Base**
    
    -   背景：Coinbase 推出，基于 OP Stack
        
    -   特点：生态支持强、开发成本低
        

* * *

## 开发入门要点

### 环境准备

-   安装平台 SDK：zkSync CLI、Starknet CLI 等
    
-   区分 L1 / L2 RPC 与钱包网络配置
    
-   使用本地 Devnet 进行测试
    

### 合约部署注意

-   部分平台不支持 `CREATE2` 或有 Gas 上限差异
    
-   使用平台专属部署工具
    
-   zkRollup 平台 **不保证地址一致性**
    

### 跨链与桥接

-   官方桥：zkSync / Arbitrum Bridge
    
-   第三方协议：LayerZero、Axelar、Wormhole
    
-   需评估 **延迟与费用**
    

### 实践案例

-   zkSync：Uniswap V3 fork 部署
    
-   Starknet：Cairo NFT 合约 + 前端交互
    
-   Arbitrum：Nitro 模式下的 Gas 竞拍逻辑

# 合约审计笔记

## 审计的必要性

-   **合约不可修改**：一旦部署，漏洞难以修复，审计是关键风险控制手段
    
-   **资金安全**：防止重入、溢出等漏洞导致资产被盗
    
-   **法律合规**：部分监管或机构要求提供审计报告
    

* * *

## 常见合约审计工具

-   **Slither**：静态分析工具，检测常见漏洞与代码规范
    
    -   用法：`slither MyContract.sol`

## 智能合约安全总结

**一、安全设计基本原则**

-   **最小权限原则**：只授予必要权限，降低被滥用风险
    
-   **模块化设计**：结构清晰，便于审计与维护
    
-   **显式错误处理与事件记录**：提升可追溯性与应急能力
    

**二、常见漏洞类型与防护思路**

-   **重入攻击（Reentrancy）**
    
    -   本质：外部合约在回调中重复调用函数
        
    -   防护：先更新状态，再进行转账（Checks-Effects-Interactions）
        
-   **预言机操纵（Oracle Manipulation）**
    
    -   风险：依赖单一、不可信价格源
        
    -   防护：使用 Chainlink、多源验证、TWAP 等机制
        
-   **整数溢出/下溢**
    
    -   风险：unchecked 使用不当
        
    -   防护：Solidity 0.8+ 内建检查或 SafeMath
        
-   **权限控制缺失**
    
    -   风险：管理函数可被任意调用
        
    -   防护：使用 onlyOwner / AccessControl
        
-   **未初始化代理合约**
    
    -   风险：被他人初始化并接管合约
        
    -   防护：部署时强制初始化，锁定实现合约
        
-   **前置交易 / 三明治攻击**
    
    -   风险：被抢跑导致极端滑点损失
        
    -   防护：设置滑点限制、使用私有交易通道

**Gas 优化总结**

Gas 是 EVM 执行指令的计量单位，每一步操作都会消耗固定 gas。Gas 优化的核心目标是**减少不必要的高成本指令（尤其是 storage 读写）**，从而降低交易费用、提升用户体验。

**常见优化要点：**

1.  **减少存储操作（Storage Write / Read）**
    
    -   第一次 `SLOAD` 约 2100 gas，后续约 100 gas；而内存读取仅约 3 gas
        
    -   `SSTORE` 写入成本高达 20,000 gas
        
    -   多次使用同一 storage 变量时，应先缓存到 memory，集中一次写回
        
    
    ```
    // ❌ 非优化
    balances[msg.sender] += msg.value;
    
    // ✅ 优化（一次读，一次写）
    uint256 current = balances[msg.sender];
    balances[msg.sender] = current + msg.value;
    ```
    
2.  **位压缩（Bit Packing）**
    
    -   将多个小类型变量组合进同一个 `uint256` 存储槽，减少 storage 占用
        
    
    ```
    struct Packed {
        uint128 a;
        uint128 b;
    }
    ```
    
3.  **循环优化**
    
    -   避免在循环中重复读取 `arr.length` 等 storage 值，提前缓存
        
    
    ```
    uint256 len = arr.length;
    for (uint256 i = 0; i < len; ++i) {
        ...
    }
    ```
    
4.  **函数可见性选择**
    
    -   仅供外部调用的函数使用 `external`，比 `public` 更省 gas
        

**一句话总结：**  
Gas 优化的本质是**减少 storage 访问、合并计算、避免重复操作**，把昂贵的链上执行尽量压缩到最少次数。

## 区块链前端整合

区块链前端整合的核心目标，是**让普通用户通过网页与链上智能合约完成交互**，形成完整的“部署 → 使用 → 反馈”闭环。

整体流程包括：前端检测并连接钱包 → 用户授权账户 → 通过 ABI 和合约地址实例化合约 → 调用合约函数 → 钱包签名交易 → 交易广播上链 → 前端获取结果并更新状态。  
其中，钱包负责身份与签名，前端负责交互与状态展示，区块链负责最终执行和确认。

技术栈方面，**Solidity 合约 + 前端区块链交互库**是基础能力。Web3.js / Ethers.js 有助于理解流程，但在新项目中更推荐使用 Viem、Wagmi 等现代方案；钱包连接、RPC 节点和部分状态管理通常由第三方组件提供。

通过本章的实例，将已部署在测试网的合约接入 Web 前端，实现留言等功能，完成了从**合约开发到用户可用产品**的关键一步，也真正体现了 Web3 应用“链上逻辑 + 链下界面”的协作模式。

![屏幕截图 2026-01-17 195825.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/mizuki258/images/2026-01-17-1768651215270-_____2026-01-17_195825.png)

本前端页面采用 **HTML + JavaScript + CSS** 构建，为单页面 DApp 结构，核心依赖 **Web3.js** 实现与以太坊区块链的交互，整体代码规模约 650 行。

前端首先通过浏览器中的以太坊钱包（如 MetaMask）提供的 `ethereum` API 连接用户钱包，请求账户授权，并校验当前网络是否为 **Sepolia 测试网**。在完成钱包连接后，前端基于合约的 **ABI（应用二进制接口）** 与合约地址，使用 Web3.js 实例化合约对象，从而具备调用链上函数的能力。

在合约交互层面，写入操作通过 `.send()` 方法发起交易，需要用户在钱包中签名并支付 Gas，用于提交留言等状态变更行为；读取操作则通过 `.call()` 方法完成，不消耗 Gas，可用于查询留言数量及具体内容。  
通过区分写操作与只读操作，前端能够高效、安全地与链上数据交互。

通过上述流程，前端实现了 **钱包连接、合约绑定、链上留言写入与读取** 等核心功能，完整串联了「合约开发 → 合约上链 → 前端交互」的 DApp 基本技术路径，形成了一个可运行的以太坊 DApp 原型。

## 《021学习以太坊》第二章总结

在了解以太坊节点和客户端软件之后，逐渐意识到“以太坊不是一条链，而是一套协同运行的软件系统”。所谓节点，本质上就是运行以太坊客户端的服务器；而真正维持网络运行的，是**多种客户端软件之间的协作**。

### Web3运行原理

**交易**本质就是：做什么 + 给多少 Gas + 防重放的 nonce。  
钱包（如 MetaMask）负责把交易组装好、用私钥签名，再广播到链上，签完就能在浏览器里查到。

**数字签名**保证两件事：  
别人能验证“是你发的”，但没人能伪造。私钥一旦泄露，等于账户完全失控。

**Gas 机制**就是手续费系统：  
操作越复杂越费 Gas，网络越堵 Gas 越贵，用来防垃圾交易、激励节点干活。

**交易流程**很简单：  
签名 → 发到网络 → 排队 → 被打包进区块 → 共识确认。  
新区块先是未最终确认，约十几分钟后 Finalized，才算真正不可逆。

**共识机制**：

-   比特币用 POW（拼算力，耗能）
    
-   以太坊用 POS（质押 ETH，当验证者，作恶会被罚）
    

**RPC**是钱包和区块链之间的“入口”，有中心化风险，但多 RPC 并存能降低单点问题。

**智能合约**就是写在链上的程序：  
规则公开、不可随意改，调用时在虚拟机里执行，结果写入区块链。  
本质是把“信任人和机构”变成“信任代码和共识”。

**NFT Mint**本质也是一次合约调用，只是比转账复杂，成功后仍需等待最终确认。

# 区块链

定义：区块链是一种去中心化的分布式账本技术，用于在网络节点之间安全、透明且不可篡改地记录事务数据。每条链由一系列按照时间顺序相连的“区块”组成，每个区块内部包含了多笔交易数据及元数据，确保了数据记录的完整性与可追溯性。

特性：**不可篡改 ：**你无法改变历史信息，因为每个区块包含了上一个区块的摘要并串联起来，如果你修改了历史的区块，你将必须修改后面的全部区块

**公开透明、匿名 ：**在区块链上的信息全部公开透明。每个人都可以顺着区块和链找到历史上所有的记录来查看你的钱包余额。但是没人知道这是你的钱包

**快速交易 ：**无论金额多少以及你在什么地方，只要你的交易记录被打包在区块链中，交易就自动完成。相比传统的跨国汇款非常快速便捷。

区块链运行的步骤：

1.  **用户发起交易**：用户通过钱包应用发起转账、智能合约调用等操作
    
2.  **交易广播**：交易信息被广播到整个网络中的各个节点
    
3.  **节点验证**：网络中的矿工节点验证交易的合法性（余额是否足够、签名是否正确等）
    
4.  **打包成块**：通过共识机制（如工作量证明），矿工将验证过的交易打包成新的区块
    
5.  **链接上链**：新区块被添加到区块链上，更新全网的账本状态
    
6.  **奖励发放**：成功打包区块的矿工获得代币奖励和交易手续费
