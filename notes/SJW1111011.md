Building agents with LLM (large language model) as its core controller is a cool concept. Several proof-of-concepts demos, such as [AutoGPT](https://github.com/Significant-Gravitas/Auto-GPT), [GPT-Engineer](https://github.com/AntonOsika/gpt-engineer) and [BabyAGI](https://github.com/yoheinakajima/babyagi), serve as inspiring examples. The potentiality of LLM extends beyond generating well-written copies, stories, essays and programs; it can be framed as a powerful general problem solver.

# Agent System Overview

In a LLM-powered autonomous agent system, LLM functions as the agent’s brain, complemented by several key components:

-   **Planning**
    
    -   Subgoal and decomposition: The agent breaks down large tasks into smaller, manageable subgoals, enabling efficient handling of complex tasks.
        
    -   Reflection and refinement: The agent can do self-criticism and self-reflection over past actions, learn from mistakes and refine them for future steps, thereby improving the quality of final results.
        
-   **Memory**
    
    -   Short-term memory: I would consider all the in-context learning (See [Prompt Engineering](https://lilianweng.github.io/posts/2023-03-15-prompt-engineering/)) as utilizing short-term memory of the model to learn.
        
    -   Long-term memory: This provides the agent with the capability to retain and recall (infinite) information over extended periods, often by leveraging an external vector store and fast retrieval.
        
-   **Tool use**
    
    -   The agent learns to call external APIs for extra information that is missing from the model weights (often hard to change after pre-training), including current information, code execution capability, access to proprietary information sources and more.

黑客松不仅是技术的比拼，更是团队协作的考验。

好的创意是成功的一半，评审更看重解决方案的现实意义。

-   **问题清晰化**：能够简练地解释正在解决的具体问题，以及该问题的真实社会需求或商业价值。

-   **活用 AI 与工具**：利用生成式 AI 加速代码编写，减少重复性劳动，专注于逻辑优化。
    
-   **分工明确**：确保每个人都有具体的任务，避免多人重叠工作，提高并行开发效率。

## 一、 核心概念：什么是 Reactive 合约？

**Reactive 合约** 是一种能够实现“跨链自动化”的智能合约 。与传统合约必须由外部用户手动调用不同，它具备以下特性：

-   **跨链监听与触发**：能够监督一条链上的合约事件（Event），并触发另一条链上指定回调合约（Callback）的执行 。
    
-   **去中心化自动化**：解决了智能合约无法自我发起的限制，无需中心化控制手段即可实现跨平台自动调用 。
    
-   **核心组件 - Event（事件）**：Reactive 通过监听区块链交易日志中的 `topics` 和 `data` 来获取信息 。其中 `topics[0]` 永远是事件的签名哈希 。
    

* * *

## 二、 运行架构：Reactive Network 与 ReactVM

Reactive 合约并非直接部署在以太坊上，而是运行在专有的并行架构中 ：

| 组件 | 功能描述 |
| Reactive Network | 类似于 EVM 运行环境，支持跨链订阅其他链上的事件 。 |
| ReactVM | 一个受限且隔离的虚拟机，仅能与其对应的 Reactive Network 地址交互，保证了安全性与扩展性 。 |

**工作流**：外部合约发出事件 $\\rightarrow$ Reactive Network 接收并传给 ReactVM $\\rightarrow$ ReactVM 发起请求 $\\rightarrow$ Reactive Network 调用目标合约交易 。

* * *

## 三、 实操案例 1：基础自动化演示

该案例展示了如何监控 L1 合约并触发回调 。

### 1\. 代码结构

-   **源合约 (Source)**：收到钱后发出 `Received` 事件 。
    
-   **目标合约 (Callback)**：包含执行逻辑的函数，受 `authorizedSenderOnly`（验证官方通道）和 `rvmIdOnly`（验证部署人身份）修饰符保护 。
    
-   **Reactive 合约 (Core)**：整个系统的核心，负责监听并在 ReactVM 中判断逻辑（如 `log.topic_3 >= 0.001 ether`）以触发回调 。
    

### 2\. 部署步骤（Forge 指令）

1.  **环境配置**：在 `.env` 中配置各链 RPC、私钥及服务地址 。
    
2.  **部署源合约**：使用 `forge create` 部署监控目标 。
    
3.  **部署目标合约**：需要传入目标链的 `callback_proxy_address` 。
    
4.  **部署 Reactive 合约**：构造函数需包含源链 ID、目标链 ID、要监听的事件哈希（topic\_0）及回调地址 。
    

* * *

## 四、 进阶案例 2：Uniswap V2 止损单触发

利用 Reactive 合约监控代币对价格，当触及阈值时自动卖出止损 。

-   **环境准备**：部署模拟代币（TK1/TK2）及 Uniswap V2 交易对 。
    
-   **流动性注入**：通过 `transfer` 和 `mint` 向资金池添加流动性 。
    
-   **设置阈值**：部署 Reactive 合约时设置 `EXCHANGE_RATE_NUMERATOR`（如 0.9），表示当汇率跌破该值时触发 。
    
-   **验证触发**：人为进行一次“吃亏”的 `swap` 改变池内比例，使汇率触及阈值，随后观察 Reactive 合约自动完成止损交易 。
    

* * *

## 五、 部署关键注意事项

-   **逻辑区分 (**`vm` **标识)**：在 Reactive 合约的代码中，通过判断地址 `0x0000...fffFfF` 上是否有代码来区分是在 Reactive Network 还是 ReactVM 上运行 。
    
-   **权限验证**：目标合约必须严格检查调用来源，确保交易是由官方代理地址且由本人部署的智能体发起的 。
    
-   **数据查询**：可以通过 `lasna.reactscan.net` 搜索 Reactive 合约地址，实时追踪触发状态和日志 。

这份笔记根据文档总结了 Uniswap 从 V2 到 V4 的演进历程、核心数学原理以及生态系统概况。

* * *

## 1\. Uniswap V2：恒定乘积 AMM 基础

Uniswap V2 奠定了自动做市商（AMM）的基础架构。

-   **核心公式**：采用恒定乘积公式 $x \\cdot y = k$，其中 $x$ 和 $y$ 代表池中两种资产的库存量，$k$ 为常数。
    
-   **价格机制**：代币价格 $p$ 定义为 $y/x$ 。交易产生的价格偏差会吸引套利者，从而推动价格向市场价靠拢。
    
-   **功能特性**：
    
    -   **闪电贷**：支持资金的先借后还，极大提升了链上资金效率。
        
    -   **TWAP 预言机**：引入时间加权平均价格机制，为其他协议提供可靠的链上价格参考。
        

* * *

## 2\. Uniswap V3：集中流动性与效率优化

V3 通过引入“集中流动性”概念，解决了 V2 资金利用率低的问题。

-   **集中流动性区间**：流动性提供者（LP）可以在自定义的价格区间内提供流动性，而非全曲线分布。
    
-   **Tick 系统**：将价格空间离散化，通过 Tick 管理流动性的变化点。
    
-   **手续费分档**：提供多种手续费层级，如稳定币对（0.01% - 0.05%）、主流对（0.3%）和高波动资产（1%）。
    
-   **数学优化**：使用 $L$（流动性）与 $\\sqrt{P}$（价格平方根）公式简化计算，提高合约执行效率。
    

* * *

## 3\. Uniswap V4：高度自定义的钩子机制

V4 的核心在于提升协议的灵活性和扩展性。

-   **钩子机制 (Hooks)**：允许开发者在交易、添加或移除流动性的特定节点插入自定义逻辑。
    
-   **应用场景**：通过钩子可以实现动态手续费、链上限价单、复杂的流动性策略调整以及交易保护。
    

* * *

## 4\. 核心风险与操作建议

### 风险管理

-   **无常损失 (IL)**：这是 AMM 机制固有的风险，指资产波动时相对于单纯持币的价值损失，需通过手续费收入来补偿。
    
-   **滑点与价格影响**：价格影响是由池子深度决定的可计算结果；滑点则是实际成交价与预期价之间的不可预测偏差。
    
-   **防范抢跑攻击**：建议使用隐私交易池（如 Flashbots）、设置合理的滑点容忍度或利用 V4 钩子进行逻辑保护。
    

### LP 操作建议

-   **区间设置**：波动大的资产建议设置更宽的区间或选择更高手续费档位。
    
-   **接入方式**：Uniswap 官方没有独立 App，用户需通过网页端或 MetaMask 等第三方钱包接入。
    

* * *

## 5\. Uniswap 生态资源

Uniswap 通过多种渠道支持开发者和社区发展：

-   **资金支持**：Uniswap Foundation 提供资助、教育计划及审计补贴。
    
-   **开发者活动**：定期举办黑客松、孵化器项目以推动技术创新。

这份笔记总结了文档中关于 2026 年 **智能体 AI (Agentic AI)** 的演进趋势、核心挑战，以及 Web3 技术如何通过底层协议栈构建“机器经济”的解决方案。

* * *

## 1\. AI 智能体的演进：从“聊天”到“打工”

2026 年，AI 正在经历从传统大语言模型（LLM）向智能体 AI 的重大突破 。

-   **传统 LLM 的局限性**：存在无状态（无记忆）、无工具（仅能生成文本）、无身份（无法代表实体行动）等局限。
    
-   **智能体 AI 的突破**：
    
    -   **私有记忆**：通过向量数据库实现。
        
    -   **工具使用**：通过函数调用（Function Calling）或 MCP 获得外部工具操作能力。
        
    -   **链上身份**：通过 ERC-8004 等标准获得唯一身份，使 AI 能够实现“思考、行动并赚钱”的闭环。
        

* * *

## 2\. AI 智能体面临的核心挑战

在迈向独立经济实体的过程中，AI 智能体面临三大瓶颈：

1.  **身份与信任缺失**：难以验证请求方是 AI 还是人类，容易产生身份欺诈风险。
    
2.  **行为不可审计**：本地存储的决策日志易被篡改，缺乏第三方验证，导致法律追责困难。
    
3.  **支付基础设施不匹配**：传统信用卡支付受限于验证码、KYC 及高昂手续费，无法支持 AI 经济所需的高频、小额、实时微支付场景。
    

* * *

## 3\. Web3 解决方案：三层协议架构

Web3 通过构建专用的协议栈，为 AI 智能体经济提供了基础设施支持。

| 层次 | 协议/标准 | 核心功能 |
| 交易层 | X402 协议 | 将支付嵌入 HTTP 协议（402 状态码），实现无需注册、实时结算的原生微支付。 |
| 身份与声誉层 | ERC-8004 标准 | 为 AI 铸造唯一 NFT 身份证；包含基于真实交易的声誉评分系统，防范刷分风险。 |
| 应用层 | Spoon OS | Web3 AI 操作系统框架，封装钱包管理、身份验证等复杂底层，提供简单 API 。 |

* * *

## 4\. Spoon OS 框架详解

Spoon OS 定位于 AI 智能体经济的操作系统，旨在让开发者专注于业务逻辑 。

四层架构分析 ：

-   1\. 输入层：负责数据采集与预处理，支持向量数据库 。
    
-   2\. 核心层：包含大模型接口、智能体编排引擎、记忆管理器与工具调用器 。
    
-   3\. 协议层：集成身份管理（ERC-8004）与支付网关（X402），提供钱包签名服务。
    
-   4\. 输出层：实现与外部世界的交互，如 DeFi 连接器和跨链桥 。
    

* * *

## 5\. 核心问题答疑 (Q&A) 摘要

-   **关于速度**：在跨境及微支付场景下，Web3 的点对点结算效率高于受合规审查限制的 Web2 支付系统。
    
-   **关于责任归属**：AI 行为可追溯至其链上身份，责任主要由开发者承担（基于其设计和训练） 。
    
-   **防范攻击**：ERC-8004 通过提高经济成本（需真实交易才能评价）来抑制“女巫攻击”刷好评的行为。

-   **局限性**：仅能降低表面风险，无法规避业务实际运营中的核心法律风险 。
    

### 2\. 深度合规（第二层面/实际运营）

-   **目标**：在业务模式、资金流向及用户交互等实际操作中规避法律红线 。
    
-   **关键行动**：
    
    -   严禁 ICO/IEO/IDO 等公开募资行为 。
        
    -   在技术和运营层面彻底隔离中国大陆用户，不提供交易或支付服务 。
        
    -   远离赌博、洗钱、传销等刑事风险，并建立具备真实技术支撑的业务闭环 。
        
-   **挑战**：需要建立内部风控体系，持续跟踪跨境法律冲突及监管动态 。

## 二、 钱包与个人主权

Web3 是“个人主权”的起点，其核心在于无需许可的账户体系 ：

-   **无需许可**：任何人都可以随机创建私钥并拥有收款地址，立刻获得管理资产的权利，无需银行柜台审批或身份证验证 。
    
-   **私钥与助记词的关系**：助记词是“种子”或“根”，与私钥是一对多的关系，可以衍生出无数个私钥 。
    
-   **唯一性**：每一个私钥对应唯一的公钥和地址 。
    

* * *

## 三、 交易执行与签名机制

交易（Transaction，简写为 TX）在区块链上的完整生命周期涉及多个环节 ：

### 1\. 交易的构成

一笔交易通常包含：你要执行的操作（如转账、投票）+ 手续费（Gas Fee）+ 防重放序号（Nonce） 。Nonce 确保交易不会在其他链上被重复执行 。

### 2\. 数字签名流程

-   **组装与签名**：钱包（如 MetaMask 或 Rabby）帮助用户组装交易内容，并使用私钥生成数字签名 。
    
-   **广播与验证**：交易通过 RPC 节点广播到网络 。其他节点使用你的原始信息、声称的地址和签名，通过密码学算法反推出签署者，从而验证信息的真实性 。
    
-   **安全性核心**：只要私钥不泄露，他人无法伪造签名；反之，拥有私钥者即可代表该账户的一切行为。
    

### 3\. Gas 费用机制

-   **定义**：Gas 可以理解为手续费、邮费或汽油 。
    
-   **计算公式**：总开支 = Gas 单价（Gas Price） $\\times$ 消耗量（Gas Usage）。
    
-   **作用**：防止垃圾交易淹没网络，并为维持网络运行的节点提供激励 。
    
-   **市场波动**：当网络拥堵（如很多人抢购）时，Gas 单价会随之上涨 。
    

* * *

## 四、 区块链运行与共识机制

### 1\. 区块打包过程

交易发出后会进入“池子”排队 。节点会优先挑选 Gas 费用高的交易进行打包。以太坊目前约每 12 秒产出一个区块 。

### 2\. 共识算法对比

| 特性 | POW (工作量证明) | POS (权益证明) |
| 原理 | 竞争解决数学难题（做题） | 质押押金（32 ETH）进入记账员列表 |
| 激励与惩罚 | 算力高者优先获得奖励 | 随机挑选记账员；若造假则没收押金（罚没机制） |
| 典型代表 | 比特币 | 以太坊（目前阶段） |

### 3\. 不可篡改性与最终性

区块链通过哈希值（Hash）将区块相连：每个区块包含指向前一个区块的 `Parent Hash` 。

-   **篡改难度**：若修改一个区块，则需修改其后所有区块。
    
-   **最终确认**：以太坊交易通常需要等待 12-13 分钟达到 `Finalized` 状态，此时被篡改的可能性几乎为零，商家才可放心发货。
    

* * *

## 五、 以太坊基础设施与治理

### 1\. 节点与客户端

-   **中心化风险**：钱包通常连接 RPC API 而非直接连入区块链，若 RPC 服务商拦截交易，则资产无法上网 。
    
-   **多样性重要性**：以太坊节点软件分为执行层和共识层，使用不同语言编写。软件多样性可防止因单一 Bug 导致的网络宕机或分叉。
    
-   **全球分布**：目前全球约有 1.3 万个节点服务器，主要分布在美国、德国和中国 。
    

### 2\. EIP 治理流程

以太坊通过以太坊改进提案（EIP）进行社区协作升级 ：

1.  **论坛讨论** 。
    
2.  **形成草案（Draft）**。
    
3.  **审核与最后通牒（Last Call）**。
    
4.  **最终确定（Final）与硬分叉升级** 。
    

* * *

## 六、 Web3 的社会学意义与未来思考

Web3 不仅是技术，更是跨学科的领域，涉及社会学、经济学和密码学：

-   **权力重分配**：实现数字资产与权力的自我控制，通过智能合约让规则自动执行 。
    
-   **核心价值**：去中心化、无许可、抗审查、开源透明 。
    

### 留待思考的问题

1.  **安全性与复杂性**：如何在不降低安全性的前提下，降低普通人管理私钥的复杂度？
    
2.  **公共基础设施维护**：在没有传统政府税收的情况下，谁来维护节点和客户端等基础设施？
    
3.  **有害信息治理**：在抗审查和隐私性极强的环境下，如何鉴定与治理黑产或有害信息？
    
4.  **公平分配**：在互不信任的去中心化协作中，如何实现公平可信的利益分配？

## Abstract

This ERC proposes a standard presentation layer transformation for Ethereum addresses containing long sequences of identical hexadecimal digits. It defines two representation formats: a Unicode-based format using superscripts for graphical user interfaces (UI), and an ASCII-safe fallback format using bracket notation for logs and terminals. This standard aims to improve human readability and safety verification without altering the underlying address data.

## Motivation

As the Ethereum ecosystem matures, addresses with long repeating sequences are becoming increasingly common due to several factors:

**Vanity Addresses & CREATE2:** Factories and developers frequently generate “vanity” addresses with specific prefixes or suffixes for branding or identification.

**Gas Optimization (EIP-7939):** With proposals like EIP-7939 aiming to reduce gas costs for zero bytes in calldata, there is an economic incentive to deploy contracts at addresses containing large sequences of zeros to optimize cross-contract call costs.

**Security Risks:** The human eye struggles to distinguish between long sequences of identical characters (e.g., counting 10 zeros vs. 11 zeros). This creates a “homoglyph-like” vulnerability where users may skim over the middle of an address, missing subtle differences in a phishing scam.

Current truncation methods (e.g., `0x1234...5678`) obscure the internal structure of the address. A standardized compression format allows users to verify the magnitude of the repeated sequence (e.g., “exactly 11 zeros”) at a glance.
