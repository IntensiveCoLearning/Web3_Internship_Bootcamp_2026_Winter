## 1\. Web3 实习手册 ｜ 智能合约开发

### 架构差异 去中心化应用 DApp

DApp (Decentralized App) 不像我们熟悉的 `Client -> Server -> Database` 架构。它更像是一个由多个分布式组件拼装的系统。

**前端 (Frontend)**：依然是 HTML/React/Vue，但它\*\*不直接请求后端 API\*\*。它通过“钱包”（如 MetaMask）注入的 Provider 或 RPC 节点与区块链交互。

**智能合约 (Smart Contract)**：这就是我们的“后端”。它定义业务逻辑，部署在 EVM（以太坊虚拟机）上。逻辑一旦部署，\*\*不可篡改\*\*。

**数据检索器 (Indexer)**：这是 Web3 特有的。因为直接在链上查询复杂数据（比如“查询某用户所有的 NFT”）非常慢且难，所以我们需要一个“中间件”（如 The Graph/Ponder），监听链上事件 (Events)，把数据同步到传统数据库（PostgreSQL）供前端查询。

**存储 (Storage)**：大文件（图片、文档）不存链上（太贵），而是存去中心化存储（IPFS/Arweave）。

这与我熟悉的 Java 后端的不同：

\- **接口调用**：以前我们写 `Controller` 给前端调接口；现在前端直接调链上的 `Function`。

\- **数据库观念**：以前我们有 CRUD（增删改查）；在区块链上，\*\*Read\*\* 是免费的（通过 RPC），但 **Write**（交易）是要花钱的（Gas 费），而且非常昂贵。所以我们不能像操作 MySQL 那样随意写入数据，必须“省着写”。

### 为什么需要 RPC 网关

区块链本质上是一个由成千上万台计算机（节点）组成的 **P2P 网络**。这些节点之间通过一种复杂的 P2P 协议（Gossip Protocol）同步数据。

前端网页（JS）并不懂这种 P2P 协议。你没办法直接通过 TCP 连接去“加入”这个网络。

你需要一个“中间人”，它既是区块链网络的一员（存了全量数据），又能听懂 HTTP 请求。这个中间人就是 **RPC 节点**。

它有什么作用呢？

• **读取数据 (eth\_call)**： 当你查询余额时，你其实是发了一个 HTTP 请求给 RPC 节点。RPC 节点本地就存了区块链的“账本副本”，所以它\*\*直接查本地数据库\*\*返回给你，不需要去问全网。这也是为什么读数据是\*\*免费\*\*且\*\*快\*\*的。

• **写入数据 (eth\_sendRawTransaction)**： 当你转账时，你把签好名的交易发给 RPC 节点。RPC 节点会把这笔交易\*\*广播\*\*给它相连的其他 P2P 节点，最终被矿工打包进区块。

\### Solidity 编程：更严格的“Java”

Solidity 是静态类型语言，语法有点像 JavaScript 和 C++ 的混合，但逻辑上更接近 Java 的强类型思维。

• **合约结构**`contract MyContract {}` 类似于 Java 的 `class MyContract {}`。它也有构造函数 `constructor()` 和状态变量。

• **状态变量 (State Variables)**：定义在函数外的变量，\*\*永久存储在区块链上\*\*。修改它们需要消耗 Gas。

• **修饰符 (Modifiers)**：类似 Java 的 AOP（面向切面编程）或注解。例如 `onlyOwner`，在执行函数前先检查权限。

• **事件 (Events)**：这是 Web3 的“日志系统”。合约触发 `emit Event()`，前端或索引器监听这些日志来更新 UI，而不是轮询数据库。

### **Java开发者的思考**

• **Map 的区别**：Solidity 里的 `mapping(address => uint)` 类似于 Java 的 `HashMap`，但它\*\*无法遍历\*\*（没有 `.keys()` 或 `.size()`）。如果你需要遍历数据，必须自己维护一个数组来记录 Key。

• **异常处理**：没有 `try-catch` 这种复杂的捕获机制。主要是 `require(condition, "Error Msg")`，一旦失败，整个交易回滚 (Revert)，就像数据库事务回滚一样。

## 1\. **Web2 开发者向 Web3 转型** （Day 2: Wallets, Mnemonics, Keypairs）

### 账户本质

看视频主要需要理解区块链账户到底是什么。 与传统银行账户需要申请和审批不同，在以太坊上创建账户是完全\*\*无许可（Permissionless）\*\* **确定性 (Deterministic)** 的。只要你有那12个单词，无论在哪里恢复，都能按顺序生成 Account 1, Account 2... 这种索引关系永远不会变

### 概率与信任

关于“会不会被暴力破解”的疑问，这涉及到我们如何信任数学。 我们的“穴居人大脑 (Caveman Brain)”可能会觉得猜出12个单词很容易，但数学告诉我们要信任概率。猜中私钥的难度相当于每次洗扑克牌都洗出完全一样的顺序。 即使动用全世界的算力每秒猜测一万亿次，猜中有效私钥所需的时间也超过宇宙目前的寿命。所以，与其担心被暴力破解，不如担心自己把助记词泄露给钓鱼网站或保存在联网的设备上。

### 资产托管进阶

钱包使用的几个阶段：

L1 **托管 (Custodial)**：像 Coinbase/币安。资产只是中心化数据库里的一个数字，你并不真正拥有它（Not your keys, not your coins），但适合新手入门。

L2 **非托管 (Non-Custodial)**：像 MetaMask。你自己掌握私钥，拥有绝对控制权，但也承担绝对责任（弄丢就完了）。

L3 **智能合约钱包 (Smart Contract Wallet)**：像 Gnosis Safe 或 Argent。这是未来的方向，通过多重签名 (Multisig) 或“监护人”机制来实现社交恢复。即使丢了私钥，也能通过家人或备用设备找回，解决了单点故障的焦虑。

作为 Web3 初学者，核心要点是从用户转变为拥有者。你不需要任何机构的许可，只需通过数学生成一串随机数即可创建账户，但这同时也意味着你必须独自承担安全责任——助记词一旦丢失或泄露，资金将永久无法找回，因为区块链世界没有客服。入门的最佳实践是下载 MetaMask，学会使用 **Etherscan**（无私的第三方）来验证链上事实，并在存入真实资金前进行“自杀式测试”\*\*（删除钱包并尝试通过助记词恢复），以此真正理解“做自己的银行”的含义。

## 1\. **Web2 开发者向 Web3 转型的入门课程介绍**（Day 1 Intro）

### 整体概念

看视频还是主要了解一下以太网的整体概念。

与普通中心化不同的是，区块链是一个由成千上万个节点组成的点对点网络，没有中心控制。智能合约一旦部署，就无法更改（Code is law），每个人都要遵守相同的规则，即使是开发者自己也无法停止它。

它主要解决了信任的问题，你不需要信任别人，你只需要信任代码的逻辑。所有规则都已经写好了，通过正确的规则与激励机制来引导行为。而且由于它是去中心化的，我们中心化的项目需要主从来备份，但是区块链天生就是去中心化的，整个p2p网络就是冗余，每个节点都会保存所有智能合约，任何更新都会同步到所有节点。

## 1\. 阅读021 学习以太坊第 3 章

记录一下比较重要的几个概念：

1\. 账户的几个重要的状态

**Nonce**：计数器。EOA 记录发了多少交易；合约记录创建了多少子合约。这里也就放孩子了重放攻击，如果重复广播，发现就得交易就会失效

**Balance**：余额。持有的 ETH 数量（单位：Wei）。

**StorageRoot**：哈希值。指向该账户的私有数据库（KV 存储）。

**CodeHash**：哈希值。指向该账户的可执行字节码。

2\. 两种账户类型：

EOA：由私钥控制的账号，我们的 okx wallet、metamask 钱包

合约账户：部署在链上的代码逻辑，而且逻辑已经写死，不可篡改。（如果要进行操作，需要使用代码的逻辑来判断 例如：require(msg.sender == admin)）

3\. 资产的形式（这个从 eth scan 上也能看出来）：

1\. ETH：这个是记录到账户的 balance 字段里面的

2\. 代币（ERC-20/NFT） 本质上是通过查询向代币合约的地址查询你的地址，最终得出来你的账户里持有多少代币，而因为你有私钥，你是全网唯一能够修改代币合约 storage 这一行 Entry 权限的人

4\. 账户之间的交流：

外部调用`EOA -> 合约`。需要签名，消耗 Gas，产生顶层交易记录。

内部调用`合约 -> 合约`。通过 `call` 或 `delegatecall` 实现。

阅读完今天的内容，也解答了我之前的疑惑。那就是为什么 okx wallet 之类的钱包，你只有一套助记词，为什么我们可以直接创建另外的小号，但是为什么没有新建新的一套助记词。因为这都是使用一套助记词派生出来的，而且使用一套助记词，按照顺序创建出来的账户都是固定的。同时也理解了为什么在交易所想要快速出售代币的时候，需要找到代币合约的地址，然后通过向合约地址提前授权然后进行交互。因为这些代币根本就不存在你的账户里，只是把你的账户交易的地址记录到合约里。

## 1\. 阅读021 学习以太坊第 2 章

1\. 节点双大脑：执行客户端EL，公式客户端CL

**执行客户端（EL）：** 它是干苦力的。专门负责算账、跑智能合约、处理交易。就像是法院里的书记员，所有的案卷细节、账户余额都在它手里。常见的有 Geth 和 Nethermind。

**共识客户端（CL）：** 它是拿主意的。负责跑 PoS 协议，决定哪条链才是正统，谁有资格出块。它就像是法官，根据书记员提供的材料做裁决。常见的有 Lighthouse 和 Prysm。

2\. 节点之间如何交流的：

**找邻居：** 新节点刚上线时，会先找几个长期在线的引导节点要一份通讯录，然后通过一种叫 Kademlia 的算法，在网络里一传十、十传百，直到把自己的好友列表填满。

**传八卦（Gossip协议）：** 当一个节点收到新交易或新区块时，它不会发给所有人，而是随机告诉几个邻居（不是全部）。邻居再告诉他们的邻居，几秒钟之内，全网就都知道这个八卦了。这种方式非常抗造，就算一部分节点掉线，信息也能传开。

3\. 运行节点的选择：

**全节点（Full Node）：** 绝大多数人的选择。它存了所有的区块数据，但会把很久以前的旧账删掉一些（修剪）来省空间。它能独立验证所有交易，不求人，安全性最高。

**归档节点（Archive Node）：** 巨无霸。它存了从创世第一天到现在所有的历史快照。如果你想查三年前某个账户在某个瞬间有多少钱，只能找它。缺点是太占硬盘，动不动就要十几个 TB。

**轻节点（Light Node）：** 极简版。它只存区块头（相当于书的目录），不存具体内容。想看细节时再去问全节点。适合手机或者配置很低的设备。

4\. 最重要的！数据分析的方法

**Dune：** 重点！只要会写 SQL 就能查全链数据，不用自己维护节点。

**Google BigQuery：** 谷歌已经把以太坊数据整理成了数据库，适合大规模的数据挖掘。

**Etherscan API：** 适合写点小脚本，查查交易记录。

原来以太坊的运行底层还是挺有趣的，同时也了解到了 Dune 这个数据分析平台，之前看到别人统计钱包的交易数据就是使用的这个，图表以及数据统计也写的很精美。之后可以深入学习使用一下，毕竟感觉用的人还是挺多的。

\- ETH 代币不只是用来支付 gas，还是 PoS 质押资产，生态资产底座（比如说很多 NFT 都是使用 ETH 来计价）

\- Gas 的计价： **Gas Fee ≈ (Base Fee + Priority Fee) × Gas Used**

\- **baseFee**：协议根据拥堵自动调节，\*\*执行时直接销毁\*\* 矿工主要收入是 PoS 的质押奖励 以及可能的 MEV

\- 与比特币的对比

\- BTC 价值存储，这就是为什么会出现囤 BTC 的思想

\- ETH 主要体现在生态方面，出块速度快，也就代表了速度快 可编程的特性有了更多的可能性

\- dApp 和普通 Java 后端的区别（一个后端程序员的思维）

\- 一个是部署在 EVM 上，一个是跑在特定的服务器上。

\- 数据就是链上的数据，一个是存在数据库里。

\- 链上我的地址就是我的身份，采用私钥来验证，私钥丢了，谁也帮不了你；采用权限管理。

\- 写入=发交易 需要等待打包和确认 速度慢，写入数据库就代表完成来

\- dApp 合约部署之后不可变 所需要的测试 审计，Java 可以发版回滚来修复

\- dApp 由用户来支付 Gas 而程序员需要优化 Gas，费用估计等，Java则是由服务商来承担

\- L2 的出现，支持来大量交易的执行，提升来 TPS，降低来交易费用

阅读完这一章，感觉以太坊不再是我之前简简单单了解的 ETH 这么简单，不再知识价格上上下下波动的那么肤浅。它其实是一台世界计算机，ETH 只是它的原生燃料和核心资产之一。它真正厉害的地方在于智能合约：规则写进代码里，上链以后大家都按同一套规则玩，这也是 dApp 能跑起来、代币标准（ERC-20、ERC-721）能形成生态的原因。我也理解来支持 ETH 价格上涨的理由，他是一个跑应用的重要基础设施。

## 4.Web3 实习手册\[入门导读\]部分

-   记账在之前是银行或者平台自己存储，区块链更像把账本复制很多份，分散到一堆节点手里，每个人都存一份一样的。这样就不会出现某个中心说你没钱你就没钱那种单点控制的感觉。
    
-   区块就是一段时间内打包的一堆交易记录，链就是把区块按时间顺序串起来的结构。后面的区块保存了前面的摘要，所以前面的内容就不能修改了。也就达到了不可篡改的母的。
    
-   而匿名则是因为地址并不能对应谁是谁，除非你公开承认，不然只能知道是哪个地址进行了操作，而不是哪个人。
    
-   51%的机制保证了可信，因为要改变一半以上的节点非常困难
