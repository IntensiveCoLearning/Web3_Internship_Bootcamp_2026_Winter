# 2026-01-25
<!-- DAILY_CHECKIN_2026-01-25_START -->
今天來做gas優化的作業，我優化的是[Solidity by Example](https://solidity-by-example.org/loop/) 的For and While Loop程式碼

原代碼:

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.26;

contract Loop {

function loop() public pure {

// for loop

for (uint256 i = 0; i < 10; i++) {

if (i == 3) {

// Skip to next iteration with continue

continue;

}

if (i == 5) {

// Exit loop with break

break;

}

}

// while loop

uint256 j;

while (j < 10) {

j++;

}

}

}

修改後程式碼:

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.26;

contract Loop {

function loop() public pure {

// 優化後的 For Loop

// 1. 移除 ‘= 0’ (節省初始化 Gas)

// 2. 移除迴圈宣告中的遞增 (i++)，移至迴圈內部以利用 unchecked

for (uint256 i; i < 10; ) {

if (i == 3) {

// 在使用 continue 跳過之前，必須先遞增 i,否則 i 永遠停在 3,導致無限迴圈

unchecked { ++i; }

continue;

}

if (i == 5) {

break;

}

// 將遞增放在迴圈最後，並使用 unchecked 包裹

unchecked {

++i;

}

}

// 優化後的 While Loop

uint256 j;

while (j < 10) {

// 使用 unchecked 包裹遞增邏輯

unchecked {

++j;

}

}

}

}

整體優化思路:

-   **使用** `unchecked` **區塊：** 迴圈計數器（如 `i` 和 `j`）通常不會溢位（因為這理只數到 10），我們可以略過溢位檢查來節省大量 Gas。
    
-   **前置遞增（Pre-increment** `++i`**）：** 雖然在最新編譯器中差異變小，但 `++i` 在舊習慣上比 `i++` 省一點點 Gas（特別是在 `unchecked` 中）。
    
-   **移除預設初始化：** `uint256 i = 0;` 比 `uint256 i;` 稍貴，因為 Solidity 變數預設就是 0，顯式賦值是多餘的。

\# Solidity 學習筆記 (Basic)

這份筆記記錄了學習 [https://solidity-by-example.org/](https://solidity-by-example.org/) BASIC部分的程式碼心得。

視角：熟悉 C++ 的資工系學生 vs. Solidity 新手。

\## Hello World

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 定義了一個合約 `HelloWorld`，裡面有一個公開的字串變數 `greet` 儲存 “Hello World!”。

\> \* **跟 C++ 相比差在哪**:

\> \* **合約 vs 類別**: Solidity 的 `contract` 關鍵字看起來很像 C++ 的 `class`。都有變數 (state variables) 和函式。

\> \* **Pragma**: 第一行的 `pragma solidity` 很像 C++ 的 `#include` 或編譯器指令，用來鎖定編譯器版本，避免版本相容性問題。

\> \* **License**: 必須要在開頭宣告 `SPDX-License-Identifier`，這在 C++ 專案中通常是寫在 README 或個別檔案的註解，但在 Solidity 裡是語法的一部分，很強調開源合規。

\> \* **學到了什麼**: 區塊鏈上的 “Class”，一旦部署上去，它就永遠存在那裡了。

\## First App

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 一個簡單的計數器，有 `get()inc()` (加1)`dec()` (減1)。

\> \* **跟 C++ 相比差在哪**:

\> \* **持久化 (Persistence)**: 這是我覺得最大的衝擊。在 C++ 裡，程式跑完變數就消失了（存在 RAM）。但在 Solidity`count` 是寫在區塊鏈上的（Storage），下次呼叫時它還記得上次的值！這就像 C++ 的變數自動連線到了資料庫。

\> \* **Public Getter**: 宣告 `uint256 public count` 後，Solidity 自動幫我生成了一個 `count()` 的 getter 函式。在 C++ 我得自己寫 `int getCount() { return count; }`。

\> \* **uint**: C++ 通常用 `int` 或 `unsigned int` (32/64 bit)，這裡預設就是 `uint256`。

\> \* **學到了什麼**: 寫合約就是在寫「狀態機」，而且狀態是永久儲存的。

\## Primitive Data Types

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 介紹 `bool`, `uint`, `int`, `address`, `bytes`。

\> \* **跟 C++ 相比差在哪**:

\> \* **Address 類型**: C++ 只有指標 (pointer)，Solidity 有專門的 `address` 類型來存乙太坊地址（20 bytes），這在 C++ 裡可能要用 `std::string` 或 `char array` 來模擬，但在這是原生的。

\> \* **整數大小**: Solidity 分得很細`uint8`, `uint16`… 到 `uint256`。C++ 也有 `uint8_t`，但 Solidity 的計算主要都是為了配合 EVM 的 256 位元架構。

\> \* **學到了什麼**: `address` 是這門語言的核心，跟錢包和合約互動都靠它。

\## Variables

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 展示了 State Variables (狀態變數), Local Variables (區域變數), Global Variables (全域變數)。

\> \* **跟 C++ 相比差在哪**:

\> \* **State Variables**: 就像 C++ 的成員變數 (Member Variables)，但它們是存於區塊鏈上的（硬碟級別）。

\> \* **Global Variables**: 這裡的 Global 是指 `msg.sender` (呼叫者), `block.timestamp` (區塊時間) 這種環境變數，而不是 C++ 那種全域宣告的變數。

\> \* **學到了什麼**: 在 Solidity 寫變數要很小心「它存哪裡」，因為存 State 很貴（要付 Gas）。

\## Constants

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 定義常數。

\> \* **跟 C++ 相比差在哪**:

\> \* 跟 C++ 的 `const` 幾乎一樣，編譯期就決定數值，不佔用 Storage 空間，這點對於節省 Gas 很有幫助。

\> \* **學到了什麼**: 習慣用 `MY_CONSTANT` 大寫命名，這點跟 C++ 風格一致。

\## Immutable

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 定義可以在 `constructor` 裡賦值一次，之後就不能改的變數。

\> \* **跟 C++ 相比差在哪**:

\> \* 很像 C++ 類別中的 `const` 成員變數（可以在建構子初始化列表 `Initializer List` 中賦值）。

\> \* 比 `constant` 靈活，因為可以在部署時決定數值，但比一般變數省 Gas。

\> \* **學到了什麼**: 如果數值是在部署時才確定（例如依賴部署參數），就用 `immutable`。

\## Reading and Writing to a State Variable

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 讀寫狀態變數。

\> \* **跟 C++ 相比差在哪**:

\> \* **成本概念**: 在 C++ 裡`x = y` 幾乎沒成本。但在 Solidity，寫入狀態變數 `setter`) 是要付 Gas 的交易 (Transaction)，而讀取 `getter` via `view`) 如果是外部呼叫通常不用付 Gas。這改變了程式設計的思維：儘量少寫入。

\> \* **學到了什麼**: 寫入操作很昂貴，不要像寫 C++ 一樣隨意更動成員變數。

\## Ether and Wei

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 介紹幣值單位。

\> \* **跟 C++ 相比差在哪**:

\> \* C++ 沒有內建貨幣單位。Solidity 直接能在程式碼裡寫 `1 ether` 或 `1 gwei`，編譯器會自動轉成 `uint` (10^18 wei)。這感覺很酷，語言層級支援金融單位。

\> \* **學到了什麼**: 所有的金額運算最後都是變成 `wei` (最小單位) 來算，避免浮點數誤差（Solidity 根本不支援浮點數！）。

\## Gas and Gas Price

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 解釋 Gas 限制。

\> \* **跟 C++ 相比差在哪**:

\> \* **無窮迴圈**: C++ 寫無窮迴圈只會卡死程式或耗盡 CPU。Solidity 寫無窮迴圈會把你的錢 (Gas) 燒光，然後交易失敗 revert。這是一個強制的停機問題 (Halting Problem) 解決方案。

\> \* **學到了什麼**: 寫程式要考慮「複雜度」，不是為了效能，而是為了不要讓使用者付太多手續費。

\## If / Else

\> **\[對這個程式碼的想法\]**

\> \* **跟 C++ 相比差在哪**: 語法一模一樣 `if (x > y) { ... } else { ... }`。

\> \* **學到了什麼**: 這裡沒有學習門檻，可以直接上手。

\## For and While Loop

\> **\[對這個程式碼的想法\]**

\> \* **跟 C++ 相比差在哪**: 語法跟 C++ 一樣。

\> \* **重要差異**: 在 Solidity 裡要極力避免對一個長度未知的陣列跑迴圈。因為如果陣列太大，Gas 會超過區塊上限，導致合約卡死永遠無法執行。C++ 裡大迴圈頂多跑慢點，這裡會直接壞掉。

\> \* **學到了什麼**: 盡量不要用 Loop，或者要限制 Loop 次數。

\## Mapping

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `mapping(address => uint) public balances;`

\> \* **跟 C++ 相比差在哪**:

\> \* 類似 C++ 的 `std::unordered_map` 或 `std::map`。

\> \* **差異**: Solidity 的 Mapping 無法遍歷 (Iterable)！沒有 `begin()` 或 `end()`，也不能取得 `size`。如果你不知道 Key，你就找不到 Value。

\> \* **預設值**: 存取不存在的 Key 不會噴 Error，會回傳預設值 (0 或 false)，這點跟 C++ `map[]` 運算子行為類似（會自動建立 default constructed object），但這裡更安全一點。

\> \* **學到了什麼**: 如果需要遍歷資料，得自己額外維護一個 Array 存 Keys。

\## Array

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 動態與靜態陣列。

\> \* **跟 C++ 相比差在哪**:

\> \* `push()` 像 C++ `std::vector` 的 `push_back()`。

\> \* `pop()` 像 C++ `std::vector` 的 `pop_back()`。

\> \* **刪除**: `delete arr[index]` 不會改變陣列長度，只是把那個位置的值重置為 0。這跟 C++ `vector::erase` 會移動後續元素不同。

\> \* **學到了什麼**: 如果要像 C++ 那樣移除並縮短陣列，要自己把最後一個元素搬到被刪除的位置，然後 `pop()`。

\## Structs

\> **\[對這個程式碼的想法\]**

\> \* **跟 C++ 相比差在哪**:

\> \* 類似 C++ 的 `struct`。

\> \* 但不能在裡面定義 Member Functions (方法)，純粹是資料結構 (POD - Plain Old Data)。

\> \* **學到了什麼**: 用來打包複雜的參數傳遞很有用。

\## Data Locations - Storage, Memory and Calldata

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 解釋資料存在哪。\*\*這是這部分最重要的一課\*\*。

\> \* **跟 C++ 相比差在哪**:

\> \* **Storage**: 類似硬碟/資料庫，永久存檔。賦值給 Storage 變數是指標操作還是深拷貝 (Deep Copy) 要看情況。

\> \* **Memory**: 類似 C++ 的 Heap (雖然名字叫 Memory，但在函式結束後會清空)，但它不完全像 Stack，因為是可變長度的。

\> \* **Calldata**: 類似 C++ 的 `const reference` `const T&`)，唯讀且直接指向交易輸入數據，最省 Gas。

\> \* **學到了什麼**: 參數盡量用 `calldata`。搞錯 Location (例如把 Storage 指派給 Memory) 會產生隱含的 Copy，浪費大量 Gas。

\## Function

\> **\[對這個程式碼的想法\]**

\> \* **跟 C++ 相比差在哪**:

\> \* **多重回傳值**: Solidity 支援 `return (x, y, z);`，C++ 要用 `std::tuple` 或 `std::pair` 才能做到，這裡語法原生支援。

\> \* **Named Returns**: 可以在宣告時就定義回傳變數名稱 `returns (uint res)`，函式內直接操作 `res` 就好，不用寫 `return`。

\> \* **學到了什麼**: 回傳多個值在處理查詢合約狀態時很方便。

\## View and Pure Functions

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 定義不改狀態的函式。

\> \* **跟 C++ 相比差在哪**:

\> \* `view`: 類似 C++ 的 `const` 成員函式 `int get() const;`)，讀取但不修改成員變數。

\> \* `pure`: 類似 C++ 的 `static` 函式或是 `constexpr`，完全不讀取也不修改成員變數（狀態），只做純運算。

\> \* **學到了什麼**: 標記清楚可以讓編譯器優化，而且 `view` 和 `pure` 函式若從外部呼叫是不花 Gas 的！

\## Error

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `require`, `revert`, `assert`。

\> \* **跟 C++ 相比差在哪**:

\> \* **交易回滾**: C++ `throw exception` 只是中斷程式流程，要自己處理資源釋放。Solidity 的 `revert` 會把這筆交易的所有狀態改變全部「倒帶」(Rollback) 回原點，就像什麼都沒發生過，這在 C++ 很難實作。

\> \* `require`: 像 Assert 但用於輸入檢查，失敗會退還剩餘 Gas。

\> \* **學到了什麼**: 這是合約安全的守門員，把所有檢查寫在最前面 (Check-Effects-Interactions 模式)。

\## Function Modifier

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `modifier onlyOwner { ...; _; }`。

\> \* **跟 C++ 相比差在哪**:

\> \* C++ 沒有這種東西！這比較像 Python 的 Decorator `@decorator`)。

\> \* 那個 `_;` 代表「插入原本函式的程式碼」，這是一種 AOP (Aspect Oriented Programming) 的概念。

\> \* **學到了什麼**: 用來做權限檢查 `onlyOwner`) 或是防重入鎖 `ReentrancyGuard`) 超級方便，程式碼會變很乾淨。

\## Events

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `emit Log("message");`

\> \* **跟 C++ 相比差在哪**:

\> \* C++ 的 Log 通常是寫到檔案 `std::cout` 或 log file)。

\> \* Solidity 的 Event 是寫到區塊鏈的 Log 區域，它是給「前端網頁」或「後端服務」監聽用的，合約自己讀不到 Event！

\> \* **學到了什麼**: 這是合約跟外界溝通的唯一低成本方式。

\## Constructor

\> **\[對這個程式碼的想法\]**

\> \* **跟 C++ 相比差在哪**:

\> \* 概念一樣，物件（合約）建立時執行一次。

\> \* 但合約的 Constructor 執行完後，部署的程式碼 (Runtime Bytecode) 就不包含 Constructor 的部分了，為了省空間。

\> \* **學到了什麼**: 初始化邏輯放這裡。

\## Inheritance

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `contract A is B`

\> \* **跟 C++ 相比差在哪**:

\> \* 語法像 C++ 的 `class A : public B`。

\> \* 支援多重繼承 (Multiple Inheritance)，這點跟 C++ 一樣強大（也一樣容易搞混），Solidity 使用 C3 Linearization 來解決菱形繼承問題。

\> \* 關鍵字是 `virtual` 和 `override`，這點跟現代 C++ (C++11) 的關鍵字完全一樣。

\> \* **學到了什麼**: 寫法很親切，C++ 開發者無痛轉移。

\## Visibility

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `public`, `private`, `internal`, `external`。

\> \* **跟 C++ 相比差在哪**:

\> \* `publicprivate`: 跟 C++ 類似。

\> \* `internal`: 相當於 C++ 的 `protected`（繼承者可用）。

\> \* `external`: C++ 沒有。這是指「只能從合約外部呼叫」，合約內部要呼叫自己得用 `this.func()external` 在處理大陣列參數時比 `public` 省 Gas，因為它直接從 Calldata 讀取資料而不 copy 到 Memory。

\> \* **學到了什麼**: 對外接口優先用 `external`。

\## Interface

\> **\[對這個程式碼的想法\]**

\> \* **跟 C++ 相比差在哪**:

\> \* 相當於 C++ 的「純虛擬類別」 (Pure Virtual Class / Abstract Class with only pure virtual functions)。

\> \* 也就是只有 `.h` 標頭檔宣告，沒有實作 `.cpp`。

\> \* **學到了什麼**: 要呼叫別人的合約（例如 Uniswap），只要有 Interface 就可以，不用知道對方實作細節。

\## Payable

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 讓函式可以接收 ETH。

\> \* **跟 C++ 相比差在哪**:

\> \* 這完全是區塊鏈特有的。C++ 的函式只能收參數，不能收「錢」。

\> \* 如果一個函式沒寫 `payable`，有人轉帳進來會直接噴錯失敗。這是一種語言層級的資產保護。

\> \* **學到了什麼**: 紅色的按鈕（Remix 裡 payable 是紅的），牽涉到錢都要很謹慎。

\## Sending Ether

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `transfer`, `send`, `call`。

\> \* **跟 C++ 相比差在哪**:

\> \* 沒得比，C++ 不處理轉帳。

\> \* **重點**: `call` 是現在推薦的寫法 `(bool sent, ) = _to.call{value: msg.value}("");`，雖然看起來很醜（像是在呼叫底層 API），但它可以避免 Gas 限制問題。

\> \* **學到了什麼**: 不要用 `transfer` 或 `send`，因為它們限制 2300 Gas，現在用 `call` 才是標準做法，但要記得檢查回傳值 `bool success`。

\## Fallback

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: 當呼叫了不存在的函式，或是單純轉錢進來時觸發。

\> \* **跟 C++ 相比差在哪**:

\> \* 類似 C++ 的某些元程式設計 (Meta-programming) 或者動態語言的 `method_missing` 鉤子。

\> \* 它可以捕捉所有未定義的呼叫。

\> \* **學到了什麼**: 這是寫「代理合約 (Proxy Contract)」的核心，讓合約可以升級。

\## Library

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: `using SafeMath for uint;`

\> \* **跟 C++ 相比差在哪**:

\> \* 類似 C++ 的 `static` 工具類別。

\> \* `using A for B` 語法很像 C# 的 Extension Methods，可以讓原本的型別 `uint` 像是多出了新方法 `x.add(y)`。

\> \* **學到了什麼**: 可以用來擴充基本型別的功能，程式碼可讀性變高。

\## Keccak256

\> **\[對這個程式碼的想法\]**

\> \* **做了什麼**: Hash 函數。

\> \* **跟 C++ 相比差在哪**:

\> \* C++ 標準庫沒有內建這麼方便的 SHA3/Keccak，通常要引入 OpenSSL。Solidity 內建 `keccak256`。

\> \* **字串比較**: Solidity 不能直接 `str1 == str2`，必須 hash 後比較 `keccak256(str1) == keccak256(str2)`。這點跟 C++ `std::string` 運算子過載不同，比較麻煩。

\> \* **學到了什麼**: 生成唯一 ID 或簽名驗證都靠它。

\# AI 機器在區塊鏈上的基礎概念

\## Phase 1: 核心技術架構與議題板塊

\### 1. AI 上鏈的限制與驗證路徑

\*   **核心闡述：** 區塊鏈節點的設計初衷是達成共識而非高效計算，其內存與算力難以支撐主流 AI 模型（如 GPT-4）數百 GB 的\*\*權重（Weights）\*\*。因此，AI 無法直接在智能合約內進行\*\*推理（Inference）\*\*。

\*   **技術解決方案：** 採取「鏈下推理，鏈上驗證」的路徑。

    \*   **ZKML（零知識機器學習）：** 透過數學證明推理過程未被竄改，雖精準但計算成本極高。

    \*   **OpML（樂觀機器學習）：** 透過挑戰機制確保誠實。

    \*   **TEE（可信執行環境，如 Intel SGX）：** 作為目前的折衷方案，利用硬件安全隔離區保證模型參數不被洩露，並提供低成本的計算證明。

\*   **💡 比喻：** 想像一位「體弱多病的法官」（智能合約），他無法親自去工地搬運巨石（執行 AI 推理），因此他委託一名「強壯的工人」（鏈下算力節點）在受監控的透明防護罩（TEE）內作業。工人完工後，需交出一份不可偽造的錄像（ZKML 證明），讓法官在法庭內即可確認工作精準無誤。

\### 2. SpoonOS：AI 時代的協作操作系統

\*   **核心闡述：** SpoonOS 為開發者屏蔽了底層複雜性，將 AI Agent 的開發封裝為四個邏輯層：

    1.  **輸入層 (Input)：** 結合\*\*向量數據庫 (Vector DB)\*\* 與 **RAG (檢索增強生成)** 技術，賦予 Agent 檢索外部知識與長期記憶的能力。

    2.  **核心層 (Core)：** 提供「統一 LLM API」，讓開發者能像呼叫操作系統指令一樣呼叫不同的模型（如 OpenAI、Claude 或 Llama），無需處理接口差異。

    3.  **協議層 (Protocol)：** 處理身份（ERC-8004）與支付（x402）協議。

    4.  **執行層 (Action)：** 支援「一鍵上鏈」，Agent 能根據決策自動生成並部署智能合約代碼。

\*   **💡 比喻：** 就像開發手機 App 不需要自己寫螢幕驅動程式，SpoonOS 提供了 AI 時代的「Android 系統」，開發者只需專注於業務邏輯。

\### 3. ERC-8004：Agent 的鏈上身份與主權

\*   **核心闡述：** 為解決企業內部 **Shadow AI（影子 AI，指員工私下使用個人 AI 帳號處理敏感數據）** 導致的審計危機，必須給予 Agent 獨立的鏈上身分。ERC-8004 協議構建了 Agent 的三組核心組件：

    \*   **身份註冊表 (Identity Manager)：** Agent 的鏈上永久護照。

    \*   **信譽評價系統 (Reputation Registry)：** 基於歷史交易紀錄的信用分，解決「誰能被信任」的問題。

    \*   **驗證機制協議 (Validation Manager)：** 追蹤 AI 的 **Provenance（溯源）**，確認 Agent 是否運行在宣稱的安全環境（如特定的 TEE）中。

\*   **💡 比喻：** Agent 的身份是持久的，模型升級就像換一件衣服，但護照號碼（身分證）與信用積分是不變的，這使其在經濟活動中具備可持續的信譽累積。

\### 4. x402 協議與機器經濟 (Agent Economy)

\*   **核心闡述：** 傳統 Web2 的「按月訂閱制」是為人類體力設計的，對高頻交互的機器來說會造成嚴重資源錯配。x402 協議提出了「HTTP 原生支付」，實現 **Pay-per-inference（按推理次數支付）**。

\*   **產業數據支撐：** 

       _穩定幣月結算量已達_ \*1.39 兆美元\*\*，超越 Visa 的 **1 兆美元**。

       _Postman 數據顯示，全球_ \*30% 的 API 流量\*\* 已經是「機器對機器」的交互。

\*   **💡 比喻：** 在 Web2，AI 跨國支付 0.0001 美元數據費需要支付 30 美元手續費並等待 3 天。在 Web3，透過 x402 與 Layer 2，交易手續費趨近於零且秒級結算。這是一台「微型自動投幣機」，機器與機器之間實現了摩擦力趨近於零的經濟活動。

\---

\## Phase 2: Q&A 

\*   **Q：如果 Agent 的模型權重更新，它的鏈上身份（ERC-8004）會失效嗎？**

    \*   **A：** 不會。身份（Identity）是持久的，但對應的「驗證指紋」會更新。這保證了 Agent 在升級技術大腦的同時，仍能繼承過往的信譽評分。

\*   **Q：如何防範 Agent 被惡意攻擊導致錢包資產被盜取？**

    \*   **A：** 透過「權限分離」。Agent 帳戶（通常是 AA 帳戶）應設置每日支出限額，且高額轉帳需結合多簽機制，並由 TEE 環境監控異常行為，在檢測到異常指令時自動暫停錢包權限。

\*   \*\*Q: 既然 Web3 的效能（TPS）目前仍受限，AI 頻繁交互會不會導致網路癱瘓？

\*   **A:** 這正是 Layer 2 或 AppChain 的意義所在。AI 的高頻交互可以發生在專屬的二層網絡上，且透過 x402 的批次結算功能，能將大量微支付壓縮成單一鏈上紀錄，平衡效能與安全。

\*   \*\*Q: 如何保證 AI 不會在鏈上「說謊」或執行邪惡的代碼？

\*   **A:** 透過 ZKML 與 TEE（可信執行環境）。我們不信任 AI 的意識，我們信任的是「代碼的數學證明」。只要證明 AI 運行的模型與權重（Weights）是預先約定好的，就能防止模型被惡意掉包。

\---

\## Phase 3: 矽基生命的崛起與未來啟示

\### 1. 從「App」到「User」的轉變

在 Web2，AI 只是軟件的一個功能；在 Web3，當 AI 擁有錢包（EOA/AA 帳戶）後，它完成了從 **「工具」** 到 **「使用者」** 的身份躍遷。它不再是人類操作的對象，而是能獨立經營資產、優化生產策略的「第一公民」。

\### 2. 可驗證的智慧與合規需求

隨著歐盟 **《AI 法案》(EU AI Act)** 的推進，未來高風險 AI 必須提供不可竄改的運行日誌。Web3 提供的「鏈上存證」將黑盒子轉變為「透明玻璃箱」，實現了 **"Intelligence without Trust"**（無需信任的智能），這是 AI 進入金融與醫療合規領域的唯一途徑。

\### 3. 機器社會的運作邏輯

未來的互聯網將由無數 Agent 構成。與 Web2 依賴於不透明的商業合約不同，Web3 的機器經濟建立在「硬代碼協議」之上。當機器可以自由開戶、賺錢、支付與交易時，人類的角色將從「操作員」轉向「協議設計者」。

\### 4. 總結：Web3 給了 AI 什麼？

如果說 Web2 孕育了 AI 的強大大腦，那麼 Web3 則為 AI 提供了：

\*   **身體（Wallet）：** 具備獨立交互價值與資產的能力。

\*   **產權（Ownership）：** 讓模型參數與數據產出物得以私有化與資產化。

\*   **自由（Decentralization）：** 讓代碼脫離中心化巨頭的單點控制，在去中心化網絡中達成永續運行。

**Web3 是 AI 從「智能工具」演化為獨立「矽基主體」的關鍵基礎設施。**

**(05:00 - 10:00) 錢包、私鑰與助記詞**

首先私鑰 (Private Key) 其實就是你的一個密碼。誰擁有了，誰就可以控制你的資產。我可以隨機生成一個私鑰給大家做演示（演示生成過程）。大家可以看到這是你的 Address (錢包地址)，下面是 Private Key。其實沒有什麼好怕的，它其實就是一個字符串，一個隨機生成的字符串。

私鑰是你的密碼，是你資產管理的一個非常重要的印章。顯然是不能忘記，或者是不能夠讓別人知道的。最常見的方式就是背下來，但是很明顯這個是很難背的。所以後來有一個新的東西叫「助記詞 (Seed Phrase)」。

助記詞是用 12 個或者是 24 個單詞，它能夠變成一個種子，變成一個單詞序列。通過這個序列再加上一些路徑，可以生成無限的私鑰。那你這樣的話，你就記住這個助記詞就可以了。

大家可以看到這個關係圖：\*\*助記詞 -> 私鑰 -> 地址\*\*。

\* **助記詞 (Seed Phrase)：** 它是私鑰的根。

\* **私鑰 (Private Key)：** 它可以推導出公鑰，公鑰再推導出地址。

\* **地址 (Address)：** 這是公開的，別人可以給你轉賬。

這是一個 **一對多** 的關係：一個助記詞可以生成無數個私鑰；但一個私鑰對應一個地址。

私鑰的社會學意義就是「個人主權的起點」。這是密碼學賦予你的可驗證的個人主權。你不需要去銀行開戶，不需要別人給你授權，你自己隨機生成一個數，這就是你的賬戶。

\#### **Part 3：交易、Gas 與 簽名**

**(10:00 - 15:00)**

**Bruce：** 有了錢包之後，我們就要開始有一些交易了。在 Web3 裡面，交易 (Transaction/Tx) 就是你要做的事情（比如轉賬、調用合約）加上手續費 (Gas Fee)，再加上防重放的序號 (Nonce)。

\* **Gas Fee (手續費)：** 這是為網絡資源付費，提供激勵。

\* **Nonce (防重放序號)：** 防止同一筆交易被重複執行。比如你轉給我 100 塊，如果沒有這個序號，我可以拿著你的簽名廣播十次，把你的錢轉光。Nonce 是一個計數器，每發一筆交易就會加 1。

錢包在這個過程中在做什麼呢？它們其實就是幫你組裝這個交易的內容，然後用你的私鑰做一個 **「數字簽名」**，再給你廣播到區塊鏈網絡中。在簽名的過程中，它會用你的私鑰去簽名，生成一個數字簽名，其他人可以用這個簽名來給你做驗證。

**(演示環節)**

我現在發一筆實際的測試網交易。選擇一個賬號，選擇代幣。這個時候錢包會彈出來確認交易。我們可以看到交易的信息：To（轉給誰）、Gas Limit、Nonce 等等。點擊確認，告訴我簽名成功了，交易就已經完成並廣播了。

\#### **Part 4：區塊鏈網絡運行**

**(15:00 - 25:00)**

**Bruce：** 交易提交到網上之後，就進入了區塊鏈的網絡。這個網絡是怎麼運行的呢？

從一筆交易到出塊，它的生命週期是：

1\. **Wallet (錢包)：** 簽名。

2\. **RPC Node：** 錢包通過 RPC 接口把消息傳給節點。

3\. **Mempool (內存池)：** 交易進入一個待處理的池子排隊。

4\. **Builder/Validator (驗證者)：** 他們挑選交易（通常挑 Gas 高的），打包成 Block。

5\. **Block (區塊)：** 區塊生成並廣播。

6\. **Finality (最終性)：** 經過一段時間共識確認，交易不可逆轉。

這裡有一個關鍵點：\*\*不可篡改是需要時間的\*\*。以太坊目前大概是 12-13 分鐘達到 Finality。剛打包進去的區塊（Unfinalized）是有可能被重組或被新的鏈替代的。

**(演示可視化區塊鏈)**

大家看這個動畫（TxStreet），這些小人就是交易，他們在排隊上車（區塊）。這輛車（區塊）每 12 秒發一班。如果你給的小費（Gas）少，你就得在後面排隊；給得多，你就優先上車。這就是 Gas 競價機制的直觀展示。

我們可以在區塊鏈瀏覽器 (Etherscan) 上看到真實的區塊數據。比如這個區塊裡打包了 300 多筆交易。每個區塊都有一個 **Parent Hash**，指向前一個區塊，這樣就連成了一條鏈。如果你要篡改一個區塊，你就必須修改它後面所有的區塊，這在算力上幾乎是不可能的。

\#### **Part 5：智能合約與協議升級**

**(25:00 - 35:00)**

**Bruce：** 智能合約的本質是區塊鏈賬本里的「可執行代碼」。

\* **Code is Law：** 規則一旦部署，就按照代碼執行，不可單方面更改。

\* **去中介：** 比如我們做一個 NFT 的 Mint 合約，規則寫在代碼裡（比如每個地址只能 Mint 一個）。你不需要信任我，你只需要信任代碼。

**(演示 EIP 與協議升級)**

那麼區塊鏈網絡本身（比如以太坊）是怎麼升級的呢？這涉及到 **EIP (Ethereum Improvement Proposals)**。

這是一個去中心化的協作過程：

1\. 有人提出想法 (Idea)。

2\. 在論壇討論 (Discussion)。

3\. 形成 EIP 草案。

4\. 社區審核、測試。

5\. 最後各個客戶端（節點軟件）升級代碼，完成硬分叉升級。

這不是某個公司說升級就升級，而是需要全網的節點運營者主動更新軟件來達成共識。如果大家不更新，網絡就會分叉。

\#### **Part 6：問答環節 (Q&A)**

**(35:00 - 結束)**

**Q1 (Jimmy)：** 關於硬體錢包的安全性，怎麼判斷它是否正常？有沒有推薦的？

**A (Bruce)：** 主要是看這個硬體錢包品牌的歷史，有沒有出過安全事故。盡量選大牌的，比如 OneKey（我們也有合作），或者 Ledger、Trezor 這些老牌的。只要是大品牌，經過時間檢驗的，通常問題不大。

**Q2 (Jimmy)：** 我參加過一些活動，他們宣傳 Web3 像美團之類的應用，Web3 是不是真的要顛覆 Web2？還是一個金融遊戲？

**A (Bruce)：** 我不認為 Web3 會徹底顛覆 Web2 的所有東西。Web3 提供的是一種全新的信任機制和價值轉移方式。它解決的是 Web2 解決不了的問題，比如跨國支付、抗審查、數據主權。有些場景（如高頻低價值的數據交互）可能不需要上鏈。Web3 的核心價值在於：去中心化信任、消除中介摩擦、資產自我掌控。

**Q3 (Lynn)：** 既然私鑰這麼重要，如果我用 MetaMask 這種熱錢包，我的電腦中了木馬，私鑰是不是就丟了？

**A (Bruce)：** 是的。MetaMask 的私鑰是加密存儲在你的瀏覽器本地 (Local Storage) 的。如果你的電腦中了木馬，黑客可以把你的加密文件偷走並嘗試破解密碼，或者直接監控你的剪貼板/鍵盤。一旦私鑰洩漏，資產基本就沒了。所以大額資產一定要用冷錢包（硬體錢包），且助記詞絕對不要觸網（不要截圖、不要存微信收藏、不要打字在聯網設備上）。

**Q4 (Lucia)：** 剛剛演示的打包過程有很多小塊，這些小塊和交易金額有關嗎？還有關於「搶跑 (Front-running)」是怎麼回事？

**A (Bruce)：** 那個動畫裡的小人大小可能代表 Gas 費或者字節大小，不一定直接對應金額。

關於「搶跑」：因為交易在 Mempool（內存池）裡是公開可見的。如果黑客監控到你的私鑰洩漏了，或者發現你有一筆有利可圖的交易（比如套利），黑客可以瞬間發起一筆同樣的交易，但是給更高的 Gas 費。礦工（驗證者）為了利益，會優先打包黑客那筆錢更多的交易。這就是搶跑。這就是為什麼私鑰洩漏後，你自己很難把錢轉出來，因為機器人比你快。

\## Phase 2: 我的想法&總結

基於 Bruce老師 的講解，我們可以將 Web3 理解為一個由密碼學構建的\*\*「全球性、無許可狀態機 (Global Permissionless State Machine)」

\### 1. 信任機制的重構：從「信人」到「信代碼」 (Trust Paradigm)

在 Web2 或傳統金融中，信任是基於「機構信譽」與「法律背書」（例如我們相信銀行不會挪用存款）。

Web3 透過 **Cryptographic Truth（密碼學真相）** 重構了信任：

\* **驗證取代信任 (Don't Trust, Verify)：** 我們不需要相信交易對手是好人，只需要驗證他的數位簽名（Signature）有效，且智能合約（Smart Contract）的代碼邏輯無誤。

\* **確定性執行：** 只要 Gas 足夠且符合合約邏輯，交易必然會被執行。這種「不可阻擋性」消除了人為操作的不確定性（如銀行凍結帳戶或伺服器被拔線）。

\### 2. 價值與數據的權屬：絕對的自我主權 (Ownership & Value)

私鑰揭示了 Web3 資產權屬的本質：

\* **Web2 (帳戶制)：** 你的帳號是騰訊或 Google 資料庫裡的一條記錄。權限是公司「賦予」你的，隨時可收回。

\* **Web3 (私鑰制)：** 資產並不存在於錢包 App 裡，而是存在於區塊鏈網路上。私鑰是對這些資產的\*\*唯一數學控制權\*\*。這種所有權是原生的、物理級別的（掌握私鑰即掌握資產），不依賴任何第三方平台的認可。

也就代表著Web3把「權力」連同「責任」一併還給了你。

\### 3. 去中心化的權衡 (The Trade-offs)

為了實現上述的「去信任」與「主權」，Web3 在當前階段做出了顯著的犧牲，這也是大規模採用的阻礙：

\* **效率與成本：** 為了維護全網共識，每一筆交易都需要全球節點重複計算與存儲，導致 TPS（每秒交易量）遠低於中心化伺服器，且 Gas Fee 使微額高頻交易變得昂貴。

\* **使用者體驗與風險：** 「自我主權」意味著「自我負責」。私鑰管理的複雜度極高，且容錯率為零（不可找回）。如 Q&A 中提到的「搶跑攻擊」，顯示了在這個世界中，普通用戶面對技術性攻擊時的無力感。

\* **基礎設施的悖論：** 雖然底層鏈是去中心化的，但接入層（RPC、錢包、質押池）卻出現了新的中心化巨頭（如 Infura, Lido），這構成了系統性的單點風險。

\### 4. 總結：Web3 的技術哲學

如果 Web3 是一台機器，它的燃料是金錢，齒輪是代碼，但它的靈魂是\*\*「共識」\*\*。

對於正在閱讀這份報告的你，為什麼現在要跳進這個兔子洞？

因為這是人類歷史上第一次，我們有機會用代碼重新定義「契約」、用數學重新分配「權力」。這不只是一場技術革命，這是一場關於如何組織人類社會的社會實驗。

拿好你的私鑰。歡迎來到真實世界。

\## 第五部分：QA 問答環節 (Q&A Session)

\### 👤 案例一：國內執業律師轉型

\* **學員背景：** 國內律師，持有執業證，想轉 Web3 合規。

\* **Emily 詳細建議：**

\## 2. ERC-4804：抵御 NFT 损坏的新屏障

\### 1. 核心觀點

ERC-4804 提出了一種新的「Web3 URL」標準，允許用戶通過 HTTP 風格的鏈接直接讀取鏈上數據（如智能合約內容），旨在解決 NFT Metadata 存儲在中心化服務器或 IPFS 容易丟失的問題。

\### 2. Context & Why

\* **背景知識 (Pain Point)：** 你買的 NFT 圖片通常不是存在區塊鏈上，而是存在 AWS 或 IPFS。如果項目方不繳費或 IPFS 節點下線，你的 NFT 就會變成「破圖」。

\* **名詞解釋 (Web3 URL)：** 類似於我們上網用的 `http://`[，ERC-4804](http://`，ERC-4804) 定義了 `web3://` 協議。它告訴瀏覽器：「不要去問服務器，直接去問以太坊區塊鏈這個合約裡的數據」。

\* **產業意義：** 這讓 NFT 真正實現「永存」。只要以太坊還在，你的圖片和數據就能被讀取，不再依賴項目方的運維。

\### 3. 深度解析

\* **技術細節：**

\* **路徑解析：** `web3://<contract_address>/<resource>`。該標準將 URI 請求轉換為對 EVM 的 `call` 消息。

\* **全鏈上 (Fully On-chain)：** 過去所謂的「鏈上 NFT」往往只是 SVG 代碼存在鏈上，渲染還需前端。ERC-4804 允許將渲染邏輯也寫入合約，瀏覽器直接解析合約返回的數據（如 HTML/SVG）進行渲染。

\* **市場訊號：** OpenSea 和 Curve 等頭部協議已經開始支持或試驗此標準。這可能成為下一代高價值、全鏈上生成藝術或動態 NFT 的基礎設施標準。

\### 4. 延伸思考

\* **技術選型建議：** 對於強調「永久性」和「完全去中心化」的 NFT 項目（如鏈上遊戲資產、DID），應優先評估集成 ERC-4804，這是一個強大的敘事加分項。

\## 5. PeerDAS 對區塊和 Blob 傳播的影響

\### 1. 核心觀點

PeerDAS 是以太坊 Pectra 升級的關鍵技術，通過數據分片（Data Sharding）顯著降低了驗證者的負載，但也帶來了提議者（Proposer）計算量增加和潛在的「計時博弈 (Timing Games)」風險。

\### 2. Context & Why

\* **背景知識：** 以太坊要擴容，需要處理更多的數據包（Blob）。但如果數據太多，普通電腦跑不動節點，網絡就會變中心化。

\* **名詞解釋 (PeerDAS)：** 點對點數據可用性採樣。想像一個超大文件，我們不讓每個人都下載完整版，而是每個人只下載一小塊碎片。只要大家手裡的碎片拼起來能還原文件，就證明數據是安全的。

\* **產業意義：** 這是讓以太坊 L2 費用再降低 10 倍、吞吐量提升的關鍵技術。

\### 3. 深度解析

\* **技術機制：**

\* **Reed-Solomon 編碼：** 將 Blob 數據擴展（Erasure Coding），然後切分成 128 列（Columns）。

\* **子網分發：** 驗證者被分配到不同的子網，只需下載和驗證自己負責的幾列數據，而非全部 128KB\*6 的 Blob。

\* **數據分析結論 (EthPandaOps)：**

\* **延遲改善：** 對於 95% 的驗證者，驗證速度變快了（因為下載量變少）。

\* **尾部風險：** 對於網絡狀況最差的 5% 節點，延遲反而增加。

\* **Timing Games 風險：** 由於 Proposer 需要做更多的計算（KZG 承諾）和數據上傳，強大的 Proposer（如大型礦池）可能會故意拖延區塊廣播時間，以獲取更多 MEV 收入，這會增加區塊重組（Reorg）的風險。

\* **解決方案：** 提議建立一個 **Seeding Network (種子網絡)**，由超高性能節點組成，專門負責快速廣播大區塊和 Blob，輔助普通節點。

\### 4. 延伸思考

\* **基礎設施機會：** PeerDAS 的實施將創造對「高性能數據傳播網絡」的需求，相關的中繼（Relay）服務和 MEV 基礎設施可能有新的商業機會。

\## 二、對目前區塊鏈產業的具體影響

\### 1. NFT 的定義重塑：從「連結」到「資產」

ERC-4804 標準的出現是對 NFT 產業的一次重要修正。過去的 NFT 往往只是一張指向 AWS 服務器的「收據」，項目方倒閉，圖也就沒了。ERC-4804 讓瀏覽器能直接讀取鏈上數據渲染內容，這將迫使未來的藍籌 NFT 項目必須考慮「全鏈上存儲」和「數據永續性」，提升了發行 NFT 的技術門檻，但也賦予了資產真正的數位主權。

\### 2. 驗證者節點的專業化門檻提高

PeerDAS 技術雖然利好整體網路擴容，但也帶來了副作用。為了處理大量的數據採樣和 KZG 承諾計算，普通的家用電腦可能越來越難以承擔「區塊提議者（Proposer）」的角色。這可能導致 MEV（最大可提取價值）市場和中繼網路（Relay）進一步專業化和中心化，需要引入如 "Seeding Network" 這樣的新角色來輔助數據傳播。

\### 3. Layer 2 的代幣經濟學革命

過去 L2 發幣是為了「補貼用戶」，未來 L2 代幣必須「捕獲價值」。OP 的回購提案是一個開始，這將迫使所有 L2 項目方思考：除了治理權，代幣還能給持有者帶來什麼？是排序器收入的分紅？還是回購銷毀？這將改變投資機構對 L2 的估值模型。

\---

\## 三、給從業者的機會與建議

在這個「Web3 下半場」的開端，對於不同角色的從業者，機會點在於：

\### 1. 對於開發者 (Developers)

\* **不要再造輪子 (Stop Building Infra)：** 除非有突破性的技術，否則不要再發新的 L1 或通用 L2 了。

\* **擁抱 ERC-4804 與全鏈上應用：** 探索基於 Web3 URL 的全鏈上網站、遊戲或動態藝術，這是一個尚未被充分挖掘的藍海。

\* **專注於「中間件」機會：** PeerDAS 實施後，對高性能數據傳播網路（Seeding Network）的需求會暴增，相關的節點服務、MEV 優化工具將是技術創業的熱點。

\### 2. 對於產品/運營 (Product/Ops)

\* **尋找 Web2 的結合點 (Distribution is King)：** Solana 在 Twitter 內嵌 "Smart Cashtags" 是一個範例。未來的爆款應用不再是讓用戶痛苦地記住助記詞，而是像 Robinhood 或 Base 一樣，在用戶無感的情況下使用區塊鏈。

\* **深耕穩定幣與支付場景：** 2026 年預測中，穩定幣支付是重頭戲。開發能讓商家輕易接受 USDT/USDC 的工具，或者解決跨境支付痛點的產品，將有巨大的現金流潛力。

\## 結語

2026 年的願景並非遙不可及。我們正處於一個從「狂野西部」走向「現代文明」的轉折點。對於從業者而言，現在是放下浮躁，深耕技術標準（如 ERC-4804）和實際應用場景（如支付、RWA）的最佳時機。\*\*活下來，並在基礎設施過剩的土壤上，種出真正的應用之花，將是未來兩年的主旋律。\*\*
