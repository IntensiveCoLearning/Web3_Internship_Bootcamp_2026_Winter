# AI与Web3融合学习笔记

## 一、AI的演进：从"嘴替"到"打工人"

### 2023年的AI（传统LLM）

本质上是一个**高级复读机**——你问它答，帮你写情书、骂前任，但做完就结束了。

### 2026年的AI（Agentic AI）

不只是动嘴，还能**动手干活，干完还能收钱**。

### 传统LLM的三个致命限制

| 限制 | 具体表现 |
| --- | --- |
| 无状态 | 每次对话从零开始，不记得你是谁，聊完就忘（上下文窗口只有32K-64K） |
| 没有工具 | 只能生成文本，不能执行代码、调用API、访问外部系统 |
| 没有身份 | 不能代表任何人行动，不能签协议、管资产 |

Rick老师的比喻很形象：_"就像一个学习特别好的朋友，考试前讲得比老师还清楚，但让他干活，只负责动嘴，不负责动腿。"_

* * *

## 二、完整AI Agent的三个核心组件

### 1\. 大脑：LLM模型

-   GPT-5.2、Deepseek、Claude 4.5等
    
-   负责：理解意图 → 制定计划 → 生成行动指令
    
-   注意：只输出"应该做什么"，不执行操作
    

### 2\. 记忆：向量数据库 + RAG

这是传统LLM和Agentic AI的**分水岭**。

**工作流程：**

```
数据采集 → 向量化(Embedding) → 存储(Pinecone/Weaviate/Marvus) → 检索(RAG)
```

当用户提问时：

1.  用问题去向量库做相似度搜索
    
2.  找出最相关的信息
    
3.  拼接到prompt里
    
4.  再让LLM生成回答
    

_"让AI从金鱼进化成大象"_——能记住很多事情了。

### 3\. 手脚：Function Call / MCP

让AI从"光说不练"变成"说到做到"。

**工作原理：**

1.  定义一组函数（发邮件、查数据库、执行代码等）
    
2.  LLM判断需要调用哪个函数
    
3.  输出结构化的函数调用请求（不是普通文本）
    
4.  程序执行函数，返回结果
    
5.  LLM继续生成
    

这就形成了 **ReAct循环**：思考 → 行动 → 观察 → 思考...

## 三、AI Agent当前面临的三大问题

### 问题1：身份欺诈

HTTP协议没有机制验证请求是人还是AI，是A公司还是B公司的AI。

_"就像路上有人说'我是外卖小哥'，但没穿制服、没工牌、没订单号，你敢开门吗？"_

### 问题2：审计困难

AI做决策时，日志存在企业本地服务器，可以被修改、删除、伪造。出事了责任很难追溯。

### 问题3：隐私AI（Shadow AI）

59%的组织缺乏统一的AI治理框架。员工自己接入ChatGPT API，IT部门都不知道这些AI访问了什么数据、做了什么决定。

**总结：** 身份不可验证 + 行为不可审计 + 部署不可管控 = AI的信任危机

## 四、支付问题：传统金融为什么不行？

### 障碍1：身份认证

-   验证码要证明"你不是机器人"——但我们恰恰需要机器人完成注册
    
-   KYC要上传身份证、人脸识别——AI没有这些
    

### 障碍2：支付颗粒度

-   信用卡每笔交易：手续费2-3% + 固定处理费$0.30
    
-   AI交易规模：一次API调用可能只要$0.0005
    
-   结果：手续费比交易金额还高
    

### 障碍3：结算速度

-   传统支付：T+1到T+3，跨境可能一周
    
-   AI交互：毫秒级，一秒内可能调用十几个API
    
-   等三天结算？黄花菜都凉了
    

### 障碍4：订阅模式错配

SaaS月度/年度订阅 vs AI高动态的工作模式——要么付费过多用不完，要么峰值时能力不足

## 六、ERC8004：AI的链上身份证

### 为什么用NFT做身份？

| 特性 | 说明 |
| --- | --- |
| 唯一性 | 每个Token ID全局唯一，不会有两个agent用同一ID |
| 可验证性 | 任何人都能在链上查询这个ID是否存在、属于谁 |
| 可转让性 | 所有权可以转让（比如公司被收购） |
| 可编程性 | 可以实现复杂的权限逻辑 |

### 三个核心组件

**组件1：身份注册表**

-   基于ERC721
    
-   包含：agent类型、能力声明、运行模型哈希、所有者地址
    

**组件2：声誉注册表**

-   0-100分评分系统
    
-   **防刷分机制**：只有链上有真实交易的地址才能评价
    
-   **不可篡改**：评分上链后永久存在，删不了差评，造不了假好评
    
-   **时间衰减**：近期评价比历史评价更重要，给AI"改过自新"的机会
    

**组件3：验证机制**

-   TEE（可信执行环境）：硬件级安全模块生成证明
    
-   ZKML（零知识机器学习）：证明使用了某个模型做推理，不泄露模型权重和输入数据

## 七、X402：HTTP原生支付协议

### 背景

HTTP状态码402（Payment Required）在HTTP1.1定义了，但几十年来一直是保留状态，没人真正实现。

### 工作流程（四次握手）

```
1. 客户端 → 服务端：发起请求
2. 服务端 → 客户端：返回402 + 支付信息（地址、金额、代币类型、链网络）
3. 客户端 → 区块链：发起交易，获取交易哈希
4. 客户端 → 服务端：提交支付证明，获取数据
```

### 验证方式

-   **链上验证**：查询区块链确认交易存在、金额正确、收款地址正确（有几秒延迟）
    
-   **签名验证**：客户端签署付款承诺，先服务后结算（更快，但需要信任机制）
    

### 解决Gas费问题

如果每次API调用都发链上交易，$0.001的服务可能要付$0.01的gas。

**解决方案：**

1.  **支付通道**：双方锁定押金，链下多次交易，最后结算净额（像和女朋友吃饭最后AA总账）
    
2.  **批量结算**：小额交易累积到一定金额后一次性结算

## 八、合规：区块链的审计价值

### 传统审计的问题

日志存在自己服务器：

-   可以修改
    
-   可以删除
    
-   可以伪造
    
-   可以改服务器时间制造不在场证明
    

企业说"这是我们的日志证明没问题"，监管凭什么信？

### 区块链审计的优势

-   **不可篡改**：写入后无法修改
    
-   **时间戳可信**：由网络共识决定，不是企业说了算
    
-   **公开透明**：任何人都能验证
    
-   **永久存储**：只要区块链在，记录就在
    

**核心理念**：不需要信任，只需要可验证。

## **✅ 第一层面：形式合规**

| 目标 | 在法律文件层面做到形式合法合规 |

## **✅ 第二层面：深度合规（实际运营层面）**

**目标：**在业务模式、资金流向、用户交互等实际运营中规避法律风险

**● 避免非法募资：**绝不涉及 ICO/IEO/IDO 等公开融资行为

**● 隔离中国大陆用户：**不向境内居民提供交易、兑换、支付等服务

**● 防范刑事风险：**远离赌博、传销、洗钱、非法经营等行为模式

**● 建立真实业务闭环：**具备真实技术或服务支撑，不以「纯金融」为目的

## **核心认知**

| 1. 深度合规不是 Web3 特有本质是金融合规问题，与传统金融行业的非法集资、洗钱、传销等风险逻辑相似2. 合规是系统工程需从业务设计、技术实现、运营管理、法律架构等多维度协同构建3. 「绝对安全」是理想状态在现行监管环境下，只能在风险控制与业务发展之间寻求平衡 |

## 一、钱包与私钥：你的数字身份起点

### 核心概念

钱包的本质是**密钥对管理工具**，包含：

-   **私钥（Private Key）**：绝对保密，相当于你的"万能钥匙"
    
-   **公钥/钱包地址**：可以公开分享，相当于你的"收款账号"
    
-   **助记词（Seed Phrase）**：私钥的人类可读形式，一组12-24个单词
    

### 重要关系

```
助记词 → 私钥 → 公钥/钱包地址
（一对多）  （一对一）
```

助记词可以派生出无数个私钥（一对多关系），而每个私钥对应唯一的钱包地址（一对一关系）。

### ⚠️ 安全警示

这是老师反复强调的重点，之前有训练营学员因为私钥/助记词泄露，刚获得的奖金就被黑客转走了：

-   **绝不截图**保存私钥或助记词
    
-   **绝不发送**给任何人
    
-   **复制时要小心**——粘贴板容易被监控，建议复制一部分+手打一部分
    
-   涉及真金白银时，安全意识要拉满
    

### 社会学意义

Web3钱包代表着**个人主权的起点**——任何人都可以无需许可地创建账号和拥有资产。对比传统银行开户需要身份证、审批、可能被拒绝，而且还有地域限制，加密钱包提供了一种全新的金融自主权。

* * *

## 二、交易与签名：区块链上的"转账"是怎么工作的

### 交易（Transaction/TX）的组成

一笔交易基本包含：

1.  **你要做的事情**：转账、调用合约方法、投票等
    
2.  **手续费（Gas）**：为网络资源付费
    
3.  **防重放序号（Nonce）**：每笔交易递增，防止被重复执行
    

### 钱包的作用

钱包（如MetaMask、Rabby）做的事情：

1.  组装交易内容
    
2.  用私钥签名
    
3.  广播到区块链网络
    

### 数字签名的原理

这是一个很精妙的密码学设计：

-   我用**私钥**对交易信息签名
    
-   任何人都可以用**公钥**验证这个签名是否由我发出
    
-   但没有私钥，任何人都无法伪造签名
    

验证流程：`原始消息 + 声称的发送地址 + 签名` → 密码学算法 → 验证签署者身份

这意味着：**只要知道你的私钥，就能以你的身份发送任何交易**，所以私钥安全至关重要。

* * *

## 三、Gas费用：区块链的"汽油费"

### 什么是Gas

可以把Gas理解为汽油费的概念：

-   **Gas Price**：当时的"油价"，由网络拥堵程度决定
    
-   **Gas Used**：完成这笔交易消耗的"汽油量"
    
-   **总费用** = Gas Price × Gas Used
    

### 类比理解

从上海开车到北京：

-   路程固定（Gas Used基本固定）
    
-   油价波动（Gas Price随网络繁忙程度变化）
    
-   总油费 = 路程 × 油价
    

### Gas的作用

-   防止垃圾交易淹没网络
    
-   为网络维护者提供激励
    
-   构成区块链的经济闭环
    

* * *

## 四、区块链网络运行原理

### 交易的完整生命周期

```
钱包签名 → RPC/节点广播 → 进入交易池排队 → 验证者打包成区块 → 共识确认 → 上链完成
```

### 区块的结构

每个区块包含：

-   区块编号（Block Number）
    
-   若干笔交易（如312笔）
    
-   父区块哈希（Parent Hash）→ 指向上一个区块
    
-   当前区块哈希
    
-   验证者投票信息
    

### 链式结构的安全性

每个区块都通过Parent Hash指向上一个区块，形成一条"链"。要篡改某个区块，必须同时修改它之后的所有区块——这在计算上几乎不可能。

### 最终确认（Finality）

刚出的区块还不够安全！以太坊需要约12-13分钟才能达到Finalized状态。在此之前，交易理论上可能被回滚。

**实际应用**：如果你是电商卖家，收到加密货币支付后不能立即发货，需要等待交易最终确认。

* * *

## 五、共识机制：去中心化网络如何达成一致

### PoW（工作量证明）- 比特币使用

类比：大家比赛解一道超难的题，谁先解出来谁就能记账并获得奖励。

-   优点：安全性经过长期验证
    
-   缺点：非常耗能，算力竞争激烈
    

### PoS（权益证明）- 以太坊使用

类比：交押金（质押32 ETH）加入"记账员"名单，随机被选中出块并获得奖励；如果作恶，押金被没收。

-   优点：节能环保
    
-   缺点：可能导致"富者越富"
    

* * *

## 六、去中心化程度分析

### 钱包创建 ✅ 高度去中心化

任何人都可以随机生成钱包，无需KYC或许可

### 交易广播 ⚠️ 相对中心化

依赖RPC服务商（如Infura、Alchemy），存在单点风险。以太坊基金会正在研究去中心化RPC方案

### 网络运行 ✅ 相对去中心化

-   全球约13,700+个节点
    
-   约97万个验证者（Validator）
    
-   多种客户端软件（Geth、Prysm等），避免单点故障
    

### 为什么多样性很重要

如果所有节点都运行同一个软件，一旦该软件出现Bug，整个网络可能分叉或宕机。多种独立开发的客户端软件分散了这种风险。

* * *

## 七、智能合约：链上的可执行代码

### 什么是智能合约

把代码部署到区块链上，代码逻辑不可篡改、可追溯、自动执行。

> "Code is Law" —— 代码即法律

### 实例：NFT Mint逻辑

```solidity
// 简化示例
function mint() {
    require(!hasMinted[msg.sender], "已经Mint过了");
    hasMinted[msg.sender] = true;
    // 铸造NFT...
}
```

通过合约代码限制每个地址只能Mint一次。

### 开源的重要性

涉及资产的合约通常应该开源，方便审计。如果一个项目的合约不开源，可能是"心虚"的表现。养成习惯：交互前先看合约是否开源。

* * *

## 八、以太坊升级机制：EIP流程

### EIP（Ethereum Improvement Proposals）

以太坊的改进通过社区协作推进，不是某个公司单方面决定：

```
论坛讨论 → 形成EIP草案 → 社区审议 → Last Call → Final → 客户端升级
```

每年大约1-2次大版本升级，每次升级都是一次"硬分叉"。

### 最近的升级

2025年的Fusaka升级包含了多个EIP，如调整交易Gas Limit等。

* * *

## 九、Web3的核心价值总结

| 特性 | 说明 |
| --- | --- |
| 无许可 | 任何人都可以读写区块链，无需审批 |
| 抗审查 | 节点全球分布，钱包/RPC可替换 |
| 开放开源 | 客户端代码、交易记录全部公开 |
| 去中心化 | 没有单一控制点 |
| 隐私 | 地址与真实身份分离（但链上记录公开） |

## 十、值得思考的问题

老师最后留了几个开放性问题，我觉得很有价值：

1.  **资产托管**：如何提高安全性同时降低管理复杂度？对于不熟悉技术的人（如老年人），如何管理私钥？
    
2.  **公共物品**：没有税收的去中心化世界，谁来维护节点、开发客户端、提供教育？资金如何分配？
    
3.  **有害信息治理**：抗审查对正常用户有利，但对黑客也一样。如何在保护隐私的同时治理恶意行为？
    
4.  **去中心化协作**：在互不认识、互不信任的环境下，如何实现公平可信的分配，让合作持续下去？

# 今日学习笔记：初探 Web3 和 NFT 的世界

今天花了差不多一个小时看完了《我的第一个 NFT》这个教程，感觉对区块链和 NFT 这些之前一直听说但没搞懂的概念，终于有了一个比较清晰的认识。

* * *

**区块链原来是这么回事**

之前总觉得区块链很神秘，今天才知道它的核心概念其实挺直观的——就是把交易记录打包成一个个"区块"，然后像串珠子一样串起来。每个区块里还包含了上一个区块的摘要信息，所以如果有人想篡改历史记录，就得把后面所有的区块都改掉，这基本上是不可能的。

更妙的是分布式记账这个设计。全球有无数个节点都在记录同一份账本，除非你能控制超过一半的节点，不然你改的东西根本不会被承认。突然觉得这个设计真的挺聪明的。

**关于 NFT 的理解**

NFT 和普通代币的区别，教程里用的比喻很好懂：普通代币像钱，一块钱和另一块钱没区别，可以随便换；但 NFT 就像画作或者收藏品，每一个都是独一无二的。

让我印象比较深的是 NFT 的"完全控制权"这个特性——你真正拥有它，不像游戏里的装备，账号被封了就什么都没了。不过教程也提醒说，国内很多所谓的"数字藏品"其实不算真正的 NFT，因为不在公链上，还是受平台控制的。这点之前完全没意识到。

**几个实用的收获**

钱包那部分挺实用的。助记词是最重要的东西，丢了就真的找不回来了；钱包地址可以公开，但私钥绝对不能告诉任何人。感觉跟银行卡号和密码的关系差不多，但后果更严重，因为被盗了基本没法追回。

还学了几个 Web3 的"黑话"：DYOR（自己做研究）、FOMO（怕错过机会）、FUD（恐惧不确定）。以后看到这些缩写至少不会一脸懵了。

**一点个人感想**

说实话，看完之后既觉得有意思，又有点担心。有意思的是这套去中心化的设计确实解决了一些现实问题，比如跨国转账、数字资产的真正所有权这些；担心的是安全问题，教程里列举的那些被骗案例看得我挺紧张的，什么钓鱼网站、假客服、恶意授权之类的，感觉这个圈子的坑还挺多。

教程最后说的那句话挺实在的：在决定买任何 NFT 之前，先问问自己"愿不愿意随便给一个陌生人送钱"，如果不愿意，就好好做功课。这个心态挺重要的，不能光看别人说能赚钱就冲进去。

总的来说，今天算是入门了，对这个领域有了基本的认知框架。接下来可能会试着装一下 MetaMask 玩玩看，但暂时不打算投真金白银进去，先观察观察再说。
