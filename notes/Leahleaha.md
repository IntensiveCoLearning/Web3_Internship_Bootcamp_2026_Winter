## **ERC-7962（Key Hash Based Tokens）**

### 概述

token ownership 从 “address-based” 迁移到 “keyHash commitment-based”，把授权从链上 msg.sender 改成链下签名，使得匿名所有权与代付 gas 成为标准层能力，它更像 ERC20/721 的补充标准而非替代品。

**即 不再直接用钱包地址来确定所有权，而是用一个“keyHash”来表示所有权**。

keyHash 是一种经过哈希算法处理后的数据，长度是32字节（bytes32）。

哈希算法是一种单向的加密算法，它可以把任意长度的数据转换成固定长度的哈希值。这种哈希值具有唯一性，很难被逆向破解。所以，通过使用 keyHash 来表示所有权，可以增加系统的安全性和灵活性。

-   **隐私**：链上看不到 owner 地址，只看到 keyHash，降低身份关联
    
-   **代付 gas**：第三方拿到你的签名就能替你执行交易并付 gas（类似 permit 思路）
    
-   **批量操作**：适合平台一次性搬运/分发多个 token
    

### **流程**

1）链上状态存 keyHash，不存地址

ownerOf(tokenId) 返回 bytes32 keyHash

balanceOf(keyHash) 返回数量

2）授权在链下完成

持有人对“我要做的动作”签名（EIP-712 结构化签名 + nonce + deadline）。

3）执行在链上完成

任何人都能提交这笔交易（代付 gas / relayer / 平台），合约只验签名，不看 msg.sender。

结论：持有人不必自己发交易，也不必暴露地址作为 owner。

### **安全设计要点**

💛 **参数不可篡改**

-   利用结构化签名，签名会覆盖交易中的关键参数，使得 relay（中继者，负责将交易广播到区块链网络）无法篡改这些参数。
    
-   确保交易的关键信息（如交易金额、接收地址等）在传输过程中保持完整性和准确性，防止恶意 relay 对交易内容进行篡改。
    

💛 **防重放攻击**

-   设计包含多层防护机制，确保每次授权都是一次性的，不同场景下的授权对应不同的唯一授权标识。
    

💛  d**ouble- check 防作恶 relay 风险**

-   虽然理论上存在 relay 可能会进行一些恶意行为，如压交易（延迟交易）、重排交易顺序 或夹带其他交易，但设计中包含了额外的双重检查（double-check）防护机制。
    
-   通过双重检查等手段，降低 relay 作恶的可能性，确保交易的顺序和完整性，维护区块链网络的正常运行
    

**💛  签名有效期**

-   为授权签名设置一个明确的截止日期（deadline），一旦超过这个时间，签名就失效，无法再用于交易。
    
-   防止签名被无限期地使用，避免用户在不知情的情况下被恶意利用签名进行交易，增强了交易的安全性和时效性。
    

**💛  强制校验**

-   对签名进行强制校验，防止签名可塑性（malleability）问题。签名可塑性是指签名可以被恶意修改但仍能通过验证的情况。
    
-   通过强制校验，确保签名的唯一性和不可变性，防止攻击者通过修改签名来发起欺诈交易，从而保障了整个交易系统的安全性和可靠性。
    

### **产品能力**

1）更强的“所有权匿名化” ,NFT/资产归属不再直接绑定地址。

2）代付 gas / 赞助交易 用户只签名，不掏 gas；平台/第三方替用户上链。

3）批量执行更自然 可以离线收集大量签名，再由执行者集中上链（实现层做 batch）。

4）更接近“UTXO 风格的所有权”,一个 keyHash 更像“可花费的凭证/承诺”，比 address 账户模型更灵活。

**数据层**

关系型数据库保存用户的基础信息（id、name、created\_at）以及一个冗余的 balance 字段，用于快速展示。Redis 作为缓存层，用 key-value 方式保存 user → balance 的映射，读速度远高于数据库。

**服务端核心逻辑**

Backend 使用 Node/Next.js/Hono 一类框架，对外提供 HTTP API。

后台有一个定时脚本（daemon job），每隔几秒批量查询所有用户的链上余额。查询通过 RPC 节点（如 Infura / Alchemy / 自建节点）访问区块链。

拿到结果后，统一写入数据库，并同步更新 Redis 缓存。

**用户访问路径**

浏览器中的前端（React / Next.js / wagmi）通过 HTTP 请求后端接口获取余额。

前端不直接打 RPC，不感知区块链延迟，只从后端拿「已经算好的结果」。

**真实链上世界**

RPC 节点连接区块链，真正读取 balances\[address\]。由于 RPC 调用慢（图中标注约 2000ms），而且如果用户多会指数级放大请求数，因此被集中到后端统一处理。

**Gas 不是手续费，是“强制约束”**

Gas 不是“平台抽成”，而是系统生存机制。没有 Gas，EVM 就会被无限循环和垃圾计算拖死。我现在对 Gas 的理解是

-   它不是用户成本视角的东西
    
-   它是协议层的“资源闸门”
    

EIP-1559 之后更清楚了：BaseFee 直接烧掉 → 网络使用成本，Tip 才是给打包者的激励

## **关于以太坊的系统 如何成立**

_以太坊通过_ **_PoS 共识保障安全，通过 EVM 统一执行，通过 Gas 强制定价，通过 ABI + RPC 连接外部世界_**_，_

_在无需许可的前提下运行一个可组合、可验证、不可篡改的全球计算系统。_

### **1\. 以太坊的本质定位**

不是“区块链应用平台”，而是**一台去中心化的、全球同步的、可验证的状态机**。

所有节点在同一套规则下，对同一输入执行同一逻辑，并对结果达成共识。

* * *

## **闭环一：执行与成本（EVM × Gas）**

### **2\. EVM（Ethereum Virtual Machine）**

EVM 是每个节点内部运行的确定性虚拟机。

智能合约不是“托管服务”，而是**被所有节点重复执行的程序**。

确定性是第一原则：相同输入 → 相同输出 → 相同状态更新。

### **3\. Gas 机制**

Gas 是对计算与存储资源的强制定价机制，用于防止滥用并分配有限算力。

每条 opcode 都有固定 Gas 成本，交易费用由执行消耗决定。

EIP-1559 之后，费用结构为 BaseFee（销毁）+ Tip（激励打包者）。

即 **合约设计 = 成本设计**，Gas 是约束，不是细节。

### **4\. 常量变量（constant / immutable）**

constant：编译期确定，不进入 storage。

immutable：构造期确定，不可再改。

二者显著降低 Gas 成本，是合约设计的基础优化手段。

* * *

## **闭环二：安全与共识（PoS × Beacon Chain）**

### **5\. PoS（Proof of Stake）**

共识不再来自算力，而来自**质押的经济风险暴露**。

验证者通过质押 ETH 获得出块与投票资格，作恶将被惩罚。

### **6\. Beacon Chain**

Beacon Chain 是共识层，不执行合约逻辑。职责包括：

-   管理验证者集合
    
-   组织投票与最终性
    
-   协调共识节奏
    

执行层（EVM）与共识层分离，是以太坊长期扩展的结构前提。

### **7\. Slashing（惩罚机制）**

对双签、恶意投票等行为直接扣除质押并强制退出。

安全来自**经济不可承受性**，而非道德约束。

* * *

## **闭环三：账户与调用（ABI × RPC × 钱包）**

### **8\. ABI（Application Binary Interface）**

ABI 是合约对外暴露能力的**精确机器说明书**。它定义

-   可调用函数
    
-   参数与返回值编码方式
    
-   事件格式
    

前端与其他合约不“理解源码”，只“遵循 ABI”。

### **9\. JSON-RPC / RPC**

RPC 是链对外提供能力的唯一入口。钱包、前端、脚本均通过 RPC：

-   查询状态
    
-   构造交易
    
-   广播交易
    

区块链不是 API 服务，RPC 只是访问通道。

### **10\. 账户抽象（Account Abstraction）**

传统 EOA 账户能力固定、逻辑贫弱。

账户抽象将签名、Gas 代付、批量操作、策略控制迁移至合约层。

这是用户体验、钱包形态与安全模型的根本升级方向。

* * *

## **闭环四：合约安全（必须先于功能）**

### **11\. 重入攻击（Reentrancy）**

在外部调用前未更新内部状态，会被递归调用反复提款。

这是合约历史上最具破坏性的漏洞类型。

### **12\. nonReentrant**

通过互斥锁阻止函数重入执行。

属于基础防线，不是可选项。

### **13\. Pull Payment Pattern**

不主动转账，只记录可提取余额。

由接收方自行领取，避免 Gas 爆炸与 DoS 风险。

* * *

## **闭环五：DeFi 的结构基础**

### **14\. AMM（自动化做市商）**

用数学公式替代订单簿。

价格由池内资产比例决定，不依赖撮合者。

这是 DeFi 流动性的基础形态。

### **15\. 借贷协议（以 Aave为例）**

资产存入即变为可复用流动性。

利率、清算、风险控制全部由合约自动执行。

资金使用权与所有权分离，是链上金融的核心创新。

### **16\. TVL（Total Value Locked）**

TVL 是规模指标，不是安全指标。

高 TVL ≠ 低风险，只代表更高的系统性暴露。

## **Uniswap V2 学习笔记**

### **模块：Swap · MEV · TWAP · CREATE2**

### **一、Uniswap V2 的两个 Swap 接口（用户侧）**

**1\. swapExactTokensForTokens**

**语义**：

固定输入数量，接受价格波动，但不接受低于阈值的输出。

**核心参数**

-   amountIn：用户确定的输入代币数量
    
-   amountOutMin：最小可接受输出（滑点保护）
    
-   path：兑换路径（支持多跳）
    
-   to：接收地址
    
-   deadline：交易过期时间
    

**使用动机**

-   防止 MEV / sandwich attack 导致成交价格恶化
    
-   若最终输出 < amountOutMin → 整笔交易回滚
    

* * *

**2\. swapTokensForExactTokens**

**语义**：

固定目标输出数量，接受输入成本浮动，但不超过上限。

**核心参数**

-   amountOut：期望得到的代币数量
    
-   amountInMax：最大可接受输入成本
    
-   path / to / deadline：含义相同
    

**本质区别**

-   一个约束输出下限
    
-   一个约束输入上限
    
-   底层撮合逻辑一致
    

* * *

### **二、path 与多跳 Swap（Router 层）**

-   swap **不是 1 对 1**
    
-   path 是 token 地址数组，例如：
    

```
WETH → USDC → DAI
```

-   当不存在直接池时，通过中间池绕路完成兑换
    

**Router 的职责**

-   根据 reserves **一次性计算整条路径的 amounts\[\]**
    
-   通过 \_swap 循环调用多个 Pair
    
-   **不持有资产，不存储状态**
    

### **三、Router.\_swap 的执行逻辑**

-   for 循环遍历 path
    
-   每一跳调用对应 Pair 的 swap
    
-   上一跳输出 → 下一跳输入
    
-   最终完成目标 token 的兑换
    

* * *

### **四、Pair 合约中的 Swap 核心逻辑**

**1\. token 排序规则**

-   地址小 → token0
    
-   地址大 → token1
    

**2\. Swap 执行流程**

1.  getReserves() 获取历史储备量
    
2.  执行代币转账
    
3.  通过 balance - reserve 计算真实输入量
    
4.  判断哪一侧是输入、哪一侧是输出
    
5.  校验恒定乘积公式
    
6.  调用 \_update 更新状态
    

* * *

### **五、MEV 与 Flash Loan 风险背**

**MEV 的来源**

-   区块构建者 / 搜索者可重排交易
    
-   可进行 frontrun / sandwich / backrun
    

**Flash Loan 的放大效应**

-   单笔交易中借入巨量资金
    
-   瞬时操纵池子价格
    
-   无需长期资本成本
    

**结论**

-   Uniswap 现货价格**极易被操纵**
    
-   不适合作为链上 Oracle
    

* * *

### **六、TWAP（Time Weighted Average Price）**

**1\. 引入目的**

-   抵抗短时间价格操纵
    
-   提高攻击成本
    

**2\. 核心思想**

> 用时间换稳定性

**3\. 实现方式（Pair.\_update）**

在每次 swap / mint / burn 时：

-   记录当前区块时间
    
-   记录当前价格
    
-   累加：
    

```
priceCumulative += price × timeElapsed
```

**4\. TWAP 计算**

外部合约取两个时间点的 priceCumulative 差值

-   再除以时间间隔
    
-   得到该时间段的平均价格
    

**关键区分**

-   TWAP ≠ Spot Price
    
-   抗操纵 ≠ 不可操纵
    

* * *

### **七、CREATE2 与 Pair 地址确定性部署**

**1\. CREATE vs CREATE2**

-   CREATE
    
    -   地址依赖部署者 nonce
        
    -   无法提前预测
        
    
-   CREATE2
    
    -   地址完全由参数决定
        
    -   可在链下提前计算
        
    

**2\. CREATE2 地址公式**

```
address = keccak256(
  0xff ++ factory_address ++ salt ++ keccak256(init_code)
)[12:]
```

* * *

### **八、Uniswap V2 中 CREATE2 的具体设计**

-   factory\_address：Factory 合约地址
    
-   salt：keccak256(token0, token1)
    
-   init\_code：Pair 合约 creation code
    

**关键点**

-   token0 / token1 顺序固定
    
-   creation code 固定
    
-   任意 token 对 → 唯一 Pair 地址
    

* * *

### **九、Factory.createPair 的完整流程**

1.  校验 tokenA ≠ tokenB
    
2.  按地址大小排序 token0 / token1
    
3.  生成 salt
    
4.  使用 CREATE2 部署 Pair
    
5.  初始化 Pair（传入 token0 / token1）
    
6.  更新映射关系
    
7.  发出 PairCreated 事件
    

* * *

### **十、主网示例理解（USDC / WETH）**

-   USDC 地址以 0xA 开头
    
-   WETH 地址以 0xC 开头
    
-   因此：
    
    -   token0 = USDC
        
    -   token1 = WETH
        
    

代入 CREATE2 公式 → 可提前计算 Pair 地址。

* * *

### **十一、整体结构总结（一句话版）**

-   **Router**：算路径、算金额、调合约
    
-   **Pair**：管资产、管价格、管状态
    
-   **MEV**：结构性风险
    
-   **TWAP**：工程缓解方案
    
-   **CREATE2**：Uniswap V2 架构级设计关键点

二、核心法律风险：你可能“只是打工/写代码”也会被牵连的点

这一部分的关键是：很多罪名不是只抓“老板”，而是抓“共同行为人/帮助者”。你做了“关键一环”，就可能被算进去。

1）代币发行与交易：最容易“包装成合规但本质仍危险”

• 项目把代币叫“积分”“凭证”“治理 Token”不等于合规。只要具备融资功能或可流通性，就可能被定性为非法金融活动。

• 技术岗位高危点：代币模型设计、空投逻辑配置、合约部署等，会被认定参与关键环节，难以用“只是写代码”切割责任。

你在入职/接活前要问清楚的最小问题集：

• 是否发 Token？Token 如何分发？是否对外出售/兑换？

• 不碰来路不明的 OTC 交易对手；不参与“高价出 U”；不帮别人代买代卖

• 能与公司协商：薪资尽量保留一部分法币，降低频繁出金需求

• 留存资金来源与交易凭证（自证清白用）

• 对 DeFi/DEX 的挑战：匿名性、跨链复杂度让风险评估、协作、报告都更难

3）稳定币监管为何重要（从风险传导角度理解）

• 稳定币是 DeFi 基础设施，体量大 → 一旦出问题会系统性冲击

• 监管目标：储备充足、金融稳定、AML 合规、消费者保护

四、Web3 入职阶段的现实风险：劳动关系与薪酬结构

这部分不是“理论”，而是直接影响你的生活保障与纠纷处理能力。

1）雇佣关系新形态的核心问题

• 项目常境外注册，线上协作；中国境内可能没有合格用工主体 → 劳动合同可能无效、社保公积金无法缴纳

• 社保公积金影响：落户、购房、贷款、子女教育、婚育福利等（实际生活层面的硬影响）

可选缓解方案：EOR（名义雇主）模式，但纠纷时劳动关系认定仍需重视。

2）薪酬“人民币+Token / 全 USDT”的双重风险

• 中国劳动法语境：工资应以法定货币支付（人民币）

• Token/USDT 作为工资可能被认定支付无效；若不计入工资又会拉低工资基数，影响补偿/社保等

• 代币波动与归零风险：你承担项目风险敞口

3）出金风险是入职后最常见的“刑事入口”

• 一旦对手资金涉案，你可能“无辜但后果严重”（银行卡冻结、退赔、调查）

最务实的策略：减少出金频次 + 只走可信渠道 + 留存证据链。

五、典型刑事风险案例的“结构化识别法”（把案例看成模式）

你不需要背案情，你要学会识别模式与岗位牵连点。

1）开设赌场：关键识别点

• 平台提供赌博玩法（掷骰子、牌类等）

• 接受投注、能提现、平台抽水盈利

• 参与分工很细：运营推广、技术开发、合约、客户端、客服、翻译都可能入罪（主从犯看作用与分赃）

你的岗位自查：只要产品有“下注/随机/兑现”，以及你负责其中任何关键模块，都要视为高风险。

2）非法经营（外汇）：关键识别点

• 以 USDT 为媒介做“外币↔人民币”汇兑

• 网站/平台组织下单、代付、赚汇差与服务费

• 明知他人非法买卖外汇仍提供虚拟币交易帮助 → 可能作为共犯

3）非法吸收公众存款：关键识别点

• 向不特定公众吸收资金

• 用“矿机/返利/积分释放”等包装

• 资金大、宣传广、去向不明 → 高危

4）组织领导传销：关键识别点

• 交门槛费（交币/交钱/买资格）

• 层级体系 + 发展下线 + 以人数/业绩计酬返利

• 技术人员若明知系统用于传销仍开发，可能被按共犯处理

5）洗钱：关键识别点

• 明知上游犯罪所得仍提供账户、兑换虚拟币、转账、跨境转移

• 洗钱罪可独立认定，上游未判不必然影响下游追责

六、网络安全：Web3 新手的“高频中招路径”与防护动作

Web3 的安全本质：一旦私钥/助记词/签名授权被滥用，资产通常不可逆且难追回。你要把安全当成日常操作纪律。

1）五类最常见攻击方式（按你最可能遇到的顺序）

A. 钓鱼（Phishing）

• 伪造邮件/网站：面试通知、账号异常、奖学金发放等“紧急信息”

• 社群钓鱼：冒充官方、HR、学长学姐发链接/空投

• 假客服/假好友：盗号后用信任关系诱导转账或泄露信息

B. 木马/恶意软件（Malware & Trojan）

• 伪装成面试软件/学习资料/破解工具

• 典型能力：盗 Cookie/凭证、录屏、键盘记录、窃文件、远控；或剪贴板劫持

C. 社工（Social Engineering）

• 冒充 HR/导师/管理员，利用权威与紧迫感逼你操作

• “好友求助”是经典：盗号后借你影响力继续扩散

D. 供应链/第三方依赖攻击

• 假浏览器插件、开源库后门、甚至“官方渠道被劫持”导致批量感染

• 特征：看起来都“正规”，所以防护要依赖校验与最小化安装

E. 地址污染/剪贴板劫持（Clipper/Scanning）

• 复制粘贴地址被替换，资金直接转给攻击者

• 常见在下载了“工具/插件/资料包”之后出现

2）学生版防护清单（最小可执行版）

面试/实习：

• 只用公开主流会议工具（Zoom/Teams/腾讯会议等）；拒绝安装“专用面试软件”

• 遇到“先下载 zip/安装包/资料包” → 默认高危：先用虚拟机测试；不要在主力电脑运行

• 核实邮箱域名、联系人，多渠道交叉确认（同公司官网、领英、官方邮箱格式）

奖学金/空投：

• 任何要求“连接钱包/签名验证/输入助记词私钥”的页面，默认骗局

• 空投只用测试钱包；主钱包尽量冷存储、隔离使用场景

社群/私聊：

• 不信“管理员/官方人员”私聊；涉及转账/下载/签名一律二次核验

• 任何“紧急转账/帮忙测试/转发链接”先电话或其他渠道核实身份

软件/插件：

• 只从官网/官方商店下载；插件数量越少越安全（钱包、密码管理器、广告拦截器足够）

• 安装前查口碑、开发者、下载量；权限能关就关（摄像头/麦克风按需授权）

账号与密码：

• 重要账户启用 2FA：优选验证器或硬件密钥，短信风险更高

• 密码管理器 + 每个平台独立强密码

• 邮箱/手机号是“总钥匙”，优先保护（定期改密、提高恢复验证强度）

钱包与授权：

• 助记词/私钥只离线保存：不截图、不上网盘、不发任何人

• 转账核对地址前 6 位 + 后 4 位（防剪贴板劫持）

• 定期检查并撤销不必要授权（[Revoke.cash](http://Revoke.cash)、Etherscan 等）

3）遇到疑似攻击的第一响应（你要能背下来）

• 立刻断网（防止继续外传数据/远控）

• 截图/拍照/记录时间线（后续溯源与申诉用）

• 更换所有关键账号密码与 2FA（从邮箱开始）

• 检查钱包授权并撤销；必要时转移资产到安全钱包

• 设备做彻底清理/重装（不要只“杀毒”就以为安全）

七、把合规与安全落到“入职/做项目的检查表”

可以把下面当作入职前的硬门槛筛查。

1）项目合规四问（任意回答含糊，就不要上车）

• 项目是否发行/分发 Token？是否对外可交易/可兑换？

• 是否对用户承诺收益、回购、保底或“稳赚模型”？

• 是否涉及出入金、OTC、换汇、提现？

• 是否面向中国大陆用户做运营增长或提供交易相关服务？

2）岗位行为红线（对你个人最重要）

• 不参与：代币融资设计、交易撮合、返佣裂变、多级分销、出金通道组织、涉嫌博彩玩法闭环

• 不帮别人：代投、代买卖、代出金、代换汇

• 不接触：陌生来源资金、不明身份的“合作伙伴”钱包/账户

3）最低安全配置（建议你按这个标准固化）

• 设备分离：日常电脑 ≠ Web3/钱包操作环境（至少账户与浏览器隔离）

• 钱包分层：主钱包冷存储 + 热钱包日常交互 + 测试钱包专门用于空投/陌生交互

• 权限最小化：插件最少、授权最少、签名前先读内容（不懂就拒绝）
