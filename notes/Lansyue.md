# web3运行原理分享会笔记

今天听完这场《Web3 和区块链技术基础》分享，终于把自己之前零散的认知串成了一张比较完整的图。​

## 关于交易和钱包

过去总觉得“交易”就是转账，这次才意识到，一笔链上交易其实是“操作 + Gas 手续费 + nonce（防重放序号）”三件事打包在一起。 操作可以是转账、调用合约、投票等等，钱包只是负责帮我把这些信息组装好、用私钥签名，然后丢给网络去处理。 实际在测试网看了一笔交易的详情，to、金额、Gas limit、nonce 一目了然，感觉之前只看余额完全是浪费区块浏览器。​

## 数字签名和安全感

数字签名这一块非常打消“我怎么证明是我发的”的焦虑：用私钥对交易做签名，任何人都能验证签名确实来自这个地址，但没人能伪造出同样的签名。 验证的时候其实就是用“原始消息 + 声称的地址 + 签名”三件套，跑一遍算法去反推出是不是同一个人签的。 结论也很现实：私钥就是一切，谁拿到私钥，谁就能发出被全网认可的“我说的话”。​

## Gas 费和经济设计

Gas 被讲成“油费”这个比喻很形象：Gas price 像油价，Gas usage 像你加了多少升，两者相乘就是最后的手续费。 操作越复杂、网络越拥堵，Gas 就越贵，这相当于用经济手段直接把垃圾交易和恶意行为的成本拉高。 这也是整个网络经济闭环的一部分，矿工 / 验证者不是在“做爱心”，而是在一个有激励、有成本的系统里工作。​

## 交易一生：从钱包到区块链

一笔交易的生命周期，第一次在脑子里跑通：钱包签名 → 通过 RPC 发到网络 → 内存池排队 → 被 builder / validator 挑选 → 打包进区块 → 共识确认 → 链上永久记录。 动画里 12 秒一个新区块、几百笔交易按 Gas 高低排队的画面，让“打包”这件事不再抽象。 每个区块之间通过 parent hash 串起来，要想篡改一个区块，就得连它后面所有区块一起改，这基本就是在跟全网算力或质押对着干。​

## POW、POS 和去中心化

共识这块，讲得很克制：POW 靠算力、POS 靠质押资金，各有优缺点，没有完美方案。 比特币用 POW，拼的是谁算得快；以太坊现在用 POS，拼的是谁愿意锁更多押金、接受作恶被罚没的风险。 本质上都是让“作恶的代价”远远大于“作恶的收益”，只是筹码不一样。​

## RPC、客户端和“隐形”的中心化

之前用钱包从没认真想过 RPC，今天才意识到：钱包其实是通过 RPC API 去连节点集群，而不是直接连区块链本体。 这意味着 RPC 服务商本身就是一个潜在的中心化风险点，拦截或拒绝转发交易，等于别人替你“掐网线”。 以太坊主网有多个 RPC 提供方、执行层和共识层也有好几个客户端，这种多样性就是在分散单点故障和软件 bug 的风险。​

## 智能合约和 “Code is law”

智能合约那部分对“Web3 = 可编程规则”这件事很加分：本质就是写死在账本里的可执行代码，调用时在虚拟机里跑一遍，把结果写回链上。 现场看了一个 MyFirst NFT 合约，每个地址只能 mint 一个，这种规则写死在代码里，任何人都能查逻辑、查历史执行记录。 “Code is law” 不是一句口号，而是把对人、机构、政府的信任，转成对代码、共识机制、治理流程的信任。​

## NFT mint、区块确认与 Finality

实际 mint NFT 的演示很直观：签名的数据比简单转账复杂得多，因为背后是对合约方法的调用。 交易一旦被打进区块，刚开始还是 unfinalized 状态，需要十几分钟等到 finality，才能说这件事“几乎不可逆”。 这种“概率上不可篡改”的设计，和传统数据库的“超级管理员说了算”形成了鲜明对比。​

## 以太坊升级、社区治理和节点分布

升级机制也挺社会学的：改协议要走 EIP 流程、社区讨论，每年 1–2 次硬分叉升级，不是谁一拍脑袋说了算。 最近的 Fusaka 升级就包含对 transaction gas limit 等参数的改动，也是社区长期博弈后的结果。 目前大概有近百万个 validator、上万台服务器分布在不同国家，这种地理和软件层面的分散，让网络具备很强韧性。​

## 写在最后：对 Web3 的再理解

这次最大的收获，是把“去中心化”从一个 buzzword 还原成一系列具体的技术和经济设计：从交易、签名、Gas，到共识、客户端、多节点部署。 Web3 不是空谈理想，而是一整套在“没人可信”的前提下，依然能协同运转的基础设施设计。 下一步想做的，就是自己跑一个节点，真正从参与者的角度去体验这套系统，而不是只站在钱包 UI 的那一层看世界。​

​​2. Web3就业市场分析与策略​​

​​技术岗建议:​​ 技术背景者应优先考虑前端或后端开发，因智能合约开发岗位极为稀缺。建议快速搭建MVP并以成果为导向进入行业。

​​非技术岗建议:​​ 转型至运营或产品岗位风险较低。建议以结果为导向，优先考虑自己擅长的方向，如学习能力、设计天赋等。应深入研究行业术语和逻辑，如DAU、UV、CMC、增长运营等，以便在面试中展现专业度。

​​3. 核心就业策略建议​​

​​主动社交:​​ 推荐多参加行业峰会、主动联系项目方（VC）、KOL和交易所高管，建立人脉关系库。

​​持续学习:​​ 通过参加黑客松、深入理解行业和公司业务模式、学习优秀公司的运营策略等方式，不断提升自己的综合能力。

Web3领域的技术岗和非技术岗两大类别

非技术岗位：
