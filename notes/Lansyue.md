# 分享会笔记

从这次分享里，大概把自己理清了几个关键点，就按「为什么要搞 7962、它在做什么、怎么做到的、能用在什么地方」这几个问题来记一下。

## 从 ERC‑721 到 ERC‑7962：要解决什么问题

听下来，7962 最核心就是：在不牺牲区块链可验证性的前提下，尽量把「身份与资产」从公开地址里抽离出来，让隐私和用户体验都更友好。​

-   传统 ERC‑721：`ownerOf(tokenId)` 直接就是一个地址，谁持有什么 NFT，一查合约就一清二楚，很适合链上透明分析，但也意味着很容易给地址画「画像」。​
    
-   现实世界的会员身份（比如星巴克会员）本质上是隐私信息：品牌不能到处广播「某某是我们会员」，但在某些场景（比如机场贵宾厅）又确实需要「按需证明」这个身份。​
    

ERC‑7962 就是试图在这个缝里做一层「隐私友好的会员 / 资产证明层」：第三方能够验证「你确实有这个资格」，但看不到你到底是哪一个链上地址。​

## 零知识与哈希：7962 的基本思路

讲者没有走那种超重型的 zk 证明体系，而是用了相对轻量的思路：用哈希 + 签名做到「等于的证明」，刚好契合「是不是会员」这种二元问题。​

-   在普通 721 中：`ownerOf(tokenId) = address`。
    
-   在 7962 里：`ownerOf(tokenId) = K_hash`，这个 `K_hash` 是由某个公钥（或者说某个「秘密」）哈希出来的值。​
    
-   交易逻辑：
    
    -   真正的持有人只需要用自己的私钥对一段「转移意图」签名。
        
    -   任意一个 relayer（中继者）拿着这个签名和相关参数发起链上交易即可，合约只检查「签名与 K\_hash 背后的公钥是否匹配」。​
        

这样带来几个效果：​

-   合约里永远只看到一堆 K\_hash，看不到具体地址，外部难以做图谱分析。
    
-   发起交易的链上地址与资产所有权解耦：谁有 gas、谁愿意帮你提交交易，都可以做 relayer。
    
-   用户侧只需「签名表达意图」，不用关心钱包地址和 gas，体验更接近 Web2 的账号体系。
    

这里用到的「零知识」其实就是在一个很窄的子问题里发挥作用：证明「某个秘密对应的哈希在集合里」，而不暴露秘密本身，特别适合「你是不是在某个会员名单里」这种场景。​

## 隐私、UTXO 风格与没有 Approve

为了提高不可关联性，7962 更像把 ERC‑20/721 做成了「UTXO 风格」：​

-   每个 K\_hash 视为一个「一次性地址」：
    
    -   比如我有 100 个 token，转你 50 个。
        
    -   这笔交易之后，合约里不会继续用原来的 K\_hash 记录剩下的 50，而是为我生成一个新的 `left K_hash`，记录剩余余额。​
        
-   因为一笔交易之后，旧的 K\_hash 就算「暴露」过一次，接下来的余额会换到新的 K\_hash 上，链上很难把整条资金流串起来。​
    

在这个模型下，Approve 机制反而会破坏隐私：​

-   Approve 意味着「提前在链上声明某个 key 可以被某个地址操作」，这会暴露本该单次使用的 K\_hash 背后的公钥关系。
    
-   所以 7962 选择直接去掉 Approve，用「签名 + relayer」模式代替授权流程。
    

这套设计本质上通过强制「密钥轮换 + 哈希持有」来尽量切断交易之间的可关联性，只留下在特定时刻、特定交易的必要信息。​

## 账户抽象、Gas 代付与 Web2 → Web3 迁移

用户体验这一块，其实是这次分享里自己最有共鸣的部分：7962 借着「所有权与发起地址解耦」，顺带天然支持了账户抽象、Relayer 和 Gas 代付。​

典型流程可以抽象成：

-   用户：只在熟悉的 App / 小程序里点「同意」、「授权」，实质上是在签一个意图。
    
-   商家（如星巴克）：
    
    -   提前部署 7962 合约，内部维护一份「会员 ↔ K\_hash」映射。
        
    -   代表用户发起链上交易，自己或和合作方承担 gas。
        
-   第三方（如机场贵宾厅）：
    
    -   只需要根据合约中的状态和用户出示的证明，验证「这个人确实是星巴克会员」，不需要知道他在链上的真实地址，也不需要拿到一整份会员名单。​
        

对 Web2 用户来说，整个过程有两个显著好处：​

-   不再强迫他们先搞明白钱包、助记词、gas 这些 Web3 概念，门槛显著降低。
    
-   同时又能享受「上链记录不可篡改」、「隐私按需暴露」这些 Web3 带来的优势。
    

从商业侧看，有动力为用户代付 gas，因为每一次「链上证明」背后都是实实在在的业务合作（比如联合权益、联合营销），代付 gas 就是运营成本的一部分。​

## 安全、监管与应用边界的一些思考

分享里也有不少关于「隐私 vs 透明」「技术中立性」的讨论，这块值得记一下，方便后续自己延伸思考。​

-   区块链天生强调透明，零知识和 7962 这类方案则是在透明账本之上，增加一层「选择性可见」：
    
    -   账本依旧公开可验证。
        
    -   但「谁在背后」不必公开给所有人，只在必要时对必要方开放。​
        
-   技术本身是中立的：
    
    -   隐私工具可以保护普通人的数据不被滥用，也可能被坏人利用。
        
    -   真正的难点在于如何在制度层面配套，例如：在某些司法场景下，是否允许「受限披露」、是否要求特定门槛下的审计能力等。​
        

从开发者视角看，一个现实问题是：如果隐私技术被滥用，会不会代码作者背锅？分享里给的态度更偏向「合理使用 + 监管与技术边界的划线」——这块本质上已经超出了协议本身的范畴，但确实会是未来隐私协议绕不开的问题。​

* * *

总的来说，自己当前的理解是：ERC‑7962 并不是要彻底用 zk 重新发明一套复杂的资产系统，而是用相对克制的技术手段（哈希 + 一次性 key + 签名 + relayer）在现有 ERC‑721 / 20 语义之上，补出一块「隐私友好且 UX 友好」的空白，尤其适合会员身份、隐私资产、企业批量发放、Gas 代付这类场景。

# 难忘今宵，误闯天家

周五分享会笔记：

今天把这一长段转录看完的感觉是：这已经不是一场普通的分享会，而是一次把「怎么学、学什么、以及我在 Web3 里要成为什么样的人」摊开来认真讨论的集体头脑风暴。​

## 学到的硬技能和认知

技术部分从 Solidity 基础语法、地址类型、modifier 这些细节，一路到 Uniswap V2 的 swap 流程、TWAP 定价、Create2 计算 pair 地址，再到 MPC 钱包签名拆分、T-of-N 门限、设备备份与恢复，基本把「链上怎么跑」拆得很细。​

另一条线是底层与宏观：有人用图灵完备、合约跑在所有节点上来解释以太坊的独特性，有人讲 RWA、IPO 上链、三层结构（资产层 / 控制层 / 合约层），也有人从法律、治理角度拆「一币一票」的问题、有限合伙逻辑和人核 vs 资核的权力结构，这些都在补「Web3 为什么值得花时间」这件事的底层理由。​

## 学习方法与工具的升级

Notion、Figma、思维导图基本成了标配：有人用 OKR 把自己当成「一家公司」来管理，把产品、市场、技术、运营拆成不同角色，对应不同学习和输出任务；有人用 Figma 的 FigJam 当高配脑图，把课程和资料转成结构化图形，再反推自己哪里没懂。​

「导图 + AI」这一套被讲得很具体：先用导图暴露盲区，再把导出的文字喂给 AI，让它针对每个节点做补充解释、举例和答疑，而不是从零让 AI 讲一整门课；还有人用 Notabook / YouTube 转录 + AI 总结，把英文长视频变成可搜索的文字，再在上面做二次加工。​

## 输出、起号和内容方法论

输出不再被当成「写作业」，而是当成学习闭环的一部分：从日更笔记、导图复盘，到写长文、写串文、做直播，大家都在用公开输出来测试自己是不是真的讲得清楚。​

起号方面，思路非常明确：

-   标题上抓「普通人」「新人入门」「未来三年最值得学的××」这类高共鸣话题，或者直击焦虑（低精力人群怎么办、不会写代码怎么进 Web3）。​
    
-   内容上用「故事 +方法」结构：先讲自己为什么会写这篇，再讲具体拆解（是什么 / 为什么 / 怎么做），最后加一点情绪和态度，让账号有辨识度，而不只是干货列表。​
    
-   工具上不避讳用 AI 辅助：用别的 KOL 文风做参考、让 AI 帮忙扩写和改写，但有意识地在文章里加入自己的经历、意外细节和「我」的视角，尽量降低 AI 味。​
    

## 职业路径与个人定位的思考

整个转录里能看到很多不同起点：有已经在交易所、项目方的，有从传统互联网、快消、咨询、媒体转过来的，有法律专业、经济学背景，也有还在读书做研究的，同一个问题是「我到底适合在 Web3 里站在哪个位置」。​

回答也挺多元：

有人强调「归零」：有职场经验的人要愿意放下原来的光环，从基础概念重新学起；​

有人把自己定位在策略、产品、运营、法律、治理等「非纯技术」岗位，用原本的经验去填补 Web3 现在严重缺乏的专业化能力；​

也有人很直白地说，Web3 现在 80% 是「朝台班子」，真正高效的人只有一小撮，这反而意味着认真打基础、持续积累的人，未来能明显跑在平均线前面。​

## 对自己后续行动的启发

学习这块：需要的不只是「听懂」，而是搭出一张自己的 Web3 知识地图，用导图 + 笔记 + AI 把概念、协议、工具、法律与治理串成一套可以不断更新的结构，而不是一堆截图和收藏夹。​

输出这块：值得尽早开始起号和写长文，不是为了一两篇的爆量，而是用「连续的小输出」做自我校验：我能不能把今天吸收的东西，讲给一个完全不懂的朋友听懂，这决定了学习是停在 60 分的掌控感，还是往 80 分的真正掌握去走。​

职业上：比起「我要不要 all in Web3」这种大问题，现在更可执行的是三件小事——明确自己想在生态里的角色（builder / 研究 / 内容 / 法律 / 产品等），围绕这个角色规划接下来几周的学习和项目尝试，然后用公开输出和社区连接，把自己一点点「长」到那个位置上去。

分享会笔记：

一、核心主题与 AI 的关键转变

本次分享核心是 AI 与 Web3 的融合，聚焦如何让 AI 从 “嘴替”（仅聊天互动）升级为 “打工人”（能自主干活、赚钱）。AI 的关键转变体现在 2023 年到 2026 年的突破：

2023 年大语言模型（LLM）本质是 “自回归 TOKEN 预测器”，像 “话痨学霸”，能生成长文本但无实际执行能力；

2026 年 Agentic AI 突破三大限制：通过向量库拥有私有记忆、通过 API / 工具调用具备执行能力、通过 ERC8004 获得链上身份，实现 “能想、能做、能收钱”。

二、传统 AI 的痛点与 Web3 的解决方案

（一）传统 AI 的三大致命问题

无状态：对话忘上下文，无法长期交互；

无工具：仅生成文本，不能执行代码、调用 API；

无身份：不能代表组织行动、签署协议或管理资产。

此外，还面临身份不可验证（易欺诈）、行为不可审计（日志可篡改）、部署不可管控（隐私安全隐患）的信任危机，以及支付层面的身份认证、交易颗粒度、结算速度、订阅模式错配等问题。

（二）Web3 的核心解决方案

Web3 通过 “不可伪造的身份证” 和 “可编程银行账户”，精准解决 AI 的信任与支付难题，核心依赖两大协议：

ERC8004（AI 链上身份证）：基于 ERC721 标准，为每个 AI 铸造唯一 FT 身份 TOKEN，包含身份注册表（基本信息、所有者、能力声明）、声誉注册表（0-100 分评分，仅付费客户可评价，防刷分、不可篡改、时间衰减）、验证机制（TEE 硬件安全模块 + ZKML 零知识证明，确保 AI 表里如一）；

X402（原生支付协议）：嵌入 HTTP 协议，返回 402 状态码时携带支付信息，支持微支付（如 0.0001 美元）、毫秒级结算，通过支付通道、批量结算降低 gas 费，适配 AI 高频小额交易场景。

三、Spoon OS 框架核心内容

Spoon OS 是 Web3 生态的 AI 操作系统，类似 “AI agent 安卓系统”，封装底层复杂技术，让开发者专注业务逻辑：

（一）核心价值

降低开发门槛：Python 开发者 2 天内即可让 AI agent 具备 Web3 能力，无需关注私钥管理、交易签名等；

全流程支持：覆盖数据采集、模型适配、任务编排、身份管理、支付处理等全环节。

（二）四层架构

输入层：数据采集（链上数据、文档等）、预处理（清洗、分块、向量化），依赖向量数据库存储与相似度检索；

核心层：统一模型接口（适配 OpenAI、Anthropic 等）、智能体编排引擎（支持多 agent 协作 / 任务分配）、记忆管理器（区分短期 / 长期记忆）、工具调用器；

协议层：实现 ERC8004（身份、声誉、验证）和 X402（支付处理），提供钱包托管与私钥安全存储；

输出层：对接 DeFi、跨链桥等外部系统，实现资产转移、交易执行等实际操作。

（三）典型应用场景

Defi 交易智能体（套利机器人等）、数据市场（出售高质量数据）、AI 协作（处理仓库 issue/PR）等。

-   GoPlus 统计到全年大约 1200 起安全事件，项目方和 C 端用户各占一半，合计经济损失超过 35 亿美元。​
    
-   攻击方式主要分为两类：一类是面向交易所、高 TVL 协议的“精准猎杀”，另一类是面向普通用户的“广撒网”式攻击，如钓鱼站、Rug Pull 和貔貅盘等。​
    

### 典型事件带来的认识

-   Bybit 案例中，黑客通过多签服务商 Safe 的前端入手，篡改界面逻辑，导致管理人员在“看起来正常”的界面下完成了被攻击者修改的交易，最终损失约 15 亿美元。​
    
-   Sonne Finance 的合约被利用后，由于团队事先搭建了相对完善的应急机制，成功追回和冻结部分资金，将实际损失从 2.23 亿美元控制在大约 6000 万左右。​
    
-   Balancer 这样的老牌协议也在 AI 辅助挖掘下暴露出深层漏洞，说明历史项目并不天然安全。​
    
-   围绕 UXLINK 等的社工攻击中，攻击者利用 Deepfake 视频、音频伪装成合作方或投资人，通过会议诱导受害者安装恶意软件或点击钓鱼链接。​
    

### 新技术放大的安全风险

-   AI 与社工的结合正在削弱“看到真人就放心”的直觉判断，远程面试、投融资路演、线上会议都可能成为攻击场景。​
    
-   以太坊 7702 为外部地址引入合约能力后，黑客开始设计新的钓鱼签名，引导用户签署恶意 delegate，从而远程控制地址并进一步发起二次钓鱼。​
    
-   在 AI 辅助编程和插件大量使用的背景下，软件供应链攻击风险上升，例如浏览器钱包插件被植入后门代码的事件已经出现。​
    
-   地址投毒通过小额转账制造“首尾相同、中间不同”的假地址，利用用户依赖历史记录复制的习惯造成误转。​
    

### 面向 C 端用户的建议

-   尽量坚持“四不”：不随意点不明链接、不盲目签任何看不懂的钱包签名、不乱装来源不清的软件、不轻易转账给风险不明的地址或所谓高收益项目。​
    
-   进一步的做法包括冷热钱包分离，将大额资产放在安全等级更高的地址，并妥善、离线保存私钥和助记词。​
    

### 面向项目方与开发者的建议

-   在开发阶段引入成熟的安全库，并在上线前进行专业审计，把安全当成全生命周期的一部分，而不是事后补救。​
    
-   对 AI 生成代码和各类插件保持警惕，关键逻辑需要人工审查，以降低供应链被利用的可能性。​
    
-   预先设计并演练应急响应流程，在真正发生攻击时能够第一时间冻结资金、通知用户并与生态伙伴联动止损。​

# web3运行原理分享会笔记

今天听完这场《Web3 和区块链技术基础》分享，终于把自己之前零散的认知串成了一张比较完整的图。​

## 关于交易和钱包

过去总觉得“交易”就是转账，这次才意识到，一笔链上交易其实是“操作 + Gas 手续费 + nonce（防重放序号）”三件事打包在一起。 操作可以是转账、调用合约、投票等等，钱包只是负责帮我把这些信息组装好、用私钥签名，然后丢给网络去处理。 实际在测试网看了一笔交易的详情，to、金额、Gas limit、nonce 一目了然，感觉之前只看余额完全是浪费区块浏览器。​

## 数字签名和安全感

数字签名这一块非常打消“我怎么证明是我发的”的焦虑：用私钥对交易做签名，任何人都能验证签名确实来自这个地址，但没人能伪造出同样的签名。 验证的时候其实就是用“原始消息 + 声称的地址 + 签名”三件套，跑一遍算法去反推出是不是同一个人签的。 结论也很现实：私钥就是一切，谁拿到私钥，谁就能发出被全网认可的“我说的话”。​

## Gas 费和经济设计

Gas 被讲成“油费”这个比喻很形象：Gas price 像油价，Gas usage 像你加了多少升，两者相乘就是最后的手续费。 操作越复杂、网络越拥堵，Gas 就越贵，这相当于用经济手段直接把垃圾交易和恶意行为的成本拉高。 这也是整个网络经济闭环的一部分，矿工 / 验证者不是在“做爱心”，而是在一个有激励、有成本的系统里工作。​

## 交易一生：从钱包到区块链

一笔交易的生命周期，第一次在脑子里跑通：钱包签名 → 通过 RPC 发到网络 → 内存池排队 → 被 builder / validator 挑选 → 打包进区块 → 共识确认 → 链上永久记录。 动画里 12 秒一个新区块、几百笔交易按 Gas 高低排队的画面，让“打包”这件事不再抽象。 每个区块之间通过 parent hash 串起来，要想篡改一个区块，就得连它后面所有区块一起改，这基本就是在跟全网算力或质押对着干。​

## POW、POS 和去中心化

共识这块，讲得很克制：POW 靠算力、POS 靠质押资金，各有优缺点，没有完美方案。 比特币用 POW，拼的是谁算得快；以太坊现在用 POS，拼的是谁愿意锁更多押金、接受作恶被罚没的风险。 本质上都是让“作恶的代价”远远大于“作恶的收益”，只是筹码不一样。​

## RPC、客户端和“隐形”的中心化

之前用钱包从没认真想过 RPC，今天才意识到：钱包其实是通过 RPC API 去连节点集群，而不是直接连区块链本体。 这意味着 RPC 服务商本身就是一个潜在的中心化风险点，拦截或拒绝转发交易，等于别人替你“掐网线”。 以太坊主网有多个 RPC 提供方、执行层和共识层也有好几个客户端，这种多样性就是在分散单点故障和软件 bug 的风险。​

## 智能合约和 “Code is law”

智能合约那部分对“Web3 = 可编程规则”这件事很加分：本质就是写死在账本里的可执行代码，调用时在虚拟机里跑一遍，把结果写回链上。 现场看了一个 MyFirst NFT 合约，每个地址只能 mint 一个，这种规则写死在代码里，任何人都能查逻辑、查历史执行记录。 “Code is law” 不是一句口号，而是把对人、机构、政府的信任，转成对代码、共识机制、治理流程的信任。​

## NFT mint、区块确认与 Finality

实际 mint NFT 的演示很直观：签名的数据比简单转账复杂得多，因为背后是对合约方法的调用。 交易一旦被打进区块，刚开始还是 unfinalized 状态，需要十几分钟等到 finality，才能说这件事“几乎不可逆”。 这种“概率上不可篡改”的设计，和传统数据库的“超级管理员说了算”形成了鲜明对比。​

## 以太坊升级、社区治理和节点分布

升级机制也挺社会学的：改协议要走 EIP 流程、社区讨论，每年 1–2 次硬分叉升级，不是谁一拍脑袋说了算。 最近的 Fusaka 升级就包含对 transaction gas limit 等参数的改动，也是社区长期博弈后的结果。 目前大概有近百万个 validator、上万台服务器分布在不同国家，这种地理和软件层面的分散，让网络具备很强韧性。​

## 写在最后：对 Web3 的再理解

这次最大的收获，是把“去中心化”从一个 buzzword 还原成一系列具体的技术和经济设计：从交易、签名、Gas，到共识、客户端、多节点部署。 Web3 不是空谈理想，而是一整套在“没人可信”的前提下，依然能协同运转的基础设施设计。 下一步想做的，就是自己跑一个节点，真正从参与者的角度去体验这套系统，而不是只站在钱包 UI 的那一层看世界。​
