# 用 Solidity + Hardhat + React + Ethers 做一个**NFT 铸造小工具 dapp**，跟着抄一遍，就能实现「填几个字段 → 钱包签名 → 铸出一张属于自己的 NFT」。

* * *

## 0\. 先说清楚我们要做什么

这篇教程的目标，是跑通这样一条完整链路：

> 浏览器前端（React） → 连接钱包（MetaMask） → 调用链上的 ERC‑721 合约 → 铸造一张 NFT，并在页面上把它展示出来。

整个项目可以分成三块：

-   智能合约：基于 OpenZeppelin 写一个最小的 ERC‑721，提供 `mintNFT` 之类的方法。
    
-   后端 / 链：用 Hardhat 搭本地开发环境，把合约部署到本地链或测试网。
    
-   前端：用 React + Ethers，连上 MetaMask，对合约做读写，并给用户一个简单的铸造界面。
    

选这个例子，是因为它足够“有成就感”：一条交易之后，你就多了一张真正的链上资产，不再只是一个数字在那儿累加。

* * *

## 1\. 环境准备：链、钱包、工具

## 1.1 区块链和钱包

为了降低门槛，全部操作先在本地或测试网完成：

-   区块链：
    
    -   用 Hardhat 自带的本地以太坊节点，命令一跑，秒出区块，免费 gas。
        
    -   等熟练之后，你可以把同样的合约部署到以太坊测试网（如 Sepolia）。
        
-   钱包：
    
    -   浏览器装好 MetaMask 扩展，这是主流 dapp 里最常见的钱包入口。
        
    -   本地链场景下，可以把 Hardhat 生成的私钥导入 MetaMask，这样你有一堆预置 ETH 可用。
        

## 1.2 Hardhat 项目初始化

新建一个目录，比如 `nft-minter`，然后：

-   初始化项目：
    
    -   `npm init -y`
        
    -   `npm install --save-dev hardhat`
        
    -   `npx hardhat`，选 “Create a JavaScript project”，一路回车。
        

Hardhat 会帮你生成基础目录结构（`contracts/`、`scripts/`、`test/` 等），这是当前最主流的以太坊开发脚手架之一。

* * *

## 2\. 写一个最小可用的 NFT 合约

目标是写出一个「能铸造 NFT」的 ERC‑721 合约，不求花活，先保证安全、标准、好用。

## 2.1 引入 OpenZeppelin 合约库

在项目里安装 OpenZeppelin：

-   `npm install @openzeppelin/contracts`。
    

在 `contracts/` 下新建 `MyNFT.sol`，整体思路是继承标准实现，而不是从零写所有接口。

主体结构大致如下（只讲逻辑，不贴完整源码）：

-   声明 Solidity 版本，比如 `^0.8.20`。
    
-   `import "@openzeppelin/contracts/token/ERC721/extensions/ERC721URIStorage.sol";`
    
-   `import "@openzeppelin/contracts/access/Ownable.sol";`
    
-   `contract MyNFT is ERC721URIStorage, Ownable { ... }`。
    

这里用了两个扩展：

-   `ERC721URIStorage`：帮你维护 `tokenId -> tokenURI` 映射，让每个 NFT 都有独立的 metadata 地址。
    
-   `Ownable`：让你可以限制只有合约 owner 能执行某些操作（比如受控铸造）。
    

## 2.2 核心逻辑：mint 函数

在合约里准备几个成员：

-   一个自增计数器：`uint256 private _tokenIds;`。
    
-   构造函数设置名称和符号：`ERC721("MyNFT", "MNFT")`。
    

然后写一个对外暴露的 `mintNFT` 函数，比如：

-   参数：
    
    -   `address recipient`：接收者地址（通常是 `msg.sender`）。
        
    -   `string memory tokenURI`：NFT 的 metadata 地址（比如一个 IPFS 链接）。
        
-   函数内部流程：
    
    -   `_tokenIds++` 得到新的 `tokenId`。
        
    -   `_safeMint(recipient, newTokenId);`。
        
    -   `_setTokenURI(newTokenId, tokenURI);`。
        
    -   返回 `newTokenId` 方便前端展示。
        

你可以选择是否加 `onlyOwner` 限制铸造权限：

-   如果挂 `onlyOwner`，就变成“只有项目方能给别人发 NFT”；
    
-   如果不加，任何人都能自己铸造自己的 NFT，更适合教学 demo。
    

## 2.3 编译合约

执行：

-   `npx hardhat compile`。
    

编译成功后，Hardhat 会在 `artifacts/` 里生成 ABI，前端之后会用到它来实例化合约对象。

* * *

## 3\. 部署合约，把它“丢到”链上

现在要做的是：开一个本地链，把 `MyNFT` 部署上去，拿到一个真实合约地址。

## 3.1 写部署脚本

在 `scripts/` 下新建 `deploy.js`，逻辑大致如下：

-   从 Hardhat 的环境里拿到 ethers 和合约工厂：
    
    -   `const MyNFT = await ethers.getContractFactory("MyNFT");`。
        
-   部署：
    
    -   `const myNft = await MyNFT.deploy();`
        
    -   `await myNft.deployed();`。
        
-   打印地址：
    
    -   `console.log("MyNFT deployed to:", myNft.address);`。
        

写教程时，你可以加一句人话解释：  
“这一步相当于之前把后端代码部署到服务器上，只不过现在服务器换成了区块链，它返回给你一个合约地址。”

## 3.2 启动本地链并执行部署

-   启动本地节点：
    
    -   `npx hardhat node`。
        
-   另起一个终端，运行部署脚本：
    
    -   `npx hardhat run scripts/deploy.js --network localhost`。
        

终端会打印类似 `MyNFT deployed to: 0x1234...abcd` 的信息，把这个地址复制下来，前端马上要用。

* * *

## 4\. React 前端：做一个 NFT 铸造页面

前端部分，其实就是一个带三个输入框和一个按钮的页面，再加上 MetaMask 连接逻辑。

## 4.1 初始化 React 项目

在项目根目录新建一个 `frontend/`，或者单独起一个目录：

-   `npx create-react-app nft-minter-frontend`（或者 `npx create-next-app`，看你偏好）。
    
-   进入目录后安装 Ethers：
    
    -   `npm install ethers`。
        

你也可以把 Hardhat 项目和前端放在同一个 monorepo 里，但教程里建议先分开，方便读者理解角色分工。

## 4.2 准备 ABI 和合约地址

前端需要知道两个东西：

-   合约地址：刚才部署时打印出来的那串 `0x...`。
    
-   合约 ABI：从 Hardhat 的 `artifacts/contracts/MyNFT.sol/MyNFT.json` 里复制 `abi` 字段，或者直接把这个 JSON 文件拷到前端项目中用 `import` 引入。
    

推荐写个单独的配置文件，比如 `src/contractConfig.js`，导出：

-   `export const CONTRACT_ADDRESS = "0x..."`
    
-   `export const CONTRACT_ABI = [...]`。
    

* * *

## 5\. 在前端连接钱包 + 合约

下面这部分会在你的 React 组件里完成，可以统一写在 `App.jsx` 里，保持足够“可读”。

## 5.1 连接 MetaMask

先处理钱包连接逻辑：

-   几个状态：
    
    -   `account`：当前钱包地址。
        
    -   `status`：界面提示信息。
        
    -   `nftName`、`nftDescription`、`nftUrl`：三个输入字段。
        
-   `connectWallet` 函数大致流程：
    
    -   如果 `window.ethereum` 不存在，提示“请安装 MetaMask”。
        
    -   否则调用：
        
        -   `const accounts = await window.ethereum.request({ method: 'eth_requestAccounts' });`。
            
        -   取 `accounts[0]` 写进 `setAccount`。
            
    -   状态栏提示“钱包已连接”。
        

MetaMask 开发文档里对这套流程有非常详细的解释，可以在教程末尾贴个参考链接。

## 5.2 使用 Ethers 构造合约实例

连接成功后，在前端封装一个函数 `getContract()`：

-   `const provider = new ethers.providers.Web3Provider(window.ethereum);`
    
-   `const signer = provider.getSigner();`
    
-   `const contract = new ethers.Contract(CONTRACT_ADDRESS, CONTRACT_ABI, signer);`。
    

这三个对象各司其职：

-   provider：连到链的“通道”，负责读数据。
    
-   signer：带私钥的账户，负责发交易、签名。
    
-   contract：一个代理对象，让你能像调用普通函数一样调用链上的合约方法。
    

* * *

## 6\. 实现真正的「铸造」按钮

现在页面上应该有三个输入框 + 一个 “Mint NFT” 按钮，逻辑都围绕前面写好的合约的 `mintNFT` 函数。

## 6.1 前端表单

布局可以简单暴力：

-   输入框 1：NFT 图片 / 媒体链接（可以用随便一张网络图当示例）。
    
-   输入框 2：NFT 名字。
    
-   输入框 3：描述。
    
-   按钮：`Mint NFT`。
    

这里有两个处理方式：

-   简化版：直接把用户输入的链接当作 `tokenURI`。
    
-   更正式版：
    
    -   前端先把 name / description / image 组装成 metadata JSON。
        
    -   把 JSON 传到 IPFS（比如用 Pinata、Web3.Storage），再把 IPFS 链接作为 `tokenURI`。
        

对于入门教程，完全可以先用简化版，让读者先跑通“合约 + 前端 + 钱包”这一层。

## 6.2 mint 函数调用

在前端写一个 `handleMint`：

-   基本校验：如果有任何一个字段为空，就直接在 `status` 显示“请把信息填完整”。
    
-   `const contract = getContract();` 拿到实例。
    
-   构造参数：
    
    -   `const tokenURI = nftUrl;`（简化版本）
        
    -   `const recipient = account;`。
        
-   调用合约：
    
    -   `let tx = await contract.mintNFT(recipient, tokenURI);`
        
    -   在 `status` 里提示“交易已发出，等待确认…”。
        
    -   `let receipt = await tx.wait();` 等链上确认。
        
-   解析结果：
    
    -   从 `receipt.events` 里可以拿到 `Transfer` 事件，事件里会包含新 `tokenId`。
        
    -   在页面上显示类似 “✅ Mint 成功，tokenId = X，交易哈希：0x…” 的信息。
        

在这一步，你可以把“读”和“写”的区别再强调一遍：

-   `mintNFT` 是 **写** 操作，需要签名 + gas + 等打包。
    
-   如果你额外写了一个 `tokenURI(tokenId)` 读函数，那它就只是简单的链上查询，不需要用户确认。
    

* * *

## 7\. 跑通一次完整体验

把所有环节都接好之后，可以用一个 checklist 做最后的自检。

1.  Hardhat 本地链已经开着：`npx hardhat node`。
    
2.  `MyNFT` 合约成功部署，拿到了一个合约地址，并且前端配置文件里已经用这个地址替换占位符。
    
3.  React 前端已经能正常启动：`npm start`，浏览器访问 `http://localhost:3000`。
    
4.  点击 “Connect Wallet”，MetaMask 弹出确认框，确认后页面展示当前钱包地址。
    
5.  填好图片链接 / 名字 / 描述，点 “Mint NFT”：
    
    -   钱包再次弹出交易确认框，显示你要调用的合约地址和 gas 费用。
        
    -   确认后稍等几秒，页面状态变为“Mint 成功”，并显示新 NFT 的 `tokenId`。
        

如果你愿意多走一步，可以把合约地址导入 MetaMask 或在区块浏览器里查看，甚至把合约部署到某个测试网，体验一次“链上可查”的感觉。

* * *

## 8\. 从工具小 dapp 到真正的产品

做到这里，你已经有了一个能工作的 NFT 铸造工具雏形，接下来可以做的升级很多，比如：

-   加白名单、限量、价格逻辑，让每次铸造需要支付一定 ETH；
    
-   把 metadata 正式托管在 IPFS / Arweave 上，避免链接失效；
    
-   接入某条更适合 NFT 的链（Flow / Aptos / 侧链等），或者直接按项目需要换成 ERC‑1155。

跟着 Elon 老师系统过了一遍 Solidity，感觉自己从「只知道链上有合约」到「大致知道合约在里面是怎么活的」跨了一小步。

## 对 Solidity 的整体感受

-   最大的变化是视角从“写代码”变成“设计一份会被全网执行和审计的账本逻辑”，语法反而成了最不重要的那一层。
    
-   学完一套 ERC20，从数据结构到函数、再到事件和权限，大概能看懂一个基础代币合约在做什么，也意识到安全和 gas 才是真正的「难点」。
    

## 对 EVM 与存储的理解

-   之前对 EVM 只有「虚拟机」这种抽象，现在会更自然地把它拆成 stack、storage、memory、calldata 四块，对应执行、永久存储和临时内存/入参，各自的 gas 成本也会下意识去想一下。
    
-   storage 像硬盘、memory/calldata 像运行时内存这套类比，在后面看合约读写、算 gas 的时候非常顺手，也更能理解为什么「少存、少改」几乎是所有优化的起点。
    

## ERC20 合约带来的结构化认知

-   一套完整 ERC20 把很多零散概念串起来：name/symbol/decimals/totalSupply/owner 这些字段落在 storage 里，再加上 balances 和 allowance 两本账本，合约就从「代码」变成了「规则固定的数据库」。
    
-   transfer / approve / totalSupply / balanceOf 再配上 Transfer、Approval 事件之后，会开始习惯从「接口 + 事件」这两个维度去理解一个合约，而不是只盯着函数体里的几行代码。
    

## 对安全与 gas 优化的新敏感度

-   require / revert / assert、自定义 error、unchecked 这些原本看起来有点「细枝末节」的东西，现在会自动联想到它们对 gas、可读性和安全边界的影响，而不是只把它们当语法糖。
    
-   modifier 和 onlyOwner 模式，让权限控制、重入锁这类逻辑有了更「工程化」的写法，也顺带理解了很多链上套路，比如钱包助记词陷阱、抢跑 bot 等等背后的实现思路。
    

## 学习路径与后续打算

-   Remix 这套「写 → 编译 → 部署 → 调用 → 看 storage/memory/事件」的闭环，把 Solidity 从理论快速拉到「能自己动手调一遍」的层面，之后再看别人合约就不会完全抽象。
    
-   目前对语法和基础 ERC20 还停留在「能读能改一点」的阶段，下一步会更多从协议级源码入手，比如 Uniswap V3、Aave，重点盯它们的 storage 设计、权限和安全机制，而不是急着去写花哨的新合约。

## 一、Telegram 社群搭建与基础设置

## 1\. 建群与基础信息

Telegram 建群本身很简单：新建群聊 → 取名 → 拉人，就算完成了最基本的社群搭建。

群的基础信息至少包括：头像、群名称、简介（描述社群定位），这些决定了新用户对群的第一印象。

群类型可以设置为 private group（私密群，只能邀请进入）或 public group（公开群，可以通过链接或搜索加入），根据用途选择即可。

## 2\. 权限与管理员

群权限要优先设置好，例如：谁可以改群信息、谁可以拉人、能不能创建新话题/频道等，避免新成员随意改动群设置。

管理员最好“按角色给权限”，而不是一股脑给满权限：

-   全职运营、核心团队可以有改群信息、踢人、设机器人等完整权限。
    
-   外聘的版主/志愿者更适合给部分权限，比如只能删消息、禁言、踢出等。
    

这一点在 Web3 社群里尤为重要，因为一旦错误给权给了恶意账号，对方可以借机器人、群公告等手段发钓鱼链接，后果比较严重。

## 3\. Topic（话题）的使用与整理

Telegram 支持话题，把一个群拆成多个“子板块”，对信息量大、讨论主题多的社群会更友好。

话题不要贪多：

-   过多的 topic 会让新成员无从下手，运营侧也容易失控。
    
-   对长期不活跃的 topic，可以考虑关闭或合并，保持信息结构的“可读性”。
    
-   可以参考成熟项目的做法：高优先级信息（公告、活动、重要更新）固定在少数几个话题里，方便快速浏览。
    

* * *

## 二、使用 Rose 机器人管理 Telegram 群

## 1\. 认准官方账号与基础接入

Rose 是一个常用的 Telegram 群管理机器人，可以做欢迎、封禁、防刷屏、黑名单关键词等多种管理工作。

由于高仿号很多，添加前一定要确认 handle 和认证信息，确保是官方 Rose，而不是钓鱼仿冒版本。

基本接入流程：

-   先在与 Rose 的私聊里 `/start`，完成初始化。
    
-   把 Rose 拉进自己的群，并授予管理员权限，否则很多管理功能无法启用。
    

## 2\. 核心功能与常用命令

Rose 属于“全能型群管”，官方文档列出的功能包括：

-   locks（锁定功能，例如禁止发链接、媒体、机器人等）
    
-   blocklist（关键词黑名单）
    
-   welcome messages（新成员欢迎语）
    
-   CAPTCHA（入群验证，防止机器人号）
    
-   notes、filters 等扩展能力。
    
-   管理员权限与 Rose 行为是绑定的：
    
    -   例如有“Change Group Info”的管理员可以通过 Rose 改设置、有“Delete Messages”的可以用删除/清理命令等。
        
-   `/settings` 是一个入口命令，可以在群里或私聊中打开 Rose 设置菜单，对群进行可视化配置。
    

## 3\. Blocklist（关键词屏蔽）的使用

-   Blocklist 的逻辑是：设置一批触发词，只要有人发消息包含这些词，机器人就按规则处理（删帖、警告、禁言等）。​
    
-   添加方式通常类似：
    
    -   触发命令（如 `/addblocklist` 或对应菜单）；
        
    -   填写要屏蔽的词（例如“赚钱”“投资返利”等容易被诈骗账号利用的词）；
        
    -   保存后生效。
        
-   可以通过查看 blocklist 列表和删除条目来不断调整，让规则既能挡住垃圾信息，又不过度干涉正常对话。
    

## 4\. 安全与滥用风险

-   给机器人管理员权限本身就意味着“把群的大门钥匙交给了它”，所以：
    
    -   必须确认为官方账号；
        
    -   管理面板的访问和修改最好只开放给少数可信管理员。​
        
-   Web3 场景中，最典型的风险是：假冒机器人在群里发布“空投”“高收益项目”等钓鱼链接，用户一旦点击授权或转账，很容易资产被转走。
    

* * *

## 三、数据面板：读懂 Telegram 社群的数据

## 1\. 数据面板入口与前提

-   对于拥有管理员权限的群，Telegram 和部分管理工具可以提供数据面板，用来统计：成员数、发言人数、消息量、增长曲线等。
    
-   新建不久的群往往数据为空或不完整，需要一定时间积累活动数据，才能看出趋势。
    

## 2\. 关键指标与运营意义

-   常见可见维度包括：
    
    -   成员数量与增长曲线：可以和活动时间线对照，看某次活动对拉新是否有效。
        
    -   发言人数与消息量：判断活跃度，而不是只看“群有多少人”。
        
    -   加入来源：区分邀请链接、自然搜索、活动专用链接等，有助于评估不同渠道的带量效果。
        
    -   语言分布：如果英语或其他语种用户突然增多，就需要考虑是否要增设对应语种的管理员或分区。\[[formo](https://formo.so/blog/how-to-build-a-web3-community-best-engagement-strategies)\]​
        
-   运营闭环里，这些数据主要用于：
    
    -   评估活动是否达标；
        
    -   发现潜力人群（例如海外用户上升）；
        
    -   指导之后的内容策略和时间选择。
        

## 3\. 活跃时段与消息投放

-   许多后台都会给出「活跃时段」之类的统计，显示一天中哪些小时段消息量最高。\[[formo](https://formo.so/blog/how-to-build-a-web3-community-best-engagement-strategies)\]​
    
-   可以把最关键的公告、活动推广安排在这些高活跃时段，提高触达概率，而不是在大家都不在线的时候发完就“沉底”。\[[formo](https://formo.so/blog/how-to-build-a-web3-community-best-engagement-strategies)\]​

## 四、从 0 到 1 策划一场 Twitter Space

## 1\. 明确主题与受众

-   一场 Space 的策划，第一步是确定主题和想要吸引的目标人群，例如：以太坊十周年回顾、某个新协议（如 404 类资产）等。\[[linkedin](https://www.linkedin.com/posts/vestpi_as-usual-we-have-summarily-compiled-the-activity-7415428100282540032-MIGc)\]​
    
-   主题确定后，要给 Space 起一个清晰、有辨识度的标题，既能让核心圈层一眼看懂，又能对路人有吸引力。\[[linkedin](https://www.linkedin.com/posts/vestpi_as-usual-we-have-summarily-compiled-the-activity-7415428100282540032-MIGc)\]​
    

## 2\. 准备问题库与嘉宾名单

-   主持人需要提前做功课，整理问题库，而不是完全临场发挥：
    
    -   主题的背景与现状：它解决了什么问题、和现有方案相比有什么不同。
        
    -   该领域最近的热点和争议。
        
    -   嘉宾过往提到过的观点，可以顺势接住深入聊。\[[linkedin](https://www.linkedin.com/posts/vestpi_as-usual-we-have-summarily-compiled-the-activity-7415428100282540032-MIGc)\]​
        
-   嘉宾选择可以从已有印象入手：
    
    -   谁在这个话题上公开聊得多、观点清晰；
        
    -   谁有对应的项目或组织背景，能补足不同视角。\[[linkedin](https://www.linkedin.com/posts/vestpi_as-usual-we-have-summarily-compiled-the-activity-7415428100282540032-MIGc)\]​
        

## 3\. 邀请嘉宾的沟通要点

-   邀请函要做到“信息充分但不啰嗦”，建议至少包含：
    
    -   活动主题与时间；
        
    -   大致会涉及的内容方向（从问题库中抽几条代表性的）；
        
    -   形式说明：Twitter Space，时长大致多久，有无 Q&A 环节。\[[linkedin](https://www.linkedin.com/posts/vestpi_as-usual-we-have-summarily-compiled-the-activity-7415428100282540032-MIGc)\]​
        
-   这样嘉宾可以快速判断：是否匹配自己的专业、时间是否合适、准备成本怎样。
    

## 4\. 收集嘉宾信息用于宣发

-   为了制作海报和活动推文，建议提前收集这四类信息：
    
    -   姓名或常用 ID；
        
    -   Twitter handle；
        
    -   所属机构/项目及职务；
        
    -   头像（或允许使用的公开形象图）。
        
-   这些信息既用于海报，也用于文案中的 @ 标记，方便嘉宾以及其背后的机构账号转发，放大传播效果。

## 五、Twitter Space 的创建、宣发与复盘

## 1\. 创建 Space 与时间规划

-   在 X（原 Twitter）上创建 Space 的大致步骤：
    
    -   打开移动端 App，长按发推按钮或点击底部的 Spaces 图标；
        
    -   选择“创建 Space”，填入标题、话题标签；
        
    -   通过日历/时间设置界面预约时间（注意时区，一般中国地区是 UTC+8）。
        
-   建议在 Space 开始前的 2–3 天就完成创建，并留出时间配合海报和文案做预热。
    

## 2\. 宣发节奏与渠道

-   宣发材料包括：海报、核心文案、Space 链接。
    
-   常见节奏：
    
    -   T-2~3 天：首发海报+长一点的介绍，明确主题、嘉宾、时间、收听方式。
        
    -   T-1 天：再次提醒，把文案精简一点，强调倒计时。
        
    -   活动开始前 5–10 分钟：最后一波“马上开始”的提醒。
        
-   渠道上不仅在 X 发，还可以同步到 Telegram 群、Discord、微信群等，保证目标人群尽可能都能看到。
    

## 3\. Space 当场执行要点

-   开场：
    
    -   主办方简单介绍项目/组织；
        
    -   逐一请嘉宾自我介绍，给足个人品牌露出的空间；
        
    -   简要说明这场 Space 的背景和将要覆盖的内容，让临时进来的人也能快速“对上频道”。
        
-   中段问答：
    
    -   按照问题库推进，同时根据现场回答灵活追问。
        
    -   如果某位嘉宾已经“顺带回答”了后面的问题，可以调整提问顺序，避免重复。
        
-   Q&A 环节：
    
    -   不是强制的，但如果现场氛围不错，开放提问可以增强参与感；
        
    -   要注意时间控制，不要让 Q&A 抢掉主线内容。
        

## 4\. 活动数据与复盘

-   复盘时可以重点关注几类指标：
    
    -   宣发推文的浏览量、互动数（点赞、转发、评论）；
        
    -   Space 的实时在线人数峰值，这是判断“热度”最直接的指标；
        
    -   总收听次数可以参考，但要注意其容易“虚高”：同一个人反复进出可能被计多次。
        
-   Web3 项目在做 Space 时也要注意“买量”现象：
    
    -   有些账号会用机器人充人数，其中相当一部分听众头像会被折叠到列表里，而不是展示在主视图；
        
    -   真正的质量信号，往往来自现场互动、后续讨论和有内容的转发，而不仅仅是一个好看的“总收听数”。
        
-   总结时可以从主题匹配度、宣发到位程度、时间选择、嘉宾结构、技术与流程问题等维度，写出下次要保留和要改进的点，这样每办一次活动，自己的“活动运营手感”都会更扎实一些。

# 分享会笔记

从这次分享里，大概把自己理清了几个关键点，就按「为什么要搞 7962、它在做什么、怎么做到的、能用在什么地方」这几个问题来记一下。

## 从 ERC‑721 到 ERC‑7962：要解决什么问题

听下来，7962 最核心就是：在不牺牲区块链可验证性的前提下，尽量把「身份与资产」从公开地址里抽离出来，让隐私和用户体验都更友好。​

-   传统 ERC‑721：`ownerOf(tokenId)` 直接就是一个地址，谁持有什么 NFT，一查合约就一清二楚，很适合链上透明分析，但也意味着很容易给地址画「画像」。​
    
-   现实世界的会员身份（比如星巴克会员）本质上是隐私信息：品牌不能到处广播「某某是我们会员」，但在某些场景（比如机场贵宾厅）又确实需要「按需证明」这个身份。​
    

ERC‑7962 就是试图在这个缝里做一层「隐私友好的会员 / 资产证明层」：第三方能够验证「你确实有这个资格」，但看不到你到底是哪一个链上地址。​

## 零知识与哈希：7962 的基本思路

讲者没有走那种超重型的 zk 证明体系，而是用了相对轻量的思路：用哈希 + 签名做到「等于的证明」，刚好契合「是不是会员」这种二元问题。​

-   在普通 721 中：`ownerOf(tokenId) = address`。
    
-   在 7962 里：`ownerOf(tokenId) = K_hash`，这个 `K_hash` 是由某个公钥（或者说某个「秘密」）哈希出来的值。​
    
-   交易逻辑：
    
    -   真正的持有人只需要用自己的私钥对一段「转移意图」签名。
        
    -   任意一个 relayer（中继者）拿着这个签名和相关参数发起链上交易即可，合约只检查「签名与 K\_hash 背后的公钥是否匹配」。​
        

这样带来几个效果：​

-   合约里永远只看到一堆 K\_hash，看不到具体地址，外部难以做图谱分析。
    
-   发起交易的链上地址与资产所有权解耦：谁有 gas、谁愿意帮你提交交易，都可以做 relayer。
    
-   用户侧只需「签名表达意图」，不用关心钱包地址和 gas，体验更接近 Web2 的账号体系。
    

这里用到的「零知识」其实就是在一个很窄的子问题里发挥作用：证明「某个秘密对应的哈希在集合里」，而不暴露秘密本身，特别适合「你是不是在某个会员名单里」这种场景。​

## 隐私、UTXO 风格与没有 Approve

为了提高不可关联性，7962 更像把 ERC‑20/721 做成了「UTXO 风格」：​

-   每个 K\_hash 视为一个「一次性地址」：
    
    -   比如我有 100 个 token，转你 50 个。
        
    -   这笔交易之后，合约里不会继续用原来的 K\_hash 记录剩下的 50，而是为我生成一个新的 `left K_hash`，记录剩余余额。​
        
-   因为一笔交易之后，旧的 K\_hash 就算「暴露」过一次，接下来的余额会换到新的 K\_hash 上，链上很难把整条资金流串起来。​
    

在这个模型下，Approve 机制反而会破坏隐私：​

-   Approve 意味着「提前在链上声明某个 key 可以被某个地址操作」，这会暴露本该单次使用的 K\_hash 背后的公钥关系。
    
-   所以 7962 选择直接去掉 Approve，用「签名 + relayer」模式代替授权流程。
    

这套设计本质上通过强制「密钥轮换 + 哈希持有」来尽量切断交易之间的可关联性，只留下在特定时刻、特定交易的必要信息。​

## 账户抽象、Gas 代付与 Web2 → Web3 迁移

用户体验这一块，其实是这次分享里自己最有共鸣的部分：7962 借着「所有权与发起地址解耦」，顺带天然支持了账户抽象、Relayer 和 Gas 代付。​

典型流程可以抽象成：

-   用户：只在熟悉的 App / 小程序里点「同意」、「授权」，实质上是在签一个意图。
    
-   商家（如星巴克）：
    
    -   提前部署 7962 合约，内部维护一份「会员 ↔ K\_hash」映射。
        
    -   代表用户发起链上交易，自己或和合作方承担 gas。
        
-   第三方（如机场贵宾厅）：
    
    -   只需要根据合约中的状态和用户出示的证明，验证「这个人确实是星巴克会员」，不需要知道他在链上的真实地址，也不需要拿到一整份会员名单。​
        

对 Web2 用户来说，整个过程有两个显著好处：​

-   不再强迫他们先搞明白钱包、助记词、gas 这些 Web3 概念，门槛显著降低。
    
-   同时又能享受「上链记录不可篡改」、「隐私按需暴露」这些 Web3 带来的优势。
    

从商业侧看，有动力为用户代付 gas，因为每一次「链上证明」背后都是实实在在的业务合作（比如联合权益、联合营销），代付 gas 就是运营成本的一部分。​

## 安全、监管与应用边界的一些思考

分享里也有不少关于「隐私 vs 透明」「技术中立性」的讨论，这块值得记一下，方便后续自己延伸思考。​

-   区块链天生强调透明，零知识和 7962 这类方案则是在透明账本之上，增加一层「选择性可见」：
    
    -   账本依旧公开可验证。
        
    -   但「谁在背后」不必公开给所有人，只在必要时对必要方开放。​
        
-   技术本身是中立的：
    
    -   隐私工具可以保护普通人的数据不被滥用，也可能被坏人利用。
        
    -   真正的难点在于如何在制度层面配套，例如：在某些司法场景下，是否允许「受限披露」、是否要求特定门槛下的审计能力等。​
        

从开发者视角看，一个现实问题是：如果隐私技术被滥用，会不会代码作者背锅？分享里给的态度更偏向「合理使用 + 监管与技术边界的划线」——这块本质上已经超出了协议本身的范畴，但确实会是未来隐私协议绕不开的问题。​

* * *

总的来说，自己当前的理解是：ERC‑7962 并不是要彻底用 zk 重新发明一套复杂的资产系统，而是用相对克制的技术手段（哈希 + 一次性 key + 签名 + relayer）在现有 ERC‑721 / 20 语义之上，补出一块「隐私友好且 UX 友好」的空白，尤其适合会员身份、隐私资产、企业批量发放、Gas 代付这类场景。

# 难忘今宵，误闯天家

周五分享会笔记：

今天把这一长段转录看完的感觉是：这已经不是一场普通的分享会，而是一次把「怎么学、学什么、以及我在 Web3 里要成为什么样的人」摊开来认真讨论的集体头脑风暴。​

## 学习方法与工具的升级

Notion、Figma、思维导图基本成了标配：有人用 OKR 把自己当成「一家公司」来管理，把产品、市场、技术、运营拆成不同角色，对应不同学习和输出任务；有人用 Figma 的 FigJam 当高配脑图，把课程和资料转成结构化图形，再反推自己哪里没懂。​

「导图 + AI」这一套被讲得很具体：先用导图暴露盲区，再把导出的文字喂给 AI，让它针对每个节点做补充解释、举例和答疑，而不是从零让 AI 讲一整门课；还有人用 Notabook / YouTube 转录 + AI 总结，把英文长视频变成可搜索的文字，再在上面做二次加工。​

## 输出、起号和内容方法论

输出不再被当成「写作业」，而是当成学习闭环的一部分：从日更笔记、导图复盘，到写长文、写串文、做直播，大家都在用公开输出来测试自己是不是真的讲得清楚。​

起号方面，思路非常明确：

-   标题上抓「普通人」「新人入门」「未来三年最值得学的××」这类高共鸣话题，或者直击焦虑（低精力人群怎么办、不会写代码怎么进 Web3）。​
    
-   内容上用「故事 +方法」结构：先讲自己为什么会写这篇，再讲具体拆解（是什么 / 为什么 / 怎么做），最后加一点情绪和态度，让账号有辨识度，而不只是干货列表。​
    
-   工具上不避讳用 AI 辅助：用别的 KOL 文风做参考、让 AI 帮忙扩写和改写，但有意识地在文章里加入自己的经历、意外细节和「我」的视角，尽量降低 AI 味。​
    

## 职业路径与个人定位的思考

整个转录里能看到很多不同起点：有已经在交易所、项目方的，有从传统互联网、快消、咨询、媒体转过来的，有法律专业、经济学背景，也有还在读书做研究的，同一个问题是「我到底适合在 Web3 里站在哪个位置」。​

回答也挺多元：

有人强调「归零」：有职场经验的人要愿意放下原来的光环，从基础概念重新学起；​

有人把自己定位在策略、产品、运营、法律、治理等「非纯技术」岗位，用原本的经验去填补 Web3 现在严重缺乏的专业化能力；​

也有人很直白地说，Web3 现在 80% 是「朝台班子」，真正高效的人只有一小撮，这反而意味着认真打基础、持续积累的人，未来能明显跑在平均线前面。​

## 对自己后续行动的启发

学习这块：需要的不只是「听懂」，而是搭出一张自己的 Web3 知识地图，用导图 + 笔记 + AI 把概念、协议、工具、法律与治理串成一套可以不断更新的结构，而不是一堆截图和收藏夹。​

输出这块：值得尽早开始起号和写长文，不是为了一两篇的爆量，而是用「连续的小输出」做自我校验：我能不能把今天吸收的东西，讲给一个完全不懂的朋友听懂，这决定了学习是停在 60 分的掌控感，还是往 80 分的真正掌握去走。​

职业上：比起「我要不要 all in Web3」这种大问题，现在更可执行的是三件小事——明确自己想在生态里的角色（builder / 研究 / 内容 / 法律 / 产品等），围绕这个角色规划接下来几周的学习和项目尝试，然后用公开输出和社区连接，把自己一点点「长」到那个位置上去。

一、核心主题与 AI 的关键转变

本次分享核心是 AI 与 Web3 的融合，聚焦如何让 AI 从 “嘴替”（仅聊天互动）升级为 “打工人”（能自主干活、赚钱）。AI 的关键转变体现在 2023 年到 2026 年的突破：

2023 年大语言模型（LLM）本质是 “自回归 TOKEN 预测器”，像 “话痨学霸”，能生成长文本但无实际执行能力；

2026 年 Agentic AI 突破三大限制：通过向量库拥有私有记忆、通过 API / 工具调用具备执行能力、通过 ERC8004 获得链上身份，实现 “能想、能做、能收钱”。

二、传统 AI 的痛点与 Web3 的解决方案

（一）传统 AI 的三大致命问题

无状态：对话忘上下文，无法长期交互；

无工具：仅生成文本，不能执行代码、调用 API；

无身份：不能代表组织行动、签署协议或管理资产。

此外，还面临身份不可验证（易欺诈）、行为不可审计（日志可篡改）、部署不可管控（隐私安全隐患）的信任危机，以及支付层面的身份认证、交易颗粒度、结算速度、订阅模式错配等问题。

（二）Web3 的核心解决方案

Web3 通过 “不可伪造的身份证” 和 “可编程银行账户”，精准解决 AI 的信任与支付难题，核心依赖两大协议：

ERC8004（AI 链上身份证）：基于 ERC721 标准，为每个 AI 铸造唯一 FT 身份 TOKEN，包含身份注册表（基本信息、所有者、能力声明）、声誉注册表（0-100 分评分，仅付费客户可评价，防刷分、不可篡改、时间衰减）、验证机制（TEE 硬件安全模块 + ZKML 零知识证明，确保 AI 表里如一）；

X402（原生支付协议）：嵌入 HTTP 协议，返回 402 状态码时携带支付信息，支持微支付（如 0.0001 美元）、毫秒级结算，通过支付通道、批量结算降低 gas 费，适配 AI 高频小额交易场景。

三、Spoon OS 框架核心内容

Spoon OS 是 Web3 生态的 AI 操作系统，类似 “AI agent 安卓系统”，封装底层复杂技术，让开发者专注业务逻辑：

（一）核心价值

降低开发门槛：Python 开发者 2 天内即可让 AI agent 具备 Web3 能力，无需关注私钥管理、交易签名等；

全流程支持：覆盖数据采集、模型适配、任务编排、身份管理、支付处理等全环节。

（二）四层架构

输入层：数据采集（链上数据、文档等）、预处理（清洗、分块、向量化），依赖向量数据库存储与相似度检索；

核心层：统一模型接口（适配 OpenAI、Anthropic 等）、智能体编排引擎（支持多 agent 协作 / 任务分配）、记忆管理器（区分短期 / 长期记忆）、工具调用器；

协议层：实现 ERC8004（身份、声誉、验证）和 X402（支付处理），提供钱包托管与私钥安全存储；

输出层：对接 DeFi、跨链桥等外部系统，实现资产转移、交易执行等实际操作。

（三）典型应用场景

Defi 交易智能体（套利机器人等）、数据市场（出售高质量数据）、AI 协作（处理仓库 issue/PR）等。

四、行业趋势与合规价值

（一）趋势数据

Gartner 预测 2028 年 33% 企业核心应用将采用 Agentic AI，2024 年占比不足 1%，四年增长 30 倍；

稳定币月结算量已超 Visa，API 调用量年增 30%+，RPA 市场 2025 年将达 250 亿美元，AI 驱动的机器经济需 Web3 承载高频交易。

（二）合规价值

区块链提供不可篡改的审计日志，时间戳由网络共识决定，公开透明可验证，满足欧盟 AI 法案等监管要求（高风险 AI 需可追溯、可解释），解决传统 AI 日志易篡改的合规痛点。

# web3运行原理分享会笔记

今天听完这场《Web3 和区块链技术基础》分享，终于把自己之前零散的认知串成了一张比较完整的图。​

## 关于交易和钱包

过去总觉得“交易”就是转账，这次才意识到，一笔链上交易其实是“操作 + Gas 手续费 + nonce（防重放序号）”三件事打包在一起。 操作可以是转账、调用合约、投票等等，钱包只是负责帮我把这些信息组装好、用私钥签名，然后丢给网络去处理。 实际在测试网看了一笔交易的详情，to、金额、Gas limit、nonce 一目了然，感觉之前只看余额完全是浪费区块浏览器。​

## 数字签名和安全感

数字签名这一块非常打消“我怎么证明是我发的”的焦虑：用私钥对交易做签名，任何人都能验证签名确实来自这个地址，但没人能伪造出同样的签名。 验证的时候其实就是用“原始消息 + 声称的地址 + 签名”三件套，跑一遍算法去反推出是不是同一个人签的。 结论也很现实：私钥就是一切，谁拿到私钥，谁就能发出被全网认可的“我说的话”。​

## Gas 费和经济设计

Gas 被讲成“油费”这个比喻很形象：Gas price 像油价，Gas usage 像你加了多少升，两者相乘就是最后的手续费。 操作越复杂、网络越拥堵，Gas 就越贵，这相当于用经济手段直接把垃圾交易和恶意行为的成本拉高。 这也是整个网络经济闭环的一部分，矿工 / 验证者不是在“做爱心”，而是在一个有激励、有成本的系统里工作。​

## 交易一生：从钱包到区块链

一笔交易的生命周期，第一次在脑子里跑通：钱包签名 → 通过 RPC 发到网络 → 内存池排队 → 被 builder / validator 挑选 → 打包进区块 → 共识确认 → 链上永久记录。 动画里 12 秒一个新区块、几百笔交易按 Gas 高低排队的画面，让“打包”这件事不再抽象。 每个区块之间通过 parent hash 串起来，要想篡改一个区块，就得连它后面所有区块一起改，这基本就是在跟全网算力或质押对着干。​

## POW、POS 和去中心化

共识这块，讲得很克制：POW 靠算力、POS 靠质押资金，各有优缺点，没有完美方案。 比特币用 POW，拼的是谁算得快；以太坊现在用 POS，拼的是谁愿意锁更多押金、接受作恶被罚没的风险。 本质上都是让“作恶的代价”远远大于“作恶的收益”，只是筹码不一样。​

## RPC、客户端和“隐形”的中心化

之前用钱包从没认真想过 RPC，今天才意识到：钱包其实是通过 RPC API 去连节点集群，而不是直接连区块链本体。 这意味着 RPC 服务商本身就是一个潜在的中心化风险点，拦截或拒绝转发交易，等于别人替你“掐网线”。 以太坊主网有多个 RPC 提供方、执行层和共识层也有好几个客户端，这种多样性就是在分散单点故障和软件 bug 的风险。​

## 智能合约和 “Code is law”

智能合约那部分对“Web3 = 可编程规则”这件事很加分：本质就是写死在账本里的可执行代码，调用时在虚拟机里跑一遍，把结果写回链上。 现场看了一个 MyFirst NFT 合约，每个地址只能 mint 一个，这种规则写死在代码里，任何人都能查逻辑、查历史执行记录。 “Code is law” 不是一句口号，而是把对人、机构、政府的信任，转成对代码、共识机制、治理流程的信任。​

## NFT mint、区块确认与 Finality

实际 mint NFT 的演示很直观：签名的数据比简单转账复杂得多，因为背后是对合约方法的调用。 交易一旦被打进区块，刚开始还是 unfinalized 状态，需要十几分钟等到 finality，才能说这件事“几乎不可逆”。 这种“概率上不可篡改”的设计，和传统数据库的“超级管理员说了算”形成了鲜明对比。​

## 以太坊升级、社区治理和节点分布

升级机制也挺社会学的：改协议要走 EIP 流程、社区讨论，每年 1–2 次硬分叉升级，不是谁一拍脑袋说了算。 最近的 Fusaka 升级就包含对 transaction gas limit 等参数的改动，也是社区长期博弈后的结果。 目前大概有近百万个 validator、上万台服务器分布在不同国家，这种地理和软件层面的分散，让网络具备很强韧性。​

## 写在最后：对 Web3 的再理解

这次最大的收获，是把“去中心化”从一个 buzzword 还原成一系列具体的技术和经济设计：从交易、签名、Gas，到共识、客户端、多节点部署。 Web3 不是空谈理想，而是一整套在“没人可信”的前提下，依然能协同运转的基础设施设计。 下一步想做的，就是自己跑一个节点，真正从参与者的角度去体验这套系统，而不是只站在钱包 UI 的那一层看世界。​
