    -   **工资待遇较高** ：相对于传统行业同类型和同阶段岗位，工资更高。
        
    -   **有暴富可能** ：期望通过研究找到合适方法论投资获利。
        
-   **投研相关内容**：

## **Vibe Coding + 黑客松经验分享**

-   **黑客松介绍**：
    
    -   **定义与意义**：黑客松是大家组建小型互联网公司，在几天内开发产品的活动。其意义包括获高额奖金、作为入行跳板、宣传创业项目、结识朋友、积累创业经验、获取福利、周游世界、学习新技术、验证想法和锻炼表达能力等。
        
    -   **参赛流程（以 is global 为例）**：包括报名、质押、check in、找桌子写代码、领取赞助商周边、参加 workshop 了解项目方、开幕仪式、开发过程中享受娱乐放松、上传项目视频和代码、向评委 pitch 项目（分主赛道和赞助赛道）、合影留念、宣布获胜者、参加 after party 等环节。
        
    -   **参赛角色**：除程序员外，还有产品经理、BD（Pitchman）、设计等角色。建议团队配置为 3 - 4 人，包括两个开发、一个产品和一个 Pitchman，且尽量提前组队。
        
    -   **项目思路**：有迁移重构（将其他电商商业模式迁移到新链上）、乐高拼接（组合赞助商组件做项目）、关注在地化（结合当地实际问题做项目）、关注少数群体（如宠物安全、女性安全等项目）、功能开发、硬件结合等思路。
        
    -   **坑点与技巧**：参赛不要踩点交项目，区分好 12PM 和 12AM，关注官方提交渠道，注意提交资料不等于提交成功，不要抱着创业心态追求项目完善，应完整呈现想法；提前布局，提前读题想项目；参加 is global 要多穿衣服。
        
-   **黑客松类型与建议**：
    
    -   **按主办方目的分类**：分为 demo 松、builder 松和黑客松，demo 松和 builder 松类似创业大赛，适合创意好的项目；黑客松是在短时间内做小项目。
        
    -   **按赛博属性分类**：分为线上和线下黑客松。
        
    -   **按规模分类**：分为地区黑客松、世界黑客松和休闲黑客松。地区黑客松普及各地，世界黑客松如 ethglobal 体验最佳，休闲黑客松如良心道的适合交朋友。
        
    -   **不同人群建议**：新手先参加离自己近且轻松的黑客松；创业选手参加 Demo 松、build 松可直接与主办方沟通；经费和时间充足可参加世界级黑客松，经费不足可关注机票赞助；英语不好可找队友帮忙，但程序员也需具备一定英语表达能力。
        
-   **黑客松相关用途**：
    
    -   **找投资**：并非找投资的最佳途径，投资公司有自己专门活动，黑客松多是项目方促进生态的活动。
        
    -   **应聘**：区块链行业看重项目经验，无经验可打黑客松积累经验。
        
-   **Web Coding 介绍**：
    
    -   **定义与趋势**：Web Coding 即氛围编程，通过自然语言让 AI 写代码，是未来趋势和必备技能，但使用者需懂一些代码。
        
    -   **发展历程**：经历了让 AI 修代码、AI IDE 编程、TUI 时代（Claude、code x、Gemini 等 terminal 端运行的 AI 代码辅助）。
        
    -   **工具与模型建议**：推荐使用 Ttmaker、Openai 的 f FM 进行语音生成，剪映进行视频剪辑，支持使用 Claude code 进行 AI 辅助，Nextjs 作为开发框架，以及一个区块链加前端的框架；不同模型各有所长，写代码 90% 场景可用 Claude code，依赖关系复杂可用 Codex，不建议使用 Gemini。
        
    -   **使用技巧**：与 AI 交流要清晰表达需求，定时监督 AI 效果，提供合适的上下文；合理利用 plan 模式，勤看 context，使用 Compact 命令，利用 cloud code 官方插件 featuredev 补全需求漏洞，多向 AI 请教；可针对项目特征创建 agent 并利用其提速，做好上下文管理；维护项目时写好 [cloud.md](http://cloud.md) 并定时更新。
        
    -   **UI 设计思路**：做好看 UI 可先找模板，让 AI 总结风格并生成 demo，基于 demo 修改；可让 UI AI 先生成 ask two 布局图，在 plan 阶段解决问题。
        
-   **问答环节**：
    
    -   **创业心态参赛**：创业为主，黑客松能参加就参加；有精力且符合目标时，可声明之前基础并在比赛期间做新功能。
        
    -   **宣发角色**：有适合的生态位，产品经理和 BD 可兼任，也可只做 BD；团队多元化可能影响获奖。
        
    -   **Web2 和 Web3 产品经理区别**：分享者认为区别不大，都需描述清楚需求、分任务和聚拢团队，Web2 产品经理可能还需管项目进度。
        
    -   **交流建议**：建议大家多在推特上与分享者交流，使用推特时避免搞互关、一天不要发太多评论。

周会学员分享区块链开发框架使用、能源市场应用、知识库搭建、内容创作及安全合规等多领域话题：

-   **Hardhat 框架使用**：
    
    -   **初始化与合约生成**：使用初始化 Hardhat 命令，选择版本和目录，默认安装依赖可生成合约工程骨架，有经典模板合约如 counter。
        
    -   **编译与测试**：编译合约，V3 模板有两套完整测试覆盖思路，包括 Solidity 测试和 TypeScript 测试，可验证合约功能和事件。
        
    -   **部署与验证**：用脚本部署合约，可调用函数初始化，使用 Eden Scan API V2 验证插件，一个 API key 可用于多链，配置好后部署可自动开源验证。
        
-   **区块链能源市场应用**：
    
    -   **市场背景**：光伏发电过剩，需进行峰谷套利和削峰填谷，区块链在能源市场有较大应用潜力。
        
    -   **论文架构**：提出由 AI agent、区块链网络和宏观协调者组成的三层架构，用机器学习预测发电曲线，区块链有预测证明激励机制。
        
    -   **实验结果**：构建模拟电网实验，峰值负载最高削减 29.6%，预测证明平均误差控制在 3% - 5%，电价波动显著下降，目前使用以太坊私有链。
        
-   **开发框架学习经验**：
    
    -   **挑战困境**：使用 Foundation 框架挑战以太坊 speed run challenge 时遇问题，按 AI 指令部署合约有空缺、简陋等情况，验证合约困难。
        
    -   **经验教训**：合理使用大模型，避免过分依赖，陷入中心化学习；遇到问题及时寻求帮助，避免钻牛角尖；
        
-   **AI 辅助知识库搭建**：
    
    -   **搭建原因**：解决知识收藏后难整理学习、传统笔记手动整理累且结构僵化问题。
        
    -   **搭建工具与实操**：推荐 Obsidian 软件和 Claude code、Codex 等 AI agent，实操包括暴力收集资料、让 AI 分析设计文件夹结构、建立笔记关联、数据清洗优化排版等。
        
    -   **设计原则与使用**：笔记单篇独立可阅读、链接融入句子、观点命名明确；日常可用于写作辅助、快速检索，需定期维护迭代知识库。
        
-   **AI 内容创作平衡**：
    
    -   **痛点与优势**：内容创作有理清上下文难、文笔不佳、知识二手三手等痛点，AI 可解决这些问题，但存在 AI 味重、不讲人话、降质严重等问题。
        
    -   **创作实践与思考**：结合自身经历引入文章，部分靠 AI 创作会使内容割裂，平台对 AI 文权重算法可能降低；创作应建立审美，表达个人经历，先让自己满意。
        
-   **区块链安全漏洞**：
    
    -   **访问控制漏洞**：开发中依赖调用者地址鉴权可能有疏漏，如敏感函数未加权限修饰符、可见性设置错误、依赖 origin 鉴权等，会导致攻击。
        
    -   **真实案例分析**：Parity 多签钱包两次因访问控制问题被攻击，一次被盗超 3000 万美元 ETH，一次合约被冻结；应检查函数可见性、初始化函数、避免用 origin 鉴权。
        
-   **区块链隐私问题**：
    
    -   **隐私重要性**：隐私是基本权利，对个人、开发者、机构和社会都很重要，以太坊希望成为承载文明信任的数位基石。
        
    -   **痛点与理想钱包**：存在交易透明、钱包用预设 RPC 有中心化风险、代币发现有隐私隐患等痛点；理想钱包应具备交易隐私、本地运行 private RPC、去中心化前端、网络隐私保护等特点。
        
    -   **隐私技术与挑战**：主流隐私技术有 UTXO 和账户模型，UTXO 并行性好，但都面临合规、清洁点资源、并发、状态膨胀等问题，调研隐私项目可看链上活跃度。
        
-   **加密货币合规**：
    
    -   **引入案例**：通过案例引出加密资产法律属性认定问题，涉及资产冻结、价值确认等困境，存在 “5 美元的扳手攻击” 悖论。
        
    -   **各国认定情况**：欧洲立法先行，明确加密资产司法地位；美国未统一财产定义；澳洲确认比特币属于财产；香港完善牌照制度和事后救济制度；中国大陆立场复杂，部分地区法院有保护案例。
        
    -   **结论**：加密资产法律属性认定是合规核心，全球达成共识尚需时日，未来监管需在多方面寻求动态平衡。

## **Uniswap工作原理解析会议纪要**

### Uniswap V2 核心机制

**自动做市商 (AMM) 基础**

-   Uniswap V2 是一个恒积做市商 (Constant Product Market Maker)，核心公式是 x × y = k
    
-   在价格曲线上，所有点的 x 乘以 y 都等于 k (恒定值)，但每个点的 y/x (价格) 不同
    
-   举例：池子中有 2 个以太坊和 6000 个 USDC，则 K = 12,000，价格 P = 3,000
    

**套利机制**

-   当不同交易所价格出现差异时，会产生套利机会
    
-   例如：如果币安价格从 3000 升至 3200，而 Uniswap 仍是 3000，套利者可以在 Uniswap 以 3000 买入以太坊，再到币安以 3200 卖出
    
-   套利过程中，需要提供 196.77 个 USDC，获得 0.0635 个以太坊，平均成交价格为 3098
    
-   套利行为会快速使不同交易所的价格趋于一致
    

**通用交易公式**

-   基础公式：(x + Δx) × (y - Δy) = k，其中 Δx 是投入量，Δy 是换出量
    
-   展开后可得：Δy = (y × Δx) / (x + Δx)，反之亦然
    
-   考虑手续费时，公式修正为：(x + Δx × r) × (y - Δy) = k，其中 r = 1 - 手续费率
    
-   Uniswap 的手续费率固定为 0.3% (千分之三)，因此 r = 0.997
    

**Swap 函数核心逻辑**

-   用户可以指定想要获得的输出量 (Δy)，甚至可以同时获取两种代币
    
-   合约先将代币转给用户，然后调用回调函数，在回调中用户需要支付应付的代币
    
-   交易完成后，合约会验证：调整后的资产乘积 ≥ 原资产乘积，确保 K 不变或增加
    
-   如果用户支付超过所需金额，合约会接受并相应调整 K 值
    

### 闪电贷 (Flash Loan)

-   Uniswap 支持闪电贷功能：先将币打给用户，用户完成操作后再归还
    
-   应用场景：用户可以用闪电贷去清算其他借贷协议的不良资产，提高资金利用效率
    
-   如果不满足还款条件，交易会回退，用户只损失 gas 费
    
-   闪电贷在智能合约中原子性执行，不存在卷款跑路的风险
    

### 流动性与 K 值变化

**多条价格曲线**

-   K 值代表流动性深度，并非只有一个 K 是正确的
    
-   当 LP (流动性提供者) 添加或移除流动性时，K 值会改变，对应不同的价格曲线
    
-   从原点发出的等价线与不同曲线的交点，价格都相同 (因为 x 和 y 的比例相同)
    

**流动性深度的影响**

-   添加或移除流动性会改变 K 值，但保持价格不变；只有交易才会影响价格
    
-   在更深的池子中，将价格从 3000 推至 3200 需要更多的交易量
    
-   在更浅的池子中，同样的交易量会导致更大的价格变化
    

### 价格影响 (Price Impact) vs 滑点 (Slippage)

**价格影响 (Price Impact)**

-   价格影响是指交易导致的价格偏差，来源于 x × y = k 曲线
    
-   池子越深，价格影响越小；池子越浅，价格影响越大
    
-   价格影响是可预期和可计算的，取决于池子深度
    

**滑点 (Slippage)**

-   滑点指预期价格与实际成交价格的差异，是不可预期的
    
-   产生原因：抢跑 (MEV)、交易竞争等因素
    
-   三明治攻击：攻击者先推高价格，让受害者在高价成交，然后反向拉回价格获利
    

**滑点保护**

-   用户可以设置最大滑点容忍度来保护交易
    
-   滑点设置越小，价格保护越好，但成交失败概率越高
    
-   对于波动大的代币，可能需要设置较高的滑点才能成交
    

**MEV 防护**

-   一些解决方案将交易隐藏在"暗池"中，只有小部分人知道，防止被抢跑
    
-   启用防护后，其他人在交易上链前看不到该交易
    

### 无常损失 (Impermanent Loss)

**无常损失的产生**

-   无常损失是 LP 因价格变动而产生的相对损失
    
-   示例：初始时 LP 提供 6000 USDC + 2 ETH (价值 12,000)，价格从 3000 变为 3200
    
-   做 LP 后：约 6196 USDC + 1.9365 ETH，总价值约 12,396
    
-   不做 LP：6000 USDC + 2 ETH (按 3200 计)，总价值 12,400
    
-   LP 收益少于单纯持有，这就是无常损失
    

**无常损失的特点**

-   本质上是作为 Maker 被逆向选择的结果
    
-   如果价格回到初始点，无常损失会消失 (暂时性损失)
    
-   目前认为无常损失无法避免，主要通过增加手续费作为补偿
    
-   对于单向波动的资产，无常损失会特别严重，尤其在 V3/V4 的集中流动性中
    
-   对于散户 LP，如果不做主动流动性管理，容易出现做 LP 收益不如直接持有的情况
    

### 时间加权平均价格 (TWAP)

**TWAP 原理**

-   V2 提供链上价格预言机功能，计算时间加权平均价格
    
-   记录累积价格而非瞬时价格：累积价格 = 交易前价格 × 时间间隔
    
-   累积价格相当于价格-时间曲线下的面积
    

**TWAP 计算方法**

-   在 T0 时刻读取累积价格 V0，在 T1 时刻读取累积价格 V1
    
-   平均价格 = (V1 - V0) / (T1 - T0)
    
-   这是一个巧妙的链上平均价格计算方法，但需要提前在 T0 读取一次值
    
-   无法只获取当下的平均价格，必须是时间加权的

第一层面：形式合规

目标：在文本、合同、协议、白皮书等法律文件上做到形式合法合规

方式：

使用符合监管要求的术语和表述

避免使用敏感词汇（如“融资”“投资回报”“保本”等）

明确风险提示和免责条款

遵循国际标准（如 FATF、MiCA 等）的披露要求

局限：仅能降低表面风险，无法完全规避实际业务中的法律风险

1\. 聘请专业律师起草法律文件

2\. 白皮书/官网避免承诺性表述

3\. 明确用户协议和风险提示

4\. 遵循国际合规标准（如 KYC/AML）

第二层面：深度合规（实际运营层面）

目标：在业务模式、资金流向、用户交互等实际运营中规避法律风险

关键点：

避免非法募资：绝不涉及 ICO/IEO/IDO 等公开融资行为

隔离中国大陆用户：不向境内居民提供交易、兑换、支付等服务

防范刑事风险：远离赌博、传销、洗钱、非法经营等行为模式

建立真实业务闭环：不以“纯金融”为目的，具备真实技术或服务支撑

挑战：

需持续跟踪监管动态和司法判例

需建立内部合规体系和风险控制流程

需考虑跨境法律冲突和属地管辖问题

“深度合规不是 Web3 特有”

本质是金融合规：Web3 的合规问题本质上仍是金融活动合规问题，与传统金融行业的非法集资、洗钱、传销等风险逻辑相似

合规是系统工程：真正的安全合规需要从业务设计、技术实现、运营管理、法律架构等多个维度协同构建，而非仅靠一纸合同

“绝对安全”是理想状态：在现行监管环境下，Web3 项目难以实现“绝对安全”，只能在风险控制与业务发展之间寻求平衡

1.业务设计避开“金融属性”红线

2\. 技术架构支持合规审计与追踪

3\. 运营中隔离中国大陆市场

4\. 建立内部合规培训与监督机制

第三层面：持续风控

1\. 定期进行合规自查与风险评估

2\. 关注监管政策与司法判例动态

3\. 准备应急预案（如监管调查、用户投诉）

-   Web3运行原理（Bruce）
    
    1.  钱包与私钥、个人主权
        
        **钱包与私钥**：私钥是一个随机生成的字符串，通过它可以生成公钥，再截取部分生成钱包地址。助记词可以派生出无限个私钥。私钥一旦泄露，资产就会被盗，且无法重置。
        
    2.  交易签名机制
        
        **交易与签名**：交易包括要做的事情、手续费(gas费)和防重放序号。钱包帮助组装交易内容，用私钥签名，然后广播到区块链网络。数字签名可以让任何人验证信息是否由特定私钥签署，但无法伪造。
        
    3.  区块链网络运行方式
        
        **Gas费机制**：Gas fee相当于手续费，Gas Price是当时的"油价"，Gas Usage是完成交易需要消耗的"汽油量"。操作越复杂，消耗越多。网络拥堵时，Gas Price会上升。
        
        **区块链网络运行**：交易从钱包签名后，通过RPC节点发送到网络，进入交易池排队，由验证者(validator)挑选打包成区块，通过共识机制加入区块链。以太坊每12秒出一个块，需要12-13分钟才能最终确认(finalized)。
        
    4.  智能合约的本质
        
        **智能合约**：是存储在区块链上的可执行代码，部署后难以修改。任何人都可以查看合约代码和执行历史，实现"代码即法律"，减少中间商和信任成本。
        
    5.  区块链协议升级机制
        
    6.  Web3的关键特性，包括：
        
        ```
        去中心化特性（从钱包创建到网络运行）
        
        无许可性（任何人都可以读写操作网络）
        
        抗审查性（节点全球分布，钱包可替换）
        
        开放开源（客户端代码、交易记录、智能合约都公开）
        
        隐私性
        ```
