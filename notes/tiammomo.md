### Uniswap V2：自动化做市商（AMM）的基础

-   **恒定乘积做市模型**：核心公式为 `x * y = k`，通过两种代币储备量（`x`和 `y`）的比值决定实时价格 `p = y / x`。任何交易必须维持乘积 `k`不变，从而自动调整价格。
    
-   **套利与价格发现**：当池内价格偏离外部市场时，套利者通过买卖操作赚取差价，同时推动池内价格回归至市场均衡水平。
    
-   **无常损失**：指流动性提供者因代币价格变动，其资产价值相较于单纯持有资产所面临的价值差额。这是由AMM机制本身决定的机会成本，价格波动越大，无常损失可能越显著。
    
-   **闪电贷**：用户可在单笔交易中无抵押借出资产，但必须在交易结束前归还本金及费用，否则交易回滚。这为套利、清算等高级策略提供了便利。
    
-   **时间加权平均价格（TWAP）预言机**：通过累加每个区块的价格数据并计算平均值，为链上应用提供抗操纵的可靠价格源。
    

### Uniswap V3：集中流动性与资本效率革新

-   **集中流动性**：流动性提供者可将资金集中在自定义的特定价格区间内，而非均匀分布在整个价格曲线，从而大幅提升资金利用效率。
    
-   **Tick 系统**：将连续的价格范围划分为离散的刻度，每个 Tick 对应一个价格点。流动性按 Tick 粒度进行分配和管理，实现了精细化的价格区间控制。
    
-   **手续费分级与分配**：提供多种手续费等级（如0.05%、0.3%、1%），以适应不同波动性的交易对。手续费根据流动性提供的价格区间和贡献度进行分配。
    
-   **流动性（L）与价格（√P）**：引入流动性 `L = √(x * y)`和当前价格的平方根 `√P`作为核心计算变量，优化了合约的运算效率。
    

### Uniswap V4：可定制化与扩展性提升

-   **钩子机制**：允许开发者在流动性池生命周期的关键节点（如交易前后、流动性调整时）注入自定义逻辑，从而实现动态手续费、链上限价单、TWAMM（时间加权自动做市商）等高级功能。
    
-   **单例合约与闪电记账**：所有流动性池共存于一个单例合约中，结合闪电记账（仅在交易最终结算时进行代币转账），显著降低了创建新池子和复杂交易的Gas成本。
    

### Uniswap 生态系统支持

Uniswap基金会及社区通过资助计划、黑客松、审计补贴等方式支持项目孵化，同时提供广泛的教育资源和全球社区活动，持续推动DeFi生态的创新与发展。

⾸先解决第⼀个问题：Reactive合约是什么？

简单来说，Reactive能够做到两件事：监督某个合约的event是否发⽣，触发提前指定的callback合约的执⾏，⽽且，这两个合约可以是部署在不同链上的

那我们为什么需要Reactive合约呢？

⼤家应该还记得，常规理解中，合约只能由⽤⼾调⽤后才能⽣效，智能合约⽆法在没有外部指令的情况下发起操作，要想实现这种⽆需⼿⼯就能调⽤合约的效果，会不可避免地引⼊中⼼化的控制⼿段，⽽Reactive合约的出现可以很好地解决这⼀限制，它能够从多条链获取信息，去中⼼化地跨平台触发合约

简单讨论一下 Solana

Solana 在 Web3 领域扮演着“高性能基础设施”的核心角色，其核心价值在于通过一系列技术创新，试图从根本上解决区块链的“不可能三角”难题——即在保持去中心化和安全性的同时，实现极高的吞吐量与极低的交易成本。其技术基石是独特的 历史证明（PoH）共识机制，它像一个全局的、不可篡改的加密时钟，预先为交易生成时间戳序列，从而使网络中的成千上万个节点无需花费大量时间相互协调交易顺序，可以专注于并行验证和出块。这项创新，结合其 并行处理引擎 Sealevel，使得 Solana 能够同时处理数万笔互不依赖的交易，实现了理论峰值超过 6.5 万 TPS 的吞吐能力和约 400 毫秒的出块速度，而单笔交易的平均成本低至约 0.00025 美元。

这种技术特性直接塑造了其独特的生态系统。极低的 gas 费用和高速度，使得 Solana 成为对用户体验和交互频率要求极高的应用的理想平台。其生态因此蓬勃发展，尤其在 高频 DeFi 交易、NFT 铸造与交易、以及 GameFi 和 DePIN（去中心化物理基础设施网络） 等领域表现突出。例如，其链上的去中心化交易所能够提供媲美中心化交易所的即时交易体验，而庞大的 NFT 和 meme 币社区也依赖其低成本进行频繁的铸造和转移。然而，这种高性能架构也带来了挑战：网络对验证节点硬件要求较高，引发了对去中心化程度的讨论；同时，在交易极端峰值时期，网络曾经历过拥堵和中断。为此，Solana 核心开发团队正持续通过引入 QUIC 网络协议、费用市场机制以及全新的 Firedancer 验证者客户端 等方案来增强网络的稳健性。

今天简述下 EVM 吧

**Ethereum Virtual Machine** 是以太坊的执行引擎，负责运行智能合约。它是一个基于栈的虚拟机，所有合约都在这个隔离环境中执行。

你可以把整个以太坊网络看作一台巨大的、分布式的**状态机**。在任一时刻，这台状态机的“状态”都包含了所有账户的余额、智能合约的代码和数据等全局信息。EVM 的核心作用，就是根据接收到的新交易（Transaction），按照确定的规则，计算出下一个新区块的**新状态**。

这确保了在全球成千上万的节点上，只要从同一个状态出发并处理相同的交易，EVM 总能产生**完全一致的结果**。这种确定性是区块链达成共识的基石。

记录个基本知识点：Rust 的核心在于其**所有权系统**，这是一套在编译时就能确保内存安全的独特规则。它通过规定每个值有且只有一个所有者、通过借用（引用）机制实现灵活的数据访问（同时严格限制可变引用的使用），并结合生命周期概念来保证所有引用始终有效，从而无需垃圾回收机制即可避免常见的内存错误（如空指针、数据竞争）。这套机制使得 Rust 能够在提供与 C/C++ 相媲美的性能和高层抽象的同时，保障代码的安全性与并发可靠性。

简单学习了一下 **ERC-7962**

**ERC-7962** 是以太坊改进提案（Ethereum Request for Comments），主要定义了 **跨层消息传递（Cross-Layer Message Passing）** 标准。

以太坊 L2 扩容方案（如 Rollups）日益普及，但 L1 和 L2 之间缺乏标准化的消息传递机制。

-   **消息格式标准化**：定义 L1 ↔ L2 之间消息的统一格式
    
-   **状态验证机制**：标准化跨层状态验证流程
    
-   **可靠性保证**：确保消息传递的最终性和安全性
    

-   **跨层代币转账**：如 ETH、ERC-20 从 L1 移到 L2 或反向
    
-   **跨层调用**：在 L1 触发 L2 上的合约执行
    
-   **跨层治理**：DAO 投票结果跨层生效
    

| 标准 | 用途 |
| --- | --- |
| ERC-721 | NFT 代币标准 |
| ERC-20 | 同质化代币标准 |
| ERC-165 | 接口查询标准 |
| ERC-7962 | 跨层消息传递标准 |

**ERC-7962** 为以太坊多链生态（主网 + 多个 L2）提供互操作性基础，让不同 L2 之间、以及 L1 与 L2 之间能够安全通信。
