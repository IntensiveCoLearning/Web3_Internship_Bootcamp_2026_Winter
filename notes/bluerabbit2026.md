黑客松强调快速迭代 + 完整闭环展示，时间极短（8-48小时），核心不是“谁做得最全”，而是“交付一个能演示的闭环产品”。

1.  选题阶段（起跑线决定成败）  
    最大坑：项目看起来高大上，但商业逻辑不闭环 / 场景说不清。  
    常见错误：做“应用级大而全产品”，没想清“谁用？为什么非你不可？解决什么具体痛点？”  
    评委第一反应往往：“东西很复杂，但你自己都没想清楚。”  
    正确路径：聚焦一个明确、单点、可演示的需求，避免一开始就“输在起跑线”。
    
2.  MVP & 执行  
    时间短 → 核心目标：交付可演示闭环（允许逻辑有瑕疵，但不允许“没产品”）。  
    允许：mock数据、砍用户系统/后台/权限、前端1页、最核心功能。  
    不允许：无Demo、无法展示。
    
3.  团队结构（配置决定打法）
    
    -   技术型团队（技术 > 产品）：偏技术创新、SDK/链/协议集成，评委可能真看代码（尤其线下），重点突出技术突破、性能/成本/安全性改进。
        
    -   非技术/运营/产品型团队：80%时间想清逻辑，20%用AI/模板快速生成Demo，路演重点讲痛点 + 场景 + 意义。
        
4.  Demo & 视频（比代码更重要）  
    评委大概率不跑代码，但一定会看Demo视频。  
    强烈建议：提前录好Demo，剪掉pending/卡顿/错误；即使现场翻车，视频也能救场。
    
5.  路演 & PPT（5分钟抓住评委）  
    目标不是“讲完一切”，而是让评委快速get“你在做什么 & 为什么值得加分”。  
    最安全结构（新手版）：
    
    -   30s：问题
        
    -   60s：方案
        
    -   90s：Demo
        
    -   30s：技术亮点
        
    -   30s：愿景
        
6.  README 必写内容
    
    1.  项目一句话介绍
        
    2.  解决什么问题
        
    3.  Demo链接
        
    4.  用了哪些sponsor技术
        
    5.  合约/Repo地址（哪怕简单）

1.  Token（ERC-721 NFT）
    
    -   最基础但最重要的起点
        
    -   直接在 L2 主网上 mint，不要怕花钱
        
    -   学会基础：mint、ownerOf、tokenURI 等
        
2.  Crowdfunding（众筹合约）
    
    -   核心：状态机（State Machine）
        
    -   典型状态：Collecting → Success / Failure
        
    -   关键设计点：需要有人主动“poke”（调用函数）来推进状态 → 必须设计激励机制让别人愿意调用
        
    -   必须做对抗性设计（adversarial thinking）：防止别人偷钱、作恶、griefing
        
    -   经典问题：如果没人调用怎么办？→ 需要经济激励
        
3.  DEX / AMM（自动做市商）
    
    -   核心公式：x \* y = k（恒定乘积）
        
    -   流动性池（LP）、手续费（fee to LP）
        
    -   approve + transferFrom 模式（为什么需要两步）
        
    -   理解滑点（slippage）、无常损失（IL）产生原因
        
4.  随机数与链上游戏
    
    -   千万不要用 blockhash / block.timestamp 做随机数（矿工可操控）
        
    -   推荐方案：commit-reveal 机制（两阶段提交）
        
    -   或者用 Chainlink VRF（但有成本）
        
5.  借贷协议基础
    
    -   超额抵押（over-collateralized）
        
    -   清算（liquidation）机制：价格下跌时谁来触发清算？
        
    -   同样需要激励：清算者获得 bonus（清算奖励）
        
    -   经典设计：健康因子（health factor）< 1 时可被清算

1\. EVM 基础EVM 是以太坊智能合约的栈式虚拟机，所有合约以字节码在 EVM 中执行。理解 EVM 是掌握 Solidity 编写、优化与安全的前提。四大存储区：

-   Stack：栈式计算模型（操作数入栈 → 计算出栈）
    
-   Storage：永久链上状态，gas 最贵（初始化 ≈20k gas，修改 ≈5k gas）
    
-   Memory：临时运行内存，一次调用结束即销毁
    
-   Calldata：只读调用数据区，用于函数参数
    

优化核心：尽量少读写 storage（少存、少改、避免不必要访问）。2. 开发环境推荐 Remix（在线编辑器）：内置私有链、编译、部署、debug，可直观查看 memory/storage 变化。  
Solidity 版本敏感，教学以 0.8.20 为基准（^0.8.20 支持 0.8.20 及以上）。阅读源码时务必确认版本。3. 数据类型

-   值类型（gas 较低）：bool、int/uint（建议用最小位宽，如 uint8）、address（带类型校验）、bytes32
    
-   引用类型（gas 较高，开辟新空间）：string、动态数组、bytes、mapping（常用于余额/授权账本）
    

4\. ERC20 核心设计链上存储：name, symbol, decimals, totalSupply, owner  
两大账本：

-   mapping(address => uint256) balances
    
-   mapping(address => mapping(address => uint256)) allowance
    

Constructor：部署时初始化（名称、符号、精度、owner = msg.sender、可 mint 初始代币）函数可见性：external / public / internal / private  
状态修饰符：view（只读）、pure（不读不改）核心方法：

-   totalSupply() / balanceOf() → view
    
-   transfer(to, amount) → 校验余额、非零地址 → 更新 balances → emit Transfer
    
-   approve(spender, amount) → 更新 allowance → emit Approval
    

事件：Transfer / Approval / Mint，用于链下监听（DEX 依赖事件而非频繁读 storage）Mint：通常设为 internal + external 权限函数封装5. 安全 & 优化要点

-   错误处理：require（带信息）、revert（省 gas）、自定义 error、assert（少用）
    
-   unchecked：0.8+ 默认防溢出，确定安全时用 {} 节省 gas（初学慎用）
    
-   modifier：复用权限校验（如 onlyOwner、非重入）
    
-   重入攻击：EVM 单线程，但存在重入风险 → 用 nonReentrant modifier 防护
    
-   常见骗局：modifier 限制转出 / 抢跑 Bot（监听事件 + 高 gas 优先转走）

1\. 传统LLM的三大致命限制

-   无状态：每次对话从零开始，上下文窗口有限（早期32K/64K），像“金鱼记忆”。
    
-   无工具：只能生成文本，无法调用API、执行代码，像“被绑住的天才”。
    
-   无身份：无法代表任何人签署协议、管理资产，像“黑户”。
    

2\. 2026 Agentic AI的突破  
三大组件让AI“有脑、有记忆、有手脚”：

-   大模型（LLM） → 推理引擎，理解意图、制定计划、输出指令（GPT系列、Claude、DeepSeek等）。
    
-   记忆（Vector DB + RAG） → 通过embedding → 向量数据库（Pinecone等）实现长期私有记忆，从“金鱼”进化到“大象”。
    
-   工具调用（Function Calling / MCP） → 支持ReAct循环（思考→行动→观察），让AI真正“说到做到”。
    

3\. Web3解决的三大痛点  
AI能力强但仍是“黑户”：身份不可验证、行为不可审计、支付颗粒度/速度/门槛不匹配（验证码、人脸、信用卡手续费高、结算慢）。

1\. 安全第一原则

私钥绝对不能截图、完整复制、全文发送或存储在易被监控的地方。剪贴板易被恶意软件窃取，建议分段复制+手动输入。涉及真金白银时，安全意识必须极度严格（曾有学员因授权泄露被黑客快速转走奖金）。

2\. 个人主权与无许可特性

任何人可随机生成私钥 → 立即拥有一个无需KYC、无需审批的钱包地址与资产控制权。这就是“个人主权”的密码学起点。与传统银行（需身份证、柜台、可能被拒、跨境受限）形成鲜明对比。

3\. 助记词 vs 私钥  助记词（seed phrase） → 一对多，可派生无数个私钥  

每个私钥 → 一对一，对应唯一地址

派生规则由确定性加密算法保证（BIP-39 / BIP-32等）。

4\. 交易（Transaction / TX）核心组成  要做的事（转账、调用合约等）  

Gas费（网络资源付费 + 激励机制）  

Nonce（防重放序号）

钱包（如MetaMask、Rabbit）负责：组装交易 → 用私钥签名 → 广播至网络。

5\. 数字签名本质

用私钥签名交易信息 → 任何人用对应公钥（地址）可验证签名有效性，但无法伪造签名（除非拥有私钥）。

核心安全：不暴露私钥的前提下证明“我就是地址拥有者”。6. Gas机制类比  Gas Price ≈ 油价  

Gas Used ≈ 路程耗油量  

总费用 = Gas Price × Gas Used

复杂操作 / 网络拥堵 → Gas价格飙升。防止垃圾交易、激励验证者。

7\. 区块确认与最终性

交易广播 → 进入内存池 → 验证者挑选（优先高Gas费） → 打包进区块 → 共识确认。

以太坊PoS下：  出块 ≈ 12秒  

需等待约12–13分钟（多区块确认）达到“Finalized”状态，才认为几乎不可篡改。

8\. 去中心化程度关键点  节点地理分散（美国、德国、中国等）  

客户端多样性（Geth、Nethermind、Besu、Erigon等）  

RPC仍存一定中心化风险（可自选或研究去中心化RPC方案）

一句话总结运行原理

私钥签名证明身份 → 共识网络确保账本可信 → 智能合约自动执行规则。
