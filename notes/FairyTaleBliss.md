## 继续认识以太坊

2.8 节点同步的目标与意义

1.  节点要同步的内容主要包括两部分： ① 区块链数据（每个区块及其头信息） ② 状态数据（每个账户余额、合约存储、代码等）
    
2.  节点同步的三种方式
    
    ① FullSync（全同步）：节点从创世区块开始，下载每个区块并重新执行其中的所有交易，验证其状态变化。优点是完全的“不要信任，要验证”，缺点是非常慢，毕竟量很大。
    
    ② SnapSync（快照同步）：节点首先下载最新区块头链（header chain），然后请求一个可信状态快照（snapshot）——主要是 Merkletrie（状态树-默克尔树）的根节点和部分分支节点。随后通过并发下载账户、存储、代码等状态数据，填充本地数据库。这个速度快，应用上普遍一些。
    
    ③ LightSync（轻节点同步）：不保存完整状态，也不执行所有交易，只下载区块头和部分区块体，用于轻量级验证。适合钱包或浏览器插件，不推荐用于验证或运行RPC服务。
    
3.  同步时使用哈希验证，无需担心伪造数据：状态同步时，节点会对所有区块头和状态数据进行哈希验证。由于以太坊的状态根是**默克尔树根（只能一路哈希，逆向算力上不可行）**，客户端能通过哈希路径验证每一个账户或存储项的真实性，避免中间人伪造状态数据。
    
4.  小项目，有条件SnapSync（如果客户端支持）或者 Light Client（如果资源非常有限）。高安全性 /不信任第三方 /做审核 /历史数据相关功能 → 用Full Sync 或归档节点。
    
5.  注意**信任与可用性的区别（不要信任，要验证，又来了）**。如果只是偶尔需要历史状态，而且能接受从外部服务查询 → 轻节点＋外部 provider 是可行
    
6.  复习区块头、区块体：区块头（Block Header）：包含区块的元数据，用于识别、验证与链接区块。区块体（Block Body）：包含实际交易数据和其他可选信息，是区块中“做事情”的部分。

## 继续认识以太坊：

第二章

2.4 开发者或机构运行全节点的必要性

1\. 什么是RPC？Remote Procedure Call

在区块链生态中，RPC 通常指区块链节点提供的一种接口，供应用程序

（比如钱包、dApp、区块浏览器等）远程调用。通过这种接口，应用可以：

• 查询区块、交易、账户余额、智能合约状态等数据；

• 提交新的事务到网络；

• 调用智能合约的方法；

• 监测交易确认状态等。

节点（RPC 节点）或 RPC 服务商就提供这些接口和 endpoint。应用端只

需要知道这个 endpoint 地址，以及支持的 RPC 方法，就能与区块链交互，而不需要每个应用都自己运行完整节点。

但是，这种模式也容易被服务商控制安全与用户数据。自己跑全节点：不求人 + 更安全 + 更隐私 + 更去中心化

2\. 开发者自己跑全节点的优势与必要性：

1\. 不泄露用户隐私。通常RPC提供商会收集 IP + 钱包地址等元数据。这很容易经过数据分析就将 IP 与地址关联起来。

2\. 运行自己的节点是获得抗审查性、可用性和主权的重要一步。不必担心限流、封锁、无法被广播。

3\. **不要信任，要验证**（这一条，我一位做区块链研究的朋友也与我反复说）。当我们自己运行全节点，所有区块和交易都会在本地被完整验证（共识 + 执行），因此\*\*不必相信任何公共 RPC 提供的结果是对的\*\*，可以自己算。尤其重要的场景：金融机构 / 交易所做入账确认、做清结算和风控

4\. 以太坊的客户端多样性以及去中心化靠全节点保持。

5\. 自托管工具与服务。

6\. 升级与分叉的选择权。

7\. 配置自由（客户端、参数、合规）

2.5 归档节点在数据查询中的优势

归档节点（Archive Node）= 全节点 + 所有历史状态快照

优势有：

1\. 查询快，因为在本地。

2\. 支持按区块高度查询的高级 RPC。主要是用于历史时间点重放调用、合约升级前后行为对比、回溯某次攻击发生前后的状态。

3\. 区块浏览器、数据服务商给开发者提供历史分析服务。包括量化研究、MEV 分析、RWA 资产监控，都需要长时间窗口的细粒度链上历史；合规与审计也需要复现链上历史数据；开发者调试也需要。

代价是：存储需求很大、首次同步需要很久、对硬件有高要求。

2.6 The Merge 之后执行与共识客户端的区别

1\. 执行客户端简记：EVM + 状态 + RPC。“世界计算机”的 CPU + 内存 + 本地数据库 + RPC 网关。其主要作用：

处理交易（转账、部署合约、调用合约函数、基本校验）

执行 EVM 操作（逐条执行交易、计算 gas 消耗、抛出异常、更新 storage）

维护状态与数据库（维护当前链的世界状态：账户余额、合约代码、storage、状态树等，并写入本地数据库）

提供 JSON-RPC API（包含eth\_\*\*、\*net\_\*、web3\_\* 等标准 RPC 方法）

执行P2P网络、配合集成 Engine API（接收共识客户端通过 Engine API 发来的“待执行区块骨架”）

2\. 共识客户端简记：运行 PoS 共识，提议 / 验证新区块、追踪链头与最终性（finality）、管理验证者集与奖励 / 惩罚、共识层 P2P 网络、Beacon API / 共识 RPC

3\. 典型“出块 / 同步”流程大致是：

1\. 共识客户端收到新区块 / 或被选中提议新区块

2\. 执行客户端执行交易，返回执行结果

3\. 共识客户端把执行结果“装进” Beacon 区块

4\. 其他节点的共识客户端接收区块 → 交给本地执行客户端复验

## x402了解：

**是什么？**

x402 是由美国第一合规交易所 Coinbase 在 2025 年 5 月推出的开源支付协议。这个协议基于 HTTP 协议（网页运行的底层规则），利用了“402 Payment Required”（需要支付）状态码，因此叫 x402。

HTTP 402 状态码：遇到付费内容，网页返回 402 状态，并一并返回费用和地址。

**解决什么问题？**

Agent自主支付、微支付、简化流程（开发者易集成）、全球通用、机器人支付

**怎么实现？**

x402 的核心是把支付嵌入 HTTP 协议，用区块链（主要是 Coinbase 的 Base 链）做结算。

**为什么流行起来？**

1.  Cloudflare 和 Coinbase 成立“x402 Foundation”，Cloudflare 的服务器开始支持 x402 交易，等于全球网站都能用。
    
2.  Agent有支付的客观需求在。
    
3.  Meme. 你支付，你铸造，我打你地址，DEX任你飞，就这么就炒起来了。
    

**x402已经有生态：**

1.  x402scan：像 Etherscan 的区块链浏览器，专门记录 x402 的支付和资源。核心功能包括查询交易（支付记录、USDC 流向、热门 API）、试用（1USDC即可体验，还能Mint Meme Tokens）、生态概览（例如热门Meme）
    
2.  [https://github.com/coinbase/x402](https://github.com/coinbase/x402) 直接看代码，几行代码即可集成。
    
3.  机器人支付。专注于去中心化机器人协调的 OpenMind与稳定币发行商Circle合作将x402协议与Circle Gateway（用户的多链统一余额账户） 集成，实现高效、安全、可跨链的支付。基本流程：
    
    （1）买家向 Gateway 合约存钱，生成 EIP-3009 签名（授权转账）。
    
    （2）买家通过 x402 向卖家支付后，卖家向 Gateway 提交签名。
    
    （3）Gateway 链下验证：检查买家余额，锁定资金，立即通知卖家到账。
    
    （4）Gateway 批量积累签名，定期链上提交（批量在链上结算可节省费用）。
    
    整个过程类似支付宝的余额支付+批量对账：扫码付钱瞬间扣款（链下），但银行后台每小时批量记账（链上）。

## 继续认识以太坊

021 第二章 2.2 节点间的连接与通信方式

节点发现 → 建立安全连接 → P2P 网络 Gossip 传播

1.  节点发现：Discovery 协议，跑在UDP上。先确认对方在线（这边发Ping，引导节点回Pong），再要通讯录（这边发 FindNode 请求，引导节点回 Neighbors 或 Nodes）。
    
    请求到新邻居后，再迭代，直到填满DHT，发现模式采用Kademlia，用节点ID的异或差异值来度量距离。
    
2.  建立安全连接：TCP + RLPx / devp2p. TCP毕竟比UDP可靠性高，带确认、可重传。然后TCP上的 devp2p 协议栈，其底层加密与多路复用由 RLPx 负责。RLPx 先通过握手交换密钥，建立加密会话，再在加密 TCP 通道上，复用多种 子协议。
    
    为什么多种子协议？区块同步、交易广播、状态请求等多种子协议。
    
3.  Gossip （广播式传播）把新交易与新块传遍全网。执行层与共识层分别通过自己的P2P网络Gossip.
    
    此外还有一种 请求-响应 的传播方式。这种适合掉线一段时间，需要补齐历史数据的场景。
    

2.3 全节点、轻节点、归档节点的区别

全节点 = 当前状态 + 全部区块（但老状态会被修剪）； 归档节点 = 全节点 + 从创世到现在的“所有历史状态”，禁用修剪； 轻节点 = 只存区块头，需要时向全节点要具体数据，不能做验证者。

其中全节点适合有能力部署的个人或机构，归档节点适合大机构、数据分析机构，轻节点适合轻量设备。

第二章讲以太坊网络结构与节点类型。主要谈的The Merge之后。

1.  节点：任何一台运行以太坊客户端软件并连接到其他节点的电脑， 就是一个以太坊节点。
    
    客户端（去中心化、非常多样）：对以太坊协议的具体实现，会按协议规则验证数据、同步区块和状态，帮助网络保持安全。The Merge之后分为：执行客户端、共识客户端，验证者客户端算是个共识客户端的插件，主要是参与出块、赚质押收益。
    
2.  执行客户端：主要负责链上的“业务逻辑和状态” • 监听网络上广播来的交易与区块 • 在 EVM（以太坊虚拟机） 中执行交易和智能合约 • 维护当前链的状态数据库（账户余额、合约存储等） • 提供 JSON-RPC 接口（eth\_\*）， 给 dApp / 钱包 / 脚本调用 • 通过自己的 P2P 网络转发交易和区块头 / 区块体
    
3.  共识客户端：主要负责权益证明 PoS 共识
    
    • 维护信标链（Beacon Chain）视 图 和 fork choice 规 则（ LMD-GHOST等） • 在每个 slot 里选择谁来提议区块，并对区块进行投票（attestation） • 跟踪验证者集合、质押余额、惩罚（slashing）等共识状态 • 把来自执行客户端的结果当作“材料”，决定哪一个区块被接受到链上
    
4.  验证者客户端：参与出块、赚质押收益。负责：管理验证者密钥、在轮到自己时提议新区块、在每个 slot 为看到的区块做 attestation（投票）
    
5.  Engine-API：The Merge之后，客户端解耦为两个独立进程，于是需要一个标准化的通信接口，就是这玩意儿。共识客户端会通过 Engine API 做几件关键的事情： • 把要提议的区块骨架（谁打包、哪些交易）发给执行客户端，请它：执行交易、计算状态变更；生成执行负载和新状态根 • 向执行客户端询问“这个 payload 合法吗？”、“这条链头能不能继续”等信息 • 在新的区块被其他节点广播时，让执行客户端验证其中的执行结果

## Uniswap V2 AMM VS Uniswap V3 AMM 机制对比与个人思考

前言：在自己做交易时，尤其是高频的Meme平台，经常会看到AMM，当时没有多想，光想着K线高低。在前天的学习中了解到了这是DEX里常见的应用，算是补上了知识点。后因一些思维断点问AI，得到了一些扩展知识。由于个人的思维模式原因，从断点中得到的知识一般都特别香，所以先输出一下。即便课内内容还差很多，但今天也在进步。

Uniswap V2 AMM，简单说遵循的就是X\*Y=K，即一特定币对池的两币数值乘积。而其最大的问题在于，由于币价呈现一个严格的反比关系，因此LP资金被均匀分布在(0, +∞)区间上。举个例子，一个池子10ETH 30000U，那么K是300000，这10个ETH，不是每一个都是可以流动起来的。何以见得？假如其中9个ETH被买走了，为了满足规则，池子的另一边是300000u，那最后这1个ETH，相当于在300000U/个以上的单价时，才有机会流出池子，参与交易，赚取手续费，这就是这个AMM模型在提供流动性的时候，资金使用低效的原因——有一些价格区间，是几乎永远不可能触及的，那么对应的Token资产，就会无法流动。

那么，Uniswap V3做了什么？

(X+X\_virtual)\*(Y+Y\_virtual)=L^2

一点点拆解，首先，为什么是L^2而不用K？都是保持一个相对的常量值，为什么要用开根号后的量？

直接上例子，池子里有 10 ETH 和 25,000 USDT，我希望池子的滑点减半，这意味着更充足的两种Token带来更充足的交易深度。没错，就是我再放10 ETH 和 25,000 USDT，池子变成20ETH和50000USDT，池子的滑点就减半了。

这个场景，K为原先的4倍，L为原先的2倍。这就是关键！L与滑点呈反比，与池子的深度呈线性的正比关系，这样子，结算LP提供者们的手续费收益也很轻松，只要线性地做除法：特定LP提供的L/总L. L这个单位使得一切的结算都变得容易。

然后，X\_virtual Y\_virtual何用只有？这就是V3的虚拟LP机制。这部分LP并不存在，是算法为你提供的”虚空配资“。这解决了资金被均匀分布在(0, +∞)区间上的问题，即便将LP资金设定为特定的区间提供交易流动性（区间外则会自动把一种资产清空，全部换成另一种资产，例如设定ETH 2800-3200的价格区间，则低于2800全部换成ETH，高于3200全部换成USDT），超出区间也不会有不满足乘积等于L^2的情况。在算法上，这降低了交易者的滑点，但是对于LP来说，这极大增强了策略博弈的烈度。这就要谈到X\_virtual Y\_virtual与真实LP的比例关系：

事实上，不是一个固定的比例关系，而是取决于LP提供者选择的区间大小，比例与区间宽度大致呈现反比关系。即，窄区间会有极大的虚拟LP杠杆，甚至达到百倍，毕竟涨跌超出区间，就吃不到手续费收益了，还要全部兑换成单类资产（这就有本金损失的风险了）；宽区间则低虚拟LP，毕竟可以吃到手续费的概率大，那就不能吃太多（赔率降）。于是，就有这样的事发生：巨大的算法杠杆促进了资本的博弈，策略最好的资本将赚到最高的手续费也承担区间窄带来的风险。理论上，优秀的LP策略，可以以最少的资金，来LP给最精确的区间，创造足够深的深度，来促进区间内交易量的提升，反过来的手续费奖励又会极大提升，形成一个正反馈。

感叹于V3的AMM在数学设计上撬动竞争、博弈、交易量之精妙。

### 以太坊概览

1.  自 The Merge 后，ETH 呈现轻微通缩趋势，年通胀率约为 0.2%-0.8%。思考：通缩的目的很明确，制造稀缺性。然而，随着生态规模的扩大，轻微的通胀是最合理的。人类经济的发展需要一定程度的通胀。
    
2.  以太坊生态：DeFi NFT DAO等。核心在于智能合约在链上运行。
    
3.  以太坊升级之路：
    
    ① The Merge, PoW→PoS：攻击成本约需控制全网 67% 的质押 ETH（价值数百亿美元），所以足够安全。思考：对于代币盘子不够大的链来说，这并不安全。
    
    ② EIP-4844：引入Blob，不占用主网的永久储存。
    
    ③ 数据分片
    
    其它关键的：zk-rollup，通过零知识证明在链下一次性处理数百笔交易。
    
4.  生态包含： L1（主网）、L2（二层扩展解决方案）、侧链（Sidechains）
    
    L1：主网、EVM、账户系统
    
    L2：Rollup，批量处理交易的核心扩容技术。逐渐由Optimistic Rollup向ZK Rollup过渡。
    
    侧链：独立运行，**桥接**来连接主网。

5.  核心机制：**账户系统、Gas 模型 和 以太坊虚拟机（EVM）**。

    EVM：运行智能合约的虚拟机，运行在每个节点上，确保结果一致、可信任。虚拟机天然有隔离性质，确保用户的数据和网络不被访问。
    
6.  一般流程：发起交易，附带Gas参数→EVM执行智能合约→扣除Gas.
