**I. 让币圈脱离横盘的真正推手**

当以下三者同时失灵时，停滞便发生了：

-   **资本**毫无兴致
    
-   **情绪**消耗殆尽
    
-   **当前共识**再也无法解释“我们为何要在乎这个圈子”
    
-   在这样的局面下，价格疲软并非因为加密“已死”，**而是因为没有新的要素能让新参与者形成合力**。
    

**共识与叙事的区别**

每一轮将加密世界拖出寒冬的，始终是同一件事：**共识的进化**。

-   “共识”，在这个圈子里，是指人类找到了一种新的方式，能以加密货币为媒介，将某种**“抽象元素”**_（比如信仰、判断力、身份认同等）_进行金融化，并围绕它进行**大规模协作**。
    

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/chiputaobutuput/images/2026-01-28-1769565842805-image.png)

-   叙事，是一个**众人共享的故事**。 共识，是**众人共同的行为**。
    
-   叙事用说的，共识用做的。叙事吸引目光，共识留住人群。
    
-   所有叙事的底层，本质都是**聚合**——而这就是**共识**。
    

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/chiputaobutuput/images/2026-01-28-1769565916035-image.png)

-   加密货币不再仅仅是在搬运资金，它在重新分配“谁说了算”的权力。
    
-   **「三燃料模型」**
    
    ![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/chiputaobutuput/images/2026-01-28-1769566114643-image.png)

-   **流动性**（宏观风险偏好、美元流动性、杠杆能力等）如同注入市场的氧气，它决定了价格能移动多快。
    
-   **叙事**（人们为何在乎、如何被解释、共同的语言）吸引注意力，它决定了有多少人会看向这里。
    
-   **共识的底层构建**（共同行为、重复动作、去中心化的协作方式）影响持久性，它决定了当价格不再给予回报时，谁会真正留下。
    

**如何辨别“回光返照”与真正的“共识升级”？——行为**

**拓展案例1：2017年 ICO 的蓬勃发展 VS 早期试水**

那时候大部分 ICO 都是彻头彻尾的骗局或庞氏。这种脏事儿2017年之前有，2026年现在也还有。**但人们协同工作和调度资金的方式，已经永远改变了，这就是所谓的“共识升级”**。

**拓展案例2：2020年DeFi之夏 VS 虚假的牛**

这个时代也是一个真正的“共识升级”，**因为即使价格没有狂暴增长，**人们也开始将加密资产当作**“金融工具”**来使用。这与ICO时代截然不同——那时价格上涨与用户行为是互相喂养的共生关系。

-   **借贷：** 把你的加密资产存进协议，就能赚取“租金”。
    
-   **抵押借款：** 不卖掉你的加密资产也能获得购买力，就像用房子抵押贷款。
    
-   **流动性挖矿：** 每周把资金挪到收益最丰厚的地方，资金左腾右挪。
    
-   **做LP：** 把你的筹码放在桌上让别人交易，然后从手续费中抽成。
    
-   **循环抵押：**抵押、借出、再抵押、再借出，把杠杆和收益层层叠加。
    
-   **治理：** 真正参与协议规则的投票，而不仅仅是赌代币的价格。
    

它打破了**“纯赌场”**的思维定式，因为加密世界第一次感觉像一个**有生产力的金融系统**，而不再只是投机玩具。

在 DeFi之夏 后，如果一个新产品不能给用户提供一个“留在链上”的**实质性理由**，它就很难再激起什么水花了。

**拓展案例3：NFT时代翻转社交习惯**

如果说 DeFi之夏 是大家埋头钻研流动性曲线的极客时代，那么2021年就是加密世界终于拥有了**“人格”**的一年。我们不再纯为了收益率去内卷，开始转而追求**氛围感、身份认同**和**归属感**。

-   取代流动性池的，是_藏品系列_。
    
-   取代总锁仓价值的，是_地板价_和_社交资本_。
    
-   取代收益的，是_归属感_。
    

**失败**的叙事、失败的“共识升级”**也值得钻研**，例如2021-2022 Metaverse 1.0、2023-2024 SocialFi 1.0等。下一次“共识升级”也许是全新的东西，但也可能是曾经失败过的“旧东西”以“某种新形式”的复兴，当这样的事发生的时候，这份认知将会是你的**大机缘**。

**“韭菜自我保护5问”**：

**1\. 是否有“圈外人”入场**？

出现了一批主要目的不是赚钱的参与者，你看到的人不再只是为了炒币而来。他们是创作者、建设者，或是寻求身份认同的人。如果房间里只有交易员，那这个房间本质上就是空的。

**2\. 它能否通过“激励衰减”测试？**

观察当奖励枯竭或价格横盘时会发生什么。如果人们留下来，说明一种习惯已经养成。如果“免费午餐”一停他们就消失，那你面对的只是一堆带价格标签的空气。

**3\. 他们是在选择“日常习惯”而非“持仓”吗？**

新手只会看k线，而高手观察人们每天都在做什么。如果他们围绕这个系统建立起了日常习惯，那这就是一个永久性升级。

**4\. 是否存在“行为>体验”的现象？**

真正的转变发生在工具还很原始、分散、低效的时候。如果人们愿意忍受糟糕的用户界面也要参与，那么这种行为就是“有效”的。等到应用变得丝滑通畅精致时，你已经晚了。

**5\. （最重要的一点！！）是否存在“用爱发电”？**

这一点很关键。当人们开始捍卫一个系统，是因为它构成了他们身份的一部分，而不仅仅是因为他们会亏钱时，转变就彻底完成了。

什么是三盘理论

分红盘Mining，互助盘Pooling，拆分盘Splitting

分红盘的类型：资金型沉没成本、时间/精力型沉没成本

分红盘崩盘条件：实际增量沉没成本（为获取每单位增量收益而产生的周期性不可回收成本，如电力和维护费用）+外部流动性（外部交易场所中此分红代币的可用流动性）＜可提取回报（可自由提取并变现的收益）

此时，系统创建者应通过停止分红和“跑路”来获利。

互助盘：一种用户提供流动性以换取每单位贡献固定回报承诺的系统。就像赌场不直接靠个人输赢获利，而是从总交易额中抽取一定比例。

互助盘的类型：纯传销分钱型、准期权型、流动性挖矿

崩盘条件：系统性债务 > 可清算资产 + 外部流动性

拆分盘：其总资金在某一特定时间点保持恒定，但每单位资金对应的权益或资产数量被成倍增加，同时新生成的权益或资产的价格按比例降低，以吸引后续资金流入。

**ROI**（投资回报率）是用来评估投资效率的指标，表示投资收益与投资成本之间的比率。其计算公式为：**ROI =（投资收益 - 投资成本）/ 投资成本 × 100%**。通过计算ROI，企业可以判断投资是否达到预期的经济效益，并比较不同投资的效率。

### 一、前期筹备：定框架、备资源、聚选手、做赋能（核心：把 “规则、资源、选手” 全部落地）

-   这是活动成功的基础，需解决 **“比什么、谁来比、谁来评、用什么比、怎么比”** 五大问题，同时针对 AI Agent 的技术门槛做**前置赋能**，避免选手因技术不熟导致参赛体验差、作品完成率低。
    
    ### 1\. 定核心框架：明确活动基本规则（第 1 周完成，公示并同步所有渠道）
    
    围绕 AI Agent 赛道特性聚焦主题，避免赛制宽泛，核心确定以下内容并形成**正式赛制手册**（图文版 + 电子版）：
    
    -   **活动主题**：聚焦 AI Agent 垂直场景 / 技术方向，比如「多智能体协作开发赛」「企业级 AI Agent 落地赛」「轻量化智能体工具创作赛」，也可结合行业（如办公、教育、电商、医疗）定主题，降低选手选题迷茫；
        
    -   **时间 / 形式**：黑客松常规 1-3 天（线下建议 2 天 1 夜，线上建议 3 天），明确开幕 / 开发 / 提交 / 路演 / 颁奖的具体时间节点；
        
    -   **参赛规则**：组队要求（3-5 人最佳，建议跨角色：开发 + 产品 + 设计，纯技术队也可）、报名条件（可开放高校 / 企业开发者 / 独立开发者，无门槛但建议有 Python / 大模型基础）、作品要求（**AI Agent 核心要求**：需体现智能体的自主性 / 交互性 / 任务拆解能力，而非单纯的大模型调用脚本）；
        
    -   **提交规范**：明确作品交付物（必须包含：可运行原型 / 演示视频、代码仓库（GitHub/Gitee）、产品说明 PPT、AI Agent 设计文档（含框架、技术栈、场景落地逻辑）），统一提交格式和渠道（如飞书问卷 / 腾讯文档）；
        
    -   **评分标准**（提前公示，确保公平）：按权重打分，贴合 AI Agent 特性，示例：落地性（30%）、创新性（25%）、技术难度（20%，如多智能体协作、本地部署、个性化定制）、演示效果（15%）、文档完整性（10%）；
        
    -   **奖项设置**：分等级奖（一二三等奖）+ 特色奖（最佳创新奖、最佳落地奖、最佳技术奖），奖品贴合开发者需求（AI Agent 开发资源包：大模型 API 额度 / LangChain Pro 权限、云服务器（阿里云 / 腾讯云）、技术书籍、周边，企业合作的话可加实习 / 孵化机会）。
        
    
    ### 2\. 资源筹备：搞定 “评委、技术、物料、后勤” 四大核心资源（第 1-2 周完成）
    
    资源筹备的核心是 **“提前测试、提前对接”**，尤其是 AI Agent 的技术资源，避免开发阶段出问题，分模块落地：
    
    -   **评委资源**：组建「技术 + 产业」双评委团（5-7 人最佳），贴合 AI Agent 赛道：
        
        -   技术评委：大模型 / AI Agent 领域工程师、LangChain/AutoGPT 等框架开发者、高校人工智能专业老师；
            
        -   产业评委：科技企业产品经理 / 技术负责人、AI 创业公司创始人、行业专家（如办公 / 电商领域）；
            
            提前对接评委，确认时间、讲解评分标准，同步赛制手册，约定巡场 / 点评环节的要求。
            
    -   **技术资源**：AI Agent 开发的核心保障，**提前测试并整理成《技术资源手册》**：
        
        -   免费开发工具：大模型 API（讯飞星火 / 百度文心 / 阿里通义的免费额度）、AI Agent 框架（LangChain、AutoGPT、AgentScope）、云服务器（主办方 / 赞助商提供的免费算力）；
            
        -   技术文档：框架使用教程、API 调用指南、常见问题解决方案（如环境配置、接口报错）；
            
        -   技术支持通道：提前搭建专属技术答疑群（邀请框架开发者 / 技术志愿者入驻）。
            
    -   **物料 / 平台资源**：
        
        -   线下：场地（会议室 / 创客空间，需配备投影、网络、插排）、签到物料（胸牌、手册）、宣传物料（海报、展架）、后勤物料（饮用水、零食、休息区物资）；
            
        -   线上：直播 / 会议平台（腾讯会议 / 飞书会议）、沟通平台（Discord / 飞书群，分通用群 + 技术答疑群）、作品提交平台（云文档 / 问卷星）、代码托管平台（GitHub）。
            
    -   **合作 / 赞助资源**（可选）：对接 AI 领域企业、云服务商、技术社区，争取奖品 / 技术资源 / 资金支持，同时为后续作品孵化做铺垫。
        
    
    ### 3\. 选手招募 + 前置预热：精准触达，降低门槛（第 2-4 周完成，招募 + 赋能并行）
    
    AI Agent 属于中高门槛技术赛道，**“精准招募” 比 “广撒网” 更重要**，且需做**前置技术赋能**，提升选手参赛质量和完成率，避免 “报名多、参赛少、完成少” 的问题：
    
    -   **精准招募渠道**（按优先级排序）：
        
        ① 技术社群：LangChain/AI Agent 中文社群、GitHub AI 开源群、CSDN / 掘金 AI 板块、飞书 / 钉钉开发者社群；
        
        ② 高校渠道：计算机 / 人工智能 / 大数据专业的学院社团、创客空间、导师推荐；
        
        ③ 企业渠道：科技公司技术部门、AI 创业公司、互联网企业的算法 / 开发团队；
        
        ④ 自媒体 / 平台：AI 领域博主（视频号 / 小红书 / 知乎）、开发者平台（开源中国、GitChat）；
        
        招募形式：线上报名（问卷星 / 飞书表单，收集组队信息、技术栈、参赛方向），报名后做简单审核（避免纯新手组队，可引导新手和老手组队），最终公布参赛名单并建专属参赛群。
        
    -   **前置预热 + 技术赋能**（招募同时进行，2-3 场即可）：
        
        ① 赛制宣讲会：讲解活动规则、奖品、技术资源、提交要求，解答选手疑问；
        
        ② 技术公开课：邀请 AI Agent 领域专家讲**开发入门**（如 LangChain 搭建简单智能体、多智能体协作逻辑）、**场景拆解**（如办公 AI Agent 的需求分析），配套发放教程文档和热身题（如 “用 LangChain 搭建一个智能问答 Agent”）；
        
        ③ 社群预热：在参赛群发布技术资料、组织选手交流选题、自由组队，技术志愿者实时解答基础问题。
        
    
    ### 4\. 运营团队分工：明确岗位职责，避免协作混乱
    
    按 “**总协调 + 专项岗**” 划分，3-5 人即可支撑一场中小型黑客松，核心岗位及职责：
    
    -   总协调：统筹全流程，对接评委 / 赞助商，处理应急问题；
        
    -   招募 / 宣传岗：负责招募渠道、宣传物料、活动直播 / 拍照 / 内容记录；
        
    -   技术运营岗：整理技术资源、维护答疑群、解决开发中的技术问题、测试 API / 框架可用性；
        
    -   现场 / 线上运营岗：负责签到、开幕 / 路演控场、计时、后勤保障、群内信息同步；
        
    -   评委对接岗：对接评委，整理评分表、统计打分结果、安排巡场 / 点评环节。
        
    
    ### 二、活动实施：保流畅、做支持、控节奏、评成果（核心：让选手专注开发，让评审公平高效）
    
    黑客松实施阶段（1-3 天）的核心是 **“少打扰、多支持”，所有运营工作围绕 “保障开发顺利、把控时间节点、高效完成评审” 展开，分开幕、开发、作品提交、路演评审、闭幕颁奖** 5 个环节，每个环节有明确的时间和运营动作。
    
    ### 1\. 开幕环节（0.5 天，线下 1-2 小时，线上 1 小时）：定调 + 答疑 + 确认组队
    
    核心是 **“重申规则、明确资源、解决最后一公里疑问”**，避免开发阶段反复答疑：
    
    -   流程：签到→主办方 / 赞助商致辞→赛制 / 评分标准重申→技术资源讲解（API / 框架使用、答疑通道）→评委致辞→自由组队（未组队选手现场匹配）→选手宣誓 / 合影；
        
    -   运营重点：现场 / 线上控场，及时收集选手最后疑问并统一解答，确认所有选手加入答疑群，发放《技术资源手册》和赛制手册。
        
    
    ### 2\. 开发阶段（核心，1-2 天）：实时支持，把控进度，轻量打卡
    
    这是选手核心创作阶段，运营**不做过多干预**，只做**基础支持和进度提醒**，核心动作：
    
    -   **技术支持**：技术运营岗 7\*24 小时在线，答疑群按 “问题类型标签”（如框架 / API / 服务器）回复，复杂问题邀请技术专家 / 志愿者解答，**及时同步共性问题的解决方案**（如某 API 调用失败的统一解决办法）；
        
    -   **进度把控**：设置**轻量打卡机制**（如每 6/12 小时在群内打卡 “当前进度 + 是否有问题”），避免选手摆烂，对进度过慢的选手做私下沟通，提供选题 / 技术建议；
        
    -   **后勤保障**：线下提供不间断餐饮、休息区，保证网络 / 电力稳定；线上定时提醒选手休息，避免过度疲劳；
        
    -   **中期巡场（可选）**：邀请评委做 1-2 次轻量巡场（线下到各小组交流，线上进会议群答疑），对选手的作品方向、技术选型给出建议，提升作品质量；
        
    -   **信息同步**：群内只发**重要信息**（如时间提醒、共性问题、资源更新），避免刷屏，设置禁言时段（如夜间）。
        
    
    ### 3\. 作品提交环节（开发结束前 1-2 小时）：明确提醒，统一收集，核对完整性
    
    核心是 **“避免选手遗漏提交、确保交付物格式统一”**：
    
    -   提前 12/6/1 小时多次提醒选手提交时间，明确提交渠道和格式要求；
        
    -   提交后专人核对：是否包含所有交付物（代码 / 演示视频 / PPT / 设计文档）、链接是否可访问、视频是否能播放；
        
    -   对提交不完整的选手做最后提醒，给予 10-30 分钟补传时间，最终确认**有效作品名单**并同步评委。
        
    
    ### 4\. 路演评审环节（0.5 天，线下 3-4 小时，线上 4-5 小时）：公平计时，高效记录，实时统分
    
    AI Agent 的作品**重演示、重交互**，路演环节是评审核心，运营重点是 **“控时、记录、统分”**：
    
    -   提前制定路演顺序（随机 / 按报名顺序），同步给选手和评委，每人路演时长**5-10 分钟讲解 + 3-5 分钟评委提问**；
        
    -   运营岗负责严格计时，线下安排主持人控场，线上开启会议录制，记录每一位选手的讲解内容和评委提问；
        
    -   评委对接岗负责发放评分表（电子档），实时收集评委打分，去掉一个最高分和一个最低分，计算平均分，**全程公开统分过程**，确保公平；
        
    -   路演中若选手演示出现技术问题（如 Agent 运行失败），允许选手播放提前录制的演示视频，避免影响评审。
        
    
    ### 5\. 闭幕颁奖环节（0.5 天，线下 1 小时，线上 0.5 小时）：公布结果，点评总结，沉淀氛围
    
    核心是 **“公布结果、做价值沉淀、打造活动收尾氛围”**：
    
    -   流程：公布获奖名单（从特色奖到等级奖）→颁奖→评委对所有作品做整体点评（亮点 + 不足 + AI Agent 发展建议）→优秀选手分享创作心得→主办方总结（感谢所有参与方，预告成果落地计划）→合影 / 线上连麦留念；
        
    -   运营重点：现场拍照 / 线上截图，实时发布活动现场内容，同步到参赛群和宣传渠道，营造活动热度。
        
    
    ### 三、复盘沉淀：析数据、收反馈、落成果、梳经验（核心：让活动 “有结果、有经验、有后续”）
    
    黑客松不是 “结束即终点”，尤其是 AI Agent 赛道，很多作品是优质原型，**复盘的核心是 “数据化总结问题、推动作品落地、沉淀活动经验”**，分为**即时复盘（活动结束后 1-3 天）** 和**深度复盘（活动结束后 1 周）**，同时完成**成果沉淀和落地**。
    
    ### 1\. 即时复盘：快速收集反馈，整理基础数据
    
    核心是 **“抓住记忆点，收集最真实的反馈”**，避免时间太久反馈失真：
    
    -   **内部复盘**：运营团队全员开会，各岗位汇报工作中的问题（如技术支持不及时、路演计时混乱、后勤保障不足）、协作中的问题（如信息传递不及时、岗位职责重叠），记录所有问题并初步分析原因；
        
    -   **外部反馈收集**：通过问卷星向**选手、评委、赞助商**发放简短反馈问卷（5-8 题），核心问题包括：赛制合理性、技术资源可用性、后勤 / 体验、作品评分公平性、对后续活动的建议；
        
    -   **基础数据统计**：整理活动核心数据，形成**数据看板**
        
    -   2\. 深度复盘：全面分析问题，形成解决方案
        
        基于即时复盘的反馈和数据，做**深度分析和总结**，形成**正式的活动复盘报告**，包含以下内容：
        
        -   **活动亮点**：如技术赋能到位、选手作品质量高、评委点评专业、成果落地性强等，提炼可复用的经验；
            
        -   **核心问题**：按 “招募、技术支持、赛制、运营、后勤” 分类，分析问题本质（如 “完成率低” 是因为技术门槛太高，而非时间不够）；
            
        -   **解决方案**：针对每个问题制定具体的解决办法（如 “技术门槛高”→下次活动提前增加 1-2 场基础赋能课，提供预制的 Agent 开发模板）；
            
        -   **数据结论**：总结渠道、选手、作品的核心特征（如本次活动优秀作品多为 “企业级落地场景”，后续可聚焦该方向）。
            
        
        ### 3\. 成果沉淀：整理活动物料，打造内容资产
        
        将活动的所有物料和成果整理沉淀，为后续品牌打造和活动做铺垫，核心动作：
        
        -   **内容沉淀**：整理活动照片、直播 / 路演视频、技术分享课件、优秀选手的创作心得，剪辑成短视频 / 整理成图文，发布到宣传渠道（公众号 / 小红书 / 技术社群）；
            
        -   **作品沉淀**：制作《AI Agent 黑客松优秀作品案例集》，包含作品介绍、演示视频、代码链接、设计思路，分享到参赛群和技术社群，供开发者学习；
            
        -   **资源沉淀**：整理《AI Agent 开发资源包》（框架教程、API 指南、常见问题解决方案），形成标准化物料，复用至后续活动。
            
        
        ### 4\. 成果落地：推动优秀作品从 “原型” 到 “产品”
        
        这是 AI Agent 黑客松的**核心价值**，区别于普通黑客松，运营需主动推动优秀作品的落地，核心动作：
        
        -   **对接孵化资源**：将优秀作品推荐给赞助商、合作企业、投融资机构，为选手提供技术 / 资金 / 场地支持，邀请选手完善作品；
            
        -   **社群落地**：在技术社群内展示优秀作品，组织选手和开发者交流，推动作品开源或商业化；
            
        -   **长期跟进**：建立优秀选手档案，长期跟进作品进展，为后续的 AI Agent 赛事 / 项目储备人才。
            
        
        ### 5\. 经验固化：形成 SOP，为后续活动铺路
        
        将本次活动的全流程经验整理成 **《AI Agent 黑客松运营 SOP》**，包含各阶段的工作内容、时间节点、岗位职责、物料清单、应急预案，固化成标准化流程，后续举办类似活动可直接复用，减少试错成本。

### 1.什么是公共物品

-   非排他性
    
-   非竞争性
    

如开源代码、协议安全、开发者工具/文档

### 2.公共物品问题——公地悲剧：谁来为植物浇水？

搭便车问题，市场失灵，高使用量≠高收入

**问题：**纯粹的市场无法为公共物品定价

**方案：**需要新的机制来捕获这些价值

### 机制一：委员会制（来源于税收）

优点：简单、快速、决策高效

缺点：权力集中、偏好固化与偏见、有限的知识

### 机制二：二次方资助（QF）人数＞金额

配捐池的资金分配由贡献者的数量决定，放大了社区的共识。

### 零知识证明：

在不暴露具体细节的情况下，能够向第三方证明数据的某些特征。如提供Proof（X，Y），向第三方证明X+Y=Z，但Proof（X，Y）无法反推出X和Y的具体值。

### ERC-7962解决的关键问题：

隐私问题：避免地址可追踪，降低身份与资产的关联性

UX问题：用户只需签名表达意图，无需持币，无需理解Gas，无需管理地址隐私，也能完成资产操作

扩展性问题：天然支持批量交易、Relayer、账户抽象场景

### 技术实现要点：

所有权由Keyhash=keccak256(publicKey)表示转账时提交。

### 安全设计：

EIP-721结构化签名，防参数篡改

per-keyHash nonce，防重放攻击

deadline控制签名有效性

### 优势

更强隐私（地址不上链）

更加用户体验（免ETH/Gas代付）

更高灵活性（批量、代付、抽象账户）

# **代币经济学**

代币经济学涵盖了代币的**供应量、发行方式及效用**。如果一个项目的代币经济学设计合理，则通常会吸引大量投资者购买和持有其代币，从而导致代币需求和价格上涨。相比之下，代币经济学设计不当的平台可能无法吸引大量投资者。

代币经济学决定了代币的分配方式和用途，其\*\*本质是供需平衡，因此会影响代币的价值。\*\*例如，与在生态系统中没有明确用途的代币相比，人们对具有实际用途的代币的需求量可能更大。

加密货币区别于普通货币的一个关键特征在于，其**规则通过代码编写而成**。这样一来，代币规则变得更加**透明**，但也更难以更改。

# **加密经济学**

加密经济学通过将**密码学与经济学**相结合以协调网络参与者的行为。

具体而言，加密经济学属**计算机科学领域**，它试图通过**密码学和经济激励**以解决数字生态系统中的**参与者协调问题**。

去中心化网络的构建离不开加密经济学，因为该机制可用于**调整参与者激励，而无需受信第三方参与。**

加密经济学并非传统经济学的子集，它融合了博弈论、机制设计、数学以及经济学领域的其他方法论。加密经济学的**主要目标是了解如何资助、设计、开发和促进去中心化网络的运作**。

# **拜占庭将军问题**

在1982年，拜占庭将军问题被认为是一个逻辑上的困境，它说明了一群拜占庭将军在试图**就下一步行动达成统一意见时可能存在的沟通问题。**

该困境假设每个将军都有自己的军队，每支军队都位于他们打算攻击的城市周围的不同位置。 这些将军**需要就攻击或撤退达成一致**。只要所有将军达成共识，即协调后决定共同执行，无论是攻击还是撤退都无关紧要。

因此，我们可以考虑以下条件：

-   每个将军必须作出决定：攻击或撤退（是或否）;
    
-   一旦做出决定后无法改变;
    
-   所有将军都必须就统一决定达成一致，并同步执行。
    

上述沟通问题涉及这样一个事实，即**一个将军只能通过情报员发送的信息与另一个将军进行通信**。因此，拜占庭将军问题的**核心挑战是信息可能以某种方式被延迟传达，破坏或丢失。**

此外，即使消息成功传递，一个或多个将军也可以选择（出于任何原因）恶意行动并发送欺诈性消息以混淆其他将军，导致行动失败。

如果我们将这个困境对应到区块链上，则每个将军代表一个网络节点，而节点需要就系统的当前状态达成共识。换句话来说，分布式网络中的大多数参与者必须同意并执行相同的操作以避免失败。

因此，在**这些类型的分布式系统中达成共识的唯一方法是至少拥有三分之二以上的可靠和诚实的网络节点**。这意味着如果网络中的大多数节点决定采取恶意行为，系统就很容易受到故障和攻击（例如51％攻击）。

# **拜占庭容错（BFT）**

简单来说，拜占庭容错（BFT）是**能够抵抗拜占庭将军问题所导致的一系列失败的系统属性。** 这意味着即使某些节点出现故障或恶意行为，拜占庭容错系统也能够继续运行。

拜占庭将军问题有多种可能的解决方案，因此，有多种方法可以构建拜占庭容错系统。同样地，区块链有各种不同的方法来实现拜占庭容错，这就是我们说的共识算法。

# 区块链共识算法

我们可以将共识算法定义为**区块链网络达成共识的机制**。最常见的例子是工作量证明（PoW）和权益证明（PoS）。这里我们以比特币为例。

比特币协议规定了系统的主要规则，工作量证明共识算法说明了如何遵循这些规则以达成共识（例如，在交易的验证和验证期间）。

虽然工作量证明的概念比数字货币更早，但中本聪在原版本上进行了修改，并开发了一种改进后的工作量证明算法，可以将比特币的产生作为一种拜占庭容错系统。

请注意，该工作量证明算法不能完全抵抗拜占庭故障，但由于高成本的挖矿过程和底层加密技术，工作量证明已被证明是区块链网络中的一种最安全可靠的方法。从这个意义上说，由中本聪设计的工作量证明共识算法被许多人认为是拜占庭容错最高明的解决方案之一。

## 一、什么是 **Dune Analytics**

**Dune Analytics**（现简称 **Dune**）是一款专注于 **区块链链上数据分析的平台**，允许任何人：

-   通过 **类 SQL 语句** 对大量链上数据进行查询与聚合；
    
-   将结果可视化成 **图表与仪表盘（Dashboard）**；
    
-   分享与复用社区中其它人的分析成果；
    
-   支持包括 Ethereum、Solana 等 **100+ 公链数据**。
    

📌 它类似于链圈的 “数据 GitHub”，用户既能使用别人的分析，也能自己深度定制。

## 二、核心概念解释

### 1) **链上数据仓库**

区块链本质上是一个去中心化账本，**每一笔交易、事件和合约调用都会写入链上**。  
这些原始数据经过 Dune 预处理并构建成可查询的 **数据表（dataset）**。

### 2) **SQL / 类 SQL**

-   Dune 使用一种兼容 SQL 的查询语言（也称 **DuneSQL**），允许你用 familiar 的语法读取数据。
    
-   SQL 是结构化查询语言的标准，通过 SELECT、FROM、WHERE、GROUP BY 等语句过滤和处理数据。
    

### 3) **Query 与 Dashboard**

-   **Query**：一条 SQL 查询语句，用来从数据仓库提取你想要的信息。
    
-   **Dashboard**：把多个 Query 的可视化图组合起来的面板，用于展示财经指标、链上行为趋势等。
    

## 三、Dune 的基本操作流程

### 1) 登录与创建 Query

Step 1 — 登录

注册一个免费账户（邮箱登录即可）。

Step 2 — 打开 New Query

在 Dune 控制台点击 **“New Query”**，就能看到侧边的数据表和编辑器。

### 2) 编写你的第一条 Query

示例：查前 5 条以太坊交易

```
SELECT * FROM ethereum.transactions
LIMIT 5
```

解释：

-   `SELECT *` 表示把所有字段都取出；
    
-   `FROM ethereum.transactions` 指定要查询的数据表；
    
-   `LIMIT 5` 是限制返回结果数量。
    

### 3) 聚合与可视化

示例：统计每日 ETH 转账数量

```
SELECT date_trunc('day', block_time) as day,
       count(*) AS txn_count
FROM ethereum.transactions
GROUP BY 1
ORDER BY day DESC
```

说明：

-   `date_trunc('day', block_time)` 按日分组；
    
-   `count(*)` 是计算每天交易数量。
    

完成查询后，你可以把结果导入图表，例如柱状图（Bar Chart）或者折线图（Line Chart）。

## 四、常见 Query 进阶技巧

### 1) **汇总求和（SUM）**

示例：统计某地址总 ETH 转账金额

```
SELECT sum(value / 1e18) AS total_eth
FROM ethereum.transactions
WHERE "from" = '\x1Db3439a222C...'  
```

📌 Dune 中 `value` 默认是 wei 单位，需要除以 `1e18` 转换成 ETH。

### 2) **分类汇总（GROUP BY）**

统计不同分类的数量和总和，有助于做更复杂的数据解读。

### 3) **累计和（Cumulative Sum）**

有时候你希望看到数据的累积趋势，这可以用窗口函数实现。

## 五、可视化 & 仪表盘构建

**创建图表**

在查询结果页面点击 **“New Visualization”**，选择：

-   条形图（Bar Chart）
    
-   折线图（Line Chart）
    
-   数值计数器（Counter）
    
-   饼图等图表样式
    

然后配置图表的 X 轴、Y 轴即可。

**组合 Dashboard**

图表做好后点击 **“Add to Dashboard”** 即可将它放进你要展示的仪表盘。

## 六、实用技巧与进阶建议

### 1) **Fork 神技能**

如果你对某个现有 Dashboard 感兴趣，可以直接 **Fork（复制）别人的 Query 或看板**，修改特定参数以快速入手。

### 2) **查找数据表**

Dune 的 **Data Catalog**（数据目录）能帮你发现各种链上数据表，例如：

-   `ethereum.transactions`（交易表）
    
-   `erc20.ERC20_evt_Transfer`（代币转账事件）
    
-   `dex.trades`（去中心化交易所交易数据）
    

### 3) **使用社区资源**

Dune 有大量免费的社区 Dashboard 和 Query，可用于学习和灵感获取。

## 女巫攻击与人格证明

1.  **女巫攻击（Sybil Attack）**
    
    -   定义：攻击者创建大量虚假链上身份，操纵 “一人一票” 的治理投票，破坏社区公平性。
        
    -   案例：若创建 1000 个类似`SanfordStout.eth`的身份，可垄断项目治理投票权。
        
2.  **解决方案：人格证明（Proof of Personhood）**
    
    -   核心思路：通过分析链上行为（如投票记录、多签操作、持有老资格 NFT 等），为账户的 “真人程度” 打分。
        
    -   代表项目：Gitcoin Passport，通过多维度行为验证，区分真实用户与虚假账号。
        

1.  **Gwei**：以太坊 Gas 计价单位，`1 Gwei = 0.000000001 ETH`，用于衡量交易消耗的计算资源成本。
    
2.  **Sybil Resistance（抗女巫攻击）**：Web3 社区治理的核心特性，指系统抵御虚假身份操纵的能力。
    
3.  **算法稳定币死亡螺旋**：以 UST 为例，当代币价格脱离锚定汇率时，算法的铸币 / 销毁机制失效，引发市场恐慌性抛售，最终导致代币崩盘。
    

## 同质化代币（ERC-20）与非同质化代币（ERC-721）的区别

1.  **数据映射逻辑差异**
    
    -   **ERC-20（同质化）**：采用 `mapping(address => uint256) balance` 映射，仅跟踪地址对应的代币余额，代币间无差异（如你有 50 个 ETH 和我有 50 个 ETH 完全等价）。
        
    -   **ERC-721（非同质化）**：采用 `mapping(uint256 => address) ownerOf` 映射，跟踪每个 Token ID 的所有者，每个 NFT 独一无二（如 Token #1 属于 Alice，Token #2 属于 Bob）。
        
2.  **NFT 转账本质**：NFT 转账并非文件在网络中传输，而是区块链账本的异步更新：所有节点将该 NFT 的 `ownerOf` 映射关系从原所有者地址修改为新所有者地址。

## 一、合规性要求与法律风险

### 1\. Web3监管环境概览

-   自 2017 年起，中国监管层发布多项针对虚拟货币的政策：
    
    -   禁止 **ICO／IEO／IDO** 形式的筹资
        
    -   禁止中心化虚拟货币交易撮合服务
        
    -   虚拟货币不具有法偿性，不可作为支付工具
        
    -   禁止境外平台向国内居民提供服务
        
    -   打击非法集资、传销、赌博、洗钱等行为
        
-   监管主基调：  
    **封堵金融属性 + 有限容忍技术创新**
    
-   技术本身不被禁止，但涉及金融行为会被严查。

a) 代币发行与交易

-   中国法律禁止任何单位或个人通过 ICO、IEO、IDO 等融资。
    
-   即便叫“积分”“治理 Token”，只要具备融资或流通属性也可能被视为非法金融行为。
    
-   即使只是写合约、设计模型等技术参与者，也可能被视为“共同行为人”。
    

b) 赌博、传销、洗钱等风险

-   Web3 经济激励结构若类似“充值→抽奖→提现”，可能被认定为**赌博业务**；
    
-   多层返佣或邀请奖励机制易触及**非法传销罪**；
    
-   虚拟币兑换与场外交易可能被认定为**洗钱或非法经营**行为。
    

c) 民商事合同风险

-   虚拟货币买卖在中国目前常被认定“合同无效”；
    
-   委托他人代为投资，出现亏损或资金不明，很可能被追究民事甚至刑事责任。

## **以太坊核心机制：从账户到执行的完整链路**

在上面我们就了解到，以太坊不仅仅是一条区块链，更是一个全球共享的“分布式计算机”。它的核心价值在于通过代码实现无需信任的自动化规则（如智能合约），而这一切的背后依赖于三个关键机制：**账户系统、Gas 模型 和 以太坊虚拟机（EVM）**。

我们平时使用 **账户系统** 里面的 **外部账户（EOA）** 与区块链中的其他用户的 **外部账户（EOA）** 或者与智能合约所在的 **合约账户（CA）** 进行交互，而其中的 **Gas 模型** 是支撑整个网络的经济基础， **Gas 模型** 和其他经济来源支撑着全球的 **以太坊虚拟机（EVM）** 来实现以太坊网络。

### **1\. 账户系统：你的数字身份**

**账户系统** 包含由私钥控制的**外部账户（EOA，**Externally Owned Accounts**）**和由智能合约代码控制的**合约账户（CA，**Contract Accounts**）**。

**外部账户（EOA）**：它由一对密钥（私钥和公钥）生成。私钥是控制账户的“钥匙”，必须严格保密；公钥通过加密算法生成一个唯一的地址，可以分享接收转账。

**合约账户（CA）:** 由代码驱动。比如，部署一个智能合约（如一个 NFT 市场），区块链会自动生成一个 CA 地址（如 `0xDef...456`）。这个账户不能主动发起交易，只能通过 EOA 触发——比如点击“购买 NFT”按钮时，EOA 向 CA 发送交易，CA 的代码自动执行出货逻辑。

每个账户都包含四个关键字段：

-   **Nonce**：防止重复交易的计数器（EOA 记录发送次数，CA 记录创建合约次数）。
    
-   **余额**：账户持有的 ETH 数量。
    
-   **CodeHash**：EOA为空哈希，CA存储合约字节码的哈希值。
    
-   **StorageRoot**：记录账户数据的 Merkle 树根哈希（如NFT归属关系）。
    

### **2\. Gas 模型：交易的燃料费**

当用钱包使用自己的EOA发起一笔交易（比如转账或操作合约），**需要支付“燃料费”——也就是 Gas。**

Gas 费用 = **用多少 × 每单位多少钱**：

-   **Gas Limit（限额）**：你最多愿意“烧”多少燃料。 比如你觉得最多可能需要 15 万单位，就设置 150,000。
    
-   **Gas Price（单价）**：每单位燃料多少钱，用 Gwei 表示（1 Gwei = 0.000000001 ETH）。 网络越拥堵，价格越贵，就像打车高峰期加价。
    

Gas 的存在有两个目的：

-   **激励矿工/验证者**：给得越多（Gas Price 越高），越愿意优先处理你的交易。
    
-   **防止资源滥用**：如果有人想让合约死循环，Gas 会用光，交易失败，系统不会被拖垮。
    

**在 EIP-1559 升级后，Gas Price 被拆分了：**

以前，Gas Price 全部给矿工。 现在，分成两部分：

-   **基础费用（Base Fee）**：每个区块都会有，自动计算，直接销毁（直接消失）来帮助 ETH 通缩，毕竟物以稀为贵。
    
-   **小费（Tip）**：你额外加的钱，用来鼓励矿工优先处理你。
    

### **3\. 以太坊虚拟机（EVM）：代码的执行引擎**

EVM（Ethereum Virtual Machine）是 **以太坊的“大脑”**，是专门用来**运行智能合约的虚拟计算机**。它运行在每个节点上，确保整个网络在处理代码时，**结果都一致、可信任**。

**EVM 的核心特点：**

-   **图灵完备**：就像真正的电脑一样，EVM 可以执行各种逻辑，比如 if 判断、循环等。
    
-   **全球同步**：每个矿工/节点都会自己执行一遍合约代码，保证结果一样。
    
-   **隔离安全**：EVM 把合约“关”在一个小房间里运行，不允许它乱访问用户的数据和网络，保护隐私和安全。
    

## **一般流程与类比**

### **1\. 一般流程**

1.  **用户通过 EOA 发起交易**（如调用合约与 **CA** 交互，或者转账与其他用户的 **EOA** 交互）。
    
2.  **交易附带 Gas 参数**，矿工 / 验证者选择打包。
    
3.  **EVM 执行合约代码**，修改存储（如更新 NFT 归属）。
    
4.  **Gas 费用按 Gas Limit × Gas Price 扣除**，保障资源合理使用。
    

### **2\. 相似类比**

-   **EOA = 个人钱包**：你用私钥“签名”来证明身份，可以主动发起任何交易
    
-   **CA = 智能保险箱**：按照预设程序自动运行，只有满足条件才执行操作（如：收到正确密码后自动转账）
    
-   **Gas = 计算资源费用**：
    
    -   Gas Limit：你愿意支付的最大计算量
        
    -   Gas Price：每单位计算的价格
        
-   **EVM = 全球共享计算机**：所有节点同时运行相同的程序，确保结果一致且可信
    

## 区块链三难困境

去中心化、安全性和可扩展性这三个主要目标之间的冲突。

1.  **去中心化**：这意味着确保区块链系统中的**所有节点拥有平等的权力**，不存在任何集中式机构控制网络。去中心化是区块链的基本特征，是系统安全和抵御攻击的基础。
    
2.  **安全性**：安全性是指确保区块链系统内**交易的真实性、完整性、不可篡改性和抵御攻击的能力**。它是区块链系统可靠性和可信度的关键保障。
    
3.  **可扩展性**：可扩展性或性能是指区块链系统**支持大量交易、节点和用户**的能力。它是满足实际应用需求的必要条件。
    

这三个目标往往需要权衡取舍。例如，追求更高的去中心化程度需要所有节点存储完整的区块链数据，但这可能会阻碍可扩展性。反之，优先考虑可扩展性可能意味着牺牲一定程度的去中心化。例如，采用分片技术来提高容量可能会导致节点间数据同步方面的挑战，并增加安全漏洞的风险。

**分片技术：**分片技术早于区块链出现，已被应用于传统数据库中，以增强大规模商业数据库的性能。其原理是**将数据库分割成多个数据分片**，并将它们**分布在不同的服务器上**，从而防止在高峰期单个服务器负载过重。在区块链领域，分片技术能够帮助**在短时间内处理大量交易，从而实现可扩展性**。然而，它也带来了一些新的挑战，例如**不同分片之间的通信问题**、由于并非所有交易记录都存储在每个分片中而导致的**信息同步问题**，以及由于验证计算能力分散而**增加的51%攻击风险**。这些挑战都需要额外的机制来解决，从而增加了共识机制的整体复杂性。

**Layer2（状态通道）：**分担第一层网络的负载，从而实现可扩展性。状态通道利用多重签名技术，允许双方将一定金额的**资金锁定**在智能合约中，从而创建一个内部通道。在这个通道内，他们可以**快速、低成本地进行大量小额交易**。经过特定时间后，他们可以通过单个交易证明提取资金。状态通道是比特币闪电网络和以太坊雷电网络的技术基础。

# 行业赛道概览

## 一、DeFi：去中心化金融

DeFi，全称为 Decentralized Finance（去中心化金融），是**基于区块链技术建立的金融体系**，目标是提供**不依赖传统银行或金融中介**的**服务**，比如**借贷、交易、支付**等，让每个人都可以**自由参与，无需审批、无需信任中介机构**。以下是 DeFi 领域的三个典型案例：

### **1\. Uniswap：去中心化交易所（DEX，**Decentralized Exchange**）**

Uniswap 是去中心化交易所（DEX）的代表，由 Hayden Adams 于 2018 年创立。其核心创新在于引入**“自动做市商”**（AMM，Automated Market Maker）模式，**通过“恒定乘积公式” 实现代币定价**。用户通过存入流动性池（如 ETH 和 DAI）成为流动性提供者（LP），赚取交易手续费。

1.  **恒定乘积公式：x \* y = k**
    
    这个公式听起来复杂，但它的核心思想其实很简单。我们可以把它想象成 Uniswap 是一个非常聪明的自动交易员，它通过一个简单的数学公式来决定价格，而不依赖人工设置或市场波动。
    
    -   **x** 和 **y** 代表流动性池中的两种加密货币，比如 ETH 和 USDC。
        
    -   比如一个流动性池里，假设有 10 个 ETH 和 5000 个 USDC。那么：
        
        x = 10（ETH） y = 5000（USDC）
        
        公式如何工作？
        
    -   恒定乘积公式就是 **x \* y = k**，其中 **k** 是一个常数。
        
        10 \* 5000 = 50,000（这就是 k 的值）
        
        当你在这个池子里进行交易时，这个公式始终要保持成立。也就是说，交易后，ETH 和 USDC 的数量会发生变化，但它们相乘的结果 **k** 必须始终是 50,000（忽略手续费的情况下）。
        
    
    **举个例子：**
    
    -   如果你想用 1 个 ETH 兑换 USDC，流动性池中的 ETH 数量就会增多，而 USDC 的数量就会减少。
        
    -   假设交易后，池中的 ETH 变成了 11 个，那么为了保持公式成立，USDC 数量需要调整到 4545.45 个（50,000 / 11 = 4545.45）。
        
    -   **具体兑换价格计算**：
        
        -   交易前：10 ETH，5000 USDC
            
        -   交易后：11 ETH，4545.45 USDC
            
        -   你用 1 ETH 兑换到了 5000 − 4545.45 = 454.55 USDC（不考虑手续费的情况）
            
        -   **兑换价格 = 454.55 USDC / 1 ETH = 454.55 USDC per ETH**（低于交易前现货价 500，体现了价格冲击）
            
    -   这样，Uniswap 就能自动算出兑换的价格，而且随着池子中资产比例的变化，价格也会自动调整。
        
    
    Uniswap 通过这个公式，自动决定每个交易的价格和兑换率。它确保了池子的总价值始终保持平衡，价格会跟随储备比例连续调整。大额单会产生滑点。
    
2.  **流动性池和流动性提供者（LP）**
    
    流动性池是 Uniswap 交易的核心。它是由很多用户共同提供的资金池，用于交易中的兑换。你不再需要通过一个中心化的交易所来等待配对和匹配订单。你只需要加入流动性池，Uniswap 就会根据池子里的资金自动帮助你完成交易。
    
    **那流动性提供者（LP）是如何赚钱的？**
    
    -   当你把资金存入流动性池时，你成为了流动性提供者（LP）。每当别人进行交易时，你提供的资金就会被用来进行兑换。
        
    -   LP 在每次交易中都会收取一定的 **交易手续费**（我们以 0.3% 为例）。这些手续费会根据你在流动性池中提供的资金比例进行分配。
        
    
    **举个例子：**
    
    -   假设你存入了 1 ETH 和 5000 USDC 到一个 ETH/USDC 的流动性池。如果别人用 1 ETH 交换 USDC，那么这个交易会产生 0.003 ETH 的手续费（0.3%），实际到手约 453.31 USDC（有效输入 0.997 ETH）。
        
    -   **具体收益计算**：
        
        -   手续费是以输入代币计价（这里是 ETH），直接留在池子里分给 LP
            
        -   交易手续费：1 ETH × 0.3% = 0.003 ETH，按照 500 USDC / ETH 的价格，手续费价值 1.5 USDC
            
        -   如果你占池子 10% 的流动性份额，你获得的手续费收入：0.003 ETH × 10% = 0.0003 ETH
            
    -   当然，实际情况会因为池子大小、交易量、市场波动等因素而不同，这里只是一个简化的例子。
