Alchemy「Road to Web3」第二周的导入与实操核心，主题是 **DeFi（去中心化金融）与一个完整的“Buy Me a Coffee”DApp 示例**。讲者首先说明背景：DeFi 被视为一种用**智能合约替代传统银行和金融机构**的技术路径。虽然这是一个宏大的说法，但在课程中会通过非常具体、可运行的代码示例来理解其最基本的原理：用代码自动化资金流动，并在此基础上构建应用。

课程一开始对学习者的前置条件做了预期管理：如果熟悉编程语言会非常有帮助，尤其是 JavaScript，但即便只理解变量、循环、类这些基础概念，也足以跟上。课程设计是“照着做就能跑起来”，同时 Alchemy 社区（Discord、Telegram、Twitter）会提供支持，强调这是一个**可复制、可扩展、以实践为导向的学习路径**。

在解释 DeFi 之前，讲者先澄清了“加密货币到底是什么”。它并不是实体的金属币，而本质上是一个**公开、分布式的账本**。通过 ERC-20 合约示例说明，核心状态其实就是一个 balances 映射：把“地址”映射到“余额”。当一笔交易发生时，区块链会记录事件、更新状态，例如一个地址余额减少，另一个增加。这个账本是去中心化维护的，没有任何单一控制者，所有节点共同验证。这种结构的关键意义在于：**因为它是软件构建的公共账本，所以可以在其上层构建非常复杂而自动化的金融逻辑**，这正是 DeFi 的基础。

随后课程明确本周的学习目标：从零编写一个自定义智能合约，允许他人用 ETH 给你“买咖啡”、附带留言，并允许合约部署者提取收到的打赏。工具链包括 Hardhat 与 ethers.js（智能合约开发与交互的主流工具），前端将使用 Replit 快速搭建，最终实现一个可在测试网上被他人真实交互的网站。同时，课程属于一个为期十周的学习计划，每周完成挑战可以获得不可转让的 Proof of Knowledge NFT，作为学习轨迹的链上凭证。

在正式编码前，讲者展示了最终 Demo：一个类似 “Buy Me a Coffee” 的网页应用，但资金流转完全通过加密钱包和智能合约完成。用户连接 MetaMask 钱包后，可以输入名字和留言，用固定数量的 ETH 发送打赏。交易被确认后，留言会从区块链读取并显示在页面上，附带时间戳。这一演示强调了几个关键点：前端通过钱包发起交易，智能合约接收 ETH 并记录数据，交易被区块链确认后，状态变更即可被读取并展示。

接下来是整体架构说明：系统分为前端网站与“后端”的智能合约层。虽然它不等同于传统 Web2 的后端，但所有业务逻辑、资金托管、数据存储都发生在合约中。合约的核心功能包括：接收咖啡打赏、存储并返回留言列表、以及只允许合约部署者提取合约中的 ETH 余额。

在环境搭建部分，课程强调这是“最痛苦但必要”的阶段。使用 npm 作为包管理工具，通过 nvm 管理 Node.js 版本，确保开发环境一致。推荐的版本是 Node 16.13.1 和 npm 8.x，以避免调试问题。所有步骤和版本信息都可以在 Alchemy 文档中查到。

随后进入 Hardhat 项目初始化流程：创建项目目录、npm init 生成 package.json，使用 npx hardhat 创建基础示例项目。Hardhat 会自动生成 contracts、scripts、tests 目录以及配置文件，并安装 ethers、waffle 等依赖。讲者详细解释了目录结构的意义：contracts 存放 Solidity 合约，scripts 用于部署和交互脚本，hardhat.config.js 控制 Solidity 版本与网络配置。

核心部分是 **BuyMeACoffee.sol 智能合约的编写**。原有示例合约被删除，改为自定义合约。合约中首先定义了一个 Memo 结构体，用于保存每一笔咖啡打赏的信息，包括发送者地址、时间戳、昵称和留言内容。同时定义了一个事件 NewMemo，用于在链上记录日志，方便前端或脚本监听。合约还保存了一个 payable 的 owner 地址，用于记录部署者身份，以及一个 Memo 数组，用来存储所有留言。

构造函数在合约部署时执行一次，将部署者地址保存为 owner。由于需要向该地址转账，必须显式将其声明为 payable 类型。核心函数 buyCoffee 被声明为 public 和 payable，允许任何人调用并附带 ETH。函数内部首先通过 require 校验打赏金额必须大于零，然后创建 Memo 结构体并存入数组，同时触发 NewMemo 事件。资金不会立即转给 owner，而是累积在合约余额中。

合约还实现了两个辅助函数：withdrawTips 用于将合约内的全部 ETH 转给 owner，getMemos 用于返回所有留言。前者虽然是 public，但无论谁调用，资金都会发送给 owner；后者是 view 函数，只读取链上状态，不消耗额外 gas。通过这一设计，课程完整覆盖了 Solidity 中的状态变量、结构体、事件、构造函数、访问修饰符、payable、以及链上存储与日志的基本概念。

在合约完成后，课程通过一个 Hardhat 脚本（buy-coffee.js）演示如何与合约交互。脚本首先定义了一些辅助函数，用 ethers.js 读取地址余额并格式化输出，以及读取并打印链上存储的 Memo 数据。主流程中，脚本获取 Hardhat 提供的测试账户，部署合约，打印初始余额，然后模拟多个账户调用 buyCoffee 发送打赏，再检查余额变化，最后调用 withdrawTips 并验证资金成功回到 owner。整个过程清晰展示了：**链上状态变化、合约余额变化、事件与数据读取之间的对应关系**。

讲者特别强调，这个脚本与未来前端调用合约的方式在逻辑上是高度一致的，只是账户来源和调用环境不同。理解这一点，意味着你已经掌握了 Web3 DApp 交互的 80% 核心概念。

最后一部分转向真实测试网部署准备。课程引导学员创建 MetaMask 钱包、注册 Alchemy 账户、配置 Goerli 测试网络 RPC，并通过水龙头领取测试 ETH。Alchemy 在这里作为区块链节点服务提供商，承担了 DApp 读写链上数据的基础设施角色。完成这些配置后，编写一个 deploy.js 脚本，将合约部署到 Goerli 测试网，使其可以被互联网上的其他人真实访问和交互。

Greenpills：[https://www.youtube.com/watch?v=T6Rp-9zaagQ](https://www.youtube.com/watch?v=T6Rp-9zaagQ) 这期播客来自 Greenpill Network 的《How to DAO》系列，嘉宾是 **Liberland 的创始人 Vít Jedlička**，一个试图在现实世界中创建“链上国家 / 初创国家（startup country）”的人。整个对话围绕着一个核心问题展开：**如果我们真的相信自由市场、最小政府和链上治理，那国家是否也可以被重新设计？**

Vít 的个人经历深刻影响了他的政治理念。他出生在前捷克斯洛伐克，在共产主义体制下成长。他认为，那四十年的共产主义对社会、经济和人的幸福感造成了巨大破坏。作为对比，他反复提到新加坡：在同一代人的时间跨度里，新加坡从远比捷克斯洛伐克贫穷，变成了人均收入高出数倍的国家。这种差异在他看来并不是文化问题，而是**治理结构、制度设计的直接结果**。

他由此得出一个结论：**人们在治理自己这件事上，往往会做出错误的选择，民主也未必总是最佳解法**。问题不只是“选谁当领导”，而是“系统本身是否鼓励创造财富、自由交换和长期增长”。

在长期推动改革、做公共传播（如他在 2010 年创立的 Reform CZ）后，他意识到，仅靠教育和舆论改变社会是一个跨世代的问题，个人生命长度不足以完成这种转变。于是，他选择了一条更激进的路径：**不再试图说服一个现有国家改变，而是直接创建一个新的国家实验**。

Liberland 正是在这种背景下诞生的。它位于克罗地亚和塞尔维亚之间一块 7 平方公里、双方都未正式声索的土地上。Vít 强调，Liberland 不是“微型国家的玩笑”，而是一个认真对待国家构建的项目。他区分了“nation”和“state”的概念：民族可以没有国家，国家的存在也并不依赖他国承认。根据《蒙得维的亚公约》，只要具备明确领土、常住人口、治理能力和对外交往能力，一个国家在法律上就已经成立。

Liberland 的核心理念是**极小政府 + 自愿税收 + 强个人自由**。政府只负责司法、安全和外交，其余事务全部交给市场和私人部门。但它并不是简单复制美国建国时期的模式，而是试图在制度上更进一步。Vít 认为，传统民主的一个根本问题在于“每个人一票”，不论其对社会的贡献如何。他提出一种结合了共和制、民主制与公司治理的混合模式。

在 Liberland，公民可以否决法律、罢免官员，但不能随意通过新法律来扩大政府权力。治理更像一个“只允许删减、不允许膨胀”的系统。同时，Liberland 引入了一种类似“功绩制股权”的机制：对国家贡献更多的人，在治理中拥有更大的话语权。这在他看来比形式上的平等更接近长期公平。

最关键的一点是，Liberland 并不是把这些理念停留在纸面上，而是**直接把国家治理写进了区块链**。他们基于 Substrate 构建了一条定制区块链，把议会、参议院、总统、总理等治理结构，直接编码进智能合约。选举、投票、权力制衡都在链上完成，选举成本低到几乎可以忽略，而且安全性远高于传统系统。

Vít 认为，一个主权国家就应该拥有主权软件，因此他们没有直接部署在以太坊或 Tron 上，而是选择了自己的链。公民运行节点，本身就是国家运作的一部分。这使 Liberland 成为他所说的第一个“去中心化自治政府（Decentralized Autonomous Government）”。

在播客后半段，对话延伸到了 Justin Sun（孙宇晨）担任 Liberland 总理这一话题。Vít 并不回避 Tron 的高度中心化问题，但他认为 Tron 本身已经是一个“准主权实体”，拥有类似外交能力的影响力。对于 Liberland 来说，这种资源、资本和全球网络的引入，是一个现实且策略性的选择。他更看重的是，这种合作能否推动“用区块链治理国家”这一理念被更多现实世界采纳。

当讨论到“其他国家是否能采用区块链治理”时，Vít 提出一个非常 Greenpill 式的路径：不要试图整体推翻现有国家，而是先在国家内部划出特殊经济区，用类似 Liberland 的治理模式运行。这些区域可以快速积累财富，并通过溢出效应反哺原有国家。他认为，真正的阻力不在技术，而在既得利益结构。

在谈到 Liberland 是否是“微型国家”时，他明确拒绝这个标签。在他看来，微型国家通常不被严肃对待，而 Liberland 更像一个“初创国家（startup country）”。目前已有数十万人申请成为 Liberland 的国民，尽管实际公民人数仍较少，但国家构建的方向已经不可逆。

播客最后给出了明确的行动路径：任何人都可以通过官网或 App 申请成为 Liberland 的居民或公民。你可以通过持有一定数量的“功绩点”参与治理、运行节点，也可以选择实际居住、建设这个国家。对很多人来说，这不是为了投票，而是为了过一种更自由、更少摩擦的生活。

### **Week 2 - Day 5: Solidity 实践与 Scaffold-Eth 入门笔记**

本节主要围绕 Solidity 的基础知识复习以及 Scaffold-Eth 工具的使用展开。课程开始时，Austin、Etta 和 Carlos 对前一天内容进行了简短回顾：前一节课使用 Hardhat 写 Solidity 并运行测试，而今天的重点是通过 Scaffold-Eth 进行 Solidity 的实践和迭代，从而更直观、更快速地观察合约部署与前端交互效果。

**Scaffold-Eth 安装与初始化**

首先，使用 Scaffold-Eth 的安装流程。通过搜索 Scaffold-Eth 并访问 GitHub 仓库，将其克隆到本地机器。Etta 展示了从头开始的安装流程，包括在终端中进入工作目录、创建项目文件夹并运行 yarn install。由于安装耗时较长，可先让安装运行，再由其他成员展示已安装完成的版本，以便同时进行实践和讲解。

安装完成后，运行 yarn chain 启动本地 Hardhat 节点，再运行 yarn start 启动前端界面。随后通过 yarn deploy 将 Solidity 合约部署到本地节点。如果需要强制重新部署合约，可使用 yarn deploy --reset。部署完成后，前端会显示合约地址，用户可以直接在 UI 中进行交互。相比 Hardhat 的开发模式，Scaffold-Eth 提供了更加可视化和即时的反馈循环，便于快速迭代合约逻辑。

**Solidity 基础实践与前端交互**

在 Scaffold-Eth 中，课程从基础 Solidity 概念开始练习，包括全局变量（如 block.timestamp）、数据类型（bool, uint256, address）和函数定义。示例中创建了一个简单的 timey 只读函数，用于返回当前区块时间，展示如何通过前端调用 Solidity 函数。同时引入了布尔变量与 toggle 函数，通过前端触发交易实现布尔值翻转，从而直观理解状态变量和函数的交互。

接着，课程讲解了 Burner Wallet 的使用。在 Scaffold-Eth 中，用户可以在不同浏览器窗口生成临时钱包，每个钱包可从水龙头（faucet）领取测试 ETH。利用这个机制，可以快速模拟多个账户，进行权限测试和交易验证。例如，设置一个 boss 地址作为管理者，仅允许其调用特定函数，并使用其他临时钱包验证权限限制。

**部署脚本与构造函数**

进一步介绍了如何通过部署脚本传入构造函数参数，实现合约初始化。示例中创建了 boss 和 deployer 两个地址，分别记录管理者和部署者。利用 immutable 修饰符确保 boss 地址不可更改。此外，通过设置 payable 函数并检查 msg.value，实现对交易的付费控制，前端自动更新交易界面，体现了 Solidity 与前端的联动机制。为了进一步练习，课程演示了动态价格机制：创建 price 变量，每次调用设置函数时增加 1%，体现了智能合约中简单的经济逻辑设计。通过 Scaffold-Eth 前端，可即时观察变化，无需编写大量测试代码。

**Mapping 与 Token 示例**

课程继续介绍 Solidity 的 mapping 类型，用于存储地址与数字的映射关系，例如记录每个账户的操作次数。通过 Scaffold-Eth 前端，可以实时查看映射数据的变化。随后，课程演示了构建简易 Token，包括：

1.  初始化每个地址的余额（mapping(address => uint256) balances）。
    
2.  实现 transfer 函数，实现代币在账户之间的转移。
    
3.  检查余额和溢出情况（Solidity 8 版本内置 SafeMath，无需额外导入库）。
    

通过多钱包测试，演示了代币从初始账户转移到其他账户的完整流程，包括交易失败场景（余额不足）。

**事件（Events）与前端显示**

课程还介绍了 Solidity 事件的使用。通过在合约中触发 SetPurpose 事件，前端通过事件监听器捕获并显示事件数据，从而实现前端与链上状态的同步。示例展示了如何在 UI 中输入新数据、调用合约函数、捕获事件并更新界面，强化了事件驱动的前端交互理念。

**高级迭代与现有合约复用**

在掌握基础操作后，课程演示了如何将已有合约（如上节课的 Samford Token）复制到 Scaffold-Eth 中进行测试。只需粘贴合约内容，部署后前端即可自动识别并生成交互界面。通过这种方式，开发者可以快速验证不同合约逻辑、调试功能，而不必从零开始编写前端。此外，还介绍了所有权模式的设计思路。通过在合约中设置 boss 地址或继承 OpenZeppelin 的 Ownable 模块，实现权限控制。使用 Scaffold-Eth 前端，可快速测试“只有老板能执行”的函数，迭代过程直观且高效。

**Workflow 总结**

总体上，本节课强调了 Scaffold-Eth 在 Solidity 开发中的优势：

-   快速迭代：修改合约逻辑后，前端立即可见，减少测试和调试时间。
    
-   前端联动：内置 UI 和事件监听机制，使链上操作可即时反映。
    
-   多账户模拟：Burner Wallet 允许快速创建多个临时账户，用于权限测试和交易验证。
    
-   灵活实验：可随意复制、修改合约内容，验证各类 Solidity 功能，如 mapping、事件、支付控制、构造函数参数等。
    

与 Hardhat 的开发模式相比，Scaffold-Eth 更适合初学者快速上手和实验。Hardhat 更适合在确定合约逻辑后进行测试驱动开发（TDD），而 Scaffold-Eth 则提供了“先玩后测”的快速迭代路径，使开发者能够更直观地理解 Solidity 的基本概念和合约行为。

# **Ethereum 开发挑战笔记（DEX、Multi-sig、SVG NFT）**

### **一、DEX（去中心化交易所）挑战**

### **1\. DEX 基本概念**

-   DEX 是智能合约运行的交易平台，允许用户在无需中心化中介的情况下交换 ERC20 代币。
    
-   核心机制是 **自动做市（AMM, Automated Market Maker）**：通过代币储备比例来计算价格。
    
-   交易会产生 **滑点（slippage）**：当交易量大时，价格会偏离理论值。
    
-   用户可 **提供流动性（LP, Liquidity Provision）**，存入代币对，获得交易手续费分成，并获得 LP 代币代表自己的池份额。
    

### **2\. DEX 操作流程**

1.  选择代币对，例如 ETH ↔ Balloon（测试代币）。
    
2.  确认连接钱包（MetaMask 或 Burner Wallet）。
    

进行 **代币交换（swap）**：

-   输入交易数量，系统根据储备比计算输出。
    
-   注意交易过大可能严重滑点。
    

**提供流动性：**

-   存入等值的两个代币（例如 ETH 和 Balloon）。
    
-   LP 代币记录占比。
    
-   可以随时取回并收取交易手续费
    

### **3\. 开发学习点**

-   理解储备比例和价格公式。
    
-   熟悉 approve → swap 的代币授权模式。
    
-   流动性池越大，交易滑点越小。
    

可以尝试修改智能合约：

-   调整手续费。
    
-   修改储备比例（50/50、25/75 等）。
    

-   高级 DEX（如 Uniswap v3）引入 **集中流动性（concentrated liquidity）**，在指定价格范围提供流动性。
    

### **4\. 学习资源**

-   Telegram 群组提供 starter kits、代码仓库、示例 DEX。
    
-   查看其他人的 DEX 实例，理解不同实现方式。
    
-   Hyperstructure 概念：智能合约自动运行，去中心化，无需中央控制。
    

### **二、Multi-sig（多签钱包）挑战**

### **1\. Multi-sig 基本概念**

-   多签钱包要求 **多个签名才能执行交易**。
    
-   形式：m-of-n，例如 2-of-3 表示至少两个签名才能执行。
    

安全性：

-   单个私钥泄露无法操作钱包。
    
-   常用于 DAO、基金会等管理大量资金
    

### **2\. Multi-sig 实操**

创建多签：

-   设置签名者地址（owners）。
    
-   设置阈值（threshold）m-of-n。
    
-   部署智能合约。
    

提出交易：

-   例如向 Vitalik 转账 100 USD 测试。
    
-   生成交易并等待签名。
    

签名

-   可以使用 hash 签名或 EIP-712 签名。
    
-   两个签名确认后，执行交易。
    

Wallet Connect：

-   多签钱包可与其他 dApp 交互。
    
-   通过 Wallet Connect 提交交易，支持多操作调用（multi-call）
    

### **3\. 开发学习点**

-   熟悉 **transaction struct**、签名验证和执行流程。
    
-   理解 **ecRecover** 的签名恢复机制。
    

UI/UX 体验：

-   交易签名和执行分离。
    
-   多签操作相对慢但更安全。
    

现实应用：

-   DAO 提案执行。
    
-   多人管理资金池。
    
-   高级多签可扩展到 5-of-12、12-of-20 等复杂配置。
    

-   **三、SVG NFT 挑战**
    

### **1\. SVG 基本概念**

-   SVG = Scalable Vector Graphics，可缩放矢量图。
    

与传统 NFT 不同：

-   图片直接在智能合约内生成，而非存储在 IPFS 或服务器。
    
-   动态可编程，可通过智能合约修改属性（颜色、形状等）。
    

优势：

-   可组合性强（composable）。
    
-   高分辨率，无论缩放大小都清晰。
    
-   灵活实现 on-chain 游戏逻辑（如 breeding）。
    

### **2\. SVG NFT 实操**

1.  创建 ERC721 智能合约。
    

编写 tokenURI 函数：

-   返回 JSON metadata。
    
-   在 image 字段中直接嵌入 SVG 代码。
    

SVG 动态生成示例：

-   颜色、形状可通过 Solidity 逻辑计算。
    
-   可实现合成 NFT（breeding）。
    

随机性处理：

-   使用区块哈希 + hash 函数生成随机值。
    
-   注意 Solidity 中 randomness 可预测，需要加入 unique ID 防止重复
    

### **3\. 学习点**

-   Solidity 处理字符串和 Base64 编码较低级，需熟练。
    

可将智能合约设计为互动性 NFT：

-   例如“Loogies”可混合颜色生成新 NFT。
    
-   “40s NFT”可通过使用和回收实现公共基金奖励。
    

实践中注意安全：

-   避免 predictable randomness。
    
-   限制 mint 时间以降低攻击风险
    

### **4\. 实例与参考**

-   Sum for Stout NFT。
    
-   Loogies。
    
-   Neolastics（Simon DLR）。
    
-   Ava Stars。
    

### **四、总结**

**DEX**

-   学习自动做市、流动性提供、滑点机制。
    
-   理解 approve → swap 流程。
    
-   可通过修改合约掌握 AMM 定价公式。
    

**Multi-sig**：

-   学习交易签名、验证、执行机制。
    
-   掌握多签钱包安全优势及应用场景。
    
-   可通过 Wallet Connect 与 dApp 联动。
    

**SVG NFT**：

-   学习 on-chain metadata、SVG 编程。
    
-   理解合成 NFT 和随机性实现方法。
    
-   可设计高度可组合性 NFT。
    

**核心技能点**：

-   Solidity 开发（ERC20、ERC721）。
    
-   智能合约与前端交互。
    
-   Web3 基础操作：钱包、签名、交易、部署。
    
-   安全与可组合性考虑：slippage、randomness、multi-signature。

**为什么很多品牌在 Web3 听起来很 cringe / 失败**

**📌 核心观点资源与讨论**

🔹 品牌进入 Web3 很容易失败，有几个常见原因：

-   误解社区文化：品牌把 Web3 当作技术工具或营销噱头，而不是社区参与空间，导致活动缺乏真实互动。
    
-   技术与文化理解差距：品牌团队缺乏链上机制、tokenomics、去中心化文化的理解，结果出现 mint 流程复杂、无法融入社区对话等失误。
    
-   没有持续参与机制：许多品牌发完 NFT 或做完一次 campaign 就离开，社区认为这只是“割韭菜式营销”，损害信任。
    
-   Narrative gap（叙事差距）：项目真正做的事情与外界理解的 narrative 之间存在巨大鸿沟，有能力的技术/产品没能讲好故事。
    

这些因素叠加起来，就会让营销显得不真实或“尬”，被社区视为“空洞噱头 + corporate speak”。

**2)Culture Fit vs Product Fit：哪个更重要在 Web3？**

这是一个在营销/叙事圈常被提及但少被系统阐述的辩题，

**观点 A — 在 Web3 文化中，Culture Fit 是基础条件**

Web3 本质是去中心化文化 + 社群参与，所以：

-   产品好不如能建立文化共识
    
-   社区规则/价值决定参与度
    
-   narrative（故事）的主人不是品牌，而是持有者/DAO
    

这点被许多长期从业者在 Reddit & 公众号中提到，例如：

“Web3 marketing isn’t growth hacks; it’s building trust from scratch” — 社区是 co-owners，不接受传统 funnel 模式。

“co-creating brand culture” — Web3 branding 其实强调参与和自我表达，而不是统一品牌口号。

**🔹**

**观点 B — Product Fit 仍是必要条件，但不足以取胜**

-   很多 Web3 项目技术很先进，但因为公开 narrative 不清晰，它们仍然失败或被低估。
    
-   Narrative gap 往往是 产品能力与市场/文化理解之间的鸿沟。
    

**🔹融合视角：Culture Fit + Narrative Fit > 纯 Product Fit**

在 Web3 中：

产品是功能性的入口线，文化是参与性的入口层； narrative 是连结用户与产品意义的桥梁。

没有文化和 narrative 的产品可能会短期增长，但缺乏留存、社区共识和长期扩张。

这恰好可以是圆桌讨论的一个逻辑线：

-   为什么有些产品即使技术好也没起量？
    
-   哪些文化因素（价值共享、身份表达）驱动了用户忠诚？
    
-   在什么场景下 Culture Fit 能超过 Product Fit？

## **📌 在 Telegram 上建组的注意事项（适合运营/社群方向）**

### **1\. 选择合适的**

### **群组类型**

-   Telegram 提供“普通群组”和“超级群组”两种模式，其中**超级群组**可以支持更多成员（最多 200,000 人）、置顶消息、细粒度权限和更强的管理工具。
    
-   如果你的社群目标是增长和公开互动，一般建议直接用 **超级群组**。
    

### **2. 明确定位与规则**

群组名称、头像和简介要清晰说明主题和目标受众，这直接影响用户是否愿意加入和停留。

-   设定清晰的群规，例如：禁止广告/刷屏、讨论范围、语言规范等，能有效避免混乱。
    
-   使用固定置顶消息说明规则与新手引导。
    

### **⚙️ 3.权限与管理策略**

-   合理分配管理员权限，将不同职责分配给不同管理员（发布公告、审核成员、内容审计等）。
    
-   可以设置加入审核、慢速模式等机制减少垃圾信息，保持讨论质量。
    

### **🤖 4. 善用 Bots 与自动化工具**

可以使用 Telegram 机器人来：

-   自动欢迎新成员
    
-   自动发布规则或 FAQ
    
-   过滤垃圾消息
    
-   提供投票、问答等交互功能
    
-   这样的自动化不仅提升效率，也帮助维护长期活跃。
    

### **📈 5. 内容策略与活跃度提升**

-   定期发布有价值内容（比如 AMA、话题讨论、每日问答）能提高成员参与感。
    
-   可使用投票、话题分区（Topics）、贴纸互动等玩法提升参与度。
    
-   内容不能只有“发布”，需要引导讨论和用户反馈。
    

### **🌍 6. 用户体验与沟通策略**

-   新成员进群时建议发送欢迎信息，并附带加入指南、规则说明等。
    
-   对于大型群组，定期进行内容回顾或数据分析（活跃用户、互动率等）来优化节奏。
    

### **🔒 7. 隐私与安全设置**

-   Telegram 提供细致的隐私控制，如隐藏电话号码、控制谁能添加成员等。合理设置这些可以保护用户隐私和群组安全。

这是以太坊「成为 Power User」第一周的第四天，主题是 **NFT**。这一天的目标不是从技术原理开始，而是像真实用户一样去 **买、铸造（mint）、发送和查看 NFT**，通过实际操作来理解 NFT 在 Web3 中是如何工作的。课程延续了之前的逻辑：钱包（MetaMask）、身份（ENS）、资产（NFT）会在不同 Web3 服务之间自然流动。

课程一开始回顾了已经完成的步骤：安装并使用 MetaMask、进行过 ETH 与 DAI 的兑换、完成过链上交易。这些都是今天操作 NFT 的基础。接着通过 OpenSea 这个 NFT 市场，展示了一个重要概念：**你的 Web3 身份和资产会跨平台存在**。即使是 ENS 域名，本质上也是一个 ERC-721 NFT，它会出现在你的钱包和 OpenSea 的资产列表中。如果把 ENS NFT 转给别人，对方就会真正拥有并控制这个域名，这一点和 Uniswap 的 LP NFT（代表流动性所有权）是同样的逻辑。

在介绍 NFT 时，课程强调了 NFT 的核心价值并不只是“图片”，而是 **链上可验证的所有权与来源（provenance）**。艺术家在链上铸造作品，任何人都可以验证这是原始合约、原始作者。即使别人复制一模一样的图片重新 mint，市场也会把它视为“仿品”，因为合约地址和历史不同。这种“真实性”与“稀缺性”来自合约中设定的最大供应量（例如 10,000 个），即便原作者重新部署一个新合约，市场通常也不会承认它的价值。

同时也明确指出，NFT 世界是一个\*\*完全无许可（permissionless）\*\*的环境，任何人都可以部署合约、发行 NFT，因此垃圾项目和投机行为不可避免。用户需要自己判断，而不是默认“下一个蓝筹”。在 OpenSea 中还能看到 NFT 的属性（traits）和稀有度比例，这些属性数据通常存储在 metadata 中，构成 NFT 价值的一部分。

NFT 不只是艺术品，也可能具有 **utility（功能性）**。课程举了几个例子，比如 token-gating（只有持有 NFT 才能进入某个 Discord 或聊天室）、治理投票（持有 NFT 才有投票权）、链上游戏资产（马、鸡、道具等）。这些 NFT 本质上是“钥匙”或“权限凭证”，而不仅是收藏品。不同 NFT 所在的链也可能不同，例如以太坊主网或 Polygon（L2），这引出了对 Layer 2 的简单提示，但当天主要仍聚焦在以太坊主网。

接下来课程选择了一个具体项目，演示 **在项目官网直接 mint NFT**，而不是在二级市场购买。通过连接 MetaMask，用户向一个智能合约发送交易，支付 mint 费用和 gas。这里重点强调了一点：在 Web3 中，用户是在“向合约打钱”，而合约代码是公开的。通过 Etherscan 可以查看合约地址、交易记录，甚至直接阅读 Solidity 源码。虽然普通用户不会逐行审计代码，但理解“你在信任一段公开代码”是非常重要的 Web3 认知。

NFT mint 成功后，虽然是在项目官网铸造的，但因为遵循 ERC-721 标准，它会自动出现在 OpenSea、Zapper 等其他平台中，再次印证了“资产与身份跨平台流动”的概念。不过课程也展示了现实中的不完美：不同平台对 NFT 的 **索引和展示是有选择性的**。有些市场（如 Foundation、Rarible）是高度策展和中心化的，它们可能不显示你合法拥有的 NFT，或者不解析 ENS。这引出了一个重要讨论：**Web3 的协议层是去中心化的，但很多应用层仍然是 Web2 公司，存在展示层面的“软审查”**。Web2 与 Web3 并不是非此即彼，而是混合存在。

随后课程深入到 NFT 的底层结构，对比了 **on-chain NFT** 和 **IPFS NFT**。大多数 NFT 实际上并不是把图片直接存到链上，而是：

-   链上合约只负责记录“谁拥有哪个 token ID”
    
-   token ID 对应一个 tokenURI
    
-   tokenURI 指向 IPFS 上的 metadata（JSON）
    
-   metadata 再指向图片、SVG 或其他资源
    

通过 Etherscan 的 readContract 功能，课程一步步演示了如何查询 tokenURI，再通过 IPFS 网关查看 metadata 和图片。这里引出了 IPFS 的核心概念：**内容寻址（content-addressed）**。在 IPFS 中，地址是内容的 hash，只要内容发生任何变化，hash 就会完全不同。这意味着：只要地址不变，内容就是不可篡改的；如果内容变了，它就“不是同一个东西”了。

这也带来了对 NFT 风险的讨论。如果 metadata 或图片只是指向某个普通 Web 服务器，创作者是可以随时替换内容的，NFT 的“不可变性”就被破坏了，这也是常见的 rug 风险之一。相比之下，把内容放在 IPFS（并确保被 pin）或直接 on-chain，会更接近 NFT 的精神。

课程还通过 Loot 和 SVG NFT 的例子，说明 **on-chain NFT 的可组合性（composability）**。当属性、颜色、形态都直接存储在合约中时，其他合约就可以读取并基于这些属性进行二次创作，例如繁殖、升级、游戏交互等。这种“无需许可的组合”是 Web3 非常独特的能力。

在技术层面，课程最后用一个极简的 Solidity 对比解释了 **Fungible Token（ERC-20）** 和 **Non-Fungible Token（ERC-721）** 的本质区别。ERC-20 是通过“地址 → 余额”的映射来表示价值，转账只是数字的加减；ERC-721 则是通过“token ID → owner”的映射来表示所有权，每一个 token 都是独一无二的。所谓“发送 NFT”，并不是物理传输，而是全网共同更新一份“谁拥有它”的账本状态。

最后回到一个重要的认知转变：区块链不是“资产在网络中移动”，而是**所有节点同时更新一份状态账本**。你签名一笔交易，网络确认后，所有人都同意“所有权发生了变化”。理解这一点，是从 Web2 思维转向 Web3 思维的关键一步。

这一天的核心并不是“买了什么 NFT”，而是通过完整链路理解：NFT 是如何被铸造的、如何在不同平台被识别、它真正“存在哪”、你究竟拥有什么，以及哪些地方仍然存在 Web2 式的中心化与风险。

**How to set up a wallet and become a power user on Ethereum**。这一节的核心目标不是写代码，而是**把钱包真正用起来**：理解 wallet 是什么、seed phrase 是什么、安全模型是怎样的，以及如何开始像一个“power user”一样使用 Ethereum。

课程从 **MetaMask** 开始。MetaMask 被明确介绍为 _go‑to browser extension_，但也提前给了一个重要的免责声明：**MetaMask 是 security‑first，而不是 UX‑first**。这意味着 Web3 世界里很多操作会显得“clunky”，但这些复杂性大多是为了安全服务的。安装时一定要确认官网是 [metamask.io](http://metamask.io)，防止 phishing（如 fake 域名 mediamask 之类）。

创建钱包时最关键的概念是 **12‑word seed phrase / mnemonic / secret recovery phrase**。它被反复强调为 **master key**：这 12 个单词按顺序决定了你整个钱包的控制权。任何人一旦拿到这 12 个词，就可以在任何地方 **recreate your entire wallet** 并立即清空资产。与之相对的是 MetaMask 设置的 **local password**，它只是用来在本地设备上加密钱包数据，如果忘了可以通过 seed phrase 恢复，所以它的安全等级远低于 seed phrase。关于 seed phrase 的保存方式，课程强调的不是“最聪明”，而是“**能恢复 + 足够安全**”。常见错误包括：截图、通过 WhatsApp/Telegram 发送、复制到明文文档、随便存在 GitHub 或文本框里。相对可接受的方法包括：手写在纸上、放在 password manager（如 1Password），但前提是资产规模不大。如果是高价值资产，就需要更高级的方案，比如 **hardware wallet + multisig smart contract wallet**。同时也提醒：不要过度“创意型安全”，比如把 seed phrase 拆得太复杂，最后连自己都无法恢复。

一个非常重要但常被忽视的实践是：**在往钱包里打真实资产之前，先验证你能不能恢复它**。例如删除 MetaMask → 用 seed phrase 重新导入 → 确认地址一致。这一步如果失败，说明你的备份方案本身是有问题的。

在确认 seed phrase 时，MetaMask 要求用户重新按顺序排列 12 个单词。这里顺带引出了一个安全问题：**为什么不能暴力枚举 12 个单词？** 理论上是可能的，但实际组合空间巨大（类似 52 张牌洗牌的例子），即使全世界一起跑脚本，也几乎不可能撞到一个已有余额的钱包。这是密码学安全的基础假设之一。

钱包创建完成后，课程强调了一个 Ethereum 非常反直觉但核心的特性：**创建账户不需要任何许可**。你不是被“注册”进某个数据库，而只是生成了一个足够随机的数（seed phrase → private key → address）。任何人都可以创建账户，这也是 Ethereum 的 permissionless 特性。接下来进入实操：发送 ETH，但首先是在 **testnets** 上完成。课程介绍了 testnet 的作用：它们是“真实运行的 Ethereum 网络”，但 ETH 没有真实价值，主要用于开发和学习。提到的 testnets 包括 **Ropsten、Kovan、Rinkeby、Goerli**，并简单区分了它们的特点（block time、使用场景、未来可持续性）。获取 testnet ETH 的方式是通过 **faucet**。在转账过程中，强调了几个 power user 的基本习惯：发送前要反复核对地址（开头几个字符 + 结尾几个字符），理解 **gas fee** 是发送交易必须支付的成本，并意识到“你需要花 ETH 才能转 ETH”。之后引入了 **Etherscan** 作为 _disinterested third party_ 和 **block explorer**。Etherscan 被定义为 Ethereum 的公共基础设施，用来验证“交易是否真的发生了”。通过 transaction hash，可以看到 from / to、value、gas、block、data 字段；也可以通过 address 页面查看所有交易历史，包括 **internal transactions**（例如 faucet 通过 smart contract 发币）。一个非常重要的 Web3 现实是：**交易不可逆，没有客服**。一旦交易被 mined，就无法撤回。唯一接近“撤销”的情况发生在交易还停留在 **mempool** 时，可以通过发送一个 **same nonce + higher gas** 的交易来 replace / speed up / cancel。但一旦上链，结局不可改变。这一节最后明确了学习成果：到此为止，你已经完成了在 Ethereum 上的“第一步”——你创建了 wallet，理解了 seed phrase 的安全模型，能在 testnet 上发送 ETH，并会用 Etherscan 验证链上行为。接下来要深入的是：**seed phrase 如何派生 private key / public address、ECDSA key pairs、以及一个 seed phrase 如何生成多个 accounts**——这些将是从“会用”走向“真正理解”的关键。

## **二、NFT 核心概念**

### **1\. NFT 是“所有权记录”，不是图片本身**

-   区块链只负责记录：\*\*谁拥有哪个 token｜\*\*真正的内容（图片 / 元数据）通常：
    
-   要么 **存在 IPFS**
    
-   要么 **直接写在智能合约里（on-chain SVG）**
    

👉 真正“拥有”的是：**一个指向内容的不可篡改凭证（receipt）**

### **2\. ENS、本身就是 NFT**

### ENS 域名是 ERC-721 NFT

-   谁持有这个 NFT，谁就控制这个名字
    
-   把 ENS 转走 = 把控制权转走
    

### **3\. NFT 的价值来自三件事**

1.  **出处 / provenance**（谁 mint 的、历史如何）
    
2.  **稀缺性**（合约里写死的 total supply）
    
3.  **社区与共识**（市场是否承认这是“正品”）
    

👉 就算复制一模一样的图片，只要不是原合约 mint，**市场不会认**

## **三、NFT 的几种形态（很重要）**

### **1️⃣ Marketplace 买的 NFT（如 OpenSea）**

-   二级市场
    
-   平台展示 ≠ 链上真实存在
    
-   平台有 **强 Web2 式的筛选 / 审查**
    

### **2️⃣ Mint 网站直接 mint**

-   直接与智能合约交互
    
-   风险更高（要“信任代码”）
    
-   更贴近 Web3 本质
    

### **3️⃣ On-chain NFT（高级玩法）**

-   SVG / metadata 直接写在合约里极贵，但：
    
-   **完全不可篡改**
    
-   高度可组合（适合游戏、机制设计）
    
-   **4️⃣ IPFS NFT（最常见）**
    
-   合约只存 token → URI
    
-   URI 指向 IPFS 上的 metadata
    
-   Metadata 再指向 image
    

## **四、IPFS 的关键思想**

### **Content-addressable（内容寻址）**

-   文件的 **hash = 地址**
    
-   内容改 1 个字节 → 地址就变
    

所以：

-   ✅ 不可篡改
    
-   ✅ 没有“假副本“
    

⚠️ 但现实问题：

-   如果没人 pin，内容可能消失
    
-   需要 Pinata / Arweave 等服务
    

## **五、Web3 的“理想 vs 现实”**

### **理想：**

-   NFT 遵循标准 → 应该在哪都能看到 / 交易
    
-   去中心化、抗审查
    

### **现实：**

-   OpenSea / Rarible / Foundation 都是 **Web2 公司**
    

**Web2 和 Web3 并不是非此即彼，而是强烈混合**

## **六、Fungible vs Non-fungible（技术本质）**

### **Fungible（ERC-20）**

-   记录的是：**地址 → 余额**
    
-   转账 = 数值加减
    

### **Non-fungible（ERC-721）**

-   记录的是：**tokenID → owner**
    
-   每个 token 独一无二
    
-   转移 = 改 owner
    

👉 NFT ≠ “在网上传文件”

👉 NFT = **全网同步更新的一本“所有权账本”**

## **七、艺术家 & 创作者的意义**

NFT 让艺术家可以：

-   直接面对支持者（而不是平台）
    
-   写入 **版税（royalties）**
    

用 NFT 做：

-   门票
    
-   投票权
    
-   社群准入
    
-   游戏资产
    

👉 从「百万粉丝」转向「百个超级支持者」

## **八、重要风险提示**

-   不看 tokenURI = 不知道你买了什么
    
-   NFT 指向个人服务器 = **可被随意替换（rug）**
    
-   未验证合约 = **极大风险**
    
-   发错地址 / 发到合约地址 = **永远找不回**
    

NFT 可以被理解为一种**策展机制而不仅是媒介**：它通过区块链把“所有权、出处与规则”写入一个不可篡改的系统，使作品不再只依赖展墙、平台或机构来获得合法性。在策展层面，这意味着策展人不只是选择作品本身，而是在设计**作品如何被生成、流通、被观看与被记忆的条件**——从智能合约、内容存储方式（on-chain / IPFS），到平台是否展示、社区是否承认，都是策展的一部分。NFT 展览因此更像是在组织一套关于价值、信任与公共性的实验场，而不是简单地把数字图像搬进展厅

_Travel Rule_（旅行规则）要求虚拟资产服务提供商在转账中收集并传递发送方和接收方信息，这对去中心化交易协议提出挑战。

### 安全与合规在 Web3 领域既包括 **法律合规风险**（如代币发行、场外交易、合同效力等），也包括 **网络安全风险**（如钓鱼、木马、社交工程等）。对 Web3 从业者、求职者和参与者来说：

-   理解国内外监管趋势并评估自身行为是否合法是必要前提。
    

**非法金融活动风险**

-   代币发行（如 ICO 等）被认定为非法融资行为；即便以“积分”或“凭证”方式发行也可能触犯法律。
    

**刑事风险类别**

-   使用闭环经济模型（如链游抽奖兑币）可能构成非法赌博；多层级推广则可能构成传销罪。
    

**洗钱与非法经营风险**

-   场外交易和跨境资金绕开监管可能被认定为非法外汇经营、洗钱或协助犯罪。
    

**合同与民商事风险**

-   由于虚拟货币不具备法偿性，与其相关的合同在法律上可能被视为无效

项目合法性审查

•加入项目前应核实其注册地、合法性和合规情况，避免参与非法经营、未经许可的金融活动等。
