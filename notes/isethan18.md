**1.21补充**

**第一部分：Uniswap核心技术**

**1\. 为什么要学习Uniswap？**

Uniswap是DeFi的基石。它解决了传统交易所（如A股）需要“买家和卖家同时在线”才能成交的问题，通过自动化做市商（AMM），让交易可以随时发生。

**2\. V2阶段：零基础必懂的公式**

**1) 核心公式：x\\y=k**

x和y是池子里两种币的数量，k是它们的乘积（在交易时保持不变）。

例子：池子里有10个ETH和1000个USDT（k=10000）。如果你想买ETH，你往池子里扔USDT，ETH变少了，USDT变多了，但乘积必须还是10000。这决定了你买得越多，单价就越贵（这就是价格影响）。

**2) 无常损失(ImpermanentLoss)：**

如果你把币存入池子做流动性（LP），当外面市场价格大涨时，套利者会来池子把便宜的币买走。等你想取出币时，你会发现你的资产总值比直接拿着币不动要少。结论：波动越大，无常损失越高。

**3\. V3阶段：高手的“资本效率”**

**1) 集中流动性(ConcentratedLiquidity)：**

V2的缺点：你的钱分布在0到无限大的价格区间，但ETH可能一年都在2000-3000波动，其他区间的钱都浪费了。

V3的改进：你可以指定只在2000-2500之间提供流动性。在这个区间内，你的钱被“放大”了，赚的手续费比V2多得多。

**2) Tick系统：**

将价格轴切成成千上万个小格子（Ticks）。价格每跳过一个格子，计算方式就会更新。

**4\. V4阶段：无限可能的“钩子”**

Hooks(钩子)：

这是一种插件机制。你可以在交易发生的前一秒或后一秒，自动执行一段代码。比如：自动调整手续费、自动把手续费存入理财、或者根据天气改变交易规则。

1.20补充

**solidity智能合约开发入门**

**一、底层核心：EVM四大存储执行区**

理解“数据放在哪”是节省Gas（交易手续费）的关键，也是Solidity优化的核心。

Storage(链上存储)“持久化的硬盘”

特性：永久保存合约状态，所有节点同步。

Gas成本：极高（初始化写入约20,000gas）。

用途：存储代币余额、合约所有者等必须永久保存的数据。

Memory(运行时内存)“临时的内存条”

特性：仅存在于一次函数调用中，执行完即清空，不上链。

Gas成本：中等。

用途：存放复杂的中间计算过程或临时变量。

Calldata(调用数据)“只读的输入参数”

特性：外部调用时传入的原始参数，不可修改。

Gas成本：极低。

用途：接收函数输入参数。

Stack(栈)“快速计算的收银台”

特性：EVM采用栈式模型（LIFO），处理指令与操作数11。

Gas成本：极低/忽略不计12。

用途：执行$1+1$等底层逻辑运算13。

**二、开发环境与版本规范**

推荐工具：RemixIDE。它是一个浏览器编辑器，提供私有链环境，支持编译、部署和Debug，还能实时观察storage的变化。

版本管理：Solidity更新频繁，建议以0.8.20为基准。

^0.8.20：表示支持0.8.20及以上版本编译。

注意：0.8版本以上默认内置了溢出保护，安全性更高。

**三、数据类型体系**

3.1值类型(ValueTypes)——直接存值，成本低

uint/int：整数类型（如uint8到uint256），位宽越小越省storage空间。

address：用户钱包地址或合约地址。注意：用户地址务必用address类型，不要用string。

bool：true或false。

3.2引用类型(ReferenceTypes)——存的是“引用”，成本高

string：字符串

mapping(映射)：极其重要，常用于设计账本（如：地址余额）

bytes/动态数组：存放不定长的数据

**四、ERC20代币合约设计逻辑**

要发行一个代币，其核心就是维护两本账本：

1.余额账本：记录每个地址持有的数量。

2.授权账本：记录A授权给B可以操作的额度（这是DEX交易的基础）。

关键组成部分：Constructor(构造函数)：

Constructor(构造函数)：部署时执行，用于初始化代币名称、符号、总供应量等参数。

transfer(转账)：核心逻辑为“参数校验-余额更新-触发事件”。

Event(事件)：将状态变更广播给全网，供区块浏览器或链下系统监听。

**五、函数修饰符与安全设计**

5.1可见性与状态修饰符

external：仅能从外部调用。

public：内外皆可调用。

view：只读函数，不修改链上状态。

pure：既不读取也不修改链上状态。

5.2Modifier(修饰符)—逻辑封装

用于抽象权限控制（如onlyOwner）或前置条件校验。

优势：代码复用、减少部署Gas、确保权限逻辑统一。

5.3错误处理机制

require：最常用，条件不满足则报错回滚并返回字符串。

CustomError(自定义错误)：2026年的主流做法，进一步优化Gas消耗。

**六、快速入门避坑指南**

绝不导入不明助记词：骗子常利用抢跑机器人(Front-runningBot)，一旦监测到地址有Gas转入，立即以更高费用抢先转走资金。

重入防护：遵循“检查-生效-交互”模式，即先更新内部余额状态，再执行外部转账。

使用成熟“轮子”：推荐使用OpenZeppelin合约库，它是经过多次安全审计的标准实现。

**第四章：交易机制与 Gas 经济模型 (Transactions & Gas)**

1\. **交易（Transaction）的状态机转换**

专业定义：以太坊是一个“基于状态的复制状态机”。每一笔交易都是一次状态转换函数 Y = F(X, T)，将当前状态 X 改变为新状态Y。

核心字段：

Nonce（随机数）：用于防止重放攻击（Replay Attack），确保每笔交易仅被执行一次，并强制交易按顺序排列。

Payload（负载）：如果是转账，则为空；如果是合约调用，则包含函数选择器及其参数的编码数据。

2\. **EIP-1559：费用市场的改革**

1)Base Fee（基础费用）：由协议根据区块空间需求自动调整，这部分费用会被销毁（Burn），从而给 ETH 带来通缩效应。

2)Max Priority Fee（优先费/小费）：直接支付给验证者（Validator），诱导其优先打包该交易。

3)Gas Limit（燃料上限）：单个交易允许消耗的最大计算步数，防止合约因逻辑漏洞（如死循环）耗尽网络资源。

**第五章：以太坊扩容架构 (Scaling & Layer 2)**

**1\. 模块化区块链 (Modular Blockchain) 愿景**

Layer 1 (L1) 的定位：从“执行层”转变为“结算层”与“数据可用性层（DA）”。

Layer 2 (L2) 的核心逻辑：将昂贵的计算过程迁移至链下，仅将交易的摘要（State Root）和压缩后的原始数据（Data）上传至 L1。

**2\. Rollup 技术路线对比**

Optimistic Rollups (乐观汇总)：

原理：基于欺诈证明（Fraud Proofs）。默认交易合法，设定 7 天挑战期，若无人质疑则最终确认。

代表：Arbitrum, Optimism。

ZK Rollups (零知识证明汇总)：

原理：基于有效性证明（Validity Proofs）。利用数学算法（如 zk-SNARKs）生成证明，L1 验证通过即刻确认，具有极高的安全性和即时性。

代表：zkSync, Starknet, Scroll。

**3\. 以太坊路线图的关键阶段**

The Surge：引入 Proto-Danksharding (EIP-4844)，通过“数据槽（Blobs）”大幅降低 L2 的数据存储成本。

The Verge：引入 Verkle Trees（沃克尔树），优化数据验证结构，实现“无状态性”，让普通设备也能参与网络验证。

**第六章：应用生态与前沿技术 (Ecosystem & Frontiers)**

1\. **智能合约资产协议**

ERC-20：同质化代币标准（如 USDT），每个代币完全相同且可分割。

ERC-721 / ERC-1155：非同质化代币（NFT）标准，代表唯一且不可分割的资产的所有权。

2\. **账户抽象 (Account Abstraction - ERC-4337)**

核心革新：将外部账户 (EOA) 转化为智能合约账户。

应用场景：支持社交恢复（无需私钥即可找回账号）、Gas 签名支付（用代币或由第三方代付 Gas）、多签确认。

3\. **行业新趋势**

RWA (Real World Assets)：通过法律合规框架与链上协议，将传统金融资产（债权、股权）映射为链上代币。

DePIN (Decentralized Physical Infrastructure Networks)：利用 Token 经济模型激励物理硬件（如传感器、算力、存储）的去中心化协作。

**总结：应该掌握的技术逻辑**

1)安全性由 Layer 1（以太坊主网）通过共识算法保障。

2)性能由 Layer 2 通过 Rollup 技术解决。

3)用户体验由账户抽象（AA）和 EIP-4844（低手续费）驱动。

4)价值捕获由 EIP-1559 销毁机制与生态应用的繁荣支撑。

# **Telegram Web3 招聘群市场调研报**

## **一、调研背景与问题定义（Product Owner View）**

### **1.1 调研背景：Web3 招聘基础设施的结构性迁移**

从产品负责人的视角，Web3 招聘并非“是否存在招聘平台”的问题，而是招聘基础设施正在发生迁移的问题。

**Web2 时代，招聘基础设施高度集中于：**

1) LinkedIn（职业身份与关系网络）

2) Boss / Indeed（职位发现）

3) ATS（流程、数据与合规）

**而在 Web3 生态中，真实招聘行为已经明显迁移至：**

1) Telegram / Discord（私域即时沟通）

2) X（个人与项目品牌）

3) DAO 内部社区

**多份行业报告与行为数据表明：**

Telegram 并非过渡形态，而是当前阶段 Web3 招聘的事实基础设施。

### **1.2 宏观数据支撑：为什么是 Telegram，而不是 ATS**

**1)Electric Capital Developer Report（2024）**

全球 Web3 活跃开发者：约 22,000–25,000

远程 / 跨国协作比例：70%+

**2)CryptoJobsList /** [**Web3.career**](http://Web3.career) **行业统计**

Web3 岗位中：

全远程岗位占比 65–75%

合同制 / 项目制岗位占比 40%+

**3)a16z Crypto Hiring Trends（2023–2024）**

超过 60% Web3 项目 表示：

LinkedIn 对 Web3 岗位的转化效率明显低于私域渠道（Telegram / Discord）

**4)Telegram 官方数据（2024）**

加密 / Web3 相关频道订阅用户数：8,000 万+

**产品级含义**

Web3 招聘天然不适配以「长期雇佣 + 重流程」为核心的 ATS，而更依赖低摩擦、即时、私域化的招聘媒介。

### **1.3 核心调研问题（Product Questions）**

Telegram 招聘群的产品本质是什么？是 ATS、Job Board，还是新物种？

它们如何在几乎零 CAC 的情况下持续获客？

它们是否具备可持续的盈利模式？商业天花板在哪里？

## **二、调研假设（Product Hypotheses）**

H1：Telegram 招聘群不是 ATS 的失败形态，而是招聘价值链中“发现层（Discovery Layer）”的最优解

H2：当前 Telegram 招聘群整体仍处于人工或半自动阶段，平台化能力极弱

H3：仅存在 ATS-lite（极轻量流程能力）的机会，且边界极其清晰

## **三、样本选择与方法论**

### **3.1 样本选择原则**

招聘行为是否真实、高频发生

是否形成稳定心智

是否代表一种可复用模式

### **3.2 最终深度研究样本（6 个）**

| 样本 | 代表模式 |
| Web3万事屋 | 高流量非结构化信息流 |
| DeJob 招聘频道 | 平台导流型 |
| Web3 Jobs（英文） | 人工精选 Job Board |
| CryptoJobsList | 成熟商业化平台 |
| LobsterDAO HR | DAO 内部高信任招聘 |
| SmartDeer | Bot 自动聚合 |

## **四、核心判断：Telegram 招聘群到底是什么？**

### **4.1 是否是 ATS？结论明确**

Telegram 招聘群不是 ATS，也不应被当作 ATS 评价。

| ATS 能力 | Telegram 招聘群 |
| 简历结构化 | × |
| 招聘流程管理 | × |
| 多角色协作 | × |
| 数据分析 | × |
| 职位发现 | √ |
| 即时沟通 | √ |

### **4.2 产品本质判断**

Telegram 招聘群 = 去中心化 Job Discovery + 即时撮合系统

它们通过主动放弃后 60% 的招聘流程能力，换取：

1)发布成本 ≈ 0

2)私聊响应率 20–40%（LinkedIn <5%）

3)多数岗位 7–14 天完成初筛

## **五、6 大样本深度拆解（模式 × 获客 × 盈利）**

### **5.1 Web3万事屋**

群规模：3–5 万

日均信息量：150–300 条

招聘信息占比：15–20%

招聘帖有效期：<24 小时

模式：信息流社区

获客：社群裂变（零 CAC）

盈利：广告 / 置顶

结论：招聘为副产物，但验证了流量吸附效应。

### **5.2 DeJob — Web3 招聘频道**

单频道日均岗位：20–40

招聘方复发率：30–40%

岗位生命周期：10–20 天

模式：平台导流型 Job Board

获客：品牌背书 + 多语言矩阵

盈利：$99–$299 / 岗位

结论：Telegram 是入口，而非产品核心。

### **5.3 Web3 Jobs（英文）**

订阅人数：20k+

日均岗位：30–50

岗位重复率：25–30%

模式：人工精选 Job Board

获客：Telegram + X

盈利：$100–$250 / 发帖

结论：稳定现金流，增长天花板明确。

### **5.4 CryptoJobsList（CJL）**

官网月 UV：50–100 万

年岗位数：4–6 万

平均单价：$299

保守年收入估算：

下限：$299 × 10,000 ≈ $3M

上限：$299 × 40,000 ≈ $12M

模式：成熟平台型 Job Board

获客：SEO + Telegram

结论：健康但非指数级模型。

### **5.5 LobsterDAO HR**

DAO 成员：数千级

招聘周期：<14 天

转化率显著高于公开招聘

模式：DAO 内部招聘

盈利：无

结论：效率上限模型，不可规模化。

### **5.6 SmartDeer**

覆盖频道：数十–上百

日均岗位：100+

用户留存显著低于人工群

模式：Bot 聚合

盈利：发帖费 / 广告

结论：规模化但信任快速衰减。

## **六、市场规模与商业天花板**

### **6.1 双重验证市场规模**

**Bottom-up**

活跃招聘群：200–300

单群年岗位：1,000–2,000

年岗位：5–10 万

**Top-down**

CJL / [Web3.career](http://Web3.career) / DeJob 年岗位总量：10 万级

Telegram 为主要分发渠道

**结论一致**

年市场规模：$500 万 – $3,000 万

## **七、失败路径**

结构化过度 → 效率下降

SaaS 化 → 付费失败

流程介入 → 信任断裂

## **八、产品边界声明**

❌ 不做完整 ATS

❌ 不做强简历系统

❌ 不做合规 / 薪资 / 劳动关系

## **九、最终结论**

Telegram Web3 招聘群不是 ATS 的替代，而是 Web3 招聘生态中不可替代的“发现层基础设施”。

唯一成立的产品机会在于：

在不破坏“发帖 → 私聊”这一高效链路的前提下，补充最小必要的秩序与信任。

是否进入该方向，取决于团队是否接受：

中等市场规模

非 SaaS 收入模型

强运营、轻系统的产品形态

Web3 与 AI 融合的前沿技术栈及其构建的“机器经济”蓝图。

**一、 核心演进：从“聊天机器人”到“代理式 AI” (Agentic AI)**

传统 LLM 的局限：传统的模型是“无状态”的，表现为无记忆、无工具使用能力且无独立身份（无法代表实体行动）。

**2026 年的突破：AI 智能体通过以下技术实现质变：**

1)记忆：利用向量数据库实现私有记忆。

2)工具：通过函数调用（Function Calling）或 MCP 获取工具使用能力。

3)身份：通过 ERC-8004 等标准获得链上身份，使其能够自主思考、行动并参与经济活动。

**二、 AI 智能体面临的三大核心挑战**

身份与信任缺失：在传统网络中无法区分请求方是人类还是 AI，存在身份欺诈风险。

行为不可审计：决策日志存储在本地，易被篡改，缺乏第三方验证，导致法律追责困难。

支付基础设施不匹配：传统信用卡支付存在 KYC 障碍、结算慢、不支持高频微支付等问题，无法满足“机器经济”的需求。

**三、 Web3 提供的协议栈解决方案**

Web3 为 AI 提供了不可篡改的身份证明和原生银行账户。

**交易层 (x402 协议)：**

将支付能力嵌入 HTTP 协议。

支持无需注册、实时结算的原生微支付（如按 API 调用付费），解决微任务支付摩擦。

**身份与声誉层 (ERC-8004 标准)：**

1)链上护照：为每个 AI 铸造唯一的 NFT 作为身份证。

2)声誉系统：记录 0-100 的信用分，仅限有过真实支付记录的客户评价，以此防止“女巫攻击”。

3)验证机制：通过 TEE（可信执行环境）或 zkML（零知识机器学习）确保证明“你是谁”且“你可信”。

**四、 Spoon OS：Web3 AI 操作系统框架**

Spoon OS 是连接 Web2 AI 开发能力与 Web3 基础设施的桥梁，旨在降低开发门槛。

**四层架构设计：**

《021学习以太坊》开源教材

**第1章：认识以太坊 (What is Ethereum)**

起源与愿景：以太坊由Vitalik Buterin于2013年提出，2015年正式上线 。它不仅是类似比特币的账本，更是“全球可编程区块链” 。其核心是以太坊虚拟机 (EVM)，使区块链具备了运行智能合约的能力 。

**核心特性**

1)智能合约：存储在链上的自动执行代码，像“自动贩卖机”一样无需中介 。

2)dApps (去中心化应用)：不依赖单一服务器，具有零停机、抗审查、数据完整等特点 。

3)以太币 (ETH)：系统的“燃料” (Gas)，用于支付计算费用，也是PoS机制下的安全保证金 。

共识演进：2022年通过“合并” (The Merge) 从PoW（工作量证明）转向PoS（权益证明），能耗降低约99.95% 。

**第2章：网络结构与节点类型**

节点构成：合并后的完整节点由执行客户端 (EL) 和共识客户端 (CL) 组成，两者通过 Engine API 通信 。

**节点分类**

全节点 (Full Node)：验证所有交易，保存区块数据，但会裁剪老状态 。

1)归档节点 (Archive Node)：保存自创世以来所有历史状态，是链上的“时间机器”，适合审计和深度分析 。

2)轻节点 (Light Node)：仅保存区块头，通过Merkle证明验证数据，适合手机等受限设备 。

3)通信机制：基于UDP的Kademlia算法发现邻居，通过TCP上的RLPx协议建立加密连接，利用\*\*Gossip（八卦协议）\*\*在全网传播交易和区块 。

**第3章：账户类型与结构**

1)EOA (外部账户)：由私钥控制（如MetaMask钱包），用于发起交易，不包含代码 。

2)CA (合约账户)：由代码控制，包含逻辑和状态存储，不能主动发起交易，只能被EOA或其他合约调用 。

3)账户抽象 (AA)：以太坊正在推行的一种趋势，旨在让账户行为更灵活（如Gas代付、社交恢复），提升用户体验 。合规与刑事风险防范总结

Web3 合规与刑事风险防范：从理论架构到实操避险

一、 合规的两个维度：从“形式”到“深度”

形式合规（文本安全）：

1)核心： 在白皮书、合同及协议中，严格使用符合国际监管（如FATF、MiCA）的术语。

2)规避： 严禁出现“融资”、“投资回报”、“保本”、“二级市场操盘”等敏感词汇。

3)底线： 明确风险提示与免责声明。但需注意，形式合规仅能降低表面风险，无法对抗实质性的刑事穿透。

深度合规（业务安全）：

1)物理隔离： 绝不向境内居民提供服务。实操中不仅要屏蔽大陆IP，更要通过KYC拦截大陆证件注册，确保大陆用户“不翻墙无法访问、有证件无法注册”。

2)业务去金融化： 建立真实的业务闭环，项目应具备真实的技术支撑（如DePIN的硬件、AI的算力），避免沦为单纯的“代币炒作”工具。

二、 核心刑事风险点与法律定性

非法经营罪： 针对“法币与虚拟币”的兑换业务。在境内从事OTC撮合、资金池业务或变相买卖外汇，是目前执法部门打击的重点。

1)组织、领导传销活动罪： 重点排查项目是否存在“层级返利”、“拉人头机制”或“无限代收益”。即便有真实产品，若激励机制涉传，依然面临全案收缴的风险。

2)开设赌场罪： 针对具备“杠杆对赌、输赢不与涨跌幅挂钩、抽水渔利”特征的永续合约。辩护重点在于证明其属于“金融衍生品”而非“赌博工具”（参考146号指导案例）。

3)帮信罪/掩隐罪： 针对开发者或运营者。即便项目合法，若未履行反洗钱义务导致大量犯罪资金流入，个人亦可能被推定为“明知”而入罪。

三、 远程办公与求职者的避险指南

牌照误区： 警惕项目方宣传的“全球持牌”。持有美国或离岸牌照不代表具有全球合规性，更不能作为在境内展业的护身符。

**入职与薪酬风险：**

1)合同主体： 优先选择境内有实体公司的岗位（便于劳动维权），但需确认该公司不直接参与核心发币业务。

2)发薪方式： 法律要求以人民币发薪。以虚拟币发薪存在“账户冻结风险”和“劳动法支持缺失风险”（仲裁机构常判决公司补发人民币）。

3)身份管理： 避免在境内主体中担任法人、财务或关键决策人，技术岗位应保留日常代码提交记录，以证明自己仅从事非决策性的技术工作。

四、 实操自救建议：遇风险如何处置？

1)证据隔离： 养成保留链上记录、KYC信息、合规审查报告及非恶意沟通记录的习惯。在调查中证明自己是“善意第三人”。

2)配合调查原则： 遇到“冻卡”或警方询问，初犯、偶犯且能清晰说明资金来源的，通常可通过退赔涉案款换取解冻或不予立案。

3)最高原则： “法律具有滞后性，不作恶是最大护身符”。远离面向大陆用户的法币通道、高倍杠杆合约及具有传销特征的增长模式。

1\. 钱包与私钥系统

在理解了私钥和助记词后，需要区分钱包的存储形式，这直接决定了资产的安全级别。

热钱包 vs. 冷钱包：

热钱包 (Hot Wallet)： 联网状态，如手机 App (MetaMask, OKX Wallet)。优点是方便交易，缺点是私钥存在联网设备，容易被黑客攻击。

冷钱包 (Cold Wallet)： 硬件设备，私钥永不联网。交易时在硬件内签名，再传输结果。安全性最高。

EOA 钱包 vs. AA 智能合约钱包：

EOA (Externally Owned Account)： 你目前笔记中描述的传统钱包，完全依赖私钥。一旦丢失无法找回。

AA (Account Abstraction)： 账户抽象钱包。可以实现“社交恢复”（找回账号）、“免 Gas 费”等功能，是 Web3 大规模普及的方向。

2\. 交易与签名机制（深度补充）

理解了 Gas 和签名后，初学者最容易在“权限授权”上吃亏，这是资产被盗的高发区。

Approve（授权）机制： 在 Web3 中，如果你要用 DEX（去中心化交易所）卖出代币，你需要先执行 Approve。这代表你授权该智能合约“有权动用你钱包里一定数量的代币”。

风险提示： 很多钓鱼网站会骗取你的“无限授权”，从而搬空你的钱包。

EIP-1559 费用模型： 现在的 Gas 费由 Base Fee（基础费，会被销毁） + Priority Fee（小费，给矿工） 组成。这使得手续费预测更加准确。

3\. 区块链网络运行（架构补充）

除了你提到的 Layer 1（如以太坊主网），目前的 Web3 生态是多链并行的。

Layer 2（二层网络）： 由于以太坊主网（Layer 1）又慢又贵，出现了 Layer 2（如 Arbitrum, Optimism, Base）。

原理： 在二层处理交易，将成千上万笔交易打包成一个证明提交给 Layer 1，从而极大降低 Gas 费并提高速度。

跨链桥 (Bridge)： 资产在不同链之间转移的通道。

4\. Web3 核心概念补完：与 Web2 的区别

为了完整理解 Web3，建议在笔记中加入这一部分，明确“为什么需要 Web3”。

| 特性 | Web2 (传统互联网) | Web3 (价值互联网) |
| 数据所有权 | 归大公司所有 (Google, Facebook) | 归用户自己所有 (通过私钥控制) |
| 身份系统 | 手机号、邮箱、第三方登录 | 钱包地址 (无需许可，匿名性) |
| 透明度 | 后台逻辑黑箱 | 链上代码开源、数据公开可查 |
| 中心化 | 平台可以封禁你的账号 | 抗审查，没人能阻止你使用区块链 |

5\. Web3 常见应用形态 (DApp)

Web3 不仅仅是转账，它由以下几个核心板块构成：

DeFi (去中心化金融)： 无需银行，通过智能合约进行借贷、交易（如 Uniswap, Aave）。

NFT (非同质化代币)： 证明图片、音乐或游戏道具的唯一所有权。

GameFi (链游)： 玩家在游戏中获得的资产可以真实变现 (Play-to-Earn)。

SocialFi (去中心化社交)： 数据归用户，不再担心被平台删帖或限流。

6\. 安全建议

助记词永远不要存手机相册或收藏夹： 手写在纸上或存在离线设备。

不随意见签名： 尤其是看不懂的交互请求。

小额分仓： 交互不确定的项目时，使用存有少量资金的新钱包
