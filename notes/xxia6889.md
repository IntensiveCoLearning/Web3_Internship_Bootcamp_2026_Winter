**一、黑客松核心流程：**

**5 大关键环节**

**1.选题阶段**：避免 “大而全、逻辑闭环” 的项目，要聚焦明确、单点、可演示的需求（需回答 “谁用、解决什么痛点、为什么是你”）。

**2.MVP 执行**：时间短（8-48 小时），核心是 “交付能演示的闭环”—— 允许 mock 数据、砍掉非核心功能（用户系统 / 后台等），但不能没有 Demo。团队结构：

**3.技术型团队**：选偏技术创新的方向，重点讲技术突破点； 非技术团队：80% 时间理清逻辑，用 AI / 模板生成 Demo，路演突出痛点 / 场景。

**4.Demo 与视频**：评委不看代码但会看 Demo 视频，需提前录好、剪去卡顿 / 错误，现场翻车时视频能救场。

**5.路演与 PPT**：5 分钟内让评委理解 “做什么 + 为什么加分”，PPT 用 10-12 页、强视觉（图片 > 文字）、无目录 / 少动画，结构建议：项目 slogan→痛点→方案→Demo→技术亮点→愿景。

**二、新人避坑与核心能力**

**5 大误区**：想做完整产品、认为代码越多越牛、不敢用 mock 数据、路演狂讲技术细节、不看赛题 / 赞助商方向。

**5 大核心能力**：

1.  闭环思维：能回答 “谁在什么场景用、前后差别、为什么是你、Demo 能否 1 分钟演示”；
    
2.  MVP 取舍：砍掉注册 / 后台 / 权限等功能，只留 “一个用户 + 一个路径 + 一个结果”；
    
3.  3.展示能力（褒义 “骗评委”）：用模拟结果 / 假数据 / 单页面演示，评委关注 “想法值不值得做” 而非 “是否是成熟产品”；
    
4.  用 AI 当队友：让 AI 帮忙推 idea、写路演脚本、模拟评委提问；
    

Demo 优先：“没 Demo = 白干，Demo 清楚 = 赢一半”。

**三、新人可直接用的实操清单**

**1.选题模板（30 分钟）**：用 4 句话明确 ——“我为【某类人】在【具体场景】解决【明确痛点】用【可演示方案】”，写不出来就换题；

**2.MVP 检查清单**：是否 1 分钟能演示、只有 1 条主路径、允许 mock、能截图录屏；

**3.Demo 视频标准**：2-3 分钟，包含 “问题→操作→结果”，无多余内容。

### **一、Web2与Web3（DApp）核心差异**

| 维度 | Web2（传统App） | Web3（DApp） |
| --- | --- | --- |
| 资产归属 | 仅租赁使用权，资产是中心化服务器代码 | 拥有真正所有权，资产为用户私有财产 |
| 账号体系 | 需注册账号 | 无需注册，钱包即万能钥匙与身份证，连接即可使用，拔钥匙带走数据 |
| 数据主权 | 平台可封号，内容/数据可能归零 | 数据存储于链上，即使前端消失，数据依然存在，用户是数据主人 |
| 运行逻辑 | 依赖中心化主体（如平台、老板），可随时终止服务 | “Code is Law”（代码即法律），无人能中断服务，无需信任他人，信任机器 |

### **二、DApp的核心构成与应用场景**

NaN.  **核心引擎**：智能合约（铁面无私的裁判，一旦运行不可修改，自动执行约定，替代Web2的中间人）、区块链（不可篡改的公开账本，匿名且透明，可查看全球任意交易记录）；配套要素包括Gas费（支付给矿工的记账费）、治理权（类似股票，可参与DApp治理）。
      
NaN.  **典型应用场景**：
      
      -   DeFi：无银行、无注册、无KYC与银行审核，1秒完成全球金融交易，用户可投入资金“做银行赚手续费”。
          
      -   NFT & GameFi：资产为用户私有财产，可跨游戏使用、卖给其他玩家兑换真金白银，而非Web2中可被运营商修改的虚拟物品。
          
      -   DAO：无传统公司架构（无CEO），由互不相识的人通过智能合约与投票决策，如众筹竞拍美国宪法副本。
          

### **三、DApp的风险与现状**

NaN.  **避坑要点**：
      
      -   贵：以太坊拥堵时，单次操作Gas费可能高达几十美元。
          
      -   险：私钥丢失即意味着资产丢失，无“忘记密码”找回功能。
          
      -   乱：代码漏洞频发，黑客攻击是常态。
          
NaN.  **当前状态**：目前的DApp又慢又难用，类似1995年的互联网，但它首次将“所有权”真正归还用户，并非单纯的新应用，而是互联网的下一个时代。

# **ERC-7962 核心分享总结（基于会议内容）**

## **1：零知识证明**

在不暴露具体细节的情况下，能够向第三⽅证明数据的某些特性。 ⽐如，有两个整数值，第三⽅关注其和或者乘积是某个值，但是不能暴露具体值。 提供 Proof(X，Y)，向第三⽅证明 X+Y = Z，但是 Proof(X，Y) ⽆法反推出 X 和 Y 的具体值 。

## **2：ERC-7962**

是在 ERC-721 的基础上应⽤ ZK 原理保证了向第 三⽅证明其归属，但是不暴露其真实地址。

## **3： 核心概念**

```
  ## 一、协议基础
  - 设计初衷：解决 ERC-721 隐私泄露+Web2 用户迁移门槛高
  - 核心定位：Key Hash 替代地址，兼容 ERC-20/ERC-721
  - 核心技术：零知识证明（哈希验证）、EIP-712 签名
  ## 二、核心特性
  - 隐私保护：Key Hash 标识、一次性密钥轮换、无 Approve 机制
  - 交易灵活：所有权与发起方解耦、第三方代付 Gas、批量交易
  - 安全设计：Nonce 防重放、签名有效期、强制 low-S/v 校验
  ## 三、适用场景
  - 隐私敏感：匿名 NFT 交易、会员身份验证
  - Web2 迁移：传统用户参与链上服务（无需管理钱包）
  - 企业需求：批量空投、隐私资产分发
  ## 四、优劣势
  - 优势：隐私更强、UX 更优、支持账户抽象
  - 局限：Gas 成本高、不向后兼容、密钥管理要求高
  ## 五、关键问题
  - 隐私与透明平衡：链上公开交易，哈希隐藏身份
  - Gas 代付主体：商业机构（如星巴克）为获商业回报承担
  - 反推风险：哈希算法不可逆，无法通过 Key Hash 反推地址
```

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/xxia6889/images/2026-01-18-1768744970847-image.png)

**隐私保护机制**

-   **Key Hash 所有权标识**：链上仅存储公钥哈希，不记录真实地址，避免身份关联。
    
-   **一次性密钥轮换**：交易后强制生成新的 Key Hash 存储剩余资产，切断交易关联性，提升不可追溯性。
    
-   **去除 Approve 机制**：避免授权过程中暴露 Key Hash 对应的公钥，防止隐私泄露

## **Co-Learning 和Web3安全：**

### 钱包

_1：_钱包授权必须谨慎，恶意协议或后门可能盗走资金，即使私钥未泄露也有风险；

_2：_安装插件辅助检测风险，同时坚持自主判断，不盲目依赖工具；

_3：_钱包只存少量 Gas 费，鸡蛋要分散开放；

### 岗位：

运营岗可通过推特发中文内容（项目分析、新手教程）积累案例。多参与社区的活动积累履历；

### 安全：

_1：_攻击覆盖 DeFi、CEX、公链、钱包等多类型项目，黑客攻击是主要损失来源，钓鱼诈骗(不明邮箱链接不要随便点，即使验证来源与准确性)和项目方跑路事件损失显著下降；

_2：_中心化交易所（CEX）成损失重灾区、个人用户面临社会工程学、地址投毒等攻击威胁；

_3：_操作钱包时需校验转账地址、保护私钥、警惕冒充客服的社会工程学攻击。 了解主流项目安全防护实践，提升风险识别与应急响应意识。

_4：尽量不要到处授权，即使取消授权避免损失。可以使用新钱包、里面存放一些gas费，从而去搞一些东西；_
