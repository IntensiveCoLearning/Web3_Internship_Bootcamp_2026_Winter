Solidity 基础语法学习笔记

1\. Solidity 是什么

Solidity 是一种面向智能合约的高级编程语言，主要用于以太坊及 EVM 兼容区块链（如 Ethereum、BSC、Polygon、Arbitrum 等）。

Solidity 编写的合约在编译后会生成 EVM 字节码，并部署在区块链上，一旦部署，代码逻辑不可随意更改。

智能合约的本质是：

运行在链上的、公开透明、自动执行的程序。

2\. 合约的基本结构

一个标准的 Solidity 合约通常包含以下部分：

SPDX License 标识（声明开源协议）

pragma 版本声明（指定编译器版本）

contract 定义

状态变量

函数（读 / 写）

合约是 Solidity 中的核心单位，类似于面向对象语言中的“类”。

3\. 数据类型

3.1 值类型（Value Types）

整数类型

uint：无符号整数（>= 0）

int：有符号整数

常用且推荐使用 uint256

Solidity 中所有整数都有固定大小，默认使用 256 位以获得最佳性能。

布尔类型

bool

取值为 true 或 false

地址类型

address

用于表示账户或合约地址

常见用法：

msg.sender：当前调用者

address(this)：当前合约地址

3.2 引用类型（Reference Types）

字符串（string）

字符串用于存储文本数据，但在链上操作成本较高，不适合复杂字符串处理。

数组（array）

数组可以是定长或不定长，用于存储同类型数据集合。

动态数组支持 push，但链上遍历数组会消耗较多 Gas。

映射（mapping）

映射是一种键值对结构，类似哈希表：

Key → Value

无法遍历

所有 key 都有默认值

映射是 Solidity 中非常常用的数据结构。

4\. 变量存储位置（Storage / Memory / Calldata）

Solidity 明确区分数据的存储位置：

storage：链上永久存储（最贵）

memory：函数执行期间临时存在

calldata：只读、不可修改、最省 Gas

实践中建议：

函数参数优先使用 calldata。

5\. 函数基础

函数由以下部分组成：

可见性修饰符

状态修饰符

参数

返回值

5.1 函数可见性

public：合约内外都可调用

external：只能从外部调用

internal：当前合约及继承合约

private：仅当前合约

5.2 状态修饰符

view：读取状态，不修改

pure：不读取也不修改状态

payable：允许接收 ETH

6\. 全局变量

Solidity 提供了一些内置全局变量：

msg.sender：调用者地址

msg.value：发送的 ETH 数量

block.timestamp：当前区块时间戳

block.number：当前区块高度

这些变量在权限控制和业务逻辑中非常重要。

7\. 条件判断与循环

条件判断

使用 if / else 进行逻辑分支判断。

循环

支持 for 和 while，但在链上应尽量避免大规模循环，因为循环会显著增加 Gas 消耗。

链上原则：能不用循环就不用循环。

8\. 错误处理机制

Solidity 中主要有三种错误处理方式：

require

用于参数校验和权限判断，最常用。

revert

主动触发回滚，通常用于复杂逻辑判断。

assert

用于检查“理论上永远不应该失败”的情况。

实际开发中 90% 使用 require。

9\. 构造函数（constructor）

构造函数在合约部署时执行，只执行一次。

常用于初始化合约状态，例如设置合约拥有者（owner）。

10\. 修饰器（modifier）

修饰器用于复用函数前后的校验逻辑，常用于权限控制。

特点：

在函数执行前或后自动插入逻辑

\_ 代表原函数主体的位置

11\. 事件（Event）

事件用于记录链上日志，主要作用是：

供前端监听

方便链上数据追踪

不会被合约内部读取

事件是合约与前端交互的重要桥梁。

12\. ETH 接收与转账

使用 payable 修饰函数或地址

合约可以接收和持有 ETH

提现时通常将余额转给指定地址

转账逻辑需要特别注意安全顺序。

13\. 常见安全注意点

重入攻击

错误做法：

先转账 → 再修改状态

正确做法：

先修改状态 → 再转账

整数溢出

从 Solidity 0.8 开始，整数溢出会自动触发回滚，不再需要 SafeMath。

14\. 最小合约模型

一个最简单、最安全的合约通常具备：

明确的状态变量

简单清晰的函数逻辑

必要的 require 校验

尽量少的链上计算

黑客松经验分享

定义：短时间（通常 24h–7 天）内，从 0 到 1 完成一个可演示的产品或原型

核心目标：

不是“写完美代码”

而是 快速验证想法 + 可展示成果

评判重点：

想法是否清晰

是否解决真实问题

Demo 是否完整可讲清

二、黑客松的正确心态

不是考试，不是交付项目

不追求：

架构完美

覆盖率

工程洁癖

要追求：

能跑

能讲

能演示

一句话原则：

黑客松不是写代码比赛，是“讲故事 + Demo”的比赛

一句话能说清楚

有明确用户

有明确痛点

Demo 可以 100% 本地跑通

太宏大（想做平台 / 协议 / 全栈生态）

强依赖外部数据 / 权限 / 审核

没 Demo 只能 PPT

建议策略：

从“工具类 / 辅助决策 / 自动化”切入

优先选 你自己会用的东西

四、团队分工（哪怕只有 1–2 个人）

推荐最小分工

产品 / 故事线：定义问题 + 输出 pitch

技术实现：能跑 + 能演示

展示包装：页面 / Demo / 演示流程

一个人也要在脑子里完成这三种角色的切换

五、时间管理（极其关键）

推荐时间分配（以 48h 为例）

0–10%：明确问题 & 定方案

10–50%：核心功能能跑

50–70%：补 UI / Demo 稳定性

70–90%：准备展示内容

铁律：

六、技术选型原则

不要为了“看起来高级”而选陌生技术

一条完整路径

无需解释就能看懂

展示结构（通用模板）

现实世界的问题

为什么现在的方案不行

我们的解决方式

评委通常关注：

这个东西有没有“继续做下去的价值”

# 安全与合规

## 1 行业监管总览（中国语境）

-   2017 以来监管主线：全面封堵虚拟货币的金融属性，技术创新相对有限容忍
    
-   高风险红线场景（最容易出事）
    
    -   ICO/IEO/IDO 等任何形式的融资发币
        
    -   交易所运营与撮合服务
        
    -   虚拟货币作为支付工具
        
    -   境外平台向境内居民提供服务
        
    -   以虚拟货币为载体的非法集资、传销、洗钱、赌博等
        

核心理解：技术不等于没风险；只要碰到 Token 发行、融资、交易闭环、资金池、推广返利，风险级别会陡增

## 2 核心法律风险地图（按最常见梳理）

### 2.1 代币发行与交易（最典型的高危区）

-   中国法律明令禁止：以 ICO/IEO/IDO 等方式公开融资
    
-   你把 Token 叫积分、凭证、治理也没用
    
    -   只要具备融资功能或可流通性，就可能被认定为非法金融行为
        
-   空投、白名单也可能被穿透认定为融资或募资的一部分
    
-   技术人员参与（代币模型、空投逻辑、合约部署等）也可能被视为共犯：只是写代码不能免责
    

### 2.2 赌博 传销 洗钱（结构决定罪名）

-   赌博风险结构：充值 → 抽奖或随机收益 → 可提现或可转售
    
    -   典型：链游盲盒、转盘、随机掉落 NFT 或 Token 且能提现
        
-   传销风险结构：邀请返利、多级推广、团队计酬、层级返佣
    
    -   常见于 NFT 项目、DAO 拉新、算力挂靠、矿池推广
        
-   洗钱风险高发点：C2C/OTC 出入金、帮人换汇、资金来源不明
    
    -   你可能不知情，但银行卡冻结、被要求退赔、甚至被刑事追责都可能发生
        

### 2.3 场外交易（OTC）与地下换汇风险

-   风险本质：用虚拟货币做桥梁绕过外汇监管
    
-   反复撮合人民币与外币或虚拟币交易，可能触发非法经营外汇业务
    
-   更糟糕：对手资金若涉诈涉赌涉毒，你会被卷入协助洗钱链条
    

### 2.4 民商事争议（合同大概率不被支持）

-   虚拟货币买卖、委托代投等纠纷常被认定为合同无效、风险自担
    
-   代投（帮朋友操作资金）容易引发亏损赔偿、资金去向不明的责任纠纷，甚至被怀疑非法经营或非法集资
    

## 3 全球监管趋势（你需要知道的大方向）

### 3.1 合规化在推进：框架越来越清晰

-   欧盟：MiCA 落地，对稳定币、服务商牌照、资本与流动性要求更明确
    
-   新加坡：更强调 AML/CFT，与 FATF 标准接轨
    
-   美国：稳定币相关立法推进中，核心围绕发行、储备、反洗钱、消费者保护
    

### 3.2 FATF 与 Travel Rule（理解为什么 Web3 合规这么难）

-   FATF：全球反洗钱与反恐怖融资标准制定者
    
-   Travel Rule（旅行规则）：转账超过一定阈值时，VASP 需收集并传递汇款人、收款人信息
    
-   对 DeFi 与 DEX 的挑战：匿名性 + 跨链复杂性 → 风险评估、协作、报告都更难
    

### 3.3 稳定币监管意义（为什么被盯得最紧）

-   稳定币是 DeFi 的基础设施，体量大，若出问题会有系统性风险（UST 崩盘是标志事件）
    
-   监管目标
    
    -   储备真实、透明（保护投资者）
        
    -   防风险传导（金融稳定）
        
    -   防被用于非法活动（AML）
        

## 4 入职 Web3 的劳动关系风险（很多人忽略）

### 4.1 雇佣关系新形态

-   项目常境外注册，远程办公，国内无用工主体
    
    -   可能无法签有效劳动合同
        
    -   五险一金难保障
        
    -   发生争议或工伤维权困难
        
-   EOR（名义雇主）：可缓解部分问题，但劳动关系认定仍要谨慎
    

### 4.2 薪酬结构风险

-   常见：人民币 + Token 或全 USDT
    
-   风险点
    
    -   工资应以法定货币支付（Token 或 USDT 工资属性存在争议）
        
    -   Token 波动甚至归零
        
    -   影响社保基数、经济补偿计算等
        

### 4.3 出金风险（最现实、最常见）

-   C2C/OTC 出金可能遇到涉诈资金 → 银行卡冻结
    
-   高价出 U、协助他人洗钱、绕监管 → 可能触发刑事风险
    
-   建议：与公司协商保留部分法币收入；留存合法来源记录
    

### 4.4 入职前做项目合法性尽调

-   重点问
    
    -   是否面向中国大陆用户
        
    -   Token 分发机制、收益模型
        
    -   是否含返利层级、投资承诺
        
    -   是否存在资金池、提现闭环
        
-   你不是主导方也可能被牵连：整体项目若涉非法金融活动，技术与运营都可能被调查
    

## 5 刑事风险案例（从结构看共性）

### 5.1 开设赌场 赌博

-   特征：平台可投注、可提现、运营推广分工明确
    
-   司法实践常强调：能提现是显著特征
    

### 5.2 非法经营（含换汇、支付结算）

-   用 USDT 做媒介，为客户提供外币与人民币兑换、赚差价与服务费 → 高风险
    

### 5.3 非法吸收公众存款

-   矿机、积分、返利、释放收益 + 可兑换虚拟币 + 可提现 → 极易被认定为变相吸储
    

### 5.4 组织、领导传销

-   层级返利 + 拉人头计酬 + 门槛费或交币入场 → 传销外观特征非常明确
    
-   技术人员若明知层级奖励用于传销仍提供支持，可能被认定为共犯
    

### 5.5 洗钱

-   提供账户、协助兑换、跨境转移、掩饰来源性质 → 都可能触发
    
-   上游犯罪未判决也不影响洗钱认定的逻辑在实践中存在
    

## 6 网络安全风险（Web3 新手的黑暗森林）

结论：多数人以为是被钓鱼，实际是系统性安全习惯缺失

### 6.1 钓鱼（Phishing）

-   伪造邮件或网站：面试通知、奖学金、账号异常
    
-   社群钓鱼：Telegram、Discord 冒充官方
    
-   假客服、假好友：盗号后精准社工
    
-   高危信号：让你点链接、装软件、连接钱包、签名、给助记词或私钥
    

### 6.2 恶意软件 木马（Malware）

-   伪装面试软件、学习资料、破解器
    
-   剪贴板劫持（地址替换）
    
-   浏览器扩展后门
    
-   远程控制、录屏、窃取 Cookie、窃取密码管理器数据
    

### 6.3 社工（Social Engineering）

-   冒充 HR、导师、学长学姐
    
-   群内管理员私聊发内推或表单或验证
    
-   好友求助要求转账或测试链接
    

### 6.4 供应链攻击（第三方依赖）

-   假插件、带后门的开源库、官方渠道被劫持
    
-   经验：只装必要插件，且核验开发者与下载量
    

### 6.5 地址污染 扫描木马

-   复制粘贴地址被替换、输入框监听
    
-   后果：资产转错地址不可逆
    

### 6.6 传统账号安全

-   密码复用、邮箱或 SIM 劫持、没开 2FA → 邮箱一丢，全盘皆输
    

## 7 防护清单（学生版，可直接照做）

### 面试 实习

-   只用 Zoom、Teams、腾讯会议等主流工具，拒装任何专用面试软件
    
-   必须安装时：先虚拟机测试 + 查安全口碑 + 核对权限与摘要
    
-   核实邀请邮箱、域名、联系人，多渠道确认
    

### 奖学金 空投 福利

-   任何要求连接钱包、签名、输入助记词的网页：默认骗局
    
-   空投用测试钱包，主钱包冷存储
    
-   涉缴费或调整先打官方热线复核，核对域名
    

### 社群 私聊

-   不信官方私聊，不点陌生链接
    
-   遇到急转账、帮测试、填表连接钱包必须二次核实
    

### 软件 插件

-   只从官网或官方商店下载
    
-   插件控制在 3 个以内（钱包、密码管理、广告拦截等必要项）
    
-   摄像头、麦克风按需授权
    

### 账号与密码

-   重要账户开启 2FA（优先验证器或硬件密钥）
    
-   密码管理器 + 每站不同强密码
    
-   不在同一浏览器里同时保存助记词与日常 Cookie

-   转账前核对地址前 6 位 + 后 4 位
    
-   助记词、私钥离线保存，绝不截图、绝不网盘、绝不发送
    
-   定期检查授权，及时 revoke 不必要授权

-   怀疑被攻击：立刻断网、记录时间点、截图取证，寻求专业支持

# AI 辅助搭建 Obsidian 个人知识库

## 一、为什么我们总是「收藏了不看」

### 1\. 信息困境

-   信息分散在各个平台（微信、网页、PDF、PPT）
    
-   难以聚合，无法形成体系
    

### 2\. 传统笔记的三个痛点

-   **整理成本高**：分类、打标签非常耗时
    
-   **结构僵化**：一旦文件夹结构定死，后期修改成本极高
    
-   **孤岛效应**：笔记之间没有连接，无法复用
    

### 3\. 最终结果

-   只有信息堆积
    
-   没有真正的「知识」
    
-   需要一个能自动整理的「外脑」
    

## 二、核心理念：像管理代码一样管理知识

### 1\. 知识库的本质

-   知识库 ≈ 代码库
    
-   本质都是 **纯文本文件**
    
-   通过「链接」而不是「文件夹」建立关系
    

### 2\. Vibe Note-taking

-   从「自己记笔记」
    
-   转变为「操作一个系统」
    

### 3\. 人与 AI 的角色分工

-   **你（Editor）**
    
    -   判断

    -   拍板

# DApp

## 一、DApp 的实用定义

-   DApp = **需要与区块链进行读 / 写交互的应用**
    
-   判断一个应用是不是 DApp，看两点：
    
    -   是否调用智能合约
        
    -   是否以链上状态作为核心数据
        
-   前端和后端 **可以是中心化的**
    
-   完全去中心化不是目标，**可用性和用户体验更重要**
    

## 二、现实世界中的 DApp 形态

### 1\. 主流 DApp（最常见）

-   智能合约：去中心化
    
-   前端 / 后端：中心化
    
-   通过中心化服务提升查询效率和体验
    
-   代表项目：Uniswap、OpenSea
    

这是当前**工业级 DApp 的主流形态**。

### 2\. 简单 DApp

-   功能集中在：
    
    -   读取资格
        
    -   mint / transfer
        
    -   展示链上资产
        
-   常见于 NFT Mint 页面
    
-   后端逻辑极少或没有
    
-   **非常适合新手入门练手**
    

### 3\. 极端 DApp

-   前端部署在 IPFS
    
-   几乎没有后端
    
-   强抗审查，但访问慢、体验差
    
-   仅用于特殊场景，不是常规选择
    

## 三、DApp 的基本架构

一个完整可用的 DApp 通常包含：

-   **前端**：用户交互界面（Web）
    
-   **后端**：索引、缓存、数据聚合
    
-   **智能合约**：核心业务逻辑（链上）
    
-   **基础设施服务**：
    
    -   IPFS
        
    -   RPC
        
    -   索引服务
        
    -   预言机
        

## 四、关键基础设施说明

### 1\. IPFS

-   去中心化文件存储系统
    
-   基于内容哈希而不是服务器地址
    
-   常用于：
    
    -   NFT 图片
        
    -   NFT Metadata
        
    -   去中心化前端资源
        
-   免费 IPFS 不保证长期存储，通常配合 Pinata / Filecoin 使用
    

### 2\. RPC 服务商

-   为 DApp 提供区块链的 API 接口（JSON-RPC）
    
-   支持：
    
    -   查询账户余额
        
    -   调用合约
        
    -   发送交易
        
-   本质：**区块链世界的 API 网关**
    
-   DApp 使用链，但不需要维护链
    

### 3\. 索引服务

-   解决链上数据「不可自由查询」的问题
    
-   功能：
    
    -   扫描区块
        
    -   解析事件
        
    -   存入数据库
        
-   提供 GraphQL / REST API
    
-   大型 DApp 几乎必不可少
    

### 4\. 预言机（Oracle）

-   将现实世界数据引入区块链
    
-   通过多数据源 + 多节点 + 经济激励保证可信
    
-   主流方案：Chainlink
    
-   常用于价格、清算条件、外部状态判断
    

## 五、推荐技术栈（新手友好）

### 总体原则

-   **优先学习 JavaScript**
    
-   一套 JS 技术栈即可跑通 DApp 全流程
    

### 前端技术

-   框架：Next.js
    
-   样式：Tailwind CSS
    
-   UI：shadcn/ui
    
-   状态管理：Zustand
    
-   表单校验：Zod
    
-   数据管理：TanStack
    

### 后端技术

-   简单后端：Next.js API Routes
    
-   复杂后端：NestJS
    
-   ORM：Prisma
    
-   数据库：PostgreSQL
    

### 部署与运维

-   前端 / Serverless：Vercel
    
-   后端服务：Northflank
    
-   自托管方案：Coolify
    

## 六、以太坊 DApp 工具链

-   合约语言：Solidity
    
-   合约框架：Foundry、Hardhat
    
-   合约交互：ethers.js、viem
    
-   钱包接入：RainbowKit + wagmi
    
-   RPC：Alchemy、Infura、QuickNode
    
-   索引服务：The Graph、Moralis
    

## 七、NFT Mint DApp 的核心学习点

通过一个 NFT Mint DApp，可以系统掌握：

-   IPFS（图片 + metadata）
    
-   测试网与 Faucet
    
-   智能合约部署与验证
    
-   钱包连接与网络检测
    
-   合约读写（mint）
    
-   交易确认状态处理
    
-   链上数据到前端展示
    

## 八、核心概念速记

### Mint

-   从零地址创建新的链上资产
    
-   区别于 transfer（地址之间转移）
    

### Gas

-   链上计算的费用
    
-   防止恶意消耗资源
    
-   激励验证者维护网络
    
-   Gas 价格随网络拥堵变化
    

### 区块确认

-   去中心化共识的成本
    
-   DApp 通常等待 1–2 个区块确认即可
    
-   多确认用于提高安全性

# Solidity 智能合约开发入门

## 1\. EVM 基础认知

### 1.1 什么是 EVM

-   EVM（Ethereum Virtual Machine）是以太坊智能合约的**运行环境**
    
-   所有 Solidity 合约最终都会被编译为 **EVM 字节码**
    
-   特点：**栈式虚拟机、单线程、确定性执行**
    

### 1.2 EVM 的四大执行区（非常重要）

-   **Stack**
    
    -   用于指令和临时计算
        
    -   只能存放 256-bit 数据
        
    -   非常快，但空间极小
        
-   **Storage**
    
    -   链上永久存储合约状态
        
    -   Gas 成本最高
        
    -   合约优化的核心目标：**少用 Storage**
        
-   **Memory**
    
    -   函数执行期间的临时内存
        
    -   不上链，函数结束即销毁
        
    -   Gas 成本低于 Storage
        
-   **Calldata**
    
    -   用户调用合约时传入的参数
        
    -   只读，不可修改
        
    -   比 Memory 更省 Gas
        

类比理解：

-   Storage ≈ 硬盘
    
-   Memory / Calldata ≈ 内存
    

## 2\. Gas 成本与优化直觉

### 2.1 Gas 的核心来源

-   计算
    
-   Storage 读写
    
-   合约体积
    

### 2.2 Storage 的 Gas 成本

-   **首次写入 storage**：≈ 20,000 gas
    
-   **修改已有 storage**：≈ 5,000 gas
    

### 2.3 基本优化原则

-   尽量少存
    
-   能用 memory / calldata 就不用 storage
    
-   避免频繁修改链上状态
    

## 3\. Solidity 开发环境

### 3.1 推荐工具

-   **Remix IDE**
    
    -   在线
        
    -   内置虚拟链
        
    -   支持编译、部署、调试
        
    -   可观察 memory / storage 变化
        

### 3.2 Solidity 版本

-   常用版本：`0.8.x`
    
-   示例：`^0.8.20`
    
-   0.8+ 默认包含 **整数溢出检查**
    
-   阅读合约源码时必须注意版本号
    

## 4\. Solidity 数据类型

### 4.1 值类型（Gas 友好）

-   `bool`
    
-   `uint / int`
    
    -   可指定位宽：`uint8 ~ uint256`
        
    -   位宽越小，占用 storage 越少
        
-   `address`
    
    -   表示 EOA 或合约地址
        
    -   不要用 string 存地址
        
-   `bytes32`
    

### 4.2 引用类型（Gas 昂贵）

-   `string`
    
-   动态数组
    
-   `bytes`
    
-   `mapping`
    

### 4.3 mapping 的典型用途

-   余额账本
    
-   授权账本
    
-   权限映射
    

## 5\. ERC20 合约核心结构

### 5.1 ERC20 是什么

-   以太坊**同质化代币标准**
    
-   定义了一组固定接口，保证生态兼容
    

### 5.2 核心链上状态

-   `name`
    
-   `symbol`
    
-   `decimals`
    
-   `totalSupply`
    
-   `owner`
    

### 5.3 两本账本（核心）

-   **余额账本**
    
    -   address → uint256
        
-   **授权账本**
    
    -   owner → spender → allowance
        

## 6\. 构造函数（Constructor）

-   合约部署时执行
    
-   用于初始化链上状态
    
-   只执行一次
    
-   工具类合约可以没有 constructor
    

## 7\. 函数修饰符

### 7.1 可见性

-   `external`：仅外部
    
-   `public`：内外均可
    
-   `internal`：内部 + 子合约
    
-   `private`：仅当前合约
    

### 7.2 状态修饰

-   `view`：读链上状态
    
-   `pure`：不读不写链上状态
    

## 8\. ERC20 核心函数逻辑

### 8.1 transfer

基本步骤：

1.  校验参数（数量、地址、余额）
    
2.  更新余额
    
3.  触发 Transfer 事件
    

### 8.2 approve

-   修改授权账本
    
-   触发 Approval 事件
    
-   是大多数 DeFi 交互的第一步
    

## 9\. Event 事件机制

### 9.1 作用

-   链上 → 链下通信
    
-   区块浏览器、DApp 监听依据
    
-   比直接读 storage 高效
    

### 9.2 常见事件

-   `Transfer`
    
-   `Approval`
    
-   `Mint`
    

### 9.3 indexed

-   最多 3 个 indexed 参数
    
-   用于链下快速过滤
    

## 10\. Mint 与权限控制

-   Mint 通常不能 public
    
-   常见设计：
    
    -   external 函数做权限校验
        
    -   internal 函数做状态修改
        

## 11\. 错误处理

-   `require`
    
    -   常用
        
    -   可带错误字符串（Gas 较高）
        
-   `revert`
    
    -   无字符串，更省 Gas
        
-   `assert`
    
    -   理论上不会失败的逻辑
        
-   **自定义 error**
    
    -   最省 Gas 的现代方案
        

## 12\. 整数溢出与 unchecked

-   Solidity 0.8+ 默认防溢出
    
-   `unchecked {}` 可省 gas
    
-   初学阶段不建议使用
    

## 13\. Modifier 修饰符

### 13.1 作用

-   抽象前置条件
    
-   权限统一
    
-   代码复用
    

### 13.2 常见场景

-   onlyOwner
    
-   nonReentrant
    
-   时间锁
    
-   角色权限控制
    

## 14\. 常见安全问题直觉

-   不明助记词 ≠ 免费钱
    
-   可能是：
    
    -   合约账户伪装
        
    -   权限限制
        
    -   抢跑机器人（Front-running Bot）
        

安全原则：

-   不导入陌生私钥
    
-   查看地址是否为合约
    
-   小额测试
    

## 15\. 扩展生态

-   **OpenZeppelin**
    
    -   审计过的标准合约库
        
-   **可升级合约**
    
    -   Proxy + Logic
        
-   **钻石架构（EIP-2535）**
    
    -   模块化升级

# Agentic AI 与 Web3 的融合

## 一、从 LLM 到 Agentic AI 的范式转变

### LLM（2023 代表）

LLM 的本质是 Token 预测器  
特点是被动响应、无状态  
只能“说”，不能“做”  
不具备长期记忆  
没有身份，不能代表任何主体行动

### Agentic AI（2026 方向）

具备推理、记忆和工具使用能力  
可以自主执行任务并持续运行  
拥有私有记忆与执行能力  
能够代表某个主体完成真实世界行为

核心转变是从“对话型 AI”走向“行动型 AI”

## 二、一个完整 AI 智能体的三大组成

### 大脑 The Brain

由 LLM 构成  
负责理解意图、规划步骤、生成行动指令  
不直接执行操作

### 记忆 The Memory

基于向量数据库  
通过 RAG 实现长期与短期记忆  
解决传统 LLM 聊完就忘的问题

### 工具 The Tools

API、Function Calling、MCP 等  
让 AI 可以执行代码、调用系统、访问外部世界  
形成“思考 → 行动 → 观察”的循环

## 三、Agentic AI 面临的三大基础设施问题

### 身份不可验证

在传统 Web2 中无法判断请求来自人还是 AI  
攻击者可伪装成合法 AI 执行操作

### 行为不可审计

AI 的日志可被篡改或删除  
出现法律责任时难以追溯

### 部署不可治理

大量 Shadow AI 在企业内部运行  
缺乏统一的治理与监管框架

本质问题是信任缺失

## 四、机器经济带来的支付挑战

AI 智能体需要为 API、算力、数据等付费  
传统金融体系存在明显不适配

信用卡和订阅制不适合微支付  
KYC、验证码对机器不友好  
结算速度无法满足毫秒级交互  
微额交易手续费过高

机器经济需要一种原生的支付方式

## 五、Web3 为 Agentic AI 提供的关键能力

### 区块链的两个核心价值

不可篡改的身份

这是传统 Web2 无法提供的

## Web3 安全与刑事风险

### 一、Web3 当前整体安全形势

1.  总体风险现状  
    2025 年 Web3 领域发生 1200 起以上较严重安全事件，总损失金额超过 35 亿美元。  
    攻击呈现两种并行趋势：  
    一是“精准猎杀”，二是“广撒网攻击”。  
    整体特点是事件频发、手段复杂，对行业安全构成长期威胁。
    

### 二、两类核心攻击模式

1.  精准猎杀（低频但高损失）  
    主要目标是 CEX、DeFi 项目、核心开发者和管理员账户。  
    常见方式包括社会工程、私钥和热钱包窃取、深度合约漏洞利用，部分案件具有国家级黑客背景。  
    特点是攻击周期长、策划精细、单笔损失巨大，往往伴随复杂洗钱流程。
    
2.  广撒网攻击（高频但单笔金额较小）  
    最常见的三类是私钥窃取、钓鱼攻击和 Rug Token 跑路项目。  
    随着 AI 技术普及，钓鱼内容高度拟真，普通用户更难分辨，Rug 项目也呈现团队化、流水线化趋势。
    

### 三、AI 时代下的攻击新变化

攻击方式呈现明显的技术融合趋势。  
攻击者会将协议机制漏洞、合约缺陷、供应链漏洞与 AI 自动化工具结合使用。  
典型表现包括：  
利用计算精度漏洞攻击 DeFi 合约；  
在 npm、GitHub 依赖中投毒；  
滥用新 EIP 特性；  
使用 Deepfake 进行身份伪装和社交工程。

### 四、Web3 在中国刑事风险高发的原因

核心原因在于法律体系与技术理念的不匹配。  
中国采用中心化法律体系，而 Web3 强调去中心化。  
同时行业认知不足、监管滞后，司法实践中容易采用“结果倒推社会危害性”的方式定性。  
本质问题是用旧框架评价新技术。

### 五、最容易触发刑事风险的行为场景

高风险行为主要集中在以下方面：  
出入金相关行为，如 OTC、U 商、搬砖套利；  
项目方发币、拉人头、返佣和邀请机制；  
KOL 荐币、带单；  
合约交易被认定为赌博；  
挖矿或 DePIN 项目涉及传销式结构。

### 六、合约交易与赌博的核心区分逻辑

赌博的本质是以赌注押注不确定结果，赌注代表预期收益或损失，输赢与真实市场价格无关。

合约交易的核心特征是：  
资金用于购买具有真实价值的虚拟资产；  
盈亏取决于市场价格涨跌、开仓价格和杠杆倍数；  
平台只收取固定手续费，不与用户对赌。

因此，合约交易在本质上更接近金融衍生品，而非赌博行为。

### 七、GoPlus 给出的安全建议要点

对个人用户而言：  
不点击不明链接、不随意签名和授权；  
私钥和助记词绝不联网；  
只通过官方渠道下载钱包；  
签名前逐字核对交易内容。

对项目和生态而言：  
开发阶段要安全前置，引入审计和自动化安全工具；  
运行阶段进行全链路监控；  
建立应急响应机制；  
在协议层面强化原生安全设计。

# Web3 运行原理

## 一、Web3 到底在解决什么问题？

**核心问题：**

在没有中心化机构的情况下，

如何确认“你是谁”

如何保证“账本可信”

如何让“规则自动执行”

**Web3 的答案：**

-   私钥 & 数字签名 → 身份与授权
    
-   去中心化网络 + 共识 → 可信账本
    
-   智能合约 → 自动执行规则
    

* * *

## 二、钱包、私钥、助记词：Web3 的身份系统

### 1\. 三者关系

-   **私钥（Private Key）**
    
    -   资产和权限的唯一控制权
        
    -   区块链世界的“终极印章”
        
    -   丢了 ≈ 永久丢失（无法重置）
        
-   **助记词（Seed Phrase）**
    
    -   私钥的可读备份
        
    -   可以派生多个账户
        
-   **地址（Address）**
    
    -   公钥哈希后的字符串
        
    -   不是隐私，只是公开收款号
        
    -   行为可被链上分析关联
        

### 2\. 核心认知

-   区块链 **不认识人**
    
-   只认识：**谁能签名**
    
-   谁掌握私钥，谁就拥有资产与权限
    

### 3\. 私钥的社会学意义

-   任何人都可以随机生成私钥
    
-   不需要审批、身份、国家、机构
    
-   **个人主权的起点**
    
-   权利不是被“发放”的，而是自己生成的
    

* * *

## 三、交易（Transaction）是什么？

### 1\. 一笔交易包含什么？

交易 = 要做的事 + Gas Fee + Nonce

-   要做的事：转账 / 调用合约 / mint / 投票
    
-   Gas Fee：为网络资源付费 + 激励
    
-   Nonce：防止同一交易被重复执行
    

### 2\. 钱包真正做的事

-   组装交易数据
    
-   用私钥签名（授权）
    
-   广播到区块链网络
    

钱包不是“记账者”，只是“签名器 + 发信人”

* * *

## 四、数字签名：Web3 信任的根基

### 数字签名能保证什么？

-   任何人都可以验证交易是你发的
    
-   任何人都无法伪造你的签名（除非有私钥）
    

信任来自 **密码学**，不是机构

* * *

## 五、Gas Fee：为什么必须存在？

### Gas 的三层作用

1.  **防垃圾交易**
    
    -   提高作恶成本
        
2.  **激励节点/验证者**
    
    -   提供算力、带宽、维护网络
        
3.  **经济闭环**
    
    -   使用资源 → 付费 → 网络可持续
        

没有激励的去中心化系统无法长期运行

* * *

## 六、一笔交易是如何“跑起来”的？

### 从交易到出块的完整流程

Wallet（签名）  
→ RPC / Node（传播）  
→ Mempool（排队）  
→ Builder / Validator（挑选）  
→ Block（出块）  
→ Explorer（可查）

### 为什么需要“确认时间”？

-   区块像账本的一页
    
-   新区块引用旧区块 hash
    
-   越往后，历史越难被推翻
    
-   PoS 下有 Finality（最终确认），约 12–13 分钟
    

**不可篡改 ≠ 立刻不可改**

* * *

## 七、共识机制：全球陌生人如何记同一本账？

### PoW（工作量证明｜BTC）

-   拼算力
    
-   能耗高
    
-   规则简单、历史最久
    

### PoS（权益证明｜ETH）

-   质押 + 随机选验证者
    
-   更节能
    
-   作恶会被罚没质押资产
    

共识的本质：让作恶成本远高于收益

* * *

## 八、钱包 ≠ 网络：RPC 是现实中的风险点

### 真实关系

-   钱包通常只连一个 RPC
    
-   RPC 背后是节点或节点集群
    
-   RPC 是常见的“中心化入口”
    

### 抗审查来自哪里？

-   可换钱包
    
-   可换前端
    
-   可换 RPC
    
-   可换节点
    

**替换性 = 抗审查能力**

* * *

## 九、智能合约：链上的“自动执行规则”

### 智能合约是什么？

-   写在区块链里的可执行代码
    
-   在 EVM 中运行
    
-   由交易触发
    
-   自动改变链上状态
    
-   一旦部署，难以随意更改
    

### Code is Law 的含义

-   规则透明、可验证
    
-   执行历史可追溯
    
-   减少中介与信任成本
    

从“信人 / 信机构”  
转向“信代码 + 共识 + 治理”

* * *

## 十、区块链协议如何升级？（以太坊）

### EIP 流程

1.  社区讨论（论坛）
    
2.  形成提案文档（EIP）
    
3.  社区共识
    
4.  硬分叉升级（每年 1–2 次）
    

关键点：

-   没有 CEO
    
-   没有拍板人
    
-   协议是被治理的代码

## Web3 行业全局介绍 & 岗位概览

### 一、行业阶段与趋势

-   **Web3 四阶段演进**：  
    1）BTC 极客实验期 →  
    2）ETH + ICO 造富期 →  
    3）NFT / DeFi / L2 应用爆发 →  
    4）**合规 + ETF + RWA + DePIN + AI 融合（当前）**
    
-   **2024–2026 核心变化**：监管转暖、BTC/ETH 现货 ETF 通过、机构与传统资本大规模入场，行业从投机走向基础设施与实体结合
    

### 二、就业市场现状

-   **人效极高**：10 人团队可管理数亿–数十亿美金 TVL
    
-   **开发者稀缺**：全球活跃开发者仅约 2.5–3 万
    
-   **2025 招聘反弹**：新增岗位约 6.6 万（+47%）
    
-   **结构性失衡**：
    
    -   初级岗位极度内卷（≈450:1）
        
    -   资深 Rust / 合规 / Research 人才严重短缺
        

### 三、岗位与薪酬梯队

-   **第一梯队**：智能合约 / 安全审计 / 合规（高壁垒 + 高薪）
    
-   **第二梯队**：Research / 核心产品（懂 Tokenomics）
    
-   **第三梯队**：运营 / 增长（普通岗位低，但增长高手天花板高）
    
-   **区域分工**：
    
    -   技术核心：北美
        
    -   交易所 / 运营 / 资管：亚洲 & 中东
        

### 四、求职关键认知

-   **Web3 不看“简历”，看“链上履历”**：  
    GitHub、DAO 治理、Snapshot 投票、Mirror 输出
    
-   **核心渠道**：Web3 垂直平台 + Twitter/X + Discord + DAO
    
-   **必备能力**：
    
    -   Dev：Solidity → Rust / Move / Cairo + 安全意识
        
    -   非技术：英语 + Tokenomics + 链上交互经验
        
-   **避坑原则**：远离“三无项目”（无融资 / 无代码 / 全匿名）
    

### 五、个人反思

Web3 本质是**高杠杆、低人数、强结果导向**的行业，真正的竞争力来自可验证的链上行为与长期输出，而不是传统投递逻辑。
