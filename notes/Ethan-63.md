# 2026-01-22
<!-- DAILY_CHECKIN_2026-01-22_START -->




在看完 Uniswap 的课程回放之后，我对“AMM 到底在做什么”这件事有了一个比之前清楚得多的整体认识。这次学习并不是在记公式，而是在理解：一个去中心化交易所是如何在没有做市商、没有订单簿的情况下长期稳定运行的。

一、从 V2 开始：AMM 的核心其实是规则，而不是价格

Uniswap V2 的核心逻辑可以非常简单地概括为一句话：

用恒定乘积公式 x \* y = k 来替代人工报价。

价格并不是被“算出来给人看的”，而是交易行为自然推出来的结果。当有人用一种资产换另一种资产时，池子里的比例变化，价格也随之变化。只要市场价格和池子价格不一致，就会有人通过套利把价格拉回合理区间。

这一点让我意识到：

Uniswap 并不试图预测市场，而是通过一套简单但自洽的规则，让市场自己去修正价格。

二、流动性提供与无常损失：收益和风险是绑定的

在回放中对无常损失的解释让我理解得更现实了一些。无常损失并不是 Bug，而是 AMM 机制的“副作用”。

LP 的收益来自手续费，但代价是：当价格剧烈波动时，资产结构会发生变化，最终价值可能低于单纯持有。

也就是说：

LP 并不是“躺赚”，而是在用承担价格风险换取手续费收益。

目前的解决方式不是消除无常损失，而是通过更高的手续费或更高的资金效率去补偿它。

三、V3：集中流动性让“效率”成为核心关键词

V3 中最重要的变化，是集中流动性的引入。LP 不再被动地在全价格区间提供流动性，而是可以选择自己认可的价格区间。

这带来的变化非常直观：

资金效率显著提高

同样的资金，可以获得更高的手续费收益

但对 LP 的主动管理能力要求也更高

Tick 系统、本质上就是把连续价格离散化，用结构化的方式管理“在哪个价格点发生流动性变化”。

这里让我第一次感受到：DeFi 协议的复杂度，已经非常接近传统金融系统。

四、V4：协议开始“模块化”

V4 的钩子（Hook）机制让我印象很深。它不再只是优化效率，而是直接开放了协议行为的“插口”，允许在交易前后、添加/移除流动性时插入自定义逻辑。

这意味着：

动态手续费

限价单

定制化流动性策略

都可以通过 Hook 实现。

Uniswap 从“一个产品”，开始演变成一个可扩展的金融基础设施。

五、安全、价格与现实世界的连接

在 Q&A 中提到的 TWAP、抢跑防护、闪电贷等内容，也让我更清楚地意识到：

链上价格不是“自然可信”的，需要设计机制保证可用性

抢跑并不是个例，而是链上公开性带来的结构性问题

协议安全更多是“博弈设计”，而不是单点防御

这些问题并不是靠某一段代码解决，而是靠整体机制设计。

六、结合前面学习的一点个人理解

把 Uniswap 放回我这一周的学习脉络里看，它刚好验证了我前面反复听到的一句话：

技术不是答案，结构和规则才是。

Uniswap 成功的原因，并不是公式多复杂，而是它把“定价、激励、风险、执行”这些事情，用一套简单但可验证的规则锁死在合约里。

今晚这节 Solidity 课程，对我来说是第一次系统地把“合约是怎么在链上跑的”这件事串起来，而不是只停留在“会写几行代码”的层面。

1、 对 EVM 的理解明显更清晰了

以前对 EVM 的理解比较模糊，只知道“合约是在 EVM 里执行的”，但今天通过对 四个执行区（Stack / Storage / Memory / Calldata） 的讲解，才真正明白：

Storage 是最贵的，也是合约设计里最核心的成本来源

Memory / Calldata 只是运行时临时用，不上链

很多安全问题和 gas 问题，本质都和 storage 设计有关

这一点让我第一次意识到：

写合约不是在“写程序”，而是在“设计一个链上数据库”。

2、 Gas 成本不再是抽象概念

通过对 storage 写入和修改成本的对比（初始化写入 vs 修改已有值），我开始理解为什么老师一直强调：

少存

少改

少碰 storage

这也让我明白，所谓的“合约优化”并不是玄学，而是结构设计阶段就已经决定了大半。

3、 ERC20 不只是模板，而是一套账本设计

通过从零拆解 ERC20 合约，我对它的理解从“标准接口”变成了“清晰的账本结构”：

一张余额账本（balances）

一张授权账本（allowance）

constructor 负责初始化状态

transfer / approve 本质都是在改账本 + 发事件

尤其是 mapping 的使用，让我很直观地理解了：

ERC20 就是链上维护的一套可公开验证的账本系统。

4、 事件（Event）的重要性被真正理解了

以前只知道“要 emit Transfer”，但今天才意识到：

事件是链上 → 链下同步的核心机制

DApp、区块浏览器、Bot 很多时候靠的不是读 storage，而是监听事件

事件设计好不好，直接影响系统是否可被观察、可被审计

这让我意识到，合约并不是“自己跑完就算了”，而是天然服务于链下世界的。

5、对安全问题的认识更现实了

通过助记词陷阱、抢跑 Bot、modifier 权限限制的讲解，我对“安全”有了更现实的理解：

很多攻击不是黑科技，而是利用你对规则的不熟

modifier 本质就是把“谁能做什么”写死在规则里

EVM 是单线程的，不存在 Web2 那种并发，但有重入风险

安全不是写完代码后再补，而是一开始就要想清楚“谁有权动账本”。

6、对 Solidity 学习路径的认知变化

今天最大的一个认知变化是：

Solidity 的难点不在语法，而在逻辑与结构设计。

语法能很快学会，但：

storage 怎么设计

权限怎么拆

事件怎么发

边界条件怎么防

这些才是真正拉开差距的地方。

老师最后提到去读 Uniswap V3 和 Aave 的源码，让我意识到，下一步学习已经不是“多写 demo”，而是开始看成熟协议是怎么做结构取舍的。

一个让我印象很深的点是对 HODLer 和 BUIDLer 的区分。持币者更在意短期价格表现，建设者更在意协议能不能长期活下去，这两类人的目标本身就不完全一致。如果简单地把“钱”和“权”绑在一起，冲突几乎是必然的。这让我意识到，很多 DAO 治理失败，并不是执行问题，而是一开始的权利设计就有问题。

LXDAO 提出的双轨制方案让我第一次意识到，“去中心化”并不等于“所有东西混在一起”。把金融资本和人力资本拆开，让 Token 更偏向收益权，让 Builder Card 代表治理权，本质上是在把公司里早就存在的 LP / GP 结构，用链上方式重新实现一遍。这种思路对我冲击挺大的，因为它并不是在否定现实世界的制度，而是在用代码把成熟的制度重新表达出来。

今天还讲到一个更大的视角，就是从传统 IPO 到 Web3 的资产编程。传统金融里，收益权、治理权、控制权高度绑定，导致效率低、合规成本高；而 Web3 的优势不在于“更自由”，而在于可以把权利拆开、精确配置。金融层、控制层、执行层分开之后，很多以前靠人拍脑袋决定的事情，就可以交给代码自动执行。

这一点也和我这一周反复听到的“代码和法律的关系”形成了呼应。不是用代码取代法律，而是把确定性的、可自动化的部分交给代码，把需要解释、裁量和兜底的部分留给法律。这种分工让我对 Web3 的认知变得更加现实，也更加理性。

如果说这一周前几天的学习让我更多是在“补概念”“听不懂但先记住”，那今天的周会更像是帮我把很多零散的东西串了起来。开始意识到 Web3 讨论的，其实不是某一个技术点，而是权力怎么分、责任怎么界定、激励怎么设计这些非常基础、但也非常现实的问题。

总体来说，这一周的学习让我从一开始的“觉得新、觉得复杂”，慢慢转向“开始理解为什么会这么设计”。虽然很多细节我现在还不懂，但至少对这个方向在解决什么问题，有了一个比之前清楚得多的整体认知。这种从“听热闹”到“看结构”的转变，本身就是这一周最大的收获。

# 2026-01-15
<!-- DAILY_CHECKIN_2026-01-15_START -->










今天这场会议主要围绕 AI 和 Web3 的结合展开，说实话内容非常多，也比较偏前沿，一开始还能跟着节奏走，后面信息密度越来越高，但整体让我对“AI agent + Web3”这个方向有了一个比之前清楚得多的轮廓。

最开始的分享让我意识到，AI 正在从一个“只会聊天的工具”，慢慢变成一个“能自己干活、能花钱、能赚钱”的角色。以前的 AI 更像是嘴替，你问一句它答一句，但现在讲的 agentic AI，是可以有记忆、有工具、有执行能力的，能真正替人完成任务。这一点让我第一次认真去想：如果 AI 能自己执行操作，那它在系统里到底算什么？

接下来逐渐引出了一个核心问题：AI 很强，但它在现实系统里其实是个“黑户”。它没有明确的身份，不能被验证是谁的 AI；它的行为日志大多存在中心化服务器里，事后可以改；它也没法像人一样注册账号、绑信用卡、完成 KYC。这些问题在 AI 只是聊天工具的时候还好，但一旦 AI 开始代表人做决策、花钱、签协议，就会变成非常现实的风险。

这时候 Web3 的价值就变得比较清楚了。区块链并不是为了“炫技”，而是刚好能解决几个 AI 当前绕不开的问题：身份、信任、审计和支付。区块链天生不可篡改、可验证、可编程，这些特性放在 AI agent 身上，刚好能补齐它缺失的那一部分。

会议里重点讲了 ERC8004 和 X402，我对它们的理解更多停留在“整体逻辑”而不是技术细节。ERC8004 给我的感觉更像是给 AI 设计的一套“身份证框架”，不是简单地说给 AI 一个 ID，而是拆成了几个 registry：谁是这个 agent、它的声誉记录、以及是否被某些官方或可信机制验证过。wachi 老师后面的补充让我理解得更清楚了一点，ERC8004 本身并不规定“声誉怎么打分”，它只是提供了一个结构，真正的评分规则、权重、是否有 dispute，其实是由具体的 registry 维护方来设计的。

wachi 老师用“大众点评”的类比让我一下子就明白了。可以把 ERC8004 想成一个点评平台：identity registry 就是“有哪些餐厅”，reputation registry 就是“评分体系”，不同用户的评分权重不同，而 validation registry 就像是官方榜单或者认证标签。协议并不关心你怎么算分，只提供一个大家都能用的底座。这让我意识到，很多 Web3 协议其实不是给答案，而是给“搭系统的积木”。

X402 更多解决的是“怎么付钱”的问题。传统互联网的支付方式对 AI 来说几乎是不可用的，不管是注册、验证码、信用卡，还是结算速度和手续费，都不适合高频、微额、机器对机器的交互。而 X402 把支付能力直接嵌进 HTTP 流程里，让“先付费、再服务”或者“承诺支付”变成协议层的一部分，这一点让我第一次觉得：原来 Web3 的支付并不是为了人，而是为了机器。

后面讲到合规和审计的时候，我的感受和之前几次合规会议其实是连在一起的。技术再酷，如果出了问题说不清“谁做的、为什么做、谁负责”，那一定会被现实世界拉回来。区块链在这里的作用，不是逃避监管，而是提供一种“可验证、不可篡改”的记录方式，让责任归属变得清晰。

Q&A 环节整体信息量也很大，但我最大的感受是：很多问题没有标准答案。比如 agent 升级后身份是否延续、声誉是否继承、防刷分是否能彻底解决，这些目前都还是“设计选择”，而不是定论。这也让我更清楚地认识到，ERC8004 这种东西本身还在非常早期的阶段，更像是一个方向，而不是成熟的行业标准。

# 2026-01-14

今晚这场关于 Web3 合规与刑事风险防范的会议，对我来说感触其实挺深的，也让我第一次比较认真地从“风险”和“边界”的角度去看 Web3 这个行业。

一开始的分享主要是从整体背景和合规框架讲起，让我意识到 Web3 并不是一个游离在现实法律体系之外的世界。很多人会觉得这是新技术、新模式，但在法律层面，很多问题并没有“消失”，只是换了一种形式出现。尤其是刑事风险这一块，并不是只有明显违法、主观作恶才会踩雷，很多时候风险来自于对规则的不清楚，或者对“灰色地带”的低估。

在讲到具体业务风险的时候，我逐渐意识到一个现实问题：很多在技术或产品视角下看起来很“中性”的事情，一旦涉及资金、交易、托管、撮合、收益分配，在法律视角下风险就会迅速放大。Web3 和现实世界之间并不存在一堵真正的墙，链上做的事，最终还是要落回到现实世界来承担结果。

个人风险这一部分也让我挺警醒的。以前可能会下意识觉得，只要不是项目的核心负责人，风险应该有限，但这次分享让我意识到，参与本身就可能带来责任。不论是技术、产品、运营，还是参与决策、推广，只要角色边界不清、职责模糊，都有可能在出问题时被牵连进去。

而对我来说，Q&A 环节可以说是本次会议的高潮。有些问题老师并没有完全展开讲，但那种“点到为止”的回答反而更让人印象深刻。很多事情确实不适合明说，但该懂的都能听懂。那一刻我明显感觉到，这个行业里有些经验不是写在 PPT 上的，而是需要自己慢慢体会、保持警惕的。

最大的感受就是一句话：一定要保护好自己的合法权益。无论未来是以什么身份参与 Web3，是做技术、做产品，还是参与项目合作，清楚边界、留好证据、保持风险意识，可能比一时的机会更重要。

结合我自己的背景，我之前做过产品经理，现在读的是计算机相关专业。这次会议让我更加清楚地认识到，如果以后真的想在 Web3 这个方向长期发展，只懂产品或者只懂技术都不够，至少要对合规和风险有基本的敬畏感，知道哪些事情能做，哪些事情需要格外谨慎。

总体来说，这次学习让我对 Web3 的理解更加现实，也更加冷静了一些。它确实有机会、有创新，但同样伴随着责任和风险。对现在的我来说，这样的认知非常重要，也算是一次很及时的提醒。

后面讲到 gas fee 的时候，也算是比较直观的概念，大概明白 gas 就是手续费，出价高更容易被打包，但并不是简单的“给得越多就一定最快”，而是和网络拥堵、区块空间有关。这部分让我第一次意识到，链上的“快慢”其实是一个资源分配问题。

再往后开始涉及 validator、服务器分布、客户端占比这些内容时，就明显开始吃力了。比如验证节点主要集中在哪些国家、为什么要降低某些客户端占比来减少风险，这些内容我知道很重要，但当下更多是“听过了”，理解还停留在表面。

真正让我开始完全懵掉的，是后面讲以太坊整体定位、分片、Dencun 升级、blob 这些内容。像是把 blob 比喻成“挂在主车上的小挂车”这个说法，其实挺形象的，但一旦开始讲它怎么帮 L2 降低成本、怎么减轻 L1 的存储压力，我就明显感觉到自己对底层机制的理解还差得很远，只能知道一个大概方向，却搞不清具体是怎么运作的。

不过有一点让我印象很深的是，Dencun 升级之后，L2 的交易成本能下降非常多，甚至能降到 90% 以上。这让我第一次比较清楚地意识到，以太坊现在做的很多升级，并不是为了让普通用户直接感知到变化，而是为了让整个生态在长期内跑得更稳、更便宜。

最后讲到 ETH 本身的作用时，反而又回到了一个相对好理解的层面。ETH 不只是“币价”，它既是手续费、是质押保证金、是安全机制的一部分，同时在应用层又是记账单位、抵押品、NFT 铸造和交易用的“链上现金”。这一点让我感觉 ETH 更像是以太坊这个系统里的“燃料 + 规则核心”，而不只是一个可以交易的资产。

整体听下来，我最大的感受是：这些东西确实不简单，也不是一两次分享就能真正搞懂的。很多底层逻辑对现在的我来说还是偏生硬，听的时候会有明显的断层感，有些地方甚至完全不知道自己没听懂的是哪一块。但与此同时，我也更清楚地意识到，以太坊和 Web3 并不是靠概念撑起来的，而是有非常复杂、严肃的技术和机制在背后支撑。
