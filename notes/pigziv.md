## Dapp 架构

-   **前端 (UI)**
    
    -   使用 React/Vue 等框架
        
    -   通过钱包 Provider 或 RPC 节点与区块链交互
        
    -   钱包（如 MetaMask）负责身份验证和交易签名
        
-   **智能合约 (Smart Contracts)**
    
    -   核心业务逻辑，部署在区块链上
        
    -   以太坊常用 Solidity 编写，运行在 EVM
        
-   **数据检索器 (Indexer)**
    
    -   捕获合约事件（如 NFT Transfer）
        
    -   将数据写入数据库（如 PostgreSQL）供前端查询
        
-   **区块链与去中心化存储**
    
    -   区块链存储状态和交易
        
    -   IPFS/Arweave 存储图片、文档等非结构化数据
        

## 🛠 Dapp 开发流程

1.  **需求分析与规划**
    
    -   确定功能需求、选择区块链平台、设计用户体验
        
2.  **智能合约开发**
    
    -   编写、测试、审计合约
        
3.  **检索器开发**
    
    -   确定前端所需数据，编写事件处理逻辑，部署运维
        
4.  **前端开发**
    
    -   使用现代框架，集成钱包，展示链上数据，处理交易签名
        
5.  **链上交互**
    
    -   使用 Viem/Ethers.js/Wagmi 读取数据、发送交易
        
6.  **部署与上线**
    
    -   合约部署（Hardhat/Foundry）
        
    -   前端部署（IPFS/Vercel）
        
    -   收集反馈并维护更新
        

## ⚙️ 以太坊开发环境

-   **基础环境**：Node.js、npm/yarn、Git
    
-   **本地开发链**
    
    -   Foundry（Rust，快）：forge、anvil、cast
        
    -   Hardhat（推荐）：合约编写、测试、部署
        
-   **钱包与前端交互**：MetaMask + Viem/Wagmi
    
-   **其他工具**：Remix IDE、OpenZeppelin 合约库、Chainlink
    

## 🌐 RPC 节点服务

-   **作用**：读取链上数据、发送交易、监听事件、切换网络
    
-   **协议**：JSON-RPC 2.0（HTTP/WebSocket）
    
-   **常用方法**：eth\_getBalance、eth\_blockNumber、eth\_sendTransaction、eth\_call、eth\_getLogs
    
-   **主流服务商对比**

    -   Infura：老牌，适合测试和中小项目
        
    -   QuickNode：高性能，多链支持
        
    -   Public Node：免费，适合学习
        
    -   Ankr：多链去中心化节点
        
-   **最佳实践**
    
    -   API Key 管理（环境变量）
        
    -   错误处理与重试
        
    -   速率限制与节流

### 一、 实战中的部署成本控制

在实战中，建议通过以下步骤预估和控制 Gas 成本：

-   **本地/测试网预演**：使用 Hardhat、Foundry 或 Truffle 在本地链或测试网部署，通过日志（如 `Contract deployment: ... gas`）查看 `gasUsed`。
    
-   **监控实时 Gas**：上线主网或 L2 前，利用 Etherscan 或 Gas tracker 检查当前的 Base Fee 和建议的 Priority Fee。
    
-   **代码优化**：
    
    -   将构造函数中的大数组/固定配置改为后续按需写入。
        
    -   使用 Library 复用逻辑以减少重复代码。
        
    -   开启 Solc optimizer（编译优化），在字节码大小和执行成本间取得平衡。
        
-   **优先 L2 部署**：如果逻辑允许，优先在 Rollup (L2) 上部署以节省费用。
    

* * *

### 二、 常见安全漏洞与防范措施

以下是核心漏洞及其修复方案的速查清单：

1\. 核心逻辑与资金安全

-   **重入攻击 (Reentrancy)**：攻击者利用回调反复提取资产。
    
    -   **防范**：采用 **CEI 模式**（检查-生效-交互）；使用 `nonReentrant` 修饰符；采用 **Pull-Payment**（让用户自己领钱）模式。
        
-   **发送 ETH 失败**：依赖 `transfer()`/`send()` 的 2300 gas 津贴已不可靠。
    
    -   **防范**：使用低级调用 `call{value: ...}("")` 并检查返回值；或使用 OpenZeppelin 的 `Address.sendValue`。
        
-   **整数溢出**：Solidity <0.8.0 存在绕回风险。
    
    -   **防范**：使用 Solidity ≥ 0.8.x（自带检查）；仅在确认安全时使用 `unchecked` 做 Gas 优化。
        
-   **业务逻辑缺陷**：如价格公式、利率模型漏洞。
    
    -   **防范**：编写覆盖边界的单元测试；使用 **Fuzzing (模糊测试)** 和 **Invariant Testing (不变式测试)**。
        

2\. 权限与外部交互

-   **访问控制错误**：
    
    -   **防范**：使用 `Ownable`/`AccessControl`，关键操作加多签和时间锁；**禁止使用** `tx.origin` 进行鉴权（防钓鱼）。
        
-   **不可信外部调用**：低级调用未检查返回值。
    
    -   **防范**：显式检查 success；与 ERC-20 交互时使用 `SafeERC20` 处理非标准返回值。
        
-   **Delegatecall 风险**：上下文共享可能导致存储被篡改。
    
    -   **防范**：严格限制目标地址为白名单或自家合约。
        

3\. 操纵与环境风险

-   **预言机操纵**：依赖单一或瞬时价格容易被闪电贷攻击。
    
    -   **防范**：使用 Chainlink 等多源预言机；采用 **TWAP**（时间加权平均价格）；设置滑点限制。
        
-   **随机数不安全**：`block.timestamp` 等链上属性可被操纵。
    
    -   **防范**：使用 Chainlink VRF 或 Commit-Reveal 方案。
        
-   **DoS 攻击**：循环转账或恶意 Revert 导致合约卡死。
    
    -   **防范**：避免无上限循环；使用 Pull-Payment 替代批量打款；妥善处理外部调用失败。
        

4\. 升级与新特性

-   **可升级合约风险**：存储布局冲突或初始化被接管。
    
    -   **防范**：遵循 OpenZeppelin 标准；构造函数中禁用初始化器。
        
-   **SELFDESTRUCT 变更**：Dencun 升级 (EIP-6780) 后，`selfdestruct` 仅在同笔交易创建的合约中销毁代码，否则仅清空余额。
    
    -   **注意**：不再依赖其作为“清除状态/升级”的手段。

### **Web3 DApp 开发核心笔记**

**1\. DApp 架构解析** 去中心化应用（DApp）与传统应用的核心区别在于其逻辑和数据由分布式网络维护，而非单一实体控制。DApp 的架构主要包含四个部分：

-   **前端 (User Interface)：** 使用 HTML/JS (React/Vue) 构建。关键区别在于前端不直接连接区块链，而是通过**钱包**（如 MetaMask）注入的 Provider 或第三方 **RPC 节点**进行交互。只读操作通过 RPC，写操作需钱包签名。
    
-   **智能合约 (Smart Contracts)：** DApp 的核心后端，通常使用 Solidity 编写并部署在 EVM 上，负责业务逻辑与资产安全。
    
-   **数据检索器 (Indexer)：** 这是一个连接链上与链下的关键组件。由于直接从合约查询复杂数据（如“某用户持有的所有 NFT”）效率极低，检索器负责监听合约释放的 **Event（事件）**，将数据清洗并存入传统数据库（如 PostgreSQL），供前端快速读取。
    
-   **去中心化存储：** 针对图片、文档等大文件，使用 IPFS 或 Arweave 存储，保证数据的去中心化与持久性。
    

**2\. 开发生命周期** DApp 的开发遵循严谨的流程：

-   **需求与规划：** 确定功能、选择公链（以太坊、Layer2 等）并设计 UX。
    
-   **合约开发：** 编写 Solidity 代码 -> 编写测试用例（单元测试） -> 安全审计（防范重入攻击等漏洞）。
    
-   **检索器与前端并行：** 后端搭建检索器处理数据；前端集成钱包，使用 **Viem** (推荐)、**Wagmi** 或 **Ethers.js** 等库与合约交互。
    
-   **部署上线：** 将合约部署至测试网（Sepolia）或主网，前端部署至 Vercel 或 IPFS。
    

**3\. 开发环境与工具栈** 搭建高效的开发环境是第一步：

-   **基础环境：** 必须安装 Node.js (推荐用 nvm 管理) 和 Git。
    
-   **开发框架选择：**
    
    -   **Hardhat：** 基于 JavaScript/TypeScript，生态成熟，是目前的行业标准框架。
        
    -   **Foundry：** 基于 Rust，速度极快。提供 `forge` (构建/测试)、`anvil` (本地节点) 和 `cast` (交互工具)，是现代开发者的热门选择。
        
-   **RPC 服务：** RPC 是连接应用与区块链的“ATM 机”。开发时可使用 Infura 或 Public Node，生产环境建议使用 Alchemy 或 QuickNode 以保证稳定性。

### 1\. 2025 年安全态势总览

**总体数据**：2025 年 Web3 领域形势严峻，共发生超过 **1200 起**较严重的安全事件，总损失金额超过 **35 亿美元** 。

### 2\. 主要攻击类型与数据

A. 精准猎杀 (针对机构/高净值目标)

**CeFi 风险**：CeFi 是黑客的主要“提款机”。在单体损失超 3000 万美元的 12 起事件中，CeFi 占了 7 起 。主要原因是管理员私钥被盗和热钱包被盗。

**DeFi 漏洞**：虽然 DeFi 抗压性有所提升，但攻击者利用 AI 技术挖掘深层漏洞的能力增强，Move 智能合约漏洞明显增多 。

**典型重灾区**：

**Bybit**：损失约 **15 亿美元**（最大单案），原因是 Safe 多签服务器遭渗透篡改前端交易指令 。

**Cetus Protocol**：损失约 2.23 亿美元，系 Move 智能合约漏洞 。

**国家级黑客**：朝鲜黑客组织（如 Lazarus Group）在 2025 年至少盗取了 20.2 亿美元，同比增长 51%。他们常伪装成招聘人员，通过面试诱导下载后门代码 。

B. 广撒网 (针对普通用户)

**私钥窃取**：发生超 300 起，损失超 18 亿美元 。

**钓鱼攻击**：发生超 400 起，损失超 7.5 亿美元。AI 生成钓鱼内容和签名陷阱让其更难辨别 。

**Rug Token (跑路)**：发生约 300 起，呈现职业化和团伙化趋势 。

### 3\. 典型案例与技术解密

文档详细分析了三个具有代表性的技术攻击案例：

**Bybit (供应链攻击 + 前端篡改)**：黑客入侵了 Safe Wallet 的前端代码。当管理员进行多签时，前端显示正常，但底层数据被篡改。管理员因“盲签”（无法在硬件钱包上完全解析复杂交易）而中招 。

**Balancer (计算精度漏洞)**：由于 Vault 在 Swap 计算时存在向下取整的精度损失，攻击者通过批量操作（batchSwap）放大这一微小误差来获利 。

**EIP-7702 新型钓鱼**：以太坊新提案被黑客武器化。攻击者利用“委托（Delegation）”机制，诱导用户将权限委托给恶意合约，一旦转账资金会被自动提走 。

5.  关于 Token Economics 的核心拆解：
    
    ### 1\. 核心三大支柱 (The Iron Triangle)
    
    一个好的代币经济模型通常围绕这三个维度进行博弈和平衡：
    
    -   **供给 (Supply)：** 钱从哪里来？发多少？
        
        -   **最大供应量 (Max Supply)：** 像比特币只有2100万枚，制造稀缺性。
            
        -   **释放/通胀机制 (Emission/Inflation)：** 代币是像水龙头一样慢慢流出（如挖矿奖励），还是一次性全部解锁？过快的释放会导致抛压（通胀），过慢则可能缺乏激励。
            
        -   **销毁机制 (Burning)：** 类似股票回购。项目方将一部分代币永久移除（打入黑洞地址），从而减少流通量，试图提升币价（通缩）。例如 Ethereum 的 EIP-1559 协议每笔交易都会销毁部分 ETH。
            
    -   **需求 (Demand)：** 为什么要买这个币？
        
        -   **效用 (Utility)：** 最基本的需求。比如必须用 ETH 支付 Gas 费，用 Uniswap 的代币投票，或者在游戏中用代币买装备。
            
        -   **价值捕获 (Value Capture)：** 持有代币能分红吗？比如持有某些 DEX 的代币可以分得交易手续费。
            
        -   **质押/锁仓 (Staking/Locking)：** 为了获得更高收益或投票权，用户愿意把币锁起来不卖，这减少了市场上的抛售压力。
            
    -   **分发 (Distribution)：** 谁拿到了筹码？
        
        -   **早期分配：** 团队、VC（风投）、社区各占多少比例？如果团队和 VC 占比过高且解锁期很短，散户很容易被“割韭菜”。
            
        -   **空投 (Airdrop)：** 免费发给早期用户以获取流量和忠诚度。
            

6、了解web2和web3的获客方式：**Web2 靠“买流量”和“数据垄断”**，而 **Web3 靠“分润”和“社区共识”**。

Web2 的逻辑是\*\*“羊毛出在羊身上”**（用户是产品，数据被卖给广告商）；而 Web3 的逻辑是**“给羊发股份”\*\*（用户是股东，持有代币，通过利益绑定一起建设）。

以下是具体的对比解析：

### 1\. 核心逻辑差异

| 维度 | Web2 获客 (漏斗模型) | Web3 获客 (飞轮模型) | | 核心手段 | 烧钱买量 (Paid Ads)：依赖 Google/Facebook 广告投放。 | 代币激励 (Token Incentives)：给用户发币，让用户通过持有资产获利。 | | 用户角色 | 消费者：被动接收服务，是平台变现的工具。 | 所有者/合伙人：既是使用者，也是利益共同体（Stakeholder）。 | | 数据基础 | 封闭数据：平台独占用户画像，谁数据多谁就是赢家。 | 链上公开数据：钱包地址和交易记录公开，可以直接去挖竞品的用户。 | | 增长引擎 | 转化漏斗：曝光 点击 注册 留存。 | 社区共识：空投预期 早期参与 治理/持币 自发传播。 |

### 2\. Web3 独有的“降维打击”获客招数

由于区块链数据的公开性，Web3 诞生了几种在 Web2 时代完全不可能实现的获客方式：

A. 吸血鬼攻击 (Vampire Attack) —— 最生猛的“挖墙脚”

-   **Web2 很难做到：** 比如拼多多想抢淘宝的高端用户，它无法精准知道谁在淘宝花了很多钱，只能漫天撒广告。
    
-   **Web3 轻松做到：** 所有的交易记录都在链上。项目方可以直接筛选出竞争对手的“大户”地址。
    
-   **经典案例 (SushiSwap vs Uniswap)：**
    
    -   SushiSwap 作为一个新平台，想抢 Uniswap 的用户。它直接宣布：**只要你把在 Uniswap 里的资金取出来，存到我这里，我就给你发 SUSHI 代币（能分红、能卖钱）。**
        
    -   **结果：** 几天之内，大量资金和用户为了利益直接“搬家”。这种**直接给竞品核心用户发钱**的精准攻击，效率极高。
        

B. 空投 (Airdrop) —— “回溯性”奖励

Web2 的补贴通常是“注册送红包”，羊毛党刷完就走。Web3 的空投往往是**事后惊喜**。

-   **逻辑：** 项目方不提前明说有奖励。等你真实使用过一段时间后，突然根据你的交互记录宣布：“感谢早期支持，给每个活跃钱包发 1000 个代币（可能价值数千美元）”。
    
-   **效果：** 这种“意外之财”会制造巨大的财富效应和忠诚度。用户会因为手里持有代币，希望币价上涨，从而自发成为“死忠粉”去向身边人宣传项目。
    

C. 任务平台与链上凭证 (Quest & Credentials)

-   **平台：** 如 Galxe, QuestN。
    
-   **玩法：** 项目方发布任务（如：关注推特、加入 Discord、进行一次跨链交易），用户完成后获得一个 NFT 勋章或 OAT（链上成就证明）。
    
-   **意义：** 这不仅仅是积分墙，这些凭证构建了用户的“链上履历”。用户为了积累履历（博取未来获得更多空投的概率），会主动去完成这些交互任务。
    

### 3\. 具体场景对比：如何从 0 到 1

-   **Web2 冷启动：**
    
    -   找 VC 拿钱 $\\rightarrow$ 购买 App Store 排名/搜索引擎关键词 $\\rightarrow$ 优化落地页转化率 $\\rightarrow$ 烧钱换用户增长（关注 CAC 获客成本）。
        
-   **Web3 冷启动：**
    
    -   写白皮书 $\\rightarrow$ 建立 Discord/Telegram 社区 $\\rightarrow$ **画饼（许诺未来价值）** $\\rightarrow$ 发布白名单 (Allowlist) 机制 $\\rightarrow$ 告诉用户“只有前 1000 个进群的人才有资格低价铸造 NFT” $\\rightarrow$ 利用 **FOMO (错失恐惧症)** 让用户自己去拉人头。
        

### 总结

-   **Web2 获客像“钓鱼”：** 需要不断买昂贵的鱼饵（流量费），目的是把鱼钓上来吃掉（变现）。
    
-   **Web3 获客像“招募传教士”：** 你分给他们一部分未来的收益（Token），他们为了让手里的币更值钱，会主动去帮你拉更多的人进来。
    

Web3 获客的核心在于**将“营销费用”直接转化为了给用户的“空投奖励”**。
