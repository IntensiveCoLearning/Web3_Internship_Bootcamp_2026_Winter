# 链上数据及分析基础

-   **链上数据类型**：
    
    -   **交易数据**：包含转账资金量、收发地址、交易金额、时间、手续费等，可追踪资金流动。如 Alice 转以太坊给 Bob 的交易包含相关信息。
        
    -   **区块数据**：包含区块时间戳、打包者、交易信息和顺序等，保证区块链顺序一致性。
        
    -   **合约数据**：链上部署的智能合约代码，由开发者预先写好，执行逻辑自动化且不可更改。
        
-   **常见数据指标**：
    
    -   **总锁仓量（TVL）**：表示 Defi 生态中某协议或系统被锁定的加密资产总价值，衡量协议市场占有率和吸引力。
        
    -   **活跃地址数**：一定时间段内至少执行一次交易的钱包地址数量，反映用户基础和增长趋势。
        
    -   **交易数和交易量**：交易数指时间段内的交易数量，交易量指交易涉及的资产价值总额，评估链上网络健康和市场情绪。
        
    -   **gas 费用**：用户交易时支付给矿工或验证者的费用，反映网络拥堵程度和交易成本。
        
    -   **协议收入**：去中心化协议提供服务收取的费用，衡量协议盈利能力和给代币持有者的价值。
        
-   **数据分析工作流**：
    
    -   **数据提取**：从区块链 RPC 节点或 ethscan 获得原始交易数据，以非结构化日志形式存储。
        
    -   **数据转换**：基于 ABI 对链上数据日志和状态信息进行转换，恢复成原始函数或参数。
        
    -   **数据加载**：确定要分析的指标，对解析的交易条目进行聚合筛选，输出分析结果。
        
-   **数据分析工具**：
    
    -   **ethscan**：查询以太坊交易的必备工具，可查看交易哈希、gas 费用、活跃地址数等指标。
        
    -   **Dune**：可对原始数据进行提取和转换，用户可写查询语句获取数据结果，还能导出为 API。
        
    -   **root data**：类似加密天眼查，可查 Web3 项目信息，提供成熟指标，如项目基本信息、投融资信息等。
        
    -   **defillama**：专门统计 Defi 协议 TVL，还包含交易量、交易数量等指标，可自定义看板。
        
    -   **TOKEN terminal**：数据指标更全，服务合规机构，可查看协议各类指标并自定义看板。
        
    -   **其他工具**：Artemis 和 [RWA.xyz](http://RWA.xyz) 关注稳定币、真实世界资产等；Coinmarketcap 和 Coin Gecko 看行情数据；flip side 和 Alien 可替代 Dune 查看新兴协议数据。

# Vibe Coding & 黑客松经验分享

* * *

-   **黑客松的定义与意义**：
    
    -   **定义**：大家组小型互联网公司，在几天内爆肝做出一个产品。
        
    -   **意义**：包括拿高额奖金、作为入行跳板、宣传创业项目、结识志同道合朋友、积累创业经验、获得投资、享受餐饮和周边、周游世界、学习新技术、验证想法、锻炼表达能力等。
        
-   **isglobal 黑客松流程**：
    
    -   **前期准备**：报名、质押保证金，比赛当天 checking，可领取赞助商周边。
        
    -   **比赛过程**：有 workshop 了解项目方，之后开幕仪式，开发期间有娱乐放松活动和特色餐饮，场地提供睡觉地方。
        
    -   **后期提交**：时间到后上传项目视频和代码，向评委 pitch 项目，奖金分主赛道和赞助赛道，最后合影留念、宣布获胜者，还有 after party。
        
-   **黑客松参与角色与团队配置**：
    
    -   **参与角色**：除程序员外，还有产品经理、Pitchman、设计师等，甚至有气氛组。
        
    -   **团队配置**：建议 3 - 4 人，两个开发加一个产品加一个 Pitchman，团队最好提前组好，找熟悉靠谱的朋友。
        
-   **黑客松项目思路**：
    
    -   **迁移重构**：将其他链上成功商业模式挪到新链上。
        
    -   **乐高拼接**：组合赞助商资源做有趣项目，如新闻套利项目。
        
    -   **关注在地化**：结合当地情况做项目，如解决堵车问题的项目。
        
    -   **关注少数群体**：做关于宠物安全、女性安全等项目。
        
    -   **硬件结合**：结合主办方提供的硬件做产品，如结合手环钱包功能做投票产品。
        
-   **黑客松注意事项**：
    
    -   **提交时间**：不要踩点交，可提前交不完整版本再补全。
        
    -   **时间区分**：区分好 12PM（中午 12 点）和 12AM（凌晨 0 点）。
        
    -   **提交渠道**：关注官方通知渠道，注意提交资料不等于提交成功，要点击保存和 submit 按钮。
        
    -   **心态调整**：不要抱着创业干翻大事业的想法，完整呈现想法比做出完善产品更重要，不要放弃。
        
-   **黑客松工具推荐**：
    
    -   **语音生成**：Ttmaker 免费，Openai 的 FM 效果更好，可将文字转语音用于剪辑。
        
    -   **视频剪辑**：剪映适合新手，操作简单。
        
    -   **AI 辅助**：支持使用 cloud code。
        
    -   **PPT 生成**：自己做白稿后用伽马 APP 或 Gemini 优化页面，也可前端生成。
        
    -   **翻译工具**：deepdel 逐句翻译，沉浸式翻译是浏览器插件。
        
    -   **开发框架**：区块链行业参赛者推荐 nextjs，还有区块链加前端的框架可用于小众链开发。
        
-   **黑客松获奖技巧**：提前布局，有些比赛可提前读题想好项目，上场再写代码。
    
-   **不同人群黑客松选择建议**：
    
    -   **新手**：先打离自己近且轻松的黑客松，多打几场再考虑其他。
        
    -   **创业选手**：直接跟主办方聊 Demo 松、build 松，了解拿奖情况。
        
    -   **经费充足者**：打世界级的黑客松 is global。
        
    -   **经费不足者**：可申请 is Panda 和 GCC 的机票赞助。
        
-   **Web Coding 的概念与趋势**：
    
    -   **概念**：翻译为氛围编程，靠自然语言操作 AI 写代码，是对人类脑力的解放和能力的辅助。
        
    -   **趋势**：是未来的趋势和必备技能，但做 Web Coding 需懂点代码。
        
-   **Web Coding 工具与模型**：
    
    -   **工具**：代码少可丢给 GPT，复杂代码用 IDE 或 TUI，如 windsurf、cursor、国内的 trae 等。
        
    -   **模型**：Claude code 适合 90% 写代码场景，code x 适合依赖关系复杂的代码，Gemini 效果一般。
        
-   **以 Claude code 为例的 Web Coding 使用方法**：
    
    -   **交流要点**：清晰表达需求，不定时查看效果，提供好的上下文，做好上下文管理。
        
    -   **上下文管理**：拆解问题，合理利用 plan 模式，多使用 context 命令查看占用情况，勤 compact，使用官方 plugin 补全需求漏洞，多向 AI 请教。
        
-   **Web Coding 相关功能介绍**：
    
    -   **MCP**：封装给 AI 方便调用的文档，可调用外部服务。
        
    -   **agent**：可独立运行，主 agent 可命令子 agent，用于不同功能操作，可分离上下文。
        
    -   **skill**：预置的 promote 加脚本，方便 AI 操作。
        
    -   **plugin**：几个功能的打包，安装方便，如 code review、context 等插件。
        
    -   **LSP 插件**：让 AI 读代码时建立内在联系，提高找相关代码的能力。
        
-   **做好看 UI 的思路**：
    
    -   **找模板**：从参考网站找适合的模板，发给 AI 总结风格生成 demo 再修改。
        
    -   **生成 ASCII 布局图**：让 UI AI 先生成 ASCII 布局图，使 AI 实现效果更符合预期

# 投研基础框架建立

* * *

-   **Web3 投研现状**：
    
    -   **投研暴富难实现**：当前市场下，靠传统投研暴富思路进入 Web3 不可行，山寨币没落，流动性不足，按以往方法论投研效果不佳。
        
    -   **行业存在问题**：全球经济处于波谷，行业代币质量差，交易所上币存在问题，川普搅局收割大量资金，导致市场流动性几乎枯竭。
        
-   **投研的本质与对象**：
    
    -   **投研本质**：主要针对山寨币，通过研究项目基本面、背景和代币经济模型等，选中有潜力项目低买高卖获利。
        
    -   **研究对象分类**：包括主流币（如比特币、以太坊）、山寨币（背后有项目主体）、稳定币（交换媒介或价值尺度）和 MEME 币（无项目主体）。主流币无需投研，稳定币和 MEME 币也不是主要投研对象。
        
-   **从业者发展建议**：
    
    -   **学习行业历史**：学习历史上优秀项目，如去中心化交易所、借贷、稳定币等赛道的龙头项目，塑造行业价值观和知识体系，提升认知水平，利于找工作。
        
    -   **建立知识体系**：按赛道研究龙头项目，由点到面，如研究 Defi、Gamefi、算法稳定币等赛道，积累行业知识，提升自身价值。
        
    -   **保持学习心态**：不仅学习 Web3 知识，还应关注新兴行业，如 AI、云原生、芯片等。
        
-   **特定问题解答**：
    
    -   **撸空投投产比**：空投本质是项目方拉新道具，不好计算投产比，且依赖项目方，大量投入此方向可能走偏。
        
    -   **VC 投资项目**：研究项目基本面后投资，若币价无表现，可能不是好项目，不应盲目等待。
        
    -   **RWA 方向**：RWA（将现实资产上链交易）是未来发展方向，从业者应兼顾传统金融知识和区块链底层技术，成为两栖人才。
        
    -   **项目研究方法**：初期可通过可视化视频、AI 辅助、看白皮书和解析文章等方法研究项目。
        
    -   **判断零和游戏项目**：先学习 Web3 基础知识，积累足够知识后再考虑判断项目是否为零和博弈，投资可调整思路，提升自身知识。

## 一、Web3 公共物品与资金分配机制

### 公共物品的核心问题

公共物品具有 **非排他性、非竞争性**，例如：

-   开源代码
    
-   协议安全
    
-   开发者工具与文档
    

在 Web3 生态中，它们就像“空气和阳光”，一旦枯竭，整个生态都会受到影响。

### 搭便车问题与市场失灵

由于所有人都可以免费使用公共物品，理性选择往往是等待他人付费，最终导致：

-   贡献不足
    
-   市场无法为公共物品合理定价
    

课程强调：  
**我们不是在做慈善，而是在投资长期价值。**

### 三种典型资助机制

1.  **委员会制**
    
    -   优点：效率高、决策快
        
    -   缺点：权力集中、偏见风险
        
2.  **二次方资助（QF）**
    
    -   资金分配更多取决于参与人数而非金额
        
    -   放大社区共识，强调公共性
        
3.  **回溯性资助**
    
    -   先创造价值，再获得奖励
        
    -   为“结果”付费，而非“承诺”

# Key Hash Based Tokens:从ERC-721到ERC-7962

* * *

## 学习总结

* * *

### NFT

（非同质化代币）是基于ERC-721的一个实践，在现实世界中通常表达为一个藏品

商业组织或者机构在发展社区的过程中，用户的标识在Web3世界里往往会以某个ETH地址体现。组织会部署ERC-721智能合约，通过mint给予用户身份。

**理解：**ERC-721 是一种“每个 Token 都不一样”的标准，NFT 只是它最常见的一种应用形式。  
NFT 真正的价值不在“图片”，而在于：**链上唯一性 + 可验证的所有权**，图片只是表现形式之一

在 Web3 中，**钱包地址就相当于账号，**地址是公开且可追踪的，这会带来隐私风险

* * *

### 现实需求

会员身份属于隐私数据，但是如果把ZK应用其中，就能够在不暴露隐私的情况下进行数据交易，在现实社会中非常有意义。

ERC-7962 就是基于这样的现实需求构思并实现。

**理解：**现实中，“你是不是会员”“你有没有资格”本身就是隐私，而不是公开信息。  
但现在很多 NFT 会员是直接绑定地址的，别人一查地址，就能知道你所有资产和行为。  
👉 所以真实需求是：**能验证资格，但不暴露身份和资产全貌**。

* * *

### **零知识证明**

在不暴露具体细节的情况下，能够向第三方证明数据的某些特性。比如，有两个整数值，第三方关注其和或者乘积是某个值，但是不能暴露具体值。

提供Proof(X，Y)，向第三方证明X+Y=Z，但是Proof (X，Y)无法反推出X和Y的具体值。

**理解**：我能证明我满足条件，但不告诉你我是怎么做到的。

* * *

### ERC-7962

是在ERC-721的基础上应用ZK原理保证了向第三方证明其归属，但是不暴露其真实地址

**理解：**改变了**资产归属的证明方式**：  
不再通过“地址持有”，而是通过“你是否能证明你掌握某个密钥”

* * *

### **核心概念**

-   以KeyHash(公钥哈希)而非地址作为资产所有权标识
    
-   所有权与交易发起地址解耦，不在链上存储地址
    
-   覆盖ERC-20 与ERC-721两类资产模型
    
-   用户只需签名表达意图，无需持币、无需理解Gas、无需管理地址隐私，也能完成资产操作
    

**理解**：链上不关心你“用哪个地址”，只关心“这是不是你的签名”

-   Gas 可以别人帮你付
    
-   交易可以别人帮你发
    
-   用户只负责“签名确认”
    

* * *

### **解决的关键问题**

-   _隐私问题:避免地址可追踪，降低身份与资产的关联性_
    
-   _UX问题:用户只需签名表达意图，无_需持币、无需理解Gas、无需管理地址隐私，也能完成资产操作。
    
-   扩展性问题:天然支持批量交易、Relayer、账户抽象场景
    

**理解**：降低 Web3 的使用门槛，可以带入更多 Web2 的人

* * *

### **技术实现要点**

所有权由keyHash=keccak256(publicKey)表示，转账时提交:

-   公钥(key)
    
-   EIP-712签名
    
-   nonce +deadline (防重放)
    
-   通过ECDSA验证签名人与keyHash 的一致性
    

* * *

### **ERC-KeyHash721 (NFT)**

-   tokenld -keyHash映射关系
    
-   ownerOf返回keyHash而非地址
    
-   无approve机制，强调一次性key/ key轮换
    

* * *

### **ERC-KeyHash20 (FT)**

-   keyHashbalance余额模型
    
-   转账采用UTXO风格拆分
    
-   转出部分 toKeyHash
    
-   剩余部分  leftKeyHash
    
-   强制key轮换，增强不可关联性
    

**理解**：类似比特币的思路：

-   用一部分
    
-   剩下的转到“新身份”
    

这样可以避免**长期使用同一身份暴露交易轨迹**

* * *

### **Gas 与交易模型**

-   任何地址都可提交交易
    
-   所有权只由签名控制
    
-   天然支持:
    
-   Gas Sponsorship
    
-   批量转账
    
-   Relayer / Paymaster
    

* * *

### **安全设计**

-   EIP-712结构化签名，防参数篡改
    
-   per-keyHash nonce，防重放攻击
    
-   deadline控制签名有效期
    
-   强制low-S/v校验，防签名可塑性
    

* * *

### **优势总结**

-   更强隐私(地址不上链)
    
-   更佳用户体验(免ETH/Gas代付)
    
-   更高灵活性(批量、代付、抽象账户)
    
-   与 ERC-5564 (StealthAddress)理念一致
    

**理解：**核心优势不是“技术更复杂”，  
而是：**用户体验更接近 Web2，但仍保持链上安全性**

* * *

### **局限与代价**

-   Gas成本高于传统ERC-20/ERC-721
    
-   公钥在 calldata 中暴露(需key轮换)
    
-   与现有ERC标准不向后兼容
    
-   对密钥管理要求更高
    

**理解：**这说明ERC-7962目前还是一种面向未来的探索，而不是能够立马替代现有标准。

# 今日Zoom会议总结：

## Web3运行原理

**目标：**学习Web3运行原理：从原理出发理解Web3怎么“跑起来”

## 钱包、私钥和个人主权

**私钥（Private key）**：相当于密码，通过私钥导入到钱包，可以得到一个钱包

（不能泄露你的私钥，私钥是你的密码）

**助记词（Seed Phrase）**：私钥的可读备份（通过派生路径可以生成多个账户）

**地址**：公钥截取后的字符串，相当于公开的收款号

**核心关系：**助记词 → 私钥 → 公钥 → 钱包地址

安全性建议：私钥是非常安全的，不可重置（丢了只能换钱包），区块链不认识你是谁，只认识谁能“签名”

大部分资产被盗:不是链被攻破，而是私钥or助记词泄露或授权被钓鱼

安全原则：不截图，不发人，不网盘

**私钥的社会学意义**：个人主权的起点

任何人都可以创建私钥→立刻拥有一个无需许可的账户与资产权利

对比传统金融：开银行账户需要各种资料

* * *

## 交易与签名

### 交易是什么：

交易（tx）=你要做的事+手续费+防重放序号

你可以做：转账、调用合约、投票等操作

手续费：为网络资源付费(Gas）

防重放序号：避免同一交易被重复执行（Nonce）

### 钱包在做什么：

1.组装交易内容

2.用私钥签名

3.广播到区块链网络

### 数字签名是什么：

核心：你用私钥签名一段信息，任何人都可以用你的地址验签，但是没有人能伪造你的签名

待签名交易信息→生成本地签名→广播到网络

### Gas是什么？

Gas就是网络费用

是激励机制之一，没有激励，网络很难长期稳定运行

* * *

## 区块链网络运行

从一笔交易到出块需要经历：**签名→传播→排队→挑选→落盘→可查**

ps：以太坊平均每 12 秒出一个区块，区块永远指向上一个区块（链式结构），这就是为什么“**不可篡改需要时间**”的原因

**共识机制**：**PoW vs PoS**

**问：**匿名且互不信任的全球节点，如何任何同一本账？

**答：**PoW：工作量证明（btc使用的）

像一道题：谁先解出谁记账，全网快速验算

PoS：权益证明（ETH使用的）

像交押金：押金越多，信用好，概率越高；作恶会罚没押金

* * *

## 钱包、RPC、节点、网络：如何连接

钱包通常不直接连全网，需要连一个RPC URL

RPC背后是节点，提供网络服务

* * *

## 智能合约

### 智能合约的本质=区块链账本里的“可执行代码”

### 区块链协议如何升级

以以太坊为例：有ETH论坛，先讨论-再形成文档-再进入升级（每一年1到2次）

每次升级都是一次硬分叉，变成另外一条链

**关键点**：不是某公司拍板，而是社区协作推进

### 节点客户端多样性

节点软件：执行层客户端+共识层客户端

客户端需要：多样性，节点多，不怕单一软件BUG/分叉风险

以太坊自上线期，从未出现过宕机

整个eth的网络目前由约1万3千台服务器支撑

* * *

## Web3关键特性回顾

### 去中心化

钱包创建高度去中心化，人人可生成

交易广播可以能去中心化

网络运行节点越分散越安全

客户端越多样越稳

无许可：任何人都可以读/写网络

抗审查：节点全球分布、入口可替换

开放开源：客户端开源、交易记录公开可查询

### 隐私（现状与演进）

现状：公开账本+伪匿名

风险：可通过关联分析追踪行为图谱

演进：隐私方案仍在发展

* * *

## 总结

Web3是跨学科的领域：包含社会学、经济学、密码学

**一句话**：Web3就是用私钥证明你是谁，用共识网络保证账本可信，用智能合约让规则自动执行
