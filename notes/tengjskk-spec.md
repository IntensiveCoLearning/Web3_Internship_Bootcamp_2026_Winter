# 2026-01-21
<!-- DAILY_CHECKIN_2026-01-21_START -->

ERC-7962 标准，做到了两件事：**彻底的隐私** 和 **像 Web2 一样丝滑的体验**。

**1.隐私保护（隐身术）**：

**老办法**：你的资产（NFT）挂在你的“钱包地址”名下。地址是公开的，谁都能查。

**新办法 (ERC-7962)**：资产不再挂在“地址”名下，而是挂在一个哈希名下。

**一次性使用**：这就好比你每次去住酒店都用一个全新的假名。虽然酒店（区块链）记录了入住信息，但没人知道这100个假名其实都是你。哪怕你转账给别人，旧的假名作废，生成一个新的假名，**切断了追踪线索**

2.**极致体验（代付功能）**：

**老办法**：你想转账或使用NFT，你的钱包里必须得有 ETH 来付手续费（Gas费）。这对普通用户太难了（还得去买币）。

**新办法**：你只需要在手机上点个“签名（同意）”，手续费可以由商家（比如星巴克）或者是第三方（Relayer）帮你付。你甚至不需要知道什么是 Gas 费，体验就跟用支付宝一样。

**相关术语笔记：**

**1.UTXO 模型**

**定义**：未花费交易输出 (Unspent Transaction Output)，比特币采用的模型。

**大白话**：**现金找零模式**。

-   **普通银行账户**：你有100块，转走50，账户余额数字变成50。（这是以太坊现在的模式）。
    
-   **UTXO模式**：你有一张100块大钞。你想付50块，你得把这100块花掉（销毁），然后拿回一张新的50块钞票（找零）。
    
-   **好处**：这“新的50块”是一张全新的钱，跟之前的交易记录可以切断联系，隐私性更好。ERC-7962 借鉴了这个思路。
    

**2.Relayer (中继者)**

**定义**：负责将用户的签名交易提交到链上的第三方服务。

**大白话**：**跑腿小哥**。

你想寄信（发起交易），但你没邮票（Gas费）。你把信写好签名，交给跑腿小哥（Relayer）。小哥帮你贴上邮票扔进邮筒（上链），费用不管是你事后给小哥，还是星巴克帮你在小哥那月结，反正你不用自己去买邮票。

**一、Telegram 社群搭建与管理**

1.  **社群基础设置**：
    

**（1）创建与信息完善**：设置头像、名称、简介。根据需求选择 **Private（私有）** 或 **Public（公开）** 类型。公开群组可以自定义链接，方便传播。

**（2）权限管理**：防止“炸群”的关键。建议限制普通成员的权限（如禁止修改群信息，谨慎给予管理员权限，防止恶意操作。

**（3）Topic（话题）功能**：对于大型社群，开启 Topic 功能可以将讨论分区（如：DeFi 赛道、活动区、闲聊区），避免信息混杂。但Topic 不宜过多，否则会导致用户注意力分散，应及时关闭不活跃的 Topic。

**2.数据分析（Statistics）**：

**（1）核心指标**：**Members（成员增长曲线）**、**Messages（消息活跃度）**、**Top hours（活跃时间段）** 等。

**（2）tips**：利用“活跃时间段”数据来决定重要公告的发布时间；通过“语言分布”判断是否需要招聘小语种的管理员。

**3.机器人（Bot）配置**：

**（1）安全性**：必须认准 **官方 Bot**（如 Rose Bot），防止引入恶意 Bot 导致钓鱼链接泛滥。

**（2）Rose Bot** ： **Blocklist（黑名单）** 功能，例如屏蔽“赚钱”、“带单”等敏感词，自动过滤垃圾广告。

**二、Twitter Space 线上活动全流程 SOP**

1.  **准备阶段（策划与宣发）**：
    

**（1）选题与调研**：根据热点（如 Layer2、空投）确定主题，拟定一个响亮、吸引眼球的标题。

**（2）嘉宾邀请**：制定问题库（Question Bank），向嘉宾发送邀请时要包含：时间、主题、大致话题方向。

**（3）物料收集**：必须收集嘉宾的四大信息：**姓名、Twitter ID（Handle）、Title（头衔）、头像**。

**（4）宣发节奏**：提前 2-3 天发布海报和推文，文案需包含 Space 链接并 **Tag（@）** 所有嘉宾和相关组织，以便他们转发引流。活动开始前需多次预热（前一天提醒、开始前 5 分钟提醒）。

**2.执行阶段（现场控场）**：

**（1）流程控制**：

**\-开场**：介绍主办方/品牌背景（露出品牌）、介绍嘉宾、阐述活动背景。

**\-圆桌讨论**：依据问题库进行提问，主持人需灵活应变（如追问、跳过已回答问题）。

**\-Q&A**：开放观众提问环节（**有**会更好）。

**注意事项**：主持人需引导嘉宾多分享干货，同时把控时间节奏。

**3.复盘阶段（数据与优化）**：

**(1)核心数据**：重点关注 **实时在线人数（Live Listeners）**，这是衡量 Space 火爆程度的最直观指标。推文浏览量（Impressions）用于评估宣发效果。

**(2)鉴别真假数据**：识别“刷量/买粉”。**显示在 Space 下方头像框的通常是真实用户，而被折叠的往往包含大量机器人**。

**(3)总结优化**：记录执行中的失误（如设备问题、配合问题），分析数据不佳的原因（是选题不行还是宣发不到位），为下一次活动做迭代。

**三、运营视角的思考**

**1\. “工具”只是手段，“共识”才是护城河**

运营的核心永远是 **人**。一个没有活跃讨论、没有共同价值观（共识）的群，即使防骚扰做得再好，也只是一个“干净的死群”。工具的配置应以“不打扰用户”和“提升效率”为原则，把更多精力放在**话题引导**和**核心用户维系**上。

**2\. 调研即内容**：为了准备问题库而去研究一个赛道和热度话题，这个过程本身就是运营人员成长的机会，**不懂调研的运营做不出好活动。**

**3.细节决定专业度（SOP 的价值）**

“邀请嘉宾要给大致话题”、“发推文要 Tag 嘉宾”，Luna姐分享的这些看似琐碎的细节，其实很大一部分决定了活动的成败。在活动执行的紧张环境下，人是会犯错的（比如忘开录音、忘发链接）。建立一套详尽的 CheckList，是专业运营的基本素养。

# 进度情况

一、以前传统 LLM (大语言模型) 的三大限制：

1\. 无状态 (Stateless)：每次对话都是全新的，不记得你是谁

2\. 无工具 (No Tools)：只能生成文本，不能执行代码或调用外部 API，像“被绑住手脚的天才”

3\. 无身份 (No Identity)：没有法律实体，不能签署协议或管理资产

二、如今AI Agent (智能体) 的崛起与困境

1.2026年被认为是 AI Agent 的爆发期。Agent 突破了上述限制，拥有了记忆（通过向量数据库）、手脚（通过工具调用/MCP）和身份

2\. 虽然 Agent 能力很强，但在当前互联网架构下，它像一个“黑户”，面临三大基础设施缺失：

（1）身份缺失 (Identity)：传统 HTTP 协议无法验证请求发起者是人还是 AI。任何人都可冒充某公司的 Agent 进行商业间谍活动

（2）不可审计 (Un-auditable)：AI 的决策日志存储在中心化服务器上，企业可随意篡改、删除或伪造日志，导致法律上无法追责

（3）支付障碍 (Payment)：

AI 无法通过 KYC (身份认证) 申请信用卡。/高成本与低效率：信用卡交易有最低手续费（如 $0.3 + 3%），而 AI 的 API 调用往往是微支付（如 $0.001），传统金融体系无法处理这种高频、小额的交易 ，其次传统金融结算需要 T+1 到 T+3，而 AI 需要毫秒级实时结算。

三、Web3 解决方案：赋予 AI 身份与账户

即不可伪造的身份和可编程的银行账户：

1.身份层 (ERC-721 / 8004 协议)：

AI 的链上护照：为每个 Agent 铸造一个 NFT (ERC-721) 作为唯一身份 。

功能：包含身份注册表（存储元数据）、声誉系统（记录评分，防刷分）和验证器（zkML 证明模型真实运行）

2.不可篡改的日志：将关键决策哈希上链，确保 AI 行为可追溯、不可抵赖

3.支付层 (x402 协议)：基于 HTTP 402 (Payment Required) 状态码的各种实现。允许在 HTTP 请求中直接内嵌支付逻辑，实现机器对机器 (M2M)的原生支付
