# 今日链上数据分析笔记：

## （1）课程结构（4 个部分）

-   **区块链数据结构基础**：链上数据有哪些类型、怎么组织
    
-   **Web3 常见分析指标**：与 Web2 数据分析不同的核心指标体系
    
-   **Web3 数据分析工作流**：链上数据如何被提取、解析、汇总（ETL）
    
-   **常用数据分析工具**：从区块浏览器到数据终端与查询平台的工具栈
    

### （2）链上数据的三大基本类型

-   **交易数据（Transaction）**
    
    -   内容：收发地址（from/to）、金额（value）、时间戳、手续费（gas）、交易哈希（tx hash）等
        
    -   作用：追踪资金流动、识别参与方与交互内容
        
-   **区块数据（Block）**
    
    -   内容：区块时间戳、打包者/验证者、区块内交易列表及顺序、区块 gas 使用与区块收入等
        
    -   作用：理解交易如何被打包确认；从交易定位到区块上下文
        
-   **合约数据（Contract）**
    
    -   内容：合约地址、合约代码、函数/事件（events）、变量、ABI
        
    -   作用：确定“交互的含义”；用 ABI 将日志/输入数据解码为可读的函数与参数
        

### （3）链上交易的生命周期（理解数据为何这样产生）

-   **创建**：钱包转账或与合约交互（如 DEX swap）生成交易
    
-   **广播**：交易发送到网络，节点接收
    
-   **验证 & 选择打包**：节点校验合法性（余额/状态是否满足）；按手续费优先级选择交易进入区块
    
-   **确认上链**：区块被大多数节点确认后追加到链上
    
-   重要现象：交易可能 **success / failed**（例如池子流动性变化导致 swap 失败）
    

### （4）交易数据常见字段（区块浏览器可直接看到）

-   **Transaction Hash**：交易唯一标识，可追溯、不可篡改
    
-   **Status**：成功/失败
    
-   **Block Number**：交易所在区块高度
    
-   **Timestamp**：发生时间
    
-   **From / To**：发送方/接收方（与合约交互时 to 常为合约地址）
    
-   **Value**：转账金额
    
-   **Transaction Fee / Gas**：交易成本
    
-   **Nonce**：防止重复交易的序号
    

### （5）区块结构与区块字段（Block 视角）

-   **区块结构**
    
    -   Block Header：元信息，保证顺序与完整性
        
    -   Block Body：交易记录等核心内容
        
-   **常见字段**
    
    -   区块是否成功上链
        
    -   slot / 打包者
        
    -   交易数量（区块内包含多少笔交易）
        
    -   gas 使用量、区块收入（费用归集到收款地址等）
        

### （6）Web3 常用分析指标（重点）

-   **TVL（Total Value Locked，总锁仓量）**
    
    -   定义：某协议/某生态中被锁定的加密资产总价值
        
    -   组成：流动性池、借贷存款、质押、收益聚合/金库资产等
        
    -   用途：衡量协议吸金能力、市场占有率、增长趋势
        
    -   计算：锁定资产的价值总和（常用 USD 或 ETH 计价）
        
-   **活跃地址数（Active Addresses）**
    
    -   定义：一定时间内至少执行过一次交易/与合约交互的钱包地址数量
        
    -   用途：反映用户基础、增长趋势（类似 DAU/MAU）
        
    -   计算：筛选相关交易后统计唯一地址数（多用 from 作为用户地址）
        
-   **交易数（Transaction Count）与交易量（Volume）**
    
    -   交易数：一段时间内发生的交易笔数（统计 tx hash 数量）
        
    -   交易量：交易涉及资产价值总和
        
    -   用途：评估网络活跃度、市场情绪、需求变化
        
    -   现象：交易数/量激增往往伴随网络拥堵、费用上升
        
-   **Gas 费用（交易成本）**
    
    -   定义：用户支付给验证者/矿工的打包费用
        
    -   计算：`Transaction Fee = Gas Price × Gas Used`
        
    -   以太坊结构：Base Fee + Priority Fee（加速费/小费）
        
    -   用途：反映网络拥堵程度与交易成本
        
-   **协议收入（Protocol Revenue）**
    
    -   费用来源：DEX swap 手续费、借贷利差/管理费、质押服务费等
        
    -   分配逻辑：部分给 LP/参与者，剩余进入国库/回购销毁/分配给持币者等
        
    -   计算：`协议收入 = 协议费用 - 参与者成本`
        
    -   用途：衡量协议盈利能力与对代币持有者的价值支撑
        

### （7）数据分析工作流（ETL）

-   **Extract（提取）**
    
    -   从 RPC 节点 / API / 索引服务获取原始交易数据、状态信息、事件日志
        
    -   数据常以非结构化日志、HEX 输入等形式存在
        
-   **Transform（转换/解析）**
    
    -   依赖 ABI 将 HEX、日志解码为可读的函数、参数、事件
        
    -   将“原始链上数据”转换成可查询的结构化表
        
-   **Load（加载/聚合输出）**
    
    -   先明确要观测的业务指标（TVL/活跃地址/收入等）
        
    -   识别协议的合约地址集合 → 找交互交易 → 筛选特定事件（如 deposit）→ 聚合计算 → 输出看板/结果
        

### （8）数据产品分层（从底到上）

-   **数据源（Source）**：节点/索引/API（提供原始区块链数据访问）
    
-   **数据开发工具（Dev Tools）**：可写 SQL/GraphQL 自定义查询（自由度高）
    
-   **数据应用（Apps/Terminal）**：直接提供指标与看板（上手快、定制较少）
    

### （9）常用工具与用途（按类型归类）

-   **区块浏览器**
    
    -   Etherscan：查交易、区块、地址、合约代码、gas tracker、标签等
        
-   **查询/开发型数据平台**
    
    -   Dune：提供已解析数据集（decoded projects）+ SQL 查询 + 导出 API
        
    -   Flipside / Allium：可作为 Dune 的补充（覆盖/解析情况不同）
        
    -   The Graph / Alchemy：更偏数据源/索引与节点服务
        
-   **指标终端/数据应用层**
    
    -   RootData：项目库（类似“加密天眼查”），项目/团队/融资/指标聚合
        
    -   DefiLlama：DeFi 协议 TVL 覆盖广，同时提供费用/收入/交易量等
        
    -   Token Terminal：偏机构视角的协议指标终端（费用、收入、用户、FDV 等）
        
    -   Artemis / [RWA.xyz](http://RWA.xyz)：更关注稳定币、RWA、Treasury 等方向的数据
        
    -   CoinMarketCap / CoinGecko：行情与宏观聚合指标（市值、ETF 流入等）

# **南塘 DAO 教材笔记总结：**

南塘 DAO 是一个把 Web3 从“线上金融实验”，带回“现实共同体”的实践样本。它诞生于安徽南塘合作社这片本就具备强自治传统的土壤——一个从 90 年代维权组织，进化为新乡建实验场的真实社区。DAO 在这里不是抽象概念，而是对既有村民自治精神的技术延展。

他们发行了南塘豆（NT），但并未走投机路线，而是把它作为**贡献计量、治理凭证与社区内流通的“本地货币”**。NT 被用于真实生活场景：餐饮、住宿、社区超市消费，并通过“人民币 + NT 混合支付”降低村民使用门槛，甚至为老人设计了实体货币卡，试图让区块链变得“可被乡村接受”。

在组织层面，南塘 DAO 借鉴合作社的“萝卜白菜规则”（本土化议事机制），把链上治理与线下协商结合，探索**线上 DAO + 线下共同生活**的混合治理模式——这本身就是一次对“原生线上社区如何落地现实”的实验。

更重要的是，他们并没有把 Web3 仅当作金融工具，而是用于**重构信任与生产关系**。例如将 PGS（公众参与式农产品认证）与区块链结合，把“谁来证明食品安全”的权力，从中心化机构转移给真实社区成员，用链上不可篡改记录，替代传统权威背书。

同时，南塘 DAO 也在做**认知与人才基础设施**：通过 Web3book、乡建宣讲、案例分享、非遗共创营等项目，把 Web3 的方法论引入乡村创新创业，并尝试构建跨地域协作网络，让分散的乡村资源具备“网络效应”。

# **南塘 DAO分享会：**

乡村里最难的不是创意，而是长期协作机制。大家在南塘做过很多真实的事情：种地、生活、办活动、协商公共事务，但“真实”不等于“可持续”。长期参与很奢侈，因为情感劳动、沟通协调、陪伴这些付出难以量化，经验也难继承，关系也难延续，最后变成“铁打的南塘，流水的年轻人”。早期很多冲突并不是价值观问题，而是规则没立起来：过程是否透明、决定权在哪、钱怎么分、贡献怎么认——这些本该在事前解决，却常被情怀和信任先盖过去，久了就消耗殆尽。

他们给出的解法不是“更热血”，而是把冲突往规则层迁移：用区块链/DAO 作为工具，把模糊的付出变成可记录的贡献，把少数人的决定变成可参与的过程。南塘豆（NT）在这里被反复强调不是炒作资产，而是一个“贡献记账 + 治理凭证 + 本地兑换媒介”。他们做了“南塘数字身份”（手机号注册网页钱包）来降低门槛——承认它不是纯去中心化（服务器托管），但现实里要先可用。更关键的是他们在链上做“备注/故事”的绑定：每一次用南塘豆兑换咖啡、住宿、农产品，不只是交易，还要留下“为什么换、换了什么”的叙事记录，让社区发生过的事情不被蒸发，经验能被看到、被追溯、被连接。

同时，南塘 DAO 的另一个主轴是**议事规则（罗伯特议事规则 / “可操作的民主”体系）**。返乡青年：传统公司是“老板拍板—下面执行”，效率高但压扁了共识；南塘早期则走到另一个极端：人人都能说，但难达成一致，提案者还容易被“先挑刺后放生”，最后大家都不想提案。议事规则的价值在于它“保护不成熟的提案”，把大家的意见从情绪对抗，导向可执行的修正：反对可以，但要说清反对点，并提出可修改的方案；团队再围绕修正案辩论、表决、逐步逼近“能往前走的最小一步”。这种机制让讨论变得可预测、可迭代，提案者感觉安全，智慧才会流动。共识不是等来的，是在一次次有效对话里被构建出来的，而构建过程本身就是凝聚力。

更“制度化”的视角：他来自基层社区治理的实践（业主大会、社区停车费、物业议题等），认为这些线下组织本质上就是最大的“分布式组织”。他想把议事规则这种高度程序化的流程进一步“上链”：把会议纪要/共识保存下来，甚至未来用智能合约自动执行规则（更激进的想象是 AI 主持、合约执行）。目前他们先做了一个 Web2 形态的会议程序化 App，帮助南塘在 2025 年 11-12 月导入规则，避免“直接上链投票”导致的信息散乱与不可整理，让讨论先完成“充分交换与修正”，再进入表决。

分享里还有几个很具体的落地例子，能看出南塘豆的“非金融”用途：共创营参与者通过简单劳务获得 NT，用于兑换三餐、住宿；有人营期结束后继续在南塘住半个月到一个月，靠 NT 支付食宿；也有人兑换生态花生、小黄姜产品带回去当伴手礼，顺带帮在地农产品传播。付章老师自己的议事规则服务也用 NT 结算，驻地期间的吃住同样用 NT 兑换——说明它更像“社区内部的记账与交换系统”，而不是对外的投机工具。

最后，大家问到最尖锐的问题也被诚实回答了：南塘 DAO 目前仍然没有完全跑通脱离资助人的自我造血。他们正在探索的方向，是把“在地资源（200 亩地、住宿空间）+营队/活动（黑客松、青年营、艺术共创营）”变成可持续的服务，形成收入，再讨论与合作社/DAO 的分润机制。也就是说，南塘 DAO 把“治理机制”先立起来了，但“商业闭环”还在路上；他们目前能看到一点苗头的，是把共创营/营队能力沉淀成一种可复制的组织服务。

# **Dapp Workshop 课程笔记总结：**

## 从 Web2 到 Web3 的迁移——核心革命是“所有权”

1.  Web2 的资产是租来的：你在王者荣耀买的皮肤，本质是腾讯服务器里的一行记录；平台倒了/封号了，你的资产、内容、粉丝都可能一起消失。
    
2.  Web3 的核心只有三个字：Ownership（所有权）：从“租客”变“房东”，资产与身份可以跟着你走。
    
3.  DApp 的比喻：  
    Web2 像“小卖部”：老板不爽就关门跑路，你只能看脸色。  
    DApp 像“广场自动贩卖机”：你付 gas（投币），它按规则执行（吐可乐），没有老板能随意拔电源。
    
4.  Code is law / Trustless：不再靠信任人，而是信任机器和规则；智能合约一旦部署就“很死板”，条件满足就自动执行，消除赖账空间。
    
5.  钱包即账号：Web2 是“一站一号+100个密码”；Web3 是“一把万能钥匙=钱包”，连接钱包即成为用户，退出钱包数据也能带走。
    

## DApp 的“三驾马车”（技术底座）

1.  智能合约：裁判，负责执行逻辑（铁面无私）。
    
2.  区块链：不可篡改账本，记录“谁拥有什么”。
    
3.  代币：燃料，让系统运转（也带来激励与治理）。
    

同时强调两个常见概念：  
Gas fee：门票/辛苦费，用来防垃圾交易拥堵网络。  
Governance（治理）：持币不只是炒，更是投票参与规则与未来。

## 应用场景：DApp 能干嘛

1.  DeFi：没有银行的银行  
    典型：Uniswap；不需要身份、不需要 KYC，有钱包就能换币、做流动性、赚手续费。
    
2.  NFT / GameFi：数字资产化  
    游戏道具从“数据库里的一行代码”变成“可转让的私有资产”，甚至可跨游戏（如果支持互操作）。
    
3.  DAO：没有老板的公司  
    例子：ConstitutionDAO 几天众筹 4000 万美金竞拍宪法副本，靠代码+社区投票协作。
    

## Web3 仍是“黑暗森林”

贵：网络拥堵时转账几十美金。  
丢私钥=丢钱：没有“找回密码”和客服。

乱：黑客、合约 bug、攻击无处不在。  
结论：安全意识第一位。  
类比：今天的 DApp 像 1995 年互联网——慢、贵、难用，但未来可期，建设者在铺铁轨。

## 全栈实战栈与工具链演进（Hybrid 架构）

Web3 迭代很快，工具链也在演进：  
过去常见：Truffle / web3.js（逐渐淡出）  
现在主流：Foundry + Hardhat + Wagmi/Viem + Next.js + Redis（缓存/体验优化）

为什么要 Hybrid（混合）？  
Foundry（Rust）：跑测试/模糊测试更快，适合合约核心逻辑与安全。  
Hardhat（JS/TS 生态）：部署脚本、任务管理、与前端对接更顺。

核心策略：计算密集型（大量测试）交给 Foundry；交互/工程集成（部署、前端、任务）交给 Hardhat。  
用插件与路径映射把两套工具在一个项目里“共存不打架”。

## 合约工程干货：省 gas + 易前端索引 + 防攻击

1.  Custom Errors：比 require 字符串便宜（链上只存 4 字节哈希），大项目能省巨额 gas。
    
2.  Events + indexed：前端靠监听事件了解链上发生了什么，indexed 能用 topic 快速过滤用户记录。
    
3.  CEI 模式（Checks-Effects-Interactions）：先检查权限→先改状态→再外部调用/转账，防重入攻击，记住一句话：先改账本再给钱。
    
4.  Foundry Fuzzing（模糊测试）：自动生成大量随机地址与边界输入，帮你发现“人脑想不到”的 bug。
    
5.  Hardhat 集成测试：用 TS + 前端交互库模拟真实调用，验证端到端数据流，不只合约逻辑正确，还要“前端能正确调用”。
    

## 部署与安全：从 Demo 到产品必须补的课

1.  声明式部署（幂等）：脚本可重复跑，不怕中断导致“发两次币/部署错位”，会自动检查链上状态、跳过已部署、必要时加速重发。
    
2.  代码验证（Etherscan verify）：不验证就是一堆乱码，没人敢用；部署时自动上传验证更专业。
    
3.  私钥管理：绝不把私钥明文放 .env，黑客会扫 GitHub；用 keystore 等方式加密存储。
    
4.  RPC：公共 RPC 免费但限速，产品上线用户多会白屏；需要 Infura/Alchemy 等专属 key。
    

## 前端关键坑：SSR 水合不一致 + BigInt

1.  Next.js SSR 水合不匹配：服务端渲染时不知道用户浏览器有没有钱包，导致“未连接→闪一下变已连接→报错”。  
    解决思路：把连接状态写入 cookie，让服务端渲染就能生成一致的 HTML。
    
2.  JS number 精度问题：以太坊 uint256 远超 JS 安全整数；要用 BigInt。  
    但 JSON.stringify 不支持 BigInt，需要在渲染前 toString。
    
3.  写入体验要有完整状态机：idle → pending（请在钱包签名）→ mining（打包中）→ success（拿到 receipt 才算真成功）。
    

## Aave——“DeFi 的心脏”从 V3 到 V4 的架构迁移

当协议管理几十亿美金时，怎么通过工程和架构“扛复杂度”。

### Aave 的底层哲学

从传统 P2P 借贷（要等借贷双方需求匹配） → P2Pool（点对池）：池里有钱就能即时借，效率大幅提升。  
可组合性：统一资金池让它成为 DeFi 乐高底座。  
自动化风控：动态利率 + 自动清算维持偿付能力。

### V3 机制亮点（偏合约工程 + 数学）

位图压缩配置：把大量参数压到一个 uint256，减少 storage 读取，省 gas。

惰性计息（Liquidity Index）：余额不是每秒链上更新，而是在交互时用“份额 × 指数”结算，降低链上计算与存储压力。

三类代币：  
aToken：存款凭证，余额会 rebase 增长  
debtToken：欠条，不可转让

利率曲线拐点（利用率 80%）：低于 80% 缓慢升（鼓励借），超过 80% 急剧升（防止流动性枯竭/挤兑），逼借款人还钱、吸引存款。

清算机制（Health Factor < 1）：协议不亲自出手，靠外部清算人“猎人系统”执行清算，赚清算奖励。

eMode / Isolation Mode：  
eMode：相关性高资产（稳定币对稳定币）允许更高 LTV（提高资本效率）  
Isolation：允许引入高风险资产但限制其对全局的传染

### V4：从“单体”到“轮毂式”模块化（Hub & Spoke）

痛点：流动性碎片化（不同链/池的 USDC 彼此不通）。  
方案：统一流动性 Hub（管钱+记账） + 外围 Spokes（承载具体业务逻辑）。

Hub 极简更安全；Spoke 可无限扩展业务（甚至 RWA/KYC）。

新特征：  
风险定价：抵押高风险资产的人付更高借款利率。  
拥抱 ERC-4626 Vault 标准：从“会变多的 aToken”转向“份额 shares”，兼容性更强。  
仓位管理层：支持在一笔交易里完成闪电贷/换币/还款等复杂操作，但不影响 Hub 安全。

工程安全：依赖库代码本地化（避免供应链攻击），CI 里做 gas snapshot，100 gas 的退化都可能被拒 PR。  
最终总结 3 个词：统一、模块化、高效。

## Foundry / Hardhat / 命令行：把链当成“可操控的世界”

命令行像“文字 RPG”：命令是咒语，回车是法杖。

Foundry 三件套  
anvil：本地区块链服务器（开着=世界存在，关掉=世界毁灭）  
forge：编译+测试  
cast：命令行钱包（读写链上）

一套固定流程：写代码 → 部署 → 读（call 免费）→ 写（send 付 gas）→ 再读验证。  
Hardhat 更像“工程化正规军”：脚本比命令更强，适合自动化部署、验证、与前端集成；Foundry 更像“特种兵”，极致快、极致测。

## 总结：

1.  Web3 的核心革命是 所有权：资产/内容/身份不再被平台锁死。
    
2.  DApp 像自动贩卖机：付 gas 就按规则执行，code is law。
    
3.  Web3 三件套：智能合约（逻辑）+ 区块链（账本）+ 代币（燃料/治理）。
    
4.  场景：DeFi（无银行的银行）、NFT/GameFi（资产私有化）、DAO（新型协作）。
    
5.  现实：贵、丢私钥不可找回、黑客与 bug 多，安全第一。
    
6.  工具链演进到 Hybrid：Foundry（快+安全）+ Hardhat（工程+集成）。
    
7.  合约干货：custom error 省 gas、events+indexed 给前端索引、CEI 防重入、fuzzing 找边界 bug。
    
8.  前端坑：SSR 水合不一致用 cookie 解决；uint256 用 BigInt，渲染前 toString。
    
9.  写操作要有状态反馈：pending→mining→receipt success。
    
10.  Aave V4 的关键词：统一流动性 Hub、模块化 Spokes、风险定价+更强可组合性。

# **Uniswap 工作原理解析笔记：**

## 二、Uniswap V2：恒定乘积做市（CPMM）机制

### （1）核心公式与定价

关键关系：x · y = k  
x、y：池子两种资产储备  
k：恒定乘积（交易中保持不变；LP 加/减流动性会改变它）  
池内即时价格：P = y / x

例：2 ETH + 6000 USDC → P = 3000 USDC/ETH，k=12000

### （2）交易如何发生（从一个点移动到另一个点）

交易者往池子加 Δx、拿走 Δy：  
（x + Δx） ·（y − Δy）= k

手续费修正：  
实际参与兑换的是 r·Δx，其中 r = 1 − fee  
不算手续费会导致计算偏差，尤其套利/大额交易会出错。

### （3）套利：谁在把池子价格“拉回市场价”

当外部市场（如 CEX）价格先变化，池子价格滞后时：

套利者在池子用“旧价格”买/卖，再去外部市场反向操作吃差价

套利的意义：  
套利者赚差价  
池子价格被推回到市场一致（价格发现/校准机制）

## 三、Price impact vs Slippage：两个常被混用但不同的概念

### （1）Price impact（价格影响）

来自 AMM 曲线本身：池子越浅，交易推动价格越多。  
特点：可计算、可预期（deterministic）。

### （2）Slippage（滑点）

你“预期成交价格/数量”与“最终成交”之间的偏差

主要来自不可预期因素：  
1.抢跑 / MEV / 你手速慢被别人先套利  
2.三明治攻击（先推高/推低让你成交，再反向拉回）

实战对应：交易界面让你设置的 Max slippage 是“最坏容忍度”  
设小：保护好，但更容易失败（revert）  
设大：更容易成交，但更容易吃亏（尤其 meme 波动/抢时间）

## 四、Uniswap V2 的工程设计亮点：Flash swap（闪电借贷式 swap）

Swap 流程特点：先把输出 token 给你 → 回调 → 检查你补回的输入是否满足 k（含手续费）

意义：  
你不需要一开始就有全部资金  
你可以拿到资产去做一串链上操作（清算、套利、搬砖），最后再补回

不满足会怎样：  
交易直接 revert，不会欠债，但会损失 gas

## 五、LP 视角的核心风险：无常损失（Impermanent Loss）

### （1）本质是什么

不是“你绝对亏钱”，而是：你做 LP 的收益，常常跑输你单纯 HODL 两个资产

直观一句话：  
上涨：LP 等于被动“卖飞”  
下跌：LP 等于被动“接刀”

### （2）为什么叫“无常”

如果价格回到原点（回到最初比例），IL 可能消失  
但如果价格单边走，IL 会长期存在，能不能覆盖主要靠手续费。

### （3）实战提醒

IL 是 AMM 结构性成本，无法彻底消除

只能靠：  
更高交易手续费收入  
主动管理仓位（尤其 V3 更明显）

## 六、Uniswap V2 的链上价格源：TWAP（时间加权平均价格）

思路：不存每一刻价格，而是存 累计价格（价格×时间的面积）

使用方式：  
T0 读一次累计值 V0  
T1 再读一次累计值 V1  
（V1 − V0）/(T1 − T0) → 得到该区间均价

优缺点：  
优：便宜、抗短期尖刺  
缺：必须跨时间T0/T1读取，没法“凭空”拿到某窗口均价

## 七、Uniswap V3：集中流动性（Concentrated Liquidity）

### （1）为什么要做 V3

V2 是全域流动性：从 0 到 ∞ 都在“站岗”，很多资本效率低  
V3 允许 LP 只在自己认为“会发生交易”的价格区间提供流动性：  
同样资金 → 在更窄区间形成更深流动性  
交易者 price impact 更小，LP 单位资本的费收入更集中

### （2）V3 的代价（也是关键点）

LP 从“躺平”变成“半主动做市”

价格跑出区间：  
你的流动性不再工作（不赚手续费）  
风险结构更尖锐（IL 更敏感）

### （3）工程落地：为什么出现 √P、L、tick、bitmap

问题：不同 LP 区间叠加 → 流动性在价格移动时忽高忽低 → 合约需要高效计算跨区间 swap

引入状态表达：  
L = √k（流动性）  
√P（价格平方根）  
让区间内 Δx/Δy 的计算更可控、便于迭代

tick（刻度，离散化价格）：  
不是每个小数位都允许改变流动性，否则遍历会爆 gas  
只允许在某些刻度点发生流动性边界变化

bitmap：  
用 bit 标记“哪些 tick 有变化”，快速跳到下一个有效 tick，减少链上轮询成本

tick spacing 与费率档位：  
稳定币对：更细 tick、更低费率  
波动大资产：更粗 tick、更高费率  
市场会在风险/费率/资金规模之间形成均衡

# 八、思考笔记

1.Uniswap 的“价格”不是拍脑袋出来的，是池子储备比例 y/x 自然决定的；交易只是把储备从曲线上一个点推到另一个点。

2.池子价格为什么会跟市场一致？不是因为池子聪明，而是因为套利者永远盯着价差，把池子当成可被吃的“落后报价”，吃完价差就把价格推回去了。

3.以前总把滑点和价格影响混着说，其实 price impact 是曲线必然结果，能算出来；slippage 是你被 MEV 或者别人抢跑导致的“额外偏差”，这才是不可控的那部分。

4.V2 的 swap 先给你币再检查，这个设计一眼看起来反直觉，但它把 flash swap 这种高效资金利用变成了“天然功能”：做不成就 revert，风险变成 gas 成本。

5.无常损失不是玄学，它就是 AMM 逼你被动再平衡：涨了你就被动卖掉一部分上涨资产，跌了你就被动拿更多下跌资产，所以你常常跑输 HODL，能不能扳回靠手续费。

6.V3 的集中流动性其实就是一句话：别把钱撒在“根本不会发生交易的价格区间”，把流动性压到你认为最可能成交的地方，交易者更爽、LP 资本效率更高；但代价是你变成需要管理区间的做市者。

7.tick/bitmap 这些看着很工程，但背后其实是一个现实：价格是连续的，但链上计算是昂贵的，所以必须离散化、索引化，让跨区间计算能跑得动。

# 二、**Web3 公共物品资金分配第一节课**总结：

## （1）公共物品是什么

-   **经济学定义**：公共物品 = **非排他性**（很难把人排除在外）+ **非竞争性**（你用不影响别人用）。  
    类比：高速公路。
    
-   **私人物品**：你拥有、能排他、会竞争，比如钱包/账户/个人物件。
    
-   **Web3 里的公共品例子**（课程举例）：  
    Solidity（智能合约语言）、Hardhat（开发框架）、各种开源技术栈/开发者工具、协议安全基础设施、文档/API 等。  
    它们共同构成 Web3 的“底层地基”：让开发更快、协议更可信、生态更稳。  
    如果这些基础设施“枯萎”，生态会出现安全与信任问题（比如代码不开源、被攻击也无法验证/审计、协议可能留后门
    

## （2）公共品的核心问题：为什么会“市场失灵”

-   公地悲剧 / 搭便车：大家都能免费用开源成果，但没人愿意为维护者持续付费。
    
-   结果：公共品往往“使用量爆炸（star/fork 很高）”，但收入不匹配，导致长期维护不可持续。
    
-   关键结论：仅靠市场定价规则，公共品很难被合理定价，所以需要新的资金分配机制，把“公共价值”转成“可持续收入”。
    

## （3）Web3 常见资助/分配机制：优点与坑

### A. 委员会制（类似政府拨款）

怎么做：一小群人/委员会决定资金库投给谁（钱来自税收/财政/金库）。  
优点：简单、快、效率高。  
缺点：权力集中、偏好固化与偏见、信息有限，面对复杂多样的提案很难做出“更像群体智慧”的判断。

### B. 二次方资助 QF（Quadratic Funding）

直觉：100 个人各捐 1 票的效果 > 1 个人捐 100 票。

目的：放大“广泛支持”，而不是放大“有钱人的声音”。

代表项目：Gitcoin、Octant 等（以太坊生态常见）。

为什么好：鼓励更多人参与，资金分配更贴近社区共识，弱化鲸鱼垄断（对比 token=票的 POS 式治理）。

主要问题：  
女巫攻击：一个人注册很多钱包伪装成“很多人”，刷参与人数来薅配捐。  
为解决女巫，引入 Passport/身份门槛，但门槛会降低普通人参与积极性（参与成本上升）。

### C. 回溯性资助 RPGF（Retroactive Public Goods Funding）

核心逻辑：先创造价值，再按影响力“事后付费”（为结果买单）。

对比 VC：VC 是“先投钱、押未来”，会让项目被条款/对赌束缚、目标扭曲。

代表案例：Optimism（OP）做了很多轮 RPGF（课上提到到第八轮、曾有单轮大规模发放）。

实践中怎么做：用一套指标衡量项目影响力，再引入评审/投票做最终分配。

演化经验：早期提案多类别混在一起、评审面对海量提案很难评，容易混乱甚至被治理攻击；后来改成更清晰的轮次和类别（开发者工具/治理工具/教育等），并引入更明确的数据指标（如 GitHub star、fork、贡献者等）来提高可评估性。

### D. “Deep Funding / AI 辅助分配”（萌芽）

想解决的问题：人类评审面对几百个项目很难做高质量比较（专业背景限制、信息过载）。

思路：先用 AI 对项目数据“提纯”生成建议分配，再用陪审团/人类做最终裁决。

状态：还在萌芽阶段。

## （4）为什么要做“良心杀”这个游戏（课程的真实目的）

这是一个“资金分配 + 治理攻击”的模拟实验：体验在 DAO 里分配资金时，信息不完全、立场冲突、话术操控、串联投票、治理冷漠等会怎么影响结果。

-   **设置**：每组约 50 人，分配 **10 万美金**给 **5 个项目**；约 30% 的组有“坏人/攻击者”角色。
    
-   **角色**
    
    -   **好人**：用理性与调研把钱分给真正有价值的项目（也可以选择不分配，类比治理冷漠）。
        
    -   **坏人**：与某个项目绑定，靠话术/共谋/引导制造“虚假共识”，让资金尽量流向绑定项目（典型治理攻击）。
        
-   **关键规则**：坏人知道队友是谁；好人不知道坏人是否存在；最终方案需要群体投票达成。
    
-   **三类提案供选择**
    
    1.  Devcon 社区活动资金怎么分；
        
    2.  以太坊生态教育/新手 onboarding 类项目资金怎么分（要做调研）；
        
    3.  DAO 治理工具资金怎么分（也要调研并提出增长/迭代思路）。  
        设计意图：逼你做“产品/项目调研 + 论据说服”，而不是拍脑袋。
        
-   **课堂提醒的点**

# 体验“零知识”（**My First ZKVote：从链上投票到零知识证明**）：

# **一、为什么要在区块链上投票？**

在传统电子投票系统中，常见的痛点包括：

-   需要信任中心化服务器正确记录和统计选票；
    
-   难以向公众证明「统计过程没有被篡改」；
    
-   审计过程高度依赖专业机构，普通选民很难独立验证。
    

区块链提供了一个自然的解决思路：

-   **公开透明**：所有投票交易记录在链上，任何人可以查询与复算；
    
-   **不可篡改**：区块一旦确认，历史记录在经济假设下难以回滚；
    
-   **自动执行**：智能合约按照预先写死的逻辑完成计票与结果发布。
    

因此，基于区块链的投票系统可以做到所谓的 **端到端可验证性（End-to-End Verifiability）**：

从选票生成、传输、计票到结果发布，每一步都可以被任何第三方独立验证。  
然而，「透明」带来可验证性的同时，也引入了一个明显问题：**隐私几乎不存在**。

![链上投票的可验证性示意](https://zkvote.0xtmp.xyz/images/onchain-verifiability.svg)![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/Suemens850/images/2026-01-21-1768982009506-image.png)

## **二、传统链上投票的隐私困境**

在传统链上投票，任何人只要能访问区块浏览器，就可以轻易得到：

-   某个地址参与过多少次投票；
    
-   每次投票在什么议题上、投给了哪一方；
    
-   如果这个地址在其他场景已经与现实身份绑定（交易所、社交账号、线下活动），就可以进一步将**真实身份与投票偏好绑定**。
    

在一些不敏感的社区治理场景，这种公开记录可能是可接受甚至有价值的；

但在涉及政治立场、公司内部表决、薪酬评审等议题时，选民往往希望：

-   自己的选票**不会被公开关联**到某个具体地址或身份；
    
-   但整个投票过程依然是**可验证的**，不会因为匿名而牺牲正确性。
    

这就形成了一个经典矛盾：

> 我们既想让选票“可验证”，又希望投票“可匿名”。

零知识证明（Zero-Knowledge Proof）正是解决这一矛盾的核心密码学工具。

![公开链上投票的隐私泄露路径](https://zkvote.0xtmp.xyz/images/privacy-leak.svg)![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/Suemens850/images/2026-01-21-1768982021186-image.png)

## **三、零知识证明基础：证明什么、向谁证明？**

### **3.1 Prover / Verifier 模型**

零知识证明通常用两个角色来描述：

-   **证明者（Prover）**：掌握某个秘密信息，希望向他人证明自己「确实知道」这个秘密；
    
-   **验证者（Verifier）**：希望确认某个断言是真的，但不想（或不允许）知道该秘密本身。
    

### **3.2 Statement / Witness：你在“证明”什么？**

我们把「要被证明的陈述」称为 **语句（statement）**，把证明者掌握的秘密称为 **见证（witness）**。

在投票场景中，一个典型的语句会是：

> “我确实是这场投票中被授权的选民之一，并且我投出的票是一个合法选项。”

而见证则包括：

-   你的私有身份秘密（如某个随机生成的 `identitySecret`）；
    
-   证明你是选民集合成员所需的 Merkle 路径；
    
-   你选择的具体投票选项等私密数据
    
    ![Public Inputs 与 Witness 的区别](https://zkvote.0xtmp.xyz/images/public-vs-witness.svg)

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/Web3_Internship_Bootcamp_2026_Winter/main/assets/Suemens850/images/2026-01-21-1768982064026-image.png)

### **3.3 零知识证明必须满足的三个性质**

形式上，一个零知识证明系统需要同时满足三个核心性质：

1.  **完备性（Completeness）**
    
    如果语句是真的，并且证明者与验证者都按照协议诚实地执行，那么验证者最终一定会被说服。
    
2.  **可靠性（Soundness）**
    
    如果语句是假的，那么任何试图伪造证明的「作弊」证明者，都几乎不可能骗过诚实的验证者。
    
3.  **零知识性（Zero-Knowledge）**
    
    验证者在整个证明过程中，除了“语句是真的”这一事实之外**不会获得任何额外信息**。
    
    换句话说，验证者无法从证明内容中，还原出证明者所掌握的秘密。
    

在投票系统中：

-   完备性保证「合法选民的合法投票不会被无故拒绝」；
    
-   可靠性保证「非选民或者试图一人多投的人无法通过验证」；
    
-   零知识性保证「验证者无法从证明中推断出你是谁、投了什么票」。  
    
    ![零知识证明三大性质示意](https://zkvote.0xtmp.xyz/images/zk-properties.svg)

### **3.4 交互式 vs 非交互式、zk-SNARK 与 zk-STARK**

最早的零知识证明是**交互式**的：证明者和验证者要来回多轮问答，才能完成一次证明。

在区块链场景下，这种方式不现实：我们希望证明能以**单个对象**的形式提交到链上，由合约一次性做出判断，因此需要使用 **非交互式零知识证明（NIZK）**。

目前在区块链上常见的两类 NIZK 系统有：  

![zk-SNARK 与 zk-STARK 概念级对比](https://zkvote.0xtmp.xyz/images/snark-stark.svg)

-   **zk-SNARK（Zero-Knowledge Succinct Non-Interactive Argument of Knowledge）**
    
    -   _Succinct_：证明大小很小，验证开销与原始计算规模无关或几乎无关；
        
    -   _Non-Interactive_：证明是单向的，不需要多轮交互；
        
    -   已被广泛用于隐私交易、zk-Rollup 等场景。
        
-   **zk-STARK（Zero-Knowledge Scalable Transparent Argument of Knowledge）**
    
    -   不依赖可信初始化（trusted setup）；
        
    -   更适合大规模、复杂计算的场景，但证明大小通常更大。  
        
        ![ZK 投票交互界面示意](https://zkvote.0xtmp.xyz/ZK_1.png)

## **四、zk-SNARK 系统的一般工作流程**

虽然不同库的实现细节不同，但大多数 zk-SNARK 都可以抽象为以下几个步骤：

### **4.1 编写约束系统 / 电路**

首先，需要把我们想要证明的逻辑，转换成一种适合电路计算的形式，比如：

-   R1CS（Rank-1 Constraint System）约束；
    
-   或者算术电路（Arithmetic Circuit）形式。
    

在 ZKVote 场景中，我们想证明的逻辑包括（简化版）：

1.  证明者确实是选民集合中的一员；
    
2.  这次投票的选项属于允许的取值集合；
    
3.  为当前投票生成的 nullifier（防重复投票标识）是由身份秘密和投票 ID 计算出来的。
    

这些条件都会被转化为一系列约束，构成一个「需要被证明的程序」。

### **4.2 Setup：生成证明密钥和验证密钥**

在 zk-SNARK 中，通常需要先执行一次 **Setup**：

-   输入：约束系统 / 电路描述；
    
-   输出：
    
    -   **Proving Key（PK）**：用于之后生成证明；
        
    -   **Verification Key（VK）**：用于之后验证证明。
        

在很多实现中，这一步需要一个可信设置（Trusted Setup）阶段，由一组参与者共同执行，多余的秘密参数需要在仪式结束后安全销毁，以避免被滥用。

在本教程的上下文里，我们假设：

-   Setup 已经通过安全的方式离线完成；
    
-   合约中已经内置或存储了对应的 `Verification Key`。
    

### **4.3 证明者本地生成证明**

当你准备进行一次 ZK 投票时，浏览器会在本地执行以下操作：

1.  从你的输入中收集**公开输入（public inputs）**：
    
    -   如投票所属的 `electionId`；
        
    -   当前选民集合的 Merkle 根 `root` 等。
        
2.  从本地或钱包中收集**私有输入（witness）**：
    
    -   你的身份秘密 `identitySecret`；
        
    -   对应的 Merkle 路径；
        
    -   你选择的投票选项 `vote`。
        
3.  调用 zk 证明库，使用 `Proving Key` 计算出一份证明 `proof`。
    

生成出的证明通常是一个非常短的结构化二进制数据，可以被序列化为 JSON 再发送给智能合约。

### **4.4 智能合约端验证证明**

前端会构造一笔交易，调用投票合约的类似函数：

```
function vote(
    Proof proof,
    PublicInputs publicInputs
) external {
    // 1. 调用 Verifier 验证 proof
    // 2. 检查 nullifier 是否已被使用
    // 3. 更新计票结果
}
```

合约内部会执行以下步骤：

1.  调用内置的 Verifier 合约（或预编译）验证 `proof` 与 `publicInputs` 是否匹配且合法；
    
2.  检查 `nullifier` 是否已经被使用（防止一人多投）；
    
3.  如果验证通过且 `nullifier` 未使用：
    
    -   记录该 `nullifier`；
        
    -   更新投票结果（计票）。
        

所有这些操作都会生成交易记录，保留在链上，任何人都能复查。

## **五、ZKVote 系统设计：从身份到选票**

之前的模拟投票就是基于 zk-SNARK 流程，这里参考包括 Semaphore 在内的多种匿名信号协议的设计思想。

### **5.1 身份注册与选民集合**

![身份承诺、Merkle 树与 nullifier 示意](https://zkvote.0xtmp.xyz/images/merkle-nullifier.svg)

root 存链上；Merkle 路径在证明里；nullifier 防止重复投票。

在投票开始前，我们需要确定谁有权投票。一个典型流程是：

1.  每个选民在本地生成一个随机秘密 `identitySecret`；
    
2.  计算对应的 **身份承诺（identity commitment）**：
    
    ```
    identityCommitment = Hash(identitySecret)
    ```
    
3.  所有 `identityCommitment` 被组织成一棵 **Merkle 树**，其根节点 `root` 存储在投票合约中；
    
4.  选民在本地保存自己的 `identitySecret` 和对应的 Merkle 路径。
    

在这个模型中：

-   链上只知道「有哪些承诺构成了选民集合」；
    
-   不知道每个承诺背后是哪一个具体的钱包地址或现实身份。
    

### **5.2 Nullifier：防止重复投票**

为了防止同一个选民多次投票，我们为每次投票定义一个 **nullifier**：

```
nullifier = Hash(identitySecret, electionId)
```

在电路中，我们会强制约束：

-   `nullifier` 是由某个合法 `identitySecret` 与当前投票 ID `electionId` 计算得到的；
    
-   但证明中不会泄露 `identitySecret` 本身，也不会暴露你在 Merkle 树中的位置。
    

合约只需要确保：

-   对同一个 `electionId`，同一个 `nullifier` 只能使用一次；
    
-   无论谁提交交易，只要带着一个还没被使用过且证明合法的 `nullifier`，就能完成投票。

下午又重读了一遍入门导读的区块链基础概念，做一些读书笔记和文章内容的总结：  
区块链本身可以理解为一条按时间顺序不断延伸的账本，由一个个区块组成，每个区块记录交易信息并包含前一个区块的哈希，因此一旦历史被修改，后续所有区块都会受到影响，修改成本极高。为了保证这条账本的可信度，区块链并不是由单一机构维护，而是运行在分布式网络中，每个节点都保存完整数据，并通过共识机制共同确认新区块，这也是“不可篡改”成立的基础。  
比特币在其中的作用并不只是支付工具，而是对维护网络的节点进行激励的奖励机制，节点通过提供算力和验证交易获得代币，从而形成一套自我运转的系统。进一步来看，区块链并非只有一种形态，按照去中心化程度可以分为公链、联盟链和私链，本质区别在于谁可以成为节点、谁能看到数据以及谁拥有决策权，这决定了它们各自适用的场景。  
Web2 以平台为中心，用户不拥有数据；Web 3.0 侧重语义和结构化数据，并不等同于区块链；而 Web3 则通过区块链、钱包和智能合约，把资产和身份的控制权交还给用户。整体来看，去中心化带来了更低的信任依赖和更强的抗审查能力，但同时也牺牲了效率、成本和使用门槛，因此它并不是对现有互联网的全面替代，而是在金融、资产和数据主权等特定领域提供另一种选择。
