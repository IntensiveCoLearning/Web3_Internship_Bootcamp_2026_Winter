### Day 15 – Vibe Coding ，黑客松实战 ， 投研基础框架  
  
一、今日学习内容概述

今天主要学习了三块内容：**Vibe Coding 的工作方式、黑客松实战经验分享，以及 Web3 投研的基础分析框架**。整体感受是——从“学知识”开始往“怎么真正干活”转变，不再只是听概念，而是开始关心**效率、结果和落地路径**。

### 二、Vibe Coding：不是写代码，是把事推进

以前我理解的 Coding 是“代码写得多不多、炫不炫”，但 Vibe Coding 强调的是：

> **先把事做出来，再慢慢优化。**

我的具体理解与操作：  
不纠结一开始写得优不优雅，**先跑起来**  
用 AI（如 ChatGPT / Copilot）快速生成基础代码  
自己重点负责：  
看逻辑对不对  
能不能跑  
有没有安全或明显漏洞  
把时间花在**功能闭环**上，而不是死磕语法细节  
这对黑客松特别重要，因为黑客松拼的不是“完美代码”，而是**能不能在有限时间内交付一个可展示的东西**。

* * *

### 三、黑客松经验分享：赢的不是技术最强，而是最会协作

这部分分享让我刷新了认知。

黑客松里真正重要的点：

1.  **第一天就定清楚方向**  
    做什么？给谁用？解决什么问题？
    
2.  **分工要狠、要明确**  
    谁写合约、谁做前端、谁写文档、谁准备 Demo
    
3.  **文档和展示非常重要**  
    GitHub README  
    Demo 流程  
    问题背景说明
    

很多项目不是输在技术，而是：

讲不清自己在干嘛  
Demo 演不出来  
评委听不懂价值点

### 四、投研基础框架：从看项目到“判断值不值得

今天也初步学习了一个**通用的 Web3 投研分析框架**，不再是只看币价或热度。  
一个最基础、但很实用的分析顺序：

1.  **这是在解决什么问题？**  
    是真需求，还是自嗨？
    
2.  **项目在生态里的位置**  
    基础设施 / 应用层 / 工具 / 公共物品
    
3.  **用户是谁，谁会用？**  
    开发者？普通用户？DAO？
    
4.  **产品现在做到哪一步**  
    只是概念？Demo？已经有人在用？
    
5.  **长期有没有持续性**  
    能不能活下去  
    有没有社区、资金、迭代能力
    

这个框架对之后看项目、做提案、甚至黑客松选题都非常有帮助。

Day 13 – 深入理解 Uniswap  
前段时间听了关于Uniswap的分享会，但只是听了皮毛，介于此，今日特意深入理解学习Uniswap。  
今天我重点学习了 Uniswap 的工作原理，不是停留在“知道它是 DEX”，而是试着搞清楚：

Uniswap 不靠买卖挂单，那它是怎么自动给出价格的？

我在页面上点一次 Swap，链上实际发生了哪些事情？

为什么很多人说“做 LP 会有无常损失”，到底亏在哪里？

Uniswap V2、V3、V4 听起来很高级，但它们**到底改进了什么现实问题**？

## 二、结合实际操作理解 Uniswap 是怎么跑起来的

### **Uniswap 是怎么定价的（AMM）**

在 Uniswap 里，没有买一卖一的订单簿，价格是“算出来的”。  
每个交易对背后都是一个资金池，比如 ETH/USDC：

池子里有两种币：ETH（x）和 USDC（y）系统只遵守一个规则：`x*y=k`  
当我用 USDC 去换 ETH 时：我往池子里塞 USDC系统必须吐出 ETH 才能保持 `k`

不变吐得越多，ETH 就越贵（价格自然变化）  
个人理解：价格不是谁定的，是池子里币的比例决定的。

一次 Swap 的完整流程（用户视角）

我在 Uniswap 页面点击 Swap，发生的流程是：

1.  连接钱包（MetaMask）
    
2.  输入要换的数量
    
3.  前端根据池子状态：计算当前价格。显示价格影响和滑点
    
4.  我确认交易，钱包用私钥签名
    
5.  交易发到链上，被矿工/验证者打包
    
6.  合约执行，更新池子资产比例
    

### 套利者在里面扮演什么角色？

Uniswap 的价格可能会偏离市场价，这时候：

套利者会在 Uniswap 低买

去中心化或中心化交易所高卖

反复操作后，Uniswap 价格被拉回正常区间

## 三、流动性提供（LP）与无常损失：终于搞懂了

### LP 本质是在做什么？

当我做 LP 时，我是：

同时存入两种币（比如 ETH + USDC）

系统帮我放进池子

别人交易，我按比例分手续费

### 无常损失为什么一定会出现？

如果 ETH 大涨：

池子为了保持平衡，会不断“卖出 ETH”

我作为 LP，ETH 数量越来越少

最后算总价值时，比我单纯拿着 ETH 要少

## 四、Uniswap V3 / V4 的进阶理解

### Uniswap V3：集中流动性

以前（V2）：

流动性铺满 0 到无穷

大多数资金是“闲置的”

现在（V3）：

我可以只在某个价格区间提供流动性

钱用得更高效

但要自己盯价格区间（更像“半主动管理”）

### Uniswap V4：钩子（Hooks）机制

V4 的思路是：

允许开发者在交易前后“插逻辑”

比如动态手续费、限价单、策略执行

个人理解：Uniswap 正在从“单一 DEX”进化成 **DeFi 基础设施平台**。

## 五、今天的收获

能**看懂 Swap 页面里价格影响和滑点的来源**

理解了 **为什么链上交易一定要付 Gas**

明白了 DeFi 协议不是靠“宣传收益”，而是靠**规则设计是否自洽**

对以后参与 **黑客松 / DeFi 项目拆解 / 合约学习**有了明确方向

## 六、下一步关于Uniswap的学习

实际查看 **Uniswap 交易的 Etherscan 页面**，对照参数理解合约执行

尝试从 LP 和用户两个角度分析同一个交易

为后续黑客松选择 **DeFi / AMM 相关方向**打基础

整体来看，Web3 所追求的“去信任”，并不等同于“无规则”，而是将信任从中心化机构，转移到代码、流程与个人判断之上。

## 3\. 新的理解（认知修正）

通过今天的学习，我对 Web3 的理解从“一系列产品和应用”，转变为“一套可以自我运行的系统”。

首先，钱包并不是传统意义上的账户，而是对私钥的控制权。助记词、私钥、公钥与地址之间的派生关系，决定了资产和身份的唯一性与不可找回性。这种“无需许可即可创建钱包”的机制，本质上是 Web3 个人主权的起点，同时也将安全责任完全交还给个人。

其次，交易并不是简单的操作指令，而是一次经过私钥签名的状态变更请求。Gas 费的存在并非单纯收费，而是为了防止资源滥用，保障整个网络的可持续运行。这让我意识到，Web3 中的很多设计，优先考虑的是系统安全与抗攻击能力，而非用户体验。

在网络层面，从交易签名、进入交易池、被验证者打包、再到最终共识确认，整个流程并非追求“快”，而是追求“可信”。以太坊需要在多个区块确认后达到 finalized 状态，本质上是去中心化为安全性付出的时间成本。

此外，智能合约“代码即法律”的特性也让我印象深刻。规则以代码形式提前写入并自动执行，降低了对人为信任的依赖，但也意味着一旦设计有误，后果难以逆转。这让我更加理解 Web3 并不是简单的“更自由”，而是用更高的理解与设计成本，换取更低的信任成本。

* * *

## 4\. 一个仍然不确定、但重要的问题

在资产完全自托管、系统高度去中心化的前提下，如何在不引入强中心化中介的情况下，降低普通用户的安全管理难度，仍然是我目前感到困惑的问题。

此外，RPC 等相对中心化的基础设施在当前阶段仍不可或缺，这是否意味着 Web3 在规模化过程中必然需要阶段性妥协，也值得进一步思考。

* * *

-   在后续学习和项目中，刻意从“钱包—交易—共识—合约”的系统视角拆解 Web3 产品
